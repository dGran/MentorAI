I — Interfaces pequeñas y por rol

Nadie debería verse obligado a depender de métodos que no usa. Una interfaz «que lo tiene todo» parece cómoda, pero arrastra a sus implementadores a cumplir cosas que no les tocan y a sus clientes a conocer más de lo necesario. La cura: varios contratos pequeños por rol.

La formulación de Uncle Bob: «los clientes no deberían verse forzados a depender de interfaces que no usan». Una interfaz grande crea un acoplamiento silencioso: quien la implementa carga con métodos que no le corresponden, y quien la consume queda atado a un contrato que cambia por motivos que no le afectan. La I es la S aplicada a los contratos.

Depender solo de lo que usas #

Un contrato debería describir un rol, no un objeto entero. «Algo que se puede imprimir», «algo que se puede enviar por fax» son roles. Cuando metes impresión, fax, escaneo y grapado en una sola interfaz Multifuncion, obligas a la impresora sencilla —que solo imprime— a fingir que sabe enviar faxes.

La interfaz gorda #

php · un contrato que pide de más 
<?php

declare(strict_types=1);

interface Multifuncion
{
    public function imprimir(Documento $doc): void;
    public function escanear(): Documento;
    public function enviarFax(Documento $doc, string $numero): void;
}

final class ImpresoraBasica implements Multifuncion
{
    public function imprimir(Documento $doc): void { /* ok */ }

    public function escanear(): Documento
    {
        throw new NoSoportado();   // no escanea, pero el contrato la obliga
    }

    public function enviarFax(Documento $doc, string $numero): void
    {
        throw new NoSoportado();   // ni envía faxes
    }
}

Esos throw new NoSoportado() son el olor delator. La clase implementa métodos que no le pertenecen solo para satisfacer el contrato, y de paso viola Liskov (lanza donde el contrato prometía funcionar). Un único síntoma, dos principios rotos.

Partir por rol #

Separa la interfaz gorda en contratos pequeños, uno por capacidad. Cada clase implementa solo los roles que de verdad cumple, y una multifunción de verdad los implementa todos.

php · una interfaz por capacidad 
<?php

declare(strict_types=1);

interface Impresora
{
    public function imprimir(Documento $doc): void;
}

interface Escaner
{
    public function escanear(): Documento;
}

interface Fax
{
    public function enviarFax(Documento $doc, string $numero): void;
}

final class ImpresoraBasica implements Impresora
{
    public function imprimir(Documento $doc): void { /* y nada más */ }
}

final class EquipoMultifuncion implements Impresora, Escaner, Fax
{
    public function imprimir(Documento $doc): void { /* ... */ }
    public function escanear(): Documento { /* ... */ }
    public function enviarFax(Documento $doc, string $numero): void { /* ... */ }
}

La ImpresoraBasica ya no miente: solo promete imprimir. Una clase puede implementar varias interfaces, así que combinar roles no cuesta nada. Esto enlaza directamente con la I como prima hermana de la S: contratos cohesionados, cada uno con una razón para cambiar.

El cliente define la interfaz #

Un giro mental que aclara ISP: las interfaces no se diseñan pensando en quién las implementa, sino en quién las consume. Si un servicio de informes solo necesita leer, declara su dependencia como una interfaz LectorPedidos con un único método, aunque el repositorio real también sepa escribir. El cliente pide lo justo; el implementador puede ofrecer más.

Interfaces estrechas = tests fáciles

Cuando un colaborador depende de una interfaz de un método, su doble de test es trivial: implementas ese método y ya. Las interfaces gordas obligan a fingir diez métodos para probar uno. ISP abarata los test doubles casi gratis.

Sin pasarse #

ISP no pide una interfaz por método ni atomizarlo todo. Pide agrupar por rol cohesionado: los métodos que siempre se usan juntos y cambian juntos pertenecen al mismo contrato. guardar() y eliminar() de un repositorio suelen ir de la mano; partirlos en dos interfaces por dogma solo añade ruido. El criterio, otra vez, es «¿estos métodos sirven al mismo cliente y cambian por el mismo motivo?».

Resumen #

Lo que te llevas

  • ISP: ningún cliente debe depender de métodos que no usa. Interfaces por rol, no por objeto entero.
  • El olor es el throw new NoSoportado(): implementar métodos ajenos para cumplir un contrato gordo (y romper Liskov de paso).
  • Parte por capacidad; una clase implementa varias interfaces finas según los roles que cumpla de verdad.
  • Diseña la interfaz desde el consumidor; agrupa por rol cohesionado, sin atomizar por dogma.
