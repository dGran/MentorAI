Eventos de dominio

Cuando algo importante pasa en el negocio —se confirma un pedido, se cancela una suscripción— a menudo hay que reaccionar en otra parte del sistema. Un evento de dominio captura ese hecho como algo de primera clase y deja que otros reaccionen sin que el origen sepa quiénes son.

Un evento de dominio nombra algo que ya ocurrió y que al negocio le importa. En pasado, porque es un hecho consumado: «PedidoConfirmado», «SuscripciónCancelada». Esa forma de pensar desacopla el sistema en piezas que reaccionan a hechos en vez de llamarse unas a otras a la cara.

Qué es un evento de dominio #

Un evento de dominio es un objeto inmutable que representa algo relevante que sucedió en el dominio, nombrado en pasado y en el lenguaje del negocio. Lleva los datos del hecho (qué pedido, cuándo) y nada más. No es una orden («confirma el pedido»), es una notificación («el pedido se confirmó»).

php · un evento es un hecho inmutable en pasado 
<?php

declare(strict_types=1);

final class OrderConfirmed
{
    public function __construct(
        public readonly OrderId $orderId,
        public readonly DateTimeImmutable $occurredOn
    ) {}
}

Evento vs comando

Un comando es una petición de que algo pase («ConfirmarPedido»): puede rechazarse, tiene un destinatario. Un evento es algo que ya pasó («PedidoConfirmado»): no se rechaza, puede tener cero o muchos interesados. Confundirlos es la fuente de la mitad de los líos con eventos.

Agregados que los emiten #

El que mejor sabe que algo pasó es el agregado donde pasó. Así que es el agregado quien registra el evento cuando cambia de estado, sin publicarlo todavía: lo guarda en una lista interna. Quien orquesta la operación (el caso de uso) lo recoge y lo publica después de persistir.

php · el agregado registra el evento al cambiar de estado 
<?php

declare(strict_types=1);

final class Order
{
    private array $events = [];

    public function confirm(): void
    {
        $this->status = Status::Confirmed;

        $this->events[] = new OrderConfirmed(
            $this->id,
            new DateTimeImmutable()
        );
    }

    public function pullEvents(): array
    {
        $events = $this->events;
        $this->events = [];

        return $events;
    }
}

Consistencia entre agregados #

Aquí está el uso más profundo. La regla de DDD dice modificar un agregado por transacción, para no acoplar sus fronteras de consistencia. Pero el negocio a veces exige que, al confirmar un pedido, se actualice el inventario (otro agregado). ¿Cómo, sin meterlos en la misma transacción?

La respuesta es el evento: el pedido se confirma y emite OrderConfirmed; otro lugar escucha y, en su transacción, actualiza el inventario. Los dos agregados quedan consistentes, pero no a la vez: con consistencia eventual.

Consistencia dentro, eventual fuera

Dentro de un agregado: consistencia inmediata y transaccional. Entre agregados: consistencia eventual vía eventos. Es la misma consistencia eventual de los sistemas distribuidos, aplicada a la coherencia entre partes de tu dominio. Aceptar ese pequeño desfase es lo que mantiene el sistema desacoplado.

Publicar y reaccionar #

El caso de uso persiste el agregado y luego publica sus eventos en un bus; los handlers suscritos reaccionan. El que confirma el pedido no sabe ni cuántos handlers hay ni qué hacen: añadir una reacción nueva (enviar un email, sumar puntos de fidelidad) es registrar otro handler, sin tocar el pedido.

1 Order.confirm()Cambia de estado y registra OrderConfirmed
2 Bus de eventosReparte el evento a los handlers suscritos
3 HandlersActualizar stock · enviar email · sumar puntos

El origen emite y se olvida; los interesados reaccionan. Acoplamiento mínimo.

Puente a la mensajería #

Los handlers pueden ejecutarse en el mismo proceso (síncronos) o, cuando cruzan bounded contexts o conviene desacoplar en el tiempo, viajar como mensajes por una cola como RabbitMQ. El evento de dominio se serializa, se publica en el broker y otro servicio lo consume. Es el salto natural del monolito modular a una arquitectura orientada a eventos.

En cuanto hay cola, hace falta idempotencia

Al pasar por un broker entras en el terreno del at-least-once: un evento puede entregarse más de una vez. Si el handler que descuenta stock se ejecuta dos veces, descuenta de más. La defensa es la idempotencia: handlers que aplicar dos veces el mismo evento no rompe nada.

Errores típicos #

  • Eventos con forma de comando: nombrarlos en presente o imperativo («ActualizarStock») y meterles lógica de destino. Un evento solo dice qué pasó; lo que se hace con ello es de los handlers.
  • Publicar antes de persistir: si emites el evento y luego falla el guardado, has notificado un hecho que no ocurrió. Persiste primero, publica después (o usa el patrón outbox para que ambas cosas sean atómicas).
  • Eventos anémicos o gigantes: ni un evento vacío que obliga a ir a buscar todo, ni uno que arrastra el agregado entero. Lleva lo justo para que el handler reaccione.
  • Cadenas de eventos imposibles de seguir: un evento dispara otro que dispara otro… y nadie entiende el flujo. El desacoplamiento tiene un coste de trazabilidad; úsalo con criterio.

Resumen #

Lo que te llevas

  • Un evento de dominio es un hecho inmutable en pasado, en el lenguaje del negocio. Es una notificación, no una orden.
  • El agregado los registra al cambiar de estado; el caso de uso los publica tras persistir.
  • Sirven para coordinar agregados sin acoplarlos: consistencia inmediata dentro, eventual entre ellos.
  • Son el puente a la mensajería (RabbitMQ); en cuanto cruzan una cola, exigen idempotencia.
