Idempotencia y entrega at-least-once

En una sola máquina, llamar a una función una vez significa que se ejecuta una vez. En cuanto hay red de por medio, esa certeza desaparece: un mensaje puede llegar dos veces, o no saber tú si llegó. La idempotencia es la herramienta que convierte ese caos en algo seguro: que repetir una operación no haga daño.

Estás pagando online, pulsas «Confirmar», la rueda gira… y no pasa nada. ¿Se ha cobrado? ¿Lo intento otra vez? Si pulsas de nuevo, ¿pagas dos veces? Toda la incomodidad de ese momento es el problema que resuelve la idempotencia, y aparece en cuanto dos máquinas hablan por una red.

El problema: la red miente #

Cuando el servicio A llama al servicio B por la red y no recibe respuesta, se enfrenta a una pregunta sin respuesta: ¿se perdió mi petición de ida, o se perdió la respuesta de vuelta? Los dos casos se ven idénticos desde A: silencio. Pero son opuestos.

Se perdió la ida

  • B nunca recibió nada
  • La operación no ocurrió
  • Reintentar es obligatorio

Se perdió la vuelta

  • B sí ejecutó la operación
  • Solo se perdió la confirmación
  • Reintentar la duplica

A no puede distinguir los dos escenarios, así que solo le quedan dos estrategias: no reintentar (y arriesgarse a perder la operación) o reintentar (y arriesgarse a ejecutarla dos veces). Esto no es un caso raro: es el día a día de las colas de mensajes, las llamadas entre microservicios y los webhooks.

Lo viste en RabbitMQ

El ack manual y el at-least-once son exactamente esto: si el consumidor cae antes de confirmar, el mensaje se reentrega. Aquí generalizamos esa idea a cualquier comunicación, no solo a una cola.

Qué es una operación idempotente #

Una operación es idempotente si ejecutarla una vez o muchas veces deja el sistema en el mismo estado. No dice que no se ejecute dos veces: dice que, aunque se ejecute dos veces, el resultado final es como si se hubiera ejecutado una.

El interruptor de la luz lo ilustra bien: pulsar «apagar» diez veces deja la luz apagada igual que pulsarlo una. En cambio, «cambia el estado de la luz» (toggle) no es idempotente: repetirlo da un resultado distinto cada vez.

Operación ¿Idempotente? Por qué
saldo = 100 Sí Asignar un valor absoluto: repetirlo no cambia nada.
saldo = saldo + 10 No Cada repetición suma otra vez: estado distinto.
DELETE pedido 42 Sí Borrar lo ya borrado no hace nada nuevo.
HTTP PUT de un recurso Sí Reemplaza el recurso por un valor fijo (por diseño del método).
HTTP POST (crear) No Cada llamada crea un recurso nuevo: dos POST, dos pedidos.
Enviar un email No Cada ejecución manda otro correo. Efecto externo, no revertible.

La regla mental rápida

Si la operación fija un valor («pon esto así»), suele ser idempotente. Si aplica un incremento o tiene un efecto externo («suma», «manda», «cobra»), no lo es y habrá que protegerla.

Las tres garantías de entrega #

Ante un mensaje que viaja por la red, un sistema puede prometer tres niveles de entrega. Conviene tenerlos claros porque marcan toda la estrategia:

Garantía Qué promete Riesgo
At-most-once Cada mensaje se entrega 0 o 1 veces. Nunca duplica. Puede perder mensajes (no reintenta).
At-least-once Cada mensaje se entrega 1 o más veces. Nunca pierde. Puede duplicar (reintenta ante la duda).
Exactly-once Exactamente 1 vez. El ideal. Imposible de garantizar de extremo a extremo solo con la red.

"Exactly-once" es marketing si lo lees mal

No existe una entrega exactamente-una-vez real solo a nivel de red: el emisor nunca sabe con certeza si el receptor procesó el mensaje. Lo que sí se consigue es el efecto de exactly-once: at-least-once en el transporte + idempotencia en el procesado. Esa es la receta real.

Por qué at-least-once es lo que quieres #

Puestos a elegir entre perder o duplicar, casi siempre es preferible duplicar: una duplicación la puedes detectar y descartar, pero un mensaje perdido a menudo no deja ni rastro de que existió. Por eso la inmensa mayoría de sistemas serios eligen at-least-once en el transporte y resuelven los duplicados arriba.

El transporte garantiza «al menos una vez»; el consumidor, gracias a la clave, lo convierte en «efecto de una sola vez».

La clave de idempotencia #

La herramienta central es la clave de idempotencia (idempotency key): un identificador único que el productor genera para cada operación lógica y manda con cada intento. Lo importante: si reintenta, manda la misma clave. El consumidor la usa para reconocer «esto ya lo hice».

Quién genera la clave importa

La clave la pone el emisor, no el receptor, y debe identificar la operación de negocio («el pago del pedido 42»), no el intento de red. Stripe, por ejemplo, acepta una cabecera Idempotency-Key que tú generas: si reenvías la misma, te devuelve el resultado del primer cobro en vez de cobrar otra vez.

Implementarlo en la práctica #

La forma más sólida de deduplicar se apoya en una restricción de unicidad de la base de datos: dejas que el motor rechace el duplicado por ti, de forma atómica, sin carreras. Primero, la tabla:

sql 
CREATE TABLE processed_operations (
    idempotency_key VARCHAR(64) NOT NULL,
    result_status   VARCHAR(20) NOT NULL,
    processed_at    DATETIME    NOT NULL,
    PRIMARY KEY (idempotency_key)
);

Y el consumidor intenta «reclamar» la clave antes de hacer el trabajo. Si la clave ya existe, la restricción salta y sabemos que es un duplicado:

php · ProcessPayment.php 
<?php

declare(strict_types=1);

final class ProcessPayment
{
    public function __construct(
        private readonly PDO $database,
        private readonly PaymentGateway $gateway,
    ) {
    }

    public function handle(string $idempotencyKey, Money $amount): void
    {
        if ($this->alreadyProcessed($idempotencyKey)) {

            return;
        }

        $this->database->beginTransaction();

        $claimed = $this->claimKey($idempotencyKey);

        if ($claimed === false) {

            $this->database->rollBack();

            return;
        }

        $this->gateway->charge($amount);
        $this->database->commit();
    }

    private function claimKey(string $idempotencyKey): bool
    {
        $statement = $this->database->prepare(
            'INSERT IGNORE INTO processed_operations
             (idempotency_key, result_status, processed_at)
             VALUES (?, ?, NOW())'
        );

        $statement->execute([$idempotencyKey, 'done']);

        return $statement->rowCount() === 1;
    }
}

Deja que la BBDD gane la carrera

La clave del patrón es que INSERT sobre una columna única es atómico: si dos copias del mensaje llegan a la vez (lo viste en concurrencia), solo una consigue insertar; la otra recibe el rechazo y se retira. No hace falta un SELECT previo que abriría una ventana de carrera.

Errores típicos #

Comprobar y actuar en dos pasos

Hacer SELECT «¿existe la clave?» y luego INSERT deja una ventana entre ambos en la que un duplicado se cuela. Resuélvelo con una operación atómica (restricción única, INSERT ... ON CONFLICT) o un lock, no con dos pasos sueltos.

Efectos externos no transaccionales

Cobrar en una pasarela o mandar un email no entra en tu transacción de BBDD: si confirmas la clave pero el proceso muere antes del cobro (o al revés), te quedas inconsistente. Por eso el efecto externo debe ser él mismo idempotente (clave en la pasarela) o registrarse con un patrón outbox.

Guardar las claves para siempre

La tabla de procesados crece sin fin si no la podas. Define una ventana de deduplicación realista (horas o días, según cuánto puede tardar un reintento) y limpia las claves viejas con un TTL o un trabajo periódico.

Resumen #

Lo que te llevas

  • En cuanto hay red, un «sin respuesta» no te dice si la operación ocurrió: reintentar puede duplicar.
  • Una operación idempotente deja el mismo estado se ejecute una vez o muchas. Fijar un valor lo es; sumar o mandar un email, no.
  • El exactly-once real = transporte at-least-once + procesado idempotente. No lo busques solo en la red.
  • La clave de idempotencia la genera el emisor por operación de negocio y se repite en cada reintento.
  • Deduplica con una restricción única atómica (no SELECT+INSERT), cuida los efectos externos y poda las claves viejas.
