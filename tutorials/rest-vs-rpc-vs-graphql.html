REST vs RPC vs GraphQL: cuándo cada uno

REST no es la única forma de diseñar una API, solo la más extendida. RPC piensa en funciones que llamas; GraphQL deja que el cliente pida exactamente los datos que necesita. Conocer los tres te da el criterio para elegir, y para no usar REST por inercia donde no toca.

REST, RPC y GraphQL no son tecnologías rivales que hagan lo mismo mejor o peor: son tres maneras de pensar la conversación entre cliente y servidor. La pregunta no es cuál es mejor, sino cuál encaja con la forma de tu problema.

Tres formas de pensar una API #

La diferencia de raíz está en torno a qué gira la API:

EstiloGira en torno a…El cliente piensa en…
RESTRecursos (sustantivos)"Quiero el pedido 1001"
RPCProcedimientos (verbos)"Llama a la función reembolsar(1001)"
GraphQLUn grafo de datos consultable"Dame estos campos del pedido y el nombre de su cliente"

REST ya lo conoces: nombras un recurso en una URL y le aplicas un verbo HTTP. Los otros dos cambian el centro de gravedad. Veámoslos.

RPC: llamar funciones #

RPC —Remote Procedure Call— es la idea más antigua y la más directa: hacer que llamar a una función en otra máquina se parezca a llamar a una función local. No hay recursos ni verbos HTTP con significado; hay procedimientos con nombre y argumentos.

http · una llamada estilo RPC 
POST /reembolsarPedido HTTP/1.1
Content-Type: application/json

{ "pedidoId": 1001, "motivo": "defectuoso" }

La URL es el nombre del procedimiento y casi todo va por POST. Esto, que en REST sería un antipatrón (recuerda el "nivel 0" de Richardson de la primera lección), en RPC es el diseño correcto: el modelo mental es distinto.

Su encarnación moderna es gRPC, de Google. Define los procedimientos y los tipos en un contrato (Protocol Buffers), genera el código de cliente y servidor a partir de él, y transmite en binario sobre HTTP/2. Es muy rápido y con tipado fuerte de extremo a extremo, lo que lo hace el estándar de facto para la comunicación entre microservicios: máquina a máquina, donde nadie abre la API con un navegador y la eficiencia manda.

La contra de RPC sobre HTTP plano

Al ignorar la semántica HTTP, pierdes lo que REST te daba gratis: cache por GET, códigos de estado expresivos, idempotencia por método. gRPC lo compensa con su propio stack; un RPC casero sobre POST a secas, no.

GraphQL: el cliente pide la forma #

GraphQL, creado por Facebook, le da la vuelta a quién decide la forma de la respuesta. En REST, el servidor define qué devuelve cada endpoint. En GraphQL hay un solo endpoint y es el cliente quien describe, en cada petición, exactamente qué campos quiere y cómo navegar las relaciones:

graphql · el cliente pide justo lo que necesita 
query {
  pedido(id: 1001) {
    total
    estado
    cliente {
      nombre
    }
  }
}

La respuesta llega con esa forma exacta: el total, el estado y el nombre del cliente, ni un campo más ni uno menos, y todo en una sola petición aunque el cliente viva en otra tabla. El servidor publica un esquema tipado (qué tipos existen, qué campos tiene cada uno, cómo se relacionan) y el cliente consulta ese grafo a su antojo.

El problema del over/under-fetching #

GraphQL nació para resolver dos dolores muy concretos de REST, sobre todo en apps móviles donde cada byte y cada ida y vuelta cuentan:

Over-fetching

  • GET /pedidos/1001 devuelve 30 campos.
  • La pantalla solo pinta 3.
  • Gastas ancho de banda en 27 que sobran.

Under-fetching

  • Pides el pedido, pero te falta el cliente.
  • Otra llamada a /clientes/42.
  • Y otra para sus líneas. Tres viajes para una pantalla.

GraphQL elimina los dos de un golpe: el cliente pide los 3 campos que pinta (sin over-fetching) y navega al cliente y a las líneas en la misma consulta (sin under-fetching). Para un frontend rico con muchas vistas distintas sobre los mismos datos, esa flexibilidad es oro: el equipo de frontend itera sin pedirle al de backend un endpoint nuevo por cada pantalla.

REST también puede mitigarlo

Con campos dispersos (?fields=total,estado) y expansión de relaciones (?include=cliente) una API REST recorta bastante el over y el under-fetching. No llega a la flexibilidad de GraphQL, pero a menudo es suficiente y mucho más simple.

Lo que GraphQL te cobra #

Esa flexibilidad no es gratis. Trasladar al cliente el poder de pedir cualquier cosa traslada al servidor un puñado de problemas nuevos:

RetoPor qué aparece
Cache HTTP perdidaTodo va por POST a un único endpoint: la cache por URL de HTTP y los proxies dejan de servirte. Cacheas a otro nivel, con más trabajo.
El N+1 acechandoUna query que navega relaciones puede disparar una consulta a BD por cada nodo. Hace falta agrupar (el patrón DataLoader) para no hundir la base de datos.
Consultas abusivasUn cliente puede pedir un grafo enorme y profundo. Hay que limitar complejidad y profundidad para que no sea un vector de denegación de servicio.
Más maquinariaEsquema, resolvers, tipos: más piezas que montar y mantener que un endpoint REST que devuelve JSON.

El N+1 merece un aviso: es el mismo problema que ya asomaba en Big-O y que volverá en el acceso a datos. GraphQL no lo causa, pero su libertad para navegar relaciones lo hace muy fácil de provocar sin darte cuenta.

Cómo elegir #

No hay un ganador; hay un encaje. Esta es la guía rápida:

1 API pública o CRUD de recursosClara, cacheable, fácil de consumir → REST
2 Servicio interno, máquina a máquinaLatencia baja, tipado fuerte, alto volumen → RPC / gRPC
3 Frontend rico, muchas vistas, datos muy enlazadosClientes que necesitan formas distintas → GraphQL

Elige por la forma del problema, no por la moda.

Para la inmensa mayoría de las APIs —un backend que expone recursos a una web o a terceros— REST sigue siendo la opción por defecto sensata: la entiende todo el mundo, aprovecha HTTP y no añade maquinaria. RPC brilla puertas adentro, entre tus propios servicios. GraphQL gana cuando tienes muchos clientes distintos pidiendo recortes distintos de un grafo de datos muy conectado. Y nada te obliga a casarte con uno: es común un núcleo REST con un servicio gRPC interno, o un gateway GraphQL por delante de varias APIs REST.

No adoptes GraphQL por hype

Montar GraphQL para un CRUD sencillo con un único cliente es pagar toda su complejidad sin cobrar su beneficio. La flexibilidad solo compensa cuando de verdad tienes el problema que resuelve. Si no lo tienes, REST te llevará más lejos con menos.

Resumen #

Lo que te llevas

  • Tres modelos mentales: REST gira en torno a recursos, RPC en torno a procedimientos, GraphQL en torno a un grafo de datos que el cliente consulta.
  • RPC/gRPC: rápido, tipado y binario sobre HTTP/2. El estándar para comunicación interna entre microservicios.
  • GraphQL resuelve el over/under-fetching: un endpoint, el cliente pide los campos exactos y navega relaciones en una sola petición.
  • El precio de GraphQL: pierdes cache HTTP, te expones al N+1 y a consultas abusivas, y montas más maquinaria.
  • Para la mayoría de APIs, REST es la opción por defecto. Elige por la forma del problema, no por la moda, y combínalos sin miedo.
