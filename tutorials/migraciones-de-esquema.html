Migraciones de esquema

Tu código vive en Git, versionado y reproducible. ¿Y el esquema de tu base de datos? Si la respuesta es "alguien le aplicó un ALTER TABLE a mano y no recuerdo cuándo", tienes un problema. Las migraciones ponen la estructura de la BD bajo el mismo control que el código.

Una migración es un script versionado que describe un cambio en el esquema de la base de datos. En conjunto, las migraciones son la historia reproducible de cómo tu BD llegó a su forma actual: cualquiera puede partir de una base vacía y, aplicándolas en orden, obtener exactamente el mismo esquema. Es Git, pero para la estructura de tus datos.

La BD también es código #

El código fuente lleva décadas resuelto: está en control de versiones, cada cambio tiene su autor y su porqué, y puedes recrear cualquier estado del pasado. El esquema de la base de datos, en cambio, es fácil que viva fuera de ese control: alguien añade una columna con un cliente gráfico, otro la modifica por consola, y nadie sabe ya qué estado tiene cada entorno.

Cambios a mano

  • ALTER TABLE suelto en producción.
  • Local, staging y producción divergen.
  • Nadie sabe el estado exacto de cada entorno.
  • Imposible reproducir o revertir.

Migraciones versionadas

  • Cada cambio es un fichero en el repo.
  • Mismo esquema en todos los entornos.
  • La BD registra qué migraciones aplicó.
  • Reproducible desde cero y reversible.

Qué es una migración #

Una migración es un fichero —con un nombre que lleva una marca temporal o un número de secuencia para ordenarlas— que describe un cambio: crear una tabla, añadir una columna, crear un índice. Va en el repositorio, junto al código que la necesita.

php · una migración con su ida (up) y su vuelta (down) 
<?php

final class Version20260629_AnadeTelefonoACliente
{
    public function up(Schema $schema): void
    {
        $this->addSql('ALTER TABLE clientes ADD telefono VARCHAR(20) NULL');
    }

    public function down(Schema $schema): void
    {
        $this->addSql('ALTER TABLE clientes DROP telefono');
    }
}

La base de datos guarda en una tabla propia (algo como migrations) qué migraciones ya ejecutó. Así, cuando lanzas el comando de migrar, la herramienta aplica solo las que faltan, en orden, y nunca repite una ya aplicada. Ese registro es lo que mantiene cada entorno sincronizado.

Up y down #

Casi todos los sistemas de migración piden dos métodos: up(), que aplica el cambio, y down(), que lo revierte. El down es el plan B: si una migración recién desplegada causa problemas, poder volver atrás de forma ordenada vale oro.

bash · aplicar y revertir desde la terminal 
# aplicar las migraciones pendientes
php bin/console doctrine:migrations:migrate

# revertir hasta una versión concreta (ejecuta los down)
php bin/console doctrine:migrations:migrate prev

El down no siempre te devuelve los datos

Revertir el esquema es fácil; recuperar lo que ya se borró, no. Si una migración hizo DROP COLUMN, su down recrea la columna pero vacía: los datos que tenía se fueron. El down protege la estructura, no el contenido. Por eso los cambios destructivos en producción se piensan dos veces.

Estructura vs datos #

Hay dos cosas distintas que a veces se mezclan, y conviene separarlas mentalmente:

TipoCambiaEjemplo
Migración de estructuraLa forma del esquema (DDL)CREATE TABLE, ADD COLUMN, CREATE INDEX
Migración de datosEl contenido de las filas (DML)Rellenar una columna nueva, normalizar valores, mover datos entre tablas

El caso clásico que las junta: añades una columna nombre_completo (estructura) y necesitas rellenarla combinando nombre y apellidos de las filas existentes (datos). Lo limpio es separarlo en pasos —y a veces en varias migraciones a lo largo de varios despliegues— para no bloquear una tabla enorme ni dejar la columna a medio poblar. La distinción importa porque una migración de datos sobre millones de filas es una operación pesada que merece el cuidado de una transacción bien pensada.

En equipo y en producción #

Las migraciones brillan precisamente cuando no trabajas solo. Como cada una es un fichero versionado, viajan en la misma rama y el mismo pull request que el código que las necesita. Quien hace pull y migra, obtiene el esquema que ese código espera. El despliegue encadena lo de siempre:

1 Llega el código nuevogit pull / deploy
2 Se aplican las migraciones pendientesEl esquema se pone al día
3 Código y esquema coincidenLa app arranca sobre la BD correcta

El paso de migrar es parte del despliegue, normalmente automatizado en el pipeline.

El conflicto típico en equipo: dos personas crean una migración a la vez. Como el orden lo da la marca temporal del nombre, rara vez chocan en el fichero, pero sí pueden chocar en el efecto (las dos tocan la misma tabla). Se resuelve como cualquier integración: comunicación y revisar el orden antes de mezclar.

Reglas que ahorran sustos #

Un puñado de costumbres que separan las migraciones tranquilas de los despliegues con incidente:

Las costumbres que evitan incidentes

No edites una migración ya aplicada en otros entornos: crea una nueva que corrija. Una migración, un cambio coherente: más fáciles de revisar y revertir. Cuidado con lo destructivo (DROP, NOT NULL sobre datos existentes): hazlo en fases. Prueba la migración con datos reales (una copia de producción), no solo con la base vacía de tu local.

La regla de oro detrás de todas: una migración, una vez aplicada en un entorno compartido, es inmutable. Cambiarla rompe la reproducibilidad —el entorno que ya la aplicó no la volverá a ejecutar—, que era justo lo que las migraciones venían a darte. ¿Te equivocaste? La corrección es otra migración encima, igual que en Git corriges con un commit nuevo, no reescribiendo la historia ya publicada.

Resumen #

Lo que te llevas

  • Una migración es un script versionado de un cambio de esquema. En conjunto, la historia reproducible de tu base de datos.
  • La BD registra qué migraciones aplicó; al migrar solo se ejecutan las pendientes, en orden, sin repetir.
  • up aplica el cambio, down lo revierte. El down protege la estructura, no los datos ya borrados.
  • Separa migraciones de estructura (DDL) de las de datos (DML); las de datos sobre tablas grandes piden cuidado y fases.
  • Una migración aplicada en un entorno compartido es inmutable: para corregir, otra migración encima. Nunca edites la ya aplicada.
