Async y el event loop: un hilo para miles de conexiones

Un servidor Node atiende diez mil conexiones con un solo hilo. No es que sea más rápido que PHP: es que casi nunca está trabajando. Está esperando a la red y al disco, y mientras espera atiende a otro. Ese truco se llama event loop.

La mayoría de los servidores no se pasan el día calculando: se lo pasan esperando. Esperan la respuesta de la base de datos, de una API, del disco. El modelo asíncrono parte de una idea simple: si vas a esperar, no bloquees a nadie; ponte a atender otra cosa mientras tanto.

El cuello de botella es esperar #

Conviene distinguir dos tipos de trabajo. El trabajo CPU-bound tiene a la máquina calculando sin parar (comprimir, ordenar, cifrar). El trabajo I/O-bound se pasa la vida esperando a algo externo: una consulta SQL, una llamada HTTP, leer un fichero. La inmensa mayoría de las webs son I/O-bound: por cada milisegundo de CPU hay decenas de espera.

Y aquí está la clave: mientras esperas a la red, la CPU está libre. Desaprovecharla teniendo un hilo parado de brazos cruzados es justo lo que el modelo asíncrono quiere evitar.

Un hilo por conexión no escala #

El modelo tradicional asigna un hilo (o proceso) a cada conexión. Es el de PHP-FPM o Apache prefork: cada petición tiene su worker, que ejecuta de arriba abajo de forma síncrona. Cuando llega a una consulta, el worker se bloquea: se queda dormido esperando la respuesta, sin hacer nada más.

php · síncrono.php 
<?php

declare(strict_types=1);

$usuario = $api->obtenerUsuario($id);
$pedidos = $api->obtenerPedidos($id);

echo render($usuario, $pedidos);

La segunda llamada no empieza hasta que la primera termina, y el worker está bloqueado durante ambas esperas. Con un hilo por conexión, atender diez mil conexiones lentas exige diez mil hilos: gigas de RAM y un sistema ahogado en cambios de contexto. El modelo es robusto y fácil de razonar, pero la espera se paga en hilos parados.

El event loop por dentro #

El modelo asíncrono le da la vuelta: un solo hilo y un bucle infinito que reparte el tiempo entre todas las conexiones. En lugar de bloquearse en cada espera, registra "cuando esto termine, llámame" y sigue. Ese bucle es el event loop.

pseudocódigo · el bucle 
mientras (haya trabajo) {
    eventos = preguntar_al_SO_qué_está_listo()

    para cada evento listo {
        ejecutar su callback   // un trozo corto, sin bloquear
    }
}

En cada vuelta, el hilo le pregunta al sistema operativo qué operaciones de I/O ya han terminado y ejecuta los callbacks asociados. Cada callback es un trozo corto de trabajo; al acabar, devuelve el control al bucle, que pasa al siguiente evento. Nadie se bloquea esperando: se espera en conjunto, y el hilo solo trabaja cuando hay algo listo que hacer.

I/O no bloqueante #

Para que el truco funcione, las operaciones de entrada/salida tienen que ser no bloqueantes. En vez de "lee este socket y duérmeme hasta que haya datos", el async dice "avísame cuando este socket tenga datos". El hilo lanza la petición y se desentiende; el sistema operativo le avisará. Mientras tanto, el event loop atiende a los demás.

1 Conexión A pide a la BBDDel loop no espera: registra un callback
2 El loop atiende a B y a Cla CPU no se queda parada
3 El loop ejecuta el callback de Aretoma justo donde lo dejó

Las esperas se solapan. Con I/O no bloqueante, un hilo da servicio a miles de conexiones casi siempre ociosas.

Los callbacks anidados se vuelven ilegibles enseguida (el famoso callback hell), así que encima se construyen abstracciones más cómodas: promesas y async/await, que dejan escribir código asíncrono con aspecto secuencial. Por debajo, siguen siendo el mismo event loop.

Cuándo brilla y cuándo no #

El async gana cuando el trabajo es I/O-bound: muchas conexiones que sobre todo esperan (websockets, chats, gateways de API, proxies). Un solo hilo las sostiene todas porque casi ninguna está trabajando a la vez.

No bloquees el event loop

Como hay un solo hilo, una tarea CPU-bound larga (un bucle pesado, un hash costoso) lo congela entero: mientras tu callback calcula, ninguna otra conexión se atiende. El pecado capital del async es meter trabajo de CPU en el loop. Eso va en otro hilo o en otro proceso (un worker).

Para trabajo CPU-bound puro, el async no aporta: ahí lo que quieres es repartir el cálculo entre varios núcleos (varios procesos o hilos), no concentrarlo en uno. Async y multiproceso no compiten: se combinan (varios event loops, uno por núcleo).

Dónde encaja PHP #

PHP nació con el modelo síncrono de un proceso por petición, y para el 95 % de las webs eso está perfectamente bien: es simple, robusto y aislado. No todo tiene que ser asíncrono.

PHP-FPM (síncrono)

  • Un proceso por petición, de arriba abajo
  • Simple de razonar, sin estado compartido
  • Ideal para webs y APIs normales
  • La espera se paga en procesos parados

Swoole / ReactPHP (async)

  • Event loop: un proceso, muchísimas conexiones
  • Brilla en websockets y alta concurrencia I/O
  • Estado en memoria entre peticiones (cuidado)
  • Si bloqueas el loop, lo paras todo

Cuando de verdad necesitas miles de conexiones persistentes o latencias muy bajas, PHP también tiene event loop con Swoole o ReactPHP. Pero entonces heredas las reglas del async: estado compartido en memoria (con sus race conditions) y la prohibición de bloquear el loop.

Errores comunes #

Creer que async es "más rápido". No acelera una operación individual: una consulta tarda lo mismo. Lo que hace es no desperdiciar la espera, así que escala mejor con muchas conexiones I/O-bound. Para una sola petición de cálculo, no cambia nada.

Meter un cálculo pesado en un callback. Congelas el loop y todas las conexiones se atascan a la vez. El trabajo de CPU se delega fuera del hilo del event loop.

Mezclar I/O bloqueante en código async. Una sola llamada síncrona (un sleep, una librería que bloquea) dentro del loop anula toda la ventaja: durante esa espera nadie más avanza.

Resumen #

Lo que te llevas

  • La mayoría de los servidores son I/O-bound: se pasan la vida esperando. El async aprovecha esa espera en vez de bloquear un hilo.
  • El event loop es un hilo que, en bucle, ejecuta los callbacks de las operaciones de I/O que ya terminaron; con I/O no bloqueante las esperas se solapan.
  • Brilla con muchas conexiones I/O-bound; para trabajo CPU-bound quieres varios núcleos, no un solo loop.
  • PHP-FPM síncrono basta para casi todo; Swoole/ReactPHP traen el event loop a PHP, con su regla de oro: nunca bloquees el loop.
