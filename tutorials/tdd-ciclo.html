El ciclo TDD: rojo, verde, refactor

Escribir el test antes que el código suena del revés, pero es el corazón del Test-Driven Development. No va de testear más: va de dejar que cada test pequeño tire del diseño, paso a paso, con una red de seguridad que crece contigo.

La mayoría aprende a programar escribiendo código y, si acaso, probándolo después a mano. TDD le da la vuelta a ese orden: primero describes lo que quieres con un test que falla, y solo entonces escribes el código justo para que pase. El test no es un examen al final; es la brújula que marca cada paso.

Qué es TDD #

Test-Driven Development (desarrollo guiado por pruebas) es una forma de programar en la que escribes un test automático antes que el código que lo hace pasar. Repites ese gesto en bucles muy cortos —de segundos a minutos— y dejas que la suma de esos tests vaya dibujando el diseño de tu código.

La idea no es nueva (Kent Beck la popularizó a finales de los 90 con Extreme Programming), pero sigue costando porque choca con el instinto: queremos resolver el problema ya, no describirlo primero. La promesa de TDD es que, describiéndolo primero, lo resuelves mejor y con menos miedo a tocarlo después.

TDD no es «escribir tests»

Puedes tener una suite de tests enorme sin hacer TDD (los escribes después), y puedes hacer TDD con pocos tests. TDD es el orden y el ritmo: test que falla → código → limpieza, una y otra vez. Lo que produce buenos tests es un efecto secundario; el objetivo es guiar el diseño.

Por qué el test primero #

Escribir el test antes te obliga a contestar tres preguntas antes de implementar nada: ¿cómo se va a llamar esto?, ¿qué recibe y qué devuelve?, ¿cómo sabré que funciona? Son justo las decisiones de diseño que, si las pospones, acabas tomando a la fuerza y mal.

Como aún no existe la implementación, te ves obligado a pensar desde fuera: desde quien va a usar tu código, no desde sus tripas. Eso empuja hacia interfaces pequeñas y honestas. El test es tu primer cliente, y un cliente exigente mejora el producto.

Test después

  • Testeas lo que ya escribiste, con sus mismos sesgos.
  • Tiendes a adaptar el test al código, no al revés.
  • Lo que es difícil de testear ya está escrito así.
  • Es fácil «dejarlo para luego» y no volver.

Test antes (TDD)

  • Diseñas la interfaz desde quien la usa.
  • Si algo es difícil de testear, lo ves y lo cambias ya.
  • Cada línea de código nace porque un test la pide.
  • Tienes la red de seguridad desde el minuto uno.

Rojo, verde, refactor #

El ciclo de TDD tiene tres fases con nombre de color, por el color que pone el ejecutor de tests al pasar o fallar:

1 RojoEscribe un test que falla: describe el siguiente trocito de comportamiento
2 VerdeEscribe el código más simple que haga pasar el test, aunque sea feo
3 RefactorLimpia el código sin cambiar su comportamiento; los tests te protegen

Y vuelta a empezar. Cada vuelta añade una pizca de comportamiento.

Las tres fases tienen reglas distintas y conviene no mezclarlas:

  • Rojo: el test debe fallar por la razón correcta (la funcionalidad no existe), no por un error de sintaxis. Ver el rojo confirma que el test puede fallar; un test que nunca falla no prueba nada.
  • Verde: aquí vale hacer trampa. Si devolver un valor fijo hace pasar el test, hazlo. La fealdad es temporal y el siguiente test la expulsará. La única meta es la barra verde.
  • Refactor: con todo en verde, mejora nombres, elimina duplicación y aclara la estructura. No añades comportamiento nuevo; si lo necesitas, eso es otro test (vuelta a rojo).

No te saltes el rojo

El error más común al empezar es escribir el código y el test a la vez, o el test después «porque ya sé que pasa». Si no has visto el test en rojo, no sabes si realmente prueba algo. Un test que pasa desde el primer momento es sospechoso.

Pasos pequeños #

La otra mitad de TDD es el tamaño del paso. Los baby steps son incrementos tan pequeños que el siguiente test casi se escribe solo. En vez de «implementar la calculadora», haces «sumar dos números», luego «sumar con un cero», luego «sumar negativos». Cada uno es un rojo→verde de un par de minutos.

¿Por qué tan pequeño? Porque si un test tarda mucho en ponerse verde, te has alejado demasiado de terreno seguro: cuando algo falle, tendrás mucho código que revisar. Con pasos diminutos, cuando el test no pasa sabes que el culpable es lo último que escribiste. El depurador casi desaparece.

Un ejemplo en PHP #

Imagina una función que aplica un descuento a un precio. Empezamos por el test, con PHPUnit, antes de que exista la clase:

php · rojo (el test que falla) 
<?php

declare(strict_types=1);

use PHPUnit\Framework\TestCase;

final class DiscountTest extends TestCase
{
    public function test_aplica_un_porcentaje_al_precio(): void
    {
        $discount = new Discount();

        $final = $discount->apply(100, 20);

        $this->assertSame(80, $final);
    }
}

// Ejecutar ahora falla: la clase Discount no existe todavía. Es el rojo.

Ahora el código mínimo para llegar a verde. No generalizamos de más: solo lo que el test exige.

php · verde (lo mínimo) 
<?php

declare(strict_types=1);

final class Discount
{
    public function apply(int $price, int $percent): int
    {
        return $price - intdiv($price * $percent, 100);
    }
}

// El test pasa. Verde.

Con el verde puesto, viene el siguiente rojo: ¿qué pasa con un descuento negativo o mayor que 100? Escribimos un test que lo exija y volvemos al ciclo. El diseño (validar la entrada, quizá lanzar una excepción tipada) emerge de los tests, no de adivinar requisitos por adelantado.

Triangular para generalizar

Una técnica clásica: si en verde hiciste «trampa» devolviendo un valor fijo, añade un segundo ejemplo distinto que obligue a generalizar. Con dos o tres casos que tiran en direcciones distintas, la solución honesta aparece sola. Se llama triangulación.

Qué te da de verdad #

TDD no es gratis: ralentiza el arranque y exige disciplina. Lo que devuelve a cambio:

Beneficio Por qué aparece
Diseño emergente Al diseñar desde quien usa el código, salen interfaces pequeñas y desacopladas casi sin pretenderlo.
Red de seguridad Cada cambio futuro corre contra cientos de tests; si algo se rompe, lo sabes en segundos, no en producción.
Documentación viva Los tests describen qué hace el código con ejemplos reales y siempre actualizados (si fallan, los arreglas).
Menos depuración Con pasos pequeños, el fallo siempre está en lo último que tocaste. El depurador deja de ser tu casa.

TDD no encaja en todo por igual: para explorar (un spike, un prototipo que vas a tirar) puede estorbar, y para código sin lógica (un getter, mapear un DTO) aporta poco. Brilla donde hay decisiones: reglas de negocio, cálculos, casos límite. Ahí es donde el diseño guiado por tests gana de calle.

Resumen #

Lo que te llevas

  • TDD es escribir el test antes que el código y repetirlo en bucles cortos. Es un orden y un ritmo, no «escribir más tests».
  • El ciclo es rojo (test que falla) → verde (lo mínimo para pasar) → refactor (limpiar sin cambiar comportamiento).
  • Avanza en pasos pequeños: si el test no pasa, el culpable es lo último que escribiste.
  • Te da diseño emergente, una red de seguridad, documentación viva y menos depuración. Brilla donde hay decisiones de negocio.
