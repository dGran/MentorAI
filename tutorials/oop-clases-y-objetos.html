Clases y objetos: el primer pilar de la OOP

Antes de herencia, polimorfismo o interfaces hay una idea más simple y más importante: juntar los datos con el código que los gobierna. Eso es un objeto. Aquí lo construyes desde cero y entiendes por qué la encapsulación es lo que de verdad hace que valga la pena.

La orientación a objetos no es una sintaxis que aprender, es una forma de repartir un programa. En vez de tener datos por un lado y funciones sueltas que los manosean por otro, los juntas: cada objeto guarda su propio estado y expone los métodos que saben operar sobre él. Suena sencillo, y lo es. Lo difícil es entender por qué cambia tanto.

Qué resuelve la OOP #

Imagina una cuenta bancaria gestionada con un simple número y funciones sueltas. El saldo es una variable que anda por ahí, y cualquier parte del programa puede sumarle o restarle lo que quiera. Nada impide que un descuido deje el saldo en negativo, porque no hay un único sitio responsable de las reglas de esa cuenta.

Procedural: datos sueltos

  • El saldo es una variable cualquiera.
  • La regla «no quedar en negativo» se repite (o se olvida) en cada sitio que toca el saldo.
  • Para saber qué puede pasarle al dato, hay que leer todo el programa.

OOP: dato + reglas juntos

  • El saldo vive dentro de un objeto Cuenta.
  • Solo los métodos de Cuenta lo modifican, y aplican la regla siempre.
  • Para entender la cuenta, lees una clase.

Ese es el cambio de fondo: la OOP te da una frontera. Dentro de esa frontera viven unos datos y las reglas que los protegen; fuera solo se puede pedir lo que la frontera permite. El resto de pilares de la OOP —herencia, polimorfismo, interfaces— son maneras de organizar esas fronteras, pero todo empieza aquí.

Clase vs objeto #

Es la distinción que más confunde al principio y la más fácil con una analogía: la clase es el molde, el objeto es lo que sale del molde. La clase Cuenta describe qué datos tiene una cuenta y qué se puede hacer con ella. Un objeto es una cuenta concreta, con su saldo concreto. De una sola clase salen tantos objetos como quieras, cada uno con su propio estado.

1 Clase CuentaEl molde: define qué es una cuenta
2 Objeto: cuenta de Anasaldo = 100
3 Objeto: cuenta de Luissaldo = 40

Una clase, muchos objetos independientes. Cada uno con su propio estado.

«Instancia» es sinónimo de objeto

Cuando alguien dice «una instancia de Cuenta» se refiere a un objeto concreto creado a partir de esa clase. Instanciar = crear un objeto a partir del molde.

Estado y comportamiento #

Un objeto tiene dos mitades. El estado son sus datos: en PHP se guardan en propiedades (variables que pertenecen al objeto). El comportamiento es lo que sabe hacer: sus métodos (funciones que pertenecen al objeto). La gracia de la OOP es que viven juntos, y el comportamiento es el único autorizado a tocar el estado.

php · Cuenta.php 
<?php

declare(strict_types=1);

final class Cuenta
{
    private int $saldo = 0;

    public function ingresar(int $importe): void
    {
        $this->saldo += $importe;
    }

    public function saldo(): int
    {
        return $this->saldo;
    }
}

$saldo es el estado; ingresar() y saldo() son el comportamiento. Fíjate en que el estado es private: nadie de fuera lo lee ni lo escribe directamente. Volvemos a eso en la sección de encapsulación, porque es la pieza clave.

Instanciar y $this #

Para usar la clase creas un objeto con new, y a partir de ahí le mandas mensajes (llamas a sus métodos) con la flecha ->. Cada objeto recuerda su propio estado entre llamadas.

php · uso 
<?php

$deAna = new Cuenta();
$deLuis = new Cuenta();

$deAna->ingresar(100);
$deAna->ingresar(50);
$deLuis->ingresar(40);

echo $deAna->saldo();   // 150
echo $deLuis->saldo();  // 40

Los dos objetos salieron del mismo molde, pero sus saldos no se mezclan: cada uno guarda el suyo. ¿Cómo sabe ingresar() a qué saldo sumar? Por $this. Dentro de un método, $this es «el objeto sobre el que me han llamado». Cuando haces $deAna->ingresar(100), dentro de ingresar() el $this es la cuenta de Ana; en la llamada de Luis, es la de Luis. Es la forma que tiene el método de referirse a su propio estado.

Léelo como «mi»

Donde veas $this->saldo, lee «mi saldo». El método no opera sobre «el saldo» en abstracto, sino sobre el del objeto concreto que lo recibió. Esa es la magia de que cada objeto sea independiente.

El constructor #

Un objeto debería nacer ya válido, no en blanco a la espera de que alguien lo configure. El constructor (__construct) es el método que se ejecuta al hacer new: ahí pones lo que el objeto necesita para existir. Crear una cuenta sin titular no tiene sentido, así que lo exigimos al construirla.

php · constructor con promoción de propiedades (PHP 8) 
<?php

declare(strict_types=1);

final class Cuenta
{
    public function __construct(
        private readonly string $titular,
        private int $saldo = 0,
    ) {
    }

    public function ingresar(int $importe): void
    {
        $this->saldo += $importe;
    }

    public function titular(): string
    {
        return $this->titular;
    }
}

$cuenta = new Cuenta('Ana', 100);

La promoción de propiedades de PHP 8 declara la propiedad y la asigna en la firma del constructor de una vez: ese private string $titular en los parámetros crea la propiedad y guarda el argumento sin que tengas que escribir $this->titular = $titular. Y readonly (PHP 8.1) hace que el titular no se pueda cambiar tras crearse: una cuenta no cambia de dueño.

Encapsulación #

Aquí está el pilar de verdad, el que justifica todo lo anterior. Encapsular es ocultar el estado interno y obligar a que todo el mundo pase por los métodos públicos. La herramienta son los modificadores de visibilidad, que decides propiedad a propiedad y método a método.

Visibilidad Quién puede acceder Para qué
public Cualquiera, desde fuera de la clase. La interfaz del objeto: lo que ofreces al mundo.
protected La propia clase y las que heredan de ella. Detalle que las subclases necesitan, pero no el exterior.
private Solo la propia clase. Estado y detalles internos que nadie más debe tocar.

La regla práctica: las propiedades son private por defecto, y solo expones public lo que de verdad forma parte del contrato del objeto. Si el saldo fuera público, cualquiera podría hacer $cuenta->saldo = -9999 y saltarse todas las reglas. Al ser privado, la única vía es ingresar() o un futuro retirar(), que pueden validar.

php · la frontera protege la regla de negocio 
<?php

declare(strict_types=1);

final class Cuenta
{
    public function __construct(private int $saldo = 0)
    {
    }

    public function retirar(int $importe): void
    {
        if ($importe > $this->saldo) {
            throw new SaldoInsuficiente($this->saldo, $importe);
        }

        $this->saldo -= $importe;
    }
}

Como $saldo es privado, es imposible dejar la cuenta en negativo desde fuera: la única puerta es retirar(), y esa puerta comprueba. La encapsulación no es burocracia; es lo que convierte un saco de datos en un objeto que cuida de sí mismo. Sin ella, tienes structs con sintaxis de clase.

Un getter público no es encapsular

Poner la propiedad privada y luego un getSaldo() y un setSaldo() para todo deja la puerta igual de abierta: el set permite cualquier valor. Encapsular bien es exponer operaciones con intención (ingresar, retirar), no abrir un grifo de lectura/escritura sobre cada campo.

Resumen #

Lo que te llevas

  • La OOP junta datos y comportamiento tras una frontera: el objeto guarda su estado y solo sus métodos lo gobiernan.
  • La clase es el molde y el objeto la instancia: una clase produce muchos objetos, cada uno con su propio estado.
  • $this es «mi» dentro de un método; el constructor hace que el objeto nazca ya válido.
  • La encapsulación (propiedades private, operaciones public con intención) es lo que hace que un objeto proteja sus reglas. Es el pilar real.
