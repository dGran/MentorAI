Concurrencia: el bug que solo pasa en producción

En tu máquina funciona siempre. En producción, una vez cada mil pedidos, el stock baja de cero o cobras dos veces. No es magia negra: es una race condition, el fallo que aparece cuando dos flujos tocan el mismo dato a la vez y el orden importa.

La concurrencia es que varias cosas avanzan a la vez: dos hilos, dos procesos, dos peticiones HTTP. Mientras cada una trabaja con sus propios datos, no hay problema. El drama empieza cuando comparten uno y lo modifican sin ponerse de acuerdo.

El bug que no se reproduce #

Lo que hace temibles a estos fallos es que dependen del momento exacto en que se cruzan dos ejecuciones. En local, con un solo usuario, las cosas pasan de una en una y todo cuadra. En producción, con cien peticiones por segundo, tarde o temprano dos caen en el milisegundo justo y el resultado se corrompe. El bug existe siempre; solo se manifiesta cuando hay carga.

Anatomía de una race condition #

El patrón culpable casi siempre es el mismo: leer, modificar y escribir (read-modify-write). Imagina vender la última unidad de un producto. El código parece inocente:

php · comprar.php 
<?php

declare(strict_types=1);

$stock = $producto->leerStock();

if ($stock > 0) {
    $producto->guardarStock($stock - 1);
    $pedido->confirmar();
}

Con una unidad en stock, dos peticiones simultáneas pueden intercalarse así:

1 Petición A lee stockobtiene 1
2 Petición B lee stocktambién obtiene 1
3 A escribe 0y confirma su pedido
4 B escribe 0y confirma: vendiste 2 con stock 1

Ambas leyeron antes de que la otra escribiera: la última en escribir pisa a la primera (lost update).

Las dos peticiones leyeron 1, las dos pasaron el if, las dos confirmaron. Vendiste dos unidades teniendo una. A esto se le llama lost update: una actualización se perdió porque otra la sobrescribió sin saber que existía.

Sección crítica y mutex #

El tramo de código que no puede ejecutarse a la vez por dos flujos sobre el mismo dato se llama sección crítica. En el ejemplo es el leer-comprobar-escribir del stock. La herramienta clásica para protegerlo es el mutex (de mutual exclusion): un candado que solo uno puede tener a la vez.

1 Adquirir el candadosi lo tiene otro, esperas
2 Sección críticalees, decides y escribes a solas
3 Liberar el candadoel siguiente ya puede entrar

Mientras A tiene el candado, B espera en la puerta. Las carreras desaparecen porque dejan de ser simultáneas.

El candado serializa la sección crítica: convierte el "a la vez" en "uno detrás de otro". Tiene un precio —los que esperan no avanzan—, así que la regla es mantener la sección crítica lo más corta posible: protege solo el dato compartido, no media función.

Operaciones atómicas #

Una operación es atómica si ocurre "de golpe": para cualquier observador, o no ha pasado o ya está entera, nunca a medias y nunca intercalada con otra. La race condition del stock nace de que leer-y-escribir no era atómico: había un hueco entre la lectura y la escritura por donde se coló la otra petición.

La forma más limpia de matar una carrera es convertir el read-modify-write en una sola operación atómica que el sistema ya garantiza. No necesitas el candado si el propio motor hace el incremento de un tirón.

Mejor evitar la carrera que sincronizarla

Un INCR de Redis o un UPDATE … SET x = x - 1 de SQL son atómicos: el motor garantiza que nadie se cuela en medio. Suele ser más simple y rápido que montar candados tú mismo.

Concurrencia en PHP: vía la base de datos #

En PHP normalmente no compartes memoria entre peticiones (cada una es su proceso), así que pensarás que te libras. Pero sí compartes la base de datos y Redis, y ahí las carreras vuelven con todas sus letras: dos peticiones distintas, en procesos distintos, tocando la misma fila. La sincronización se delega al motor.

La primera opción es hacer el cambio atómico en la propia consulta, sin leer antes en PHP:

sql · stock atómico 
UPDATE productos
SET stock = stock - 1
WHERE id = 42 AND stock > 0;

El motor resta y comprueba en una sola operación, fila bloqueada por dentro. Si la sentencia afecta a 0 filas, es que el stock ya era cero: no había unidad y no la vendes. Sin if previo en PHP, sin hueco para la carrera.

Cuando la lógica no cabe en un solo UPDATE (necesitas leer, decidir con reglas y luego escribir), bloquea la fila explícitamente dentro de una transacción con SELECT … FOR UPDATE:

sql · bloqueo pesimista 
BEGIN;

SELECT stock FROM productos
WHERE id = 42
FOR UPDATE;

UPDATE productos SET stock = stock - 1 WHERE id = 42;

COMMIT;

El FOR UPDATE bloquea esa fila hasta el COMMIT: cualquier otra petición que quiera el mismo FOR UPDATE espera su turno. Es el mutex, pero gestionado por la base de datos. Lo viste en el tutorial de transacciones; aquí es la herramienta para la sección crítica.

Deadlock: el abrazo mortal #

Los candados resuelven las carreras, pero traen su propio peligro. Un deadlock ocurre cuando dos flujos se quedan esperándose mutuamente para siempre: A tiene el candado 1 y quiere el 2; B tiene el 2 y quiere el 1. Ninguno suelta, ninguno avanza.

El antídoto: orden de bloqueo consistente

Si todos los flujos adquieren los candados siempre en el mismo orden (primero el de menor id, luego el mayor), el abrazo mortal no puede formarse. Las bases de datos además detectan el deadlock y abortan una transacción para romperlo; tu código debe estar preparado para reintentar.

Errores comunes #

Leer en PHP, decidir y escribir. El clásico $x = leer(); $x++; guardar($x); sobre un dato compartido es una carrera esperando a ocurrir. Hazlo atómico en la consulta o bloquea la fila.

Confiar en que "es muy improbable". Con suficiente tráfico, lo improbable pasa cada día. Si el dato es dinero o stock, la probabilidad no es una defensa.

Bloquear de más. Un candado demasiado amplio (toda la tabla, toda la función) serializa peticiones que no chocaban y mata el rendimiento. Protege el dato justo, durante el tramo justo.

Resumen #

Lo que te llevas

  • Una race condition aparece cuando dos flujos comparten un dato y el resultado depende del orden en que se cruzan; el patrón culpable suele ser leer-modificar-escribir.
  • La sección crítica se protege con un mutex: solo uno entra a la vez. Mantenla corta.
  • Mejor que sincronizar es evitar la carrera con una operación atómica (INCR, UPDATE … SET x = x - 1).
  • En PHP las carreras llegan por la base de datos y Redis: resuélvelas con UPDATE atómico o SELECT … FOR UPDATE. Cuidado con los deadlocks: orden de bloqueo consistente.
