SAPIs y PHP-FPM a fondo

El mismo PHP corre de formas muy distintas según quién lo arranque: un script de consola, Apache o el servidor que sirve tu web. Esa "puerta de entrada" es el SAPI. Veamos qué es, por qué FPM se comió a los demás y qué implica su modelo de que cada petición empiece de cero.

Imagina un cocinero (PHP) que sabe preparar cualquier plato. Da igual si trabaja en un food truck que abre y cierra para cada cliente, en un restaurante con varios cocineros fijos esperando comandas o en tu casa cocinando una sola cena. El cocinero es el mismo; lo que cambia es el local y cómo le llegan los pedidos. Ese "local" es el SAPI.

Qué es un SAPI #

SAPI son las siglas de Server API. Es la capa que conecta el intérprete de PHP con el mundo que lo invoca: de dónde lee la petición, cómo le pasa las variables de entorno, dónde escribe la salida. El núcleo que ejecuta tu código es siempre el mismo; el SAPI es la interfaz de entrada y salida que lo envuelve.

Por eso una misma instalación de PHP trae varios binarios: uno para la consola, otro para el servidor web. No son "PHP distintos": son el mismo motor con un SAPI distinto delante.

Cómo saber qué SAPI corre

La función php_sapi_name() (o la constante PHP_SAPI) te dice bajo qué SAPI se está ejecutando el código: cli en consola, fpm-fcgi bajo PHP-FPM, apache2handler con mod_php. Útil para ramificar comportamiento entre web y comandos.

El zoo de SAPIs #

Históricamente ha habido varias formas de poner PHP detrás de una web. Conviene conocerlas porque explican de dónde viene FPM:

SAPI Cómo funciona Problema
cli Ejecuta un script y muere. Para comandos y workers. No sirve para la web.
CGI clásico Arranca un proceso PHP nuevo por cada petición. Arrancar un proceso por request es carísimo.
mod_php PHP incrustado dentro de cada proceso de Apache. Ata PHP al servidor web; gasta RAM hasta sirviendo imágenes.
PHP-FPM Un pool de procesos PHP persistentes que hablan FastCGI. El estándar actual: rápido y desacoplado.

El gran salto está entre CGI y FPM. CGI tenía la idea correcta (separar PHP del servidor web) pero la pagaba carísima: levantar e inicializar un proceso de PHP entero por cada petición. FastCGI arregló justo eso.

FastCGI y por qué FPM #

FastCGI es un protocolo entre el servidor web (Nginx, Apache) y el proceso que ejecuta el lenguaje. La idea: en vez de arrancar un proceso por petición, mantén un conjunto de procesos vivos y a la espera, y reutilízalos. El servidor web recibe el HTTP, se lo pasa a un worker libre por FastCGI, y este devuelve la respuesta.

PHP-FPM (FastCGI Process Manager) es la implementación oficial de ese gestor para PHP. "Process Manager" porque su trabajo es justo administrar ese conjunto de workers: cuántos hay, cuándo nacen, cuándo se reciclan.

1 NavegadorPetición HTTP
2 NginxSirve estáticos; lo dinámico lo delega
3 PHP-FPM (master)Asigna a un worker libre
4 Worker PHPEjecuta y devuelve la respuesta

Nginx no sabe ejecutar PHP: reparte. Sirve los ficheros estáticos él mismo y manda lo dinámico a FPM por FastCGI.

Por eso Nginx + FPM es el estándar

El servidor web hace lo que sabe (HTTP, TLS, estáticos) y PHP hace lo suyo, cada uno en su proceso. Puedes escalar, reiniciar o actualizar PHP sin tocar Nginx. Ese desacoplamiento es justo lo que mod_php no permitía.

El ciclo de vida de una petición #

Dentro de un worker, cada petición pasa por las mismas fases. Esto es clave para entender el modelo de ejecución de PHP:

1 El worker recibe la peticiónPor FastCGI desde Nginx
2 RINITInicializa el request: superglobales, sesión…
3 Ejecuta tu códigoTu app responde
4 RSHUTDOWNLibera la memoria del request entera
5 Worker libre otra vezListo para la siguiente

El worker no muere entre peticiones (eso lo hacía el CGI clásico): se limpia y espera la siguiente. Lo que sí se tira es la memoria de cada request.

Pools y los modos de pm #

FPM organiza los workers en pools. Un pool es un grupo de procesos con su propia configuración (usuario, límites, número de workers). Puedes tener varios: por ejemplo uno para la web y otro, aislado y con más memoria, para el panel de administración.

La directiva más importante de un pool es pm: decide cómo se gestiona el número de workers. Hay tres modos:

Modo Comportamiento Cuándo
static Número fijo de workers, siempre vivos. Carga estable y alta; máximo rendimiento.
dynamic Mantiene un mínimo; crea y destruye según demanda. Por defecto; buen equilibrio.
ondemand Cero workers en reposo; los crea solo al llegar tráfico. Sitios con poco tráfico; ahorrar RAM.
ini · www.conf 
[www]
pm = dynamic

; Tope absoluto de procesos atendiendo a la vez
pm.max_children = 20

; Workers vivos al arrancar
pm.start_servers = 4

; Colchón de workers libres esperando
pm.min_spare_servers = 2
pm.max_spare_servers = 6

; Reciclar un worker tras N peticiones (evita fugas de memoria)
pm.max_requests = 500

Para qué sirve pm.max_requests

Reinicia el worker tras N peticiones. Aunque PHP libera la memoria de cada request, alguna extensión en C puede ir dejando pequeñas fugas que se acumulan en un proceso de larga vida. Reciclar cada cierto número de peticiones es el cinturón de seguridad barato contra eso.

Dimensionar max_children #

pm.max_children es el número de peticiones PHP que puedes atender a la vez. Si se llena, las demás esperan en cola: la web "se nota lenta" aunque la CPU esté ociosa. Pero pasarse tampoco es gratis: cada worker consume RAM, y si abres más de los que caben en memoria, el servidor empieza a usar swap y todo se desploma.

La cuenta de servilleta es sencilla: RAM disponible para PHP dividida entre la memoria media de un worker.

bash 
# Ejemplo: 4 GB para PHP, cada worker ~80 MB de media
#   4096 MB / 80 MB ≈ 50 workers
#
# Deja margen para el sistema, Nginx y la base de datos:
#   pm.max_children = 40

# Mira el consumo real de un worker en ejecución
ps --no-headers -o rss -C php-fpm8.3 | sort -n | tail -5

Más workers no es más velocidad

Subir max_children a lo bestia es el error típico. Si no hay RAM, entras en swap y vas más lento que con menos workers. Y si tus peticiones esperan a la base de datos, abrir 200 workers solo traslada el atasco a la BBDD. Ajusta a la RAM real y mide.

El modelo shared-nothing #

Aquí está la idea que define a PHP y que sorprende a quien viene de Node o Java: cada petición empieza desde cero y no comparte estado con las demás. Cuando un request termina, PHP tira toda su memoria. Las variables, los objetos, lo que cargaste… nada sobrevive a la siguiente petición.

La cara buena

  • Una petición no puede corromper a otra
  • Una fuga de memoria se limpia sola al acabar
  • Modelo mental simple: empiezas limpio siempre
  • Un worker que peta no se lleva a los demás

La consecuencia

  • No hay variables "globales" entre peticiones
  • El estado compartido va fuera
  • Sesiones, caché y colas: Redis o la BBDD
  • Cargar el framework se repite en cada request

Por qué Redis aparece tan pronto

Si el estado no vive entre peticiones, tiene que vivir en algún sitio al que todos los workers accedan: ahí entran Redis (sesiones, caché, contadores) y la base de datos. El modelo shared-nothing es la razón de fondo de por qué casi toda app PHP acaba con un Redis al lado.

Este modelo enlaza con dos cosas que ya hemos visto en otros tutoriales: que PHP-FPM es multiproceso (un worker por petición, no hilos) y que OPcache es justo lo que evita pagar la compilación del framework en cada uno de esos arranques desde cero.

Resumen #

Lo que te llevas

  • El SAPI es la interfaz por la que se arranca PHP: el mismo motor con distinta puerta de entrada (CLI, mod_php, FPM).
  • FastCGI reutiliza procesos vivos en vez de arrancar uno por petición; PHP-FPM es su gestor oficial.
  • Los pools agrupan workers; pm (static/dynamic/ondemand) decide cómo se crean y se destruyen.
  • max_children = peticiones a la vez. Se dimensiona por RAM ÷ memoria por worker, no a ojo.
  • Shared-nothing: cada petición empieza limpia y no comparte estado. Por eso el estado vive fuera, en Redis o la BBDD.
