Hashing: cómo un hashmap busca en O(1)

El hashmap parece magia: le das una clave y te devuelve el valor al instante, da igual que guarde diez elementos o diez millones. No es magia, es hashing: convertir la clave en una dirección. Abrir esa caja explica también por qué a veces el truco falla.

En el tutorial anterior el hashmap era una caja negra que buscaba por clave en O(1). Aquí la abrimos. La idea central es simple y elegante: en vez de buscar la clave, calcular dónde tendría que estar.

El problema: de la clave a la posición #

Un array accede a la posición i al instante, pero solo sabe de índices numéricos. ¿Y si quiero guardar por una clave como 'ana'? La idea del hashing es convertir esa clave en un número, y usar ese número como índice del array interno. Si la conversión es siempre la misma, para guardar y para recuperar calculo el mismo sitio: no busco, voy directo.

1 Clave"ana"
2 Númerocrc32 → 1402437568
3 Posicióníndice 0 del array interno

La clave se transforma en un número y este se ajusta al tamaño de la tabla con el módulo. Mismo camino al guardar y al leer.

La función hash #

Una función hash toma datos de cualquier tamaño y devuelve un número de tamaño fijo. Para servir a un hashmap necesita tres propiedades: ser determinista (la misma clave da siempre el mismo número), repartir uniforme (claves distintas caen en posiciones distintas, sin amontonarse) y ser rápida (se llama en cada acceso). Con el número, el módulo sobre la capacidad lo encaja en un índice válido.

php · de la clave a la posición 
<?php

declare(strict_types=1);

function bucketIndex(string $key, int $capacity): int
{
    return crc32($key) % $capacity;
}

$capacity = 8;

echo bucketIndex('ana', $capacity);   // siempre la misma posición para 'ana'
echo bucketIndex('leo', $capacity);   // otra posición, repartida

Hash de tabla ≠ hash criptográfico

El hash de un hashmap (crc32, FNV, el interno de PHP) busca velocidad y buen reparto, no seguridad. Los hashes criptográficos (SHA-256, bcrypt) buscan justo lo contrario: ser lentos e imposibles de revertir. Misma palabra, propósitos opuestos; no uses uno donde toca el otro. La parte criptográfica es un tutorial del pilar de seguridad.

Colisiones: dos claves, un sitio #

Como infinitas claves se reducen a un número finito de posiciones, tarde o temprano dos claves caen en la misma: es una colisión. No es un error, es inevitable, y toda tabla hash necesita un plan para resolverlas. Hay dos familias:

Encadenamiento

  • Cada posición guarda una pequeña lista de entradas.
  • Si dos claves chocan, conviven en el mismo "cubo" (bucket).
  • Simple y robusto; es lo que usa PHP.

Direccionamiento abierto

  • Si la posición está ocupada, se busca la siguiente libre.
  • Todo vive en el array, sin listas aparte.
  • Más rápido en caché, pero sensible al factor de carga.
php · mini tabla hash con encadenamiento 
<?php

declare(strict_types=1);

function put(array &$buckets, string $key, mixed $value): void
{
    $index = crc32($key) % count($buckets);
    $buckets[$index][$key] = $value;        // el bucket guarda varias entradas
}

function get(array $buckets, string $key): mixed
{
    $index = crc32($key) % count($buckets);

    return $buckets[$index][$key] ?? null;  // dentro del bucket, por clave
}

$buckets = array_fill(0, 8, []);
put($buckets, 'ana', 30);
echo get($buckets, 'ana');                  // 30

Factor de carga y rehashing #

El factor de carga es cuán llena está la tabla: elementos dividido entre capacidad. Cuanto más se llena, más colisiones y más largas las cadenas, así que el O(1) se degrada. Por eso, al superar un umbral (típicamente 0,75), la tabla hace rehashing: crea un array interno más grande (suele doblar) y recoloca todos los elementos con el nuevo módulo.

El rehashing es O(n), pero amortizado O(1)

Recolocar todo cuesta O(n), pero ocurre muy de vez en cuando (al doblar). Repartido entre todas las inserciones, el coste medio por inserción sigue siendo O(1). A esto se le llama coste amortizado: caro de golpe, barato de media.

Por qué O(1)… casi siempre #

El O(1) del hashmap es el caso medio, y depende de que el hash reparta bien. En el peor caso —si todas las claves colisionan en el mismo bucket— la tabla degenera en una lista y buscar pasa a ser O(n). Con un buen hash y un factor de carga sano, ese peor caso no se da por azar… pero sí puede provocarse a propósito.

Hash flooding: el peor caso como ataque

Si un atacante conoce la función hash, puede enviar miles de claves diseñadas para colisionar todas en el mismo bucket (por ejemplo, en los campos de un formulario o un JSON). El hashmap se vuelve O(n²) al procesarlas y el servidor se ahoga: un denial of service. Por eso los lenguajes modernos (PHP incluido) aleatorizan el hash con una semilla por proceso, para que el atacante no pueda predecir las colisiones.

Más allá del hashmap #

El hashing no vive solo dentro de los diccionarios. La misma idea — resumir un dato en un número manejable— aparece por todas partes:

Errores comunes #

ErrorPor qué duele
Usar crc32/md5 para contraseñas Son rápidos: justo lo que un atacante quiere para probar millones por segundo. Las contraseñas piden hash lento (bcrypt/argon2).
Asumir O(1) garantizado Es el caso medio. Con un hash malo o un ataque de colisiones, degrada a O(n).
Hashear objetos mutables como clave Si la clave cambia tras insertarla, su hash ya no apunta a su bucket y "desaparece".
Confiar en el orden de un hashmap Salvo que el lenguaje lo garantice (PHP sí, por inserción), el orden de recorrido no es de fiar.

Resumen #

Lo que te llevas

  • Un hashmap no busca la clave: calcula su posición con una función hash. Por eso accede en tiempo constante.
  • Una buena función hash es determinista, uniforme y rápida; el módulo encaja su número en la capacidad de la tabla.
  • Las colisiones son inevitables; se resuelven con encadenamiento o direccionamiento abierto.
  • Al pasar el factor de carga (~0,75) hay rehashing: O(n) puntual, pero O(1) amortizado.
  • El O(1) es el caso medio: el peor es O(n), y puede forzarse (hash flooding). Por eso se aleatoriza el hash.
