Imagen vs contenedor: qué es Docker

"En mi máquina funciona" es la frase que mató mil tardes. Docker la jubiló empaquetando tu aplicación con todo lo que necesita para correr, igual aquí que en producción. Pero antes de usarlo hay que entender dos palabras que la gente confunde a diario: imagen y contenedor.

Un contenedor empaqueta tu aplicación junto a sus dependencias, su versión de PHP, sus librerías del sistema y su configuración, en una unidad que corre igual en cualquier máquina que tenga Docker. Deja de importar qué tienes instalado en tu portátil: lo que se ejecuta es el contenedor, idéntico en tu mesa y en el servidor.

El problema que resuelve #

Sin contenedores, tu aplicación depende del entorno donde corre: la versión de PHP instalada, las extensiones activas, las librerías del sistema, variables de configuración. Tu portátil, el de tu compañera y el servidor de producción son tres entornos sutilmente distintos, y esas diferencias son justo donde anidan los bugs que "solo pasan en producción".

La idea en una metáfora

Un contenedor de transporte marítimo cabe igual en un barco, un tren o un camión: el contenido viaja idéntico sin importar el vehículo. Docker hace lo mismo con software: empaqueta la app de forma que el "vehículo" (tu máquina, el servidor) deje de importar.

El resultado es reproducibilidad: el entorno deja de ser algo que se configura a mano y pasa a ser algo que se declara en un fichero y se construye igual siempre. Onboarding de un nuevo compañero: docker compose up y a trabajar, sin una tarde instalando dependencias.

Contenedor no es máquina virtual #

La confusión más común. Una máquina virtual emula un ordenador entero: lleva su propio sistema operativo completo encima del tuyo, con su kernel. Es pesada (gigabytes) y lenta de arrancar (minutos). Un contenedor, en cambio, comparte el kernel del sistema anfitrión y solo empaqueta lo que está por encima: tu app y sus dependencias.

Máquina virtual

  • Lleva un sistema operativo completo, con kernel propio.
  • Pesa gigabytes, arranca en minutos.
  • Aislamiento fuerte, pero caro en recursos.
  • Pocas por máquina.

Contenedor

  • Comparte el kernel del anfitrión.
  • Pesa megabytes, arranca en milisegundos.
  • Aislamiento bueno, ligerísimo.
  • Decenas o cientos por máquina.

Por eso un contenedor arranca casi instantáneo: no hay un sistema operativo que iniciar, solo tu proceso corriendo en un espacio aislado que el kernel del anfitrión le da. Esa ligereza es lo que hizo viable tener muchos servicios pequeños, cada uno en su contenedor, en una misma máquina.

El truco está en el kernel

Compartir kernel es lo que hace al contenedor ligero, y también su límite: un contenedor usa funciones del kernel de Linux (namespaces, cgroups) para aislarse. Por eso en Mac y Windows Docker corre en realidad sobre una pequeña VM de Linux por debajo. El aislamiento de procesos que sostiene todo esto es el de procesos e hilos.

Imagen vs contenedor #

Y llegamos a la distinción clave, la que hace que todo lo demás encaje:

ImagenContenedor
Plantilla inmutable de solo lecturaInstancia en ejecución de una imagen
El moldeLa galleta
La claseEl objeto
Se construye una vezSe crean y destruyen a voluntad
Se comparte (registry)Vive y muere en una máquina

La imagen es el paquete congelado: tu app, su runtime, sus dependencias, todo en capas de solo lectura. No se ejecuta; se guarda y se comparte. El contenedor es lo que obtienes cuando ejecutas una imagen: un proceso vivo, aislado, con su propia capa de escritura encima de la imagen.

De una sola imagen puedes arrancar diez contenedores idénticos —igual que de una clase instancias diez objetos— y cada uno corre por su cuenta. Si entendiste clases y objetos, ya entendiste imagen y contenedor: la imagen define, el contenedor ejecuta.

El ciclo básico #

Cuatro comandos cubren el día a día. Las imágenes se descargan de un registry (Docker Hub es el público por defecto) y de ahí arrancas contenedores:

bash · del registry al contenedor corriendo 
# descargar una imagen del registry
docker pull php:8.3-fpm

# arrancar un contenedor a partir de una imagen
docker run -d --name api php:8.3-fpm

# ver los contenedores en ejecución
docker ps

# parar y eliminar el contenedor
docker stop api && docker rm api

Fíjate en el patrón: pull trae la imagen (el molde), run crea un contenedor a partir de ella (la galleta). Puedes lanzar run varias veces sobre la misma imagen y tendrás varios contenedores independientes. Borrar un contenedor no toca la imagen: el molde sigue ahí para hacer más.

Qué aísla y qué comparte #

Un contenedor vive en su propia burbuja: su sistema de ficheros, su red, su lista de procesos. Desde dentro, parece tener la máquina para él solo. Pero hay dos cosas importantes que entender sobre esa burbuja:

Un contenedor es efímero por diseño

Cuando borras un contenedor, su capa de escritura se va con él: todo lo que escribió dentro desaparece. Eso es una virtud (arrancas siempre limpio), pero significa que los datos que deben sobrevivir no pueden vivir dentro del contenedor. Para eso están los volúmenes, que veremos con docker-compose. La base de datos no guarda sus datos "en el contenedor", sino en un volumen que persiste aunque el contenedor muera.

La otra cara: como el contenedor está aislado, para que sirva de algo hay que abrir puertas a propósito —exponer un puerto para que le llegue tráfico, montar un volumen para compartir ficheros, conectarlo a una red para que hable con otros contenedores—. Ese "abrir puertas con intención" es lo que configuras en el Dockerfile y en compose, las dos siguientes lecciones.

Resumen #

Lo que te llevas

  • Docker empaqueta tu app con todo lo que necesita para correr igual en cualquier sitio: mata el "en mi máquina funciona".
  • Un contenedor no es una VM: comparte el kernel del anfitrión, así que pesa megabytes y arranca en milisegundos.
  • Imagen = molde inmutable de solo lectura; contenedor = instancia en ejecución. Como clase y objeto.
  • Ciclo básico: pull trae la imagen, run crea contenedores; de una imagen, tantos contenedores como quieras.
  • El contenedor es efímero: lo que escribe dentro muere con él. Los datos que deben persistir van en volúmenes, fuera.
