Workers y modelos de ejecución en PHP

"Worker" se usa para dos cosas distintas que conviene no mezclar: los procesos de PHP-FPM que atienden la web y los procesos de cola que procesan tareas en segundo plano. Y por encima, una pregunta de fondo: ¿el proceso muere por petición o vive para siempre?

Cuando alguien dice "súbele workers", para saber qué tocar primero hay que preguntar: ¿workers de qué? El de FPM que atiende peticiones web y el que vacía una cola de emails son criaturas distintas, con configs y problemas distintos. Empecemos por separarlos.

Dos sentidos de "worker" #

Worker de FPM

  • Atiende peticiones HTTP
  • Una request, una respuesta, a esperar
  • Lo gestiona PHP-FPM (los pools)
  • Vive entre peticiones, pero cada una empieza limpia

Worker de cola

  • Procesa tareas en segundo plano
  • Saca un "job" de una cola y lo ejecuta
  • Lo lanzas tú (queue:work, messenger:consume)
  • Un proceso de PHP CLI en bucle

El de FPM ya lo vimos en SAPIs y PHP-FPM: request-per-process, modelo shared-nothing. El de cola es otra historia: es un script de consola que arranca una vez y se queda dando vueltas, esperando trabajo. Ese cambio (de "muere por petición" a "vive en bucle") es el que lo cambia todo.

Workers de cola: por qué existen #

Hay trabajo que no debe hacerse durante la petición web: enviar un email, generar un PDF, llamar a una API externa lenta. Si lo haces en línea, el usuario espera. La solución es encolar la tarea y responder ya; un worker aparte la procesa después.

1 La web encola un job"Enviar email a X" → a la cola
★ Cola (Redis / BBDD / RabbitMQ)Guarda los jobs pendientes
2 Worker de colaProcesa el job en segundo plano

La web no espera: encola y sigue. El worker, en otro proceso (a menudo otra máquina), consume la cola a su ritmo.

bash 
# Laravel: un worker que consume la cola
php artisan queue:work --max-jobs=1000 --max-time=3600

# Symfony Messenger: equivalente
php bin/console messenger:consume async --limit=1000 --time-limit=3600

Si la cola es RabbitMQ, todo lo de enrutado, durabilidad y acks que se cuenta en RabbitMQ a fondo aplica aquí: el worker es justo el consumidor del otro lado.

Mantenerlos vivos: supervisor #

Un worker de cola es un proceso de larga vida, y los procesos de larga vida se caen: un error no capturado, un kill, un deploy. Nadie va a estar relanzándolo a mano. Para eso se usa un supervisor de procesos (Supervisor, systemd) que lo vuelve a arrancar si muere.

ini · supervisor.conf 
[program:queue-worker]
command=php /app/artisan queue:work --max-jobs=1000 --max-time=3600
numprocs=4
autostart=true
autorestart=true
stopwaitsecs=3600

numprocs es tu paralelismo

Con numprocs=4 tienes cuatro workers consumiendo la misma cola a la vez: cuádruple capacidad de procesar jobs. Es el equivalente a max_children en FPM, pero para tareas de fondo. Súbelo si la cola se acumula, vigilando la RAM.

El modelo long-running: Swoole y RoadRunner #

Hasta aquí, incluso los workers de cola arrancan PHP, hacen un job y (con --max-jobs) acaban reiniciándose. Hay un modelo más radical para servir la propia web: procesos que arrancan el framework una vez y atienden miles de peticiones sin morir. Es lo que hacen Swoole, RoadRunner o ReactPHP.

Clásico (FPM)

  • Cada petición arranca el framework
  • OPcache evita recompilar, pero hay que bootear
  • Memoria limpia siempre (shared-nothing)
  • Simple y a prueba de fugas

Long-running

  • El framework se carga una sola vez
  • Peticiones muy rápidas: ya está todo en memoria
  • Permite corrutinas e I/O no bloqueante
  • Pero el estado persiste: cuidado

La ganancia es real: te ahorras el arranque del framework en cada petición. Pero rompes la red de seguridad del modelo clásico. Lo que antes se limpiaba solo al morir el request, ahora se acumula. Esto conecta de lleno con async y el event loop: en Swoole no puedes bloquear el loop, y con la memoria de PHP ya no cuentas con que la arena se tire sola.

Por qué un worker se reinicia cada N jobs #

Aquí está el peligro central de cualquier proceso de larga vida en PHP: las fugas de memoria. En el modelo clásico no importan porque el request muere y se lleva todo. En un worker que vive horas, cada pequeña fuga (un array estático que crece, referencias que no se sueltan, ciclos no recogidos) se acumula hasta que el proceso choca con memory_limit y muere.

1 Worker arranca40 MB de memoria
2 Va subiendoPequeñas fugas se acumulan: 120 MB…
3 Reinicio controlado--max-jobs lo recicla antes de petar

El reinicio periódico no es un parche feo: es la estrategia estándar. Mata el proceso "limpio" cada N jobs o cada X tiempo y supervisor lo relanza desde cero.

Reiniciar es la red de seguridad recuperada

--max-jobs=1000 y --max-time=3600 devuelven, a propósito, la limpieza que el modelo clásico te daba gratis por petición. En vez de cazar cada fuga, aceptas que las habrá y reinicias antes de que importen. Barato y robusto.

El bug del estado que persiste

En un worker de larga vida, un singleton con datos del job anterior, una conexión que se quedó en mal estado o una entidad cacheada del ORM pueden filtrarse al siguiente job. En el modelo clásico esto era imposible (memoria limpia por petición). Aquí hay que limpiar a conciencia entre tareas, o reiniciar a menudo.

Cuándo usar cada modelo #

Necesitas… Usa
Servir una web normal, simple y robusta PHP-FPM clásico
Tareas de fondo (emails, PDFs, APIs lentas) Workers de cola + supervisor
Exprimir latencia o miles de conexiones simultáneas Swoole / RoadRunner (long-running)
I/O concurrente sin hilos (muchas APIs a la vez) ReactPHP / corrutinas de Swoole

La regla práctica: empieza por lo clásico, porque es el que menos te puede morder. Pasa a colas en cuanto tengas trabajo que no debe correr durante la petición. Y reserva el long-running para cuando tengas un problema concreto de rendimiento que justifique perder la simplicidad del shared-nothing.

Resumen #

Lo que te llevas

  • "Worker" son dos cosas: los de FPM (web, request-per-process) y los de cola (tareas de fondo en bucle).
  • Las colas sacan trabajo lento fuera de la petición; un supervisor mantiene vivos a los consumidores.
  • El modelo long-running (Swoole, RoadRunner) carga el framework una vez: muy rápido, pero el estado persiste.
  • En procesos de larga vida las fugas se acumulan; por eso se reinician cada N jobs o X tiempo (--max-jobs, --max-time).
  • Empieza por lo clásico; añade colas cuando haga falta; reserva long-running para problemas concretos de rendimiento.
