Redis como caché de aplicación

Cachear es fácil de empezar y fácil de hacer mal. Guardar el resultado de una consulta cara en Redis te da un salto de rendimiento brutal… hasta que sirves datos obsoletos o toda la caché expira a la vez. Veamos los patrones que funcionan y las tres formas clásicas de pegarte un tiro.

Este tutorial asume que ya sabes qué es Redis y sus tipos: si no, pasa antes por Redis a fondo. Aquí nos centramos en un solo uso, el más común de todos: poner Redis delante de tu base de datos para no repetir trabajo caro.

Por qué cachear #

Hay datos que cuestan mucho de calcular y cambian poco: el top de productos, el perfil de un usuario, el resultado de un informe. Pedirlos a la base de datos en cada petición es desperdiciar: la respuesta sería casi siempre idéntica. La caché guarda ese resultado en memoria y lo sirve sin tocar la BBDD.

Caché = trato: velocidad por frescura

Toda caché es el mismo pacto: aceptas que el dato pueda estar ligeramente desactualizado a cambio de servirlo mucho más rápido. Todo el arte de cachear está en gestionar ese desfase para que nadie lo note. Si el dato no puede desactualizarse nunca, no lo caches.

Cache-aside, el patrón base #

El patrón más usado se llama cache-aside (o lazy loading): la caché se llena bajo demanda, solo cuando alguien pide un dato que no estaba.

1 Pregunta a Redis¿Tienes este dato?

HIT

  • Redis lo tiene
  • Lo devuelves directo
  • La BBDD ni se entera

MISS

  • No está en Redis
  • Lo lees de la BBDD
  • Lo guardas en Redis con TTL
  • Y lo devuelves

La primera petición paga el coste (miss); las siguientes, hasta que expire, son baratas (hit).

php · cache-aside.php 
<?php

declare(strict_types=1);

final class ProductRepository
{
    private const TTL_SECONDS = 600;

    public function __construct(
        private \Redis $redis,
        private \PDO $database,
    ) {
    }

    public function find(int $productId): ?array
    {
        $cacheKey = "product:{$productId}";
        $cached = $this->redis->get($cacheKey);

        if ($cached !== false) {
            return json_decode($cached, true);
        }

        $product = $this->fetchFromDatabase($productId);

        if ($product === null) {
            return null;
        }

        $this->redis->setex($cacheKey, self::TTL_SECONDS, json_encode($product));

        return $product;
    }
}

El TTL no es opcional #

Fíjate en que arriba usamos setex (set con expiración), no un set a secas. Esto es clave: toda entrada de caché debe tener TTL. El TTL es tu red de seguridad. Aunque te equivoques al invalidar, aunque olvides borrar una clave, el dato obsoleto desaparece solo pasado ese tiempo.

Caché sin TTL = fuga de memoria con datos viejos

Una clave sin expiración se queda para siempre. Multiplícalo por millones de productos y Redis se llena hasta que empieza a expulsar claves (o a fallar). Y si la invalidación tiene un bug, sirves un dato incorrecto indefinidamente. Pon TTL siempre, aunque también invalides a mano.

Invalidar: el problema difícil #

Hay una frase famosa: "solo hay dos cosas difíciles en informática: invalidar la caché y nombrar las cosas". La parte de la caché es real. El TTL te da frescura "eventual", pero cuando un dato cambia y quieres que se vea ya, tienes que invalidar.

La estrategia más limpia con cache-aside es borrar la clave al escribir (no actualizarla): cuando modificas el producto en la BBDD, eliminas su entrada de caché. El siguiente lector tendrá un miss y la recargará fresca.

php · invalidar.php 
<?php

declare(strict_types=1);

public function update(int $productId, array $changes): void
{
    $this->saveToDatabase($productId, $changes);

    // Borrar, NO reescribir: que el próximo lector recargue
    $this->redis->del("product:{$productId}");
}

Borrar es más seguro que reescribir

Si actualizas la caché al escribir, dos escrituras concurrentes pueden dejar en Redis el valor de la que terminó primero en la BBDD: un dato viejo pegado. Borrar la clave esquiva esa carrera: el valor siempre se reconstruye desde la fuente de verdad.

El dato obsoleto (y dónde es aceptable) #

Por bien que invalides, siempre habrá una ventana en la que la caché y la BBDD discrepan. La pregunta correcta no es "¿cómo lo evito del todo?" (no puedes), sino "¿cuánto desfase tolera este dato?".

Dato Desfase tolerable TTL razonable
Saldo de una cuenta bancaria Ninguno No cachear (o invalidar estricto)
Stock de un producto Segundos 5–30 s + invalidar al vender
Ficha de producto Minutos 5–15 min
Lista de categorías Horas 1–24 h

Los tres fallos clásicos #

Tres problemas tienen nombre propio porque tumban sistemas en producción. Conviene reconocerlos:

Penetration (penetración)

Se piden claves que no existen (IDs inventados). Cada una falla en caché y golpea la BBDD. Atacable a propósito. Solución: cachear también el "no existe" (un valor nulo con TTL corto) para que el segundo intento ya no llegue a la BBDD.

Avalanche (avalancha)

Muchísimas claves expiran a la vez (porque las cargaste juntas con el mismo TTL). De golpe, todo es miss y la BBDD recibe una marea. Solución: añadir un margen aleatorio al TTL (jitter) para repartir las expiraciones.

Breakdown (caída de una clave caliente)

Una sola clave muy popular expira y, en ese instante, miles de peticiones simultáneas fallan y van todas a recalcularla a la BBDD. Es el stampede, que vemos ahora.

Cache stampede y cómo evitarlo #

El stampede (o dogpile) es el caso peligroso de breakdown: cuando expira una clave caliente, todas las peticiones que la querían tienen un miss a la vez y se lanzan en tropel a reconstruirla. La consulta cara que la caché evitaba se ejecuta cientos de veces de golpe y puede tumbar la BBDD.

1 Clave caliente expiraJusto en hora punta
2 Todas tienen MISSNinguna encuentra el dato
3 La BBDD se ahoga500× la misma query cara

El daño no es el dato que falta: es la estampida de recálculos simultáneos sobre la base de datos.

La defensa habitual combina dos ideas:

  • Lock (mutex): que solo la primera petición que ve el miss recalcule; las demás esperan brevemente o devuelven el valor anterior. Un SET NX con TTL hace de cerrojo.
  • Recalcular antes de expirar: refrescar la clave de forma proactiva cuando le queda poco TTL, en lugar de esperar a que caiga a cero.
php · lock.php 
<?php

declare(strict_types=1);

// Solo el primero en poner el lock recalcula; vence solo por TTL
$gotLock = $this->redis->set(
    "lock:product:{$productId}",
    "1",
    ['nx', 'ex' => 10],
);

if ($gotLock) {
    $product = $this->fetchFromDatabase($productId);
    $this->redis->setex("product:{$productId}", 600, json_encode($product));
    $this->redis->del("lock:product:{$productId}");
}

No te montes Redlock para esto

Un lock distribuido perfecto es un problema serio (de los de sistemas distribuidos). Para evitar un stampede no necesitas esa garantía: un SET NX con TTL que "casi siempre" deja pasar a uno solo ya reduce el tropel de 500 a 1. Suficiente.

Resumen #

Lo que te llevas

  • Cachear es cambiar frescura por velocidad; si un dato no puede desactualizarse nunca, no lo caches.
  • Cache-aside: mira Redis; si hay miss, lee de la BBDD y guarda con TTL. La primera paga, las demás vuelan.
  • Pon TTL siempre y, al escribir, borra la clave en vez de reescribirla (evita carreras).
  • Reconoce los tres fallos: penetration (cachea el nulo), avalanche (TTL con jitter), breakdown (lock).
  • El stampede se ataca con un SET NX de cerrojo y/o refrescando la clave antes de que expire.
