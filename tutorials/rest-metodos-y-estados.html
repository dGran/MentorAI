Verbos y códigos de estado con criterio

Si la URL nombra el recurso, el método dice qué le haces y el código de estado dice cómo fue. Elegir bien los dos es lo que hace que un cliente entienda tu API sin preguntarte. Aquí está el criterio para no dudar.

Diseñar bien la capa HTTP de una API se reduce a dos decisiones que se repiten en cada endpoint: qué método expresa lo que el cliente quiere hacer, y qué código de estado resume lo que pasó. Las dos tienen respuestas con criterio, no a gusto.

Un verbo para cada intención #

HTTP trae un puñado de métodos con significado acordado. En REST cada uno expresa una intención sobre el recurso de la URL. Estos son los cinco del día a día:

MétodoIntenciónEjemplo
GETLeer la representación de un recurso. Nunca modifica nada.GET /pedidos/1001
POSTCrear un recurso nuevo en una colección, o disparar un proceso.POST /pedidos
PUTReemplazar por completo un recurso (o crearlo en una URL conocida).PUT /usuarios/42
PATCHModificar parcialmente un recurso: solo los campos que mandas.PATCH /usuarios/42
DELETEBorrar el recurso.DELETE /pedidos/1001

La regla de oro: el método es el verbo; la URL nunca lo es. En cuanto veas /crearPedido o /usuarios/borrar, sabes que algo se ha torcido: ese verbo pertenece al método HTTP. Fíjate también en que POST va contra la colección (/pedidos) porque aún no existe el recurso, mientras que PUT, PATCH y DELETE van contra un recurso concreto que ya tiene URL.

¿Y las acciones que no son CRUD?

Un "publicar", un "reembolsar", un "enviar". Lo más limpio es modelarlas como un recurso (POST /pedidos/1001/reembolsos) o como un cambio de estado vía PATCH. Cuando de verdad no encaje, un POST a una sub-URL de acción (POST /pedidos/1001/envio) es un compromiso aceptado. Lo vemos en la lección de diseño de URLs.

PUT reemplaza, PATCH parchea #

Es la confusión más común. PUT manda la representación completa del recurso: lo que envías es el nuevo estado entero, y los campos que omitas se entienden como borrados o reseteados. PATCH manda solo el cambio: los campos que incluyes se actualizan y el resto se queda como estaba.

PATCH /usuarios/42 — cambiar el email

  • { "email": "nuevo@ej.com" }
  • El nombre y el resto siguen intactos.
  • Mandas lo que cambia.

PUT /usuarios/42 — con el mismo cuerpo

  • { "email": "nuevo@ej.com" }
  • El nombre ausente: ¡se borra!
  • PUT espera el recurso entero.

De ahí la regla práctica: usa PATCH para cambios parciales (lo habitual en formularios de edición) y reserva PUT para cuando de verdad reemplazas el recurso completo. Mandar medio recurso por PUT es una fuente clásica de "se me han borrado campos sin querer".

Seguro e idempotente #

Dos propiedades de los métodos que el diseño debe respetar, porque clientes, proxies y caches cuentan con ellas:

  • Seguro: no modifica el estado del servidor. Solo lee. Un buscador puede recorrer todos los enlaces GET de tu sitio sin miedo a romper nada.
  • Idempotente: repetir la misma petición varias veces deja el mismo resultado que hacerla una vez. Si se pierde la respuesta y el cliente reintenta, no pasa nada malo.
Método¿Seguro?¿Idempotente?Por qué
GETSíSíSolo lee; leer mil veces no cambia nada.
PUTNoSíDejar el recurso en el estado X repetido = estado X.
DELETENoSíBorrar lo ya borrado deja el mismo resultado (no existe).
PATCHNoDependeIdempotente si fija valores; no lo es si hace "suma 1".
POSTNoNoRepetirlo crea un recurso nuevo cada vez (otro pedido).

La consecuencia de diseño es directa: como POST no es idempotente, un cliente que reintenta tras un timeout puede crear pedidos duplicados. Por eso las operaciones de crear con dinero de por medio necesitan una clave de idempotencia explícita. Es un tema con suficiente miga propia: lo desarrolla el tutorial Idempotencia, y volvemos a aplicarlo en la lección de errores y validación.

No hagas un GET con efectos

El error más caro contra "seguro": GET /pedidos/1001/cancelar. Un prefetch del navegador, un crawler o un acelerador de enlaces lo dispararán solos y cancelarán pedidos sin que nadie pulse nada. Cualquier cosa que cambie estado va por POST, PUT, PATCH o DELETE; nunca por GET.

Códigos de estado con criterio #

El código de estado es el resumen de tres dígitos de cómo fue la petición. La primera cifra ya cuenta la historia, y es lo primero que mira cualquier cliente:

FamiliaSignificaDe quién es la "culpa"
2xxÉxito. Se hizo lo que pedías.—
3xxRedirección. El recurso está en otro sitio.—
4xxError del cliente. La petición está mal.Del que llama (corregir y reintentar tiene sentido).
5xxError del servidor. La petición estaba bien, el servidor falló.Del servidor (reintentar más tarde puede funcionar).

Esa frontera 4xx/5xx es la más importante de respetar: define quién tiene que actuar. Un dato mal validado es 4xx (el cliente lo arregla); una excepción no capturada o una base de datos caída es 5xx (el cliente no puede hacer nada salvo reintentar). Confundirlas manda a todo el mundo a buscar el problema en el sitio equivocado.

La elección del código, caso a caso #

Dentro de cada familia, estos son los códigos que cubren casi todo lo que devuelve una API. No necesitas más para empezar:

CódigoCuándo
200 OKGET, PATCH o PUT que salen bien y devuelven cuerpo.
201 CreatedPOST que crea un recurso. Acompáñalo de la cabecera Location con su URL.
204 No ContentÉxito sin cuerpo que devolver. Típico de DELETE.
400 Bad RequestLa petición está malformada o falla la validación de entrada.
401 UnauthorizedNo te has autenticado (falta o caduca el token). Realmente es "no autenticado".
403 ForbiddenEstás autenticado, pero no tienes permiso para esto.
404 Not FoundEl recurso no existe (o no quieres revelar que existe).
409 ConflictChoque con el estado actual: un email ya registrado, una edición concurrente.
422 Unprocessable EntitySintaxis correcta pero semántica inválida. Muy usado para errores de validación de campos.
500 Internal Server ErrorAlgo reventó en el servidor. Es tu bug, no el del cliente.

El par 201 + Location es la respuesta canónica de un POST que crea: cuenta que se creó y dónde vive lo creado.

http · crear un recurso: 201 + Location 
POST /pedidos HTTP/1.1
Content-Type: application/json

{ "producto": "teclado", "cantidad": 1 }

HTTP/1.1 201 Created
Location: /pedidos/1001
Content-Type: application/json

{ "id": 1001, "estado": "pendiente_de_pago" }

401 vs 403, la pareja que todos confunden

Truco mnemotécnico: 401 es "no sé quién eres" (autenticación: identifícate). 403 es "sé quién eres y aun así no puedes" (autorización: permisos). El detalle de tokens y permisos está en Autenticación.

Errores que se cuelan #

Tres antipatrones de la capa HTTP que aparecen una y otra vez:

Así no

  • 200 OK con {"error":"no encontrado"} dentro: el cliente cree que fue bien.
  • POST para todo, incluso para leer.
  • Un 500 cuando el cliente mandó un campo inválido (es culpa suya, va 4xx).
  • Inventar códigos raros (299, 420) que nadie interpreta.

Así sí

  • El código de estado dice la verdad; el cuerpo da el detalle.
  • Cada verbo para su intención (lees con GET).
  • 4xx para errores del cliente, 5xx para fallos del servidor.
  • Códigos estándar; el cliente ya sabe interpretarlos.

El primero es el más dañino: devolver 200 con un error metido en el cuerpo rompe a todos los que se fían del código de estado —monitorización, reintentos, caches— y obliga a cada cliente a leer el JSON para saber si algo falló. El código de estado es la señal; que diga la verdad. Cómo dar el detalle del error de forma consistente es justo lo que veremos en la lección de errores y validación.

Resumen #

Lo que te llevas

  • El método es el verbo (GET lee, POST crea, PUT reemplaza, PATCH parchea, DELETE borra); la URL nunca lleva el verbo.
  • PUT manda el recurso completo (lo que omitas se pierde); PATCH manda solo los campos a cambiar.
  • GET es seguro (no modifica); GET, PUT y DELETE son idempotentes; POST no lo es: repetirlo crea duplicados.
  • La frontera 4xx/5xx dice de quién es el problema: 4xx lo arregla el cliente, 5xx es fallo del servidor.
  • Nunca devuelvas 200 con un error dentro: el código de estado es la señal y tiene que decir la verdad.
