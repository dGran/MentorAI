Ramas y flujo de trabajo

Las ramas son lo que convierte Git en una herramienta de equipo: líneas de trabajo paralelas que luego se combinan. Aquí ves cómo crearlas y moverte, la diferencia entre merge y rebase (la pregunta clásica), cómo resolver un conflicto sin pánico y el flujo de feature branches y pull requests.

Una rama te deja desarrollar una funcionalidad aislada del código estable, sin romper lo que ya funciona ni chocar con lo que hacen los demás. Cuando está lista, la combinas. Esa mecánica —separar para trabajar, juntar al terminar— es la columna vertebral de cómo se programa en equipo.

Qué es una rama #

Técnicamente, una rama es un puntero ligero a un commit. Cuando creas una rama y haces commits, ese puntero avanza, dejando la rama principal (main) intacta. Por eso crear ramas en Git es instantáneo y barato: no copia el proyecto, solo mueve una etiqueta. HEAD es otro puntero especial: marca en qué rama estás ahora.

Crear y moverse #

ComandoQué hace
git branchLista las ramas; marca con * en la que estás.
git branch nueva-funcionCrea una rama nueva (pero no te cambia a ella).
git switch nueva-funcionTe cambia a esa rama.
git switch -c nueva-funcionCrea la rama y te cambia a ella, de una vez.
git switch mainVuelves a la rama principal.
git branch -d nueva-funcionBorra una rama ya integrada.
bash · empezar una funcionalidad 
# Crear la rama y saltar a ella
git switch -c feature/factura-pdf

# ...trabajas y haces commits aquí, sin tocar main...

# Volver a la principal cuando quieras
git switch main

switch y restore, los nuevos checkout

Verás mucho git checkout para cambiar de rama: es el comando clásico, hace muchas cosas distintas. Git lo dividió en dos más claros: git switch (cambiar de rama) y git restore (descartar cambios). Ambos siguen funcionando; switch es el recomendado hoy.

Merge vs rebase #

Llega el momento de combinar tu rama con main. Hay dos formas, y entender la diferencia te ahorra muchos sustos.

Merge crea un commit nuevo que une las dos historias: conserva exactamente lo que pasó, ramas incluidas. Rebase reescribe tus commits como si los hubieras hecho a partir de la última versión de main: deja una historia lineal, más limpia, pero cambia tus commits (tienen hashes nuevos).

Merge

  • Une con un commit de merge.
  • Historia real, con sus ramas y bifurcaciones.
  • No reescribe nada: seguro siempre.
  • El grafo puede quedar enmarañado.

Rebase

  • Reaplica tus commits sobre la punta de main.
  • Historia lineal y limpia, fácil de leer.
  • Reescribe commits (hashes nuevos).
  • Nunca sobre commits ya compartidos.
bash · las dos maneras de integrar 
# MERGE: traer mi rama a main con un commit de unión
git switch main
git merge feature/factura-pdf

# REBASE: poner mis commits encima de la última main
git switch feature/factura-pdf
git rebase main

La regla de oro del rebase

Nunca hagas rebase de commits que ya has compartido (subido al remoto y que otros pueden tener). Como reescribe la historia, dejarías a tus compañeros con una versión incompatible. La norma simple: rebase solo sobre tu trabajo local, merge para lo que ya es público.

Resolver conflictos #

Un conflicto ocurre cuando dos ramas tocan las mismas líneas de un fichero y Git no sabe cuál ganar. No es un error ni un drama: es Git pidiéndote que decidas tú. Marca el fichero con las dos versiones y espera tu elección.

bash · así marca Git un conflicto en el fichero 
<<<<<<< HEAD
$total = $base * 1.21;        # lo que hay en tu rama
=======
$total = $base * self::IVA;   # lo que viene de la otra rama
>>>>>>> feature/factura-pdf

Para resolverlo: editas el fichero y dejas la versión correcta (borrando las marcas <<<, ===, >>>), lo añades a staging y confirmas. El IDE de PhpStorm te da una vista de tres paneles para esto, pero por debajo hace exactamente lo mismo.

bash · cerrar el conflicto 
# Tras editar y dejar la versión buena
git add src/Factura.php
git commit          # cierra el merge (o: git rebase --continue)

El flujo de trabajo #

La forma más extendida de organizar todo esto en equipo es el feature branch workflow: main siempre estable y desplegable, y cada tarea en su propia rama que se integra mediante una pull request (o merge request).

1 Rama desde maingit switch -c feature/x
2 Push y Pull Requestotros la revisan
3 Merge a mainy se borra la rama

Cada tarea vive en su rama hasta que pasa la revisión y se integra. main nunca se rompe.

La pull request es donde entra la revisión de código y, normalmente, los tests automáticos: nadie mete nada en main sin que otra persona (y la máquina) le den el visto bueno. Es el punto donde Git deja de ser una herramienta personal y se vuelve el motor del trabajo en equipo.

Resumen #

Lo que te llevas

  • Una rama es un puntero ligero a un commit; crearlas es barato. git switch -c crea y salta.
  • Merge une conservando la historia (un commit de unión); rebase la reescribe lineal y limpia.
  • Regla de oro: nunca rebase de commits ya compartidos. Un conflicto no es un error: editas, add y confirmas.
  • El feature branch workflow: main estable, una rama por tarea, integrada por pull request con revisión y tests.
