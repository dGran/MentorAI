Índices y B-tree: por qué tu query es lenta

La misma consulta tarda 2 milisegundos con diez filas y 8 segundos con diez millones. No es que la base de datos sea lenta: es que sin un índice no le has dado más opción que leerlo todo. Entender el B-tree que hay detrás es entender por qué.

Un índice de base de datos es, en el fondo, el mismo truco que el índice alfabético de un diccionario: en vez de leer página a página, saltas directo a la letra. La estructura que lo hace posible casi siempre es un B-tree, primo del árbol que viste en estructuras de datos.

El problema: leerlo todo #

Cuando consultas una columna sin índice, la base de datos no tiene por dónde empezar: debe mirar todas las filas para ver cuáles cumplen. Es un full table scan, y su coste es O(n): dobla las filas y dobla el tiempo. Con mil filas no lo notas; con diez millones, cada consulta se arrastra.

Un índice cambia el juego: mantiene los valores de una columna ordenados en una estructura aparte que apunta a la fila real. Buscar deja de ser O(n) y pasa a O(log n): en una tabla de diez millones de filas son unos 23 saltos, no diez millones de lecturas.

Anatomía de un B-tree #

Un B-tree es un árbol ordenado y equilibrado, pensado para el disco. Cada nodo no guarda un solo valor sino muchos, y reparte el rango entre sus hijos. Buscar es descender preguntando «¿en qué rango cae mi valor?» hasta llegar a la hoja, que apunta a la fila.

1 Raízrangos: …–M, M–Z
2 Ramaacota: P–R
3 Hojapuntero a la fila

Cada salto descarta una porción enorme de los datos. Por eso un árbol de pocos niveles cubre millones de filas.

La clave es que el árbol es poco profundo y muy ancho: como cada nodo ramifica en cientos de hijos, tres o cuatro niveles bastan para millones de filas. Y las hojas están enlazadas entre sí en orden, así que un BETWEEN o un ORDER BY sobre la columna indexada se sirve recorriéndolas en fila, sin reordenar nada.

B-tree para rangos, hash para igualdad

Un índice hash (el del tutorial anterior) es O(1) para =, pero no sabe de orden: no sirve para <, BETWEEN ni ORDER BY. El B-tree es algo más lento en la igualdad pero mantiene el orden, y por eso es el índice por defecto en casi todos los motores relacionales.

Leer un EXPLAIN #

No adivines si un índice se usa: pregúntaselo a la base de datos. EXPLAIN delante de una consulta te muestra el plan de ejecución: cómo piensa resolverla, qué índice elige y cuántas filas estima leer.

sql · antes y después del índice 
-- Sin índice: escanea la tabla entera
EXPLAIN SELECT * FROM usuarios WHERE email = 'ana@example.com';
-- type: ALL      rows: 1000000   (lee el millón de filas)

CREATE INDEX idx_usuarios_email ON usuarios (email);

-- Con índice: baja por el árbol hasta la fila
EXPLAIN SELECT * FROM usuarios WHERE email = 'ana@example.com';
-- type: ref      rows: 1         (un solo recorrido del B-tree)

La columna type es la que delata el plan. De peor a mejor, estos son los valores que más verás:

typeQué significa
ALLFull scan: lee toda la tabla. La señal de alarma.
indexRecorre el índice entero (mejor que la tabla, pero aún todo).
rangeUsa el índice para un rango (BETWEEN, >, IN).
refUsa el índice para una igualdad no única. Muy bueno.
eq_ref / constUna sola fila por clave única o primaria. Lo óptimo.

Índices compuestos y orden #

Un índice puede cubrir varias columnas. El detalle que casi nadie tiene claro: el orden de las columnas importa, y mucho. Un índice compuesto se ordena por la primera columna; dentro de cada valor, por la segunda; y así. Es como una guía telefónica ordenada por apellido y luego por nombre.

sql · prefijo izquierdo 
CREATE INDEX idx_pedidos_cliente_fecha
    ON pedidos (cliente_id, fecha);

-- USA el índice: arranca por cliente_id (la primera columna)
SELECT * FROM pedidos
WHERE cliente_id = 42 AND fecha >= '2026-01-01';

-- USA el índice: filtra por cliente_id (el prefijo izquierdo basta)
SELECT * FROM pedidos WHERE cliente_id = 42;

-- NO lo usa: salta la primera columna y filtra solo por fecha
SELECT * FROM pedidos WHERE fecha >= '2026-01-01';

La regla del prefijo izquierdo

Un índice compuesto (a, b, c) sirve para filtrar por a, por a, b o por a, b, c, pero no por b suelto ni por c. Pon primero la columna por la que siempre filtras, y deja para el final la que usas en rangos.

Cuándo un índice NO se usa #

Tener el índice no garantiza usarlo. El error más común: envolver la columna indexada en una función o un cálculo. El índice guarda el valor crudo de la columna, no el resultado de la función, así que el motor no puede aprovecharlo y vuelve al full scan.

sql · que la consulta sea sargable 
-- NO usa el índice: la función envuelve la columna
SELECT * FROM eventos WHERE YEAR(creado_en) = 2026;

-- SÍ lo usa: el mismo filtro reescrito como rango sobre la columna cruda
SELECT * FROM eventos
WHERE creado_en >= '2026-01-01'
  AND creado_en <  '2027-01-01';

Una consulta que deja la columna «desnuda» a un lado del operador se llama sargable (de Search ARGument ABLE): es la que el optimizador puede resolver con el índice. Otros casos en los que el índice se ignora: comodín al principio de un LIKE '%texto', un OR sobre columnas distintas, o una columna con poquísimos valores distintos (un booleano) donde el full scan sale más barato.

El índice no es gratis #

Un índice acelera las lecturas, pero hay que pagarlo. Ocupa espacio en disco y, sobre todo, cada INSERT, UPDATE o DELETE debe actualizar también el árbol para mantenerlo ordenado. Indexar de más no es gratis: ralentiza justo las escrituras.

Indexa cuando

  • La columna aparece mucho en WHERE, JOIN u ORDER BY.
  • Tiene muchos valores distintos (alta cardinalidad: email, DNI).
  • La tabla se lee mucho más de lo que se escribe.

Piénsatelo cuando

  • La columna tiene pocos valores (un estado con 3 opciones).
  • La tabla recibe escrituras a ráfagas y se lee poco.
  • Ya hay cinco índices: cada uno frena cada escritura.

Errores comunes #

ErrorPor qué duele
Envolver la columna en una función WHERE YEAR(fecha) = 2026 ignora el índice. Reescríbelo como rango.
Orden equivocado en un índice compuesto Si filtras siempre por b pero el índice es (a, b), no se usa.
Indexarlo todo «por si acaso» Cada índice frena las escrituras y ocupa espacio. Indexa lo que se consulta.
Confiar en la query, no en el EXPLAIN El optimizador decide con estadísticas; lo único fiable es mirar el plan real.
LIKE '%texto' para buscar El comodín inicial impide usar el B-tree; eso pide un índice de texto completo.

Resumen #

Lo que te llevas

  • Sin índice, filtrar es un full scan O(n): la base de datos lee la tabla entera.
  • Un B-tree es ancho y poco profundo: encuentra una fila en O(log n) y mantiene el orden para rangos y ORDER BY.
  • EXPLAIN te dice la verdad: mira type y huye de ALL.
  • En un índice compuesto manda el prefijo izquierdo: pon primero la columna por la que siempre filtras.
  • Mantén las consultas sargable (columna desnuda) y no indexes de más: cada índice frena las escrituras.
