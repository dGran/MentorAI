Hashing vs cifrado: cómo se guardan las contraseñas

«Hashear» y «cifrar» suenan parecido y se confunden a diario, pero resuelven problemas opuestos: uno está pensado para poder deshacerse, el otro para no poder. Entender la diferencia es lo que separa guardar contraseñas bien de filtrar la base de datos entera el día de una brecha.

Una caja fuerte tiene llave: guardas algo dentro y, con la llave correcta, lo recuperas tal cual. Una trituradora de papel, no: metes un documento y sale confeti del que nadie reconstruye el original. El cifrado es la caja fuerte; el hashing, la trituradora. Y las contraseñas, sorprendentemente, quieres triturarlas.

Dos herramientas, dos objetivos #

La confusión nace de que ambos «transforman» un dato en algo ilegible. Pero la pregunta que los separa es simple: ¿quieres poder recuperar el original?

Cifrado

  • Reversible con una clave
  • Objetivo: confidencialidad
  • «Guárdalo de forma que solo yo pueda leerlo luego»
  • Ej.: datos de tarjeta, mensajes, ficheros

Hashing

  • Irreversible por diseño
  • Objetivo: verificación / integridad
  • «No me importa recuperarlo, solo comprobar si coincide»
  • Ej.: contraseñas, checksums, huellas de fichero

Cifrado: reversible a propósito #

Cifrar transforma un texto claro en texto cifrado usando una clave. Con la clave correcta, el proceso se deshace y recuperas el original. Sin ella, el resultado es ruido. Hay dos familias:

Tipo Claves Uso típico
Simétrico La misma clave cifra y descifra (AES). Cifrar datos en reposo: una columna sensible, un fichero.
Asimétrico Clave pública cifra, privada descifra (RSA, curvas elípticas). Intercambiar secretos sin compartir clave antes; base de TLS.

Ya usas cifrado asimétrico

El handshake de TLS que protege cada web «https» usa criptografía asimétrica para acordar una clave de sesión, y luego cifra el tráfico con ella. La reversibilidad ahí es el objetivo: el servidor tiene que poder leer lo que le mandas.

Hashing: de un solo sentido #

Una función hash criptográfica toma una entrada de cualquier tamaño y devuelve una huella de tamaño fijo. Tiene tres propiedades que la hacen útil para seguridad:

  • Determinista: la misma entrada da siempre la misma huella.
  • Irreversible: de la huella no se puede volver al original.
  • Resistente a colisiones: es inviable encontrar dos entradas con la misma huella, y un cambio mínimo en la entrada cambia la huella por completo (efecto avalancha).

No es el hash de una tabla hash

Ojo con el nombre: el hash de una tabla hash busca ser rápido y repartir uniforme, y le da igual ser predecible. Un hash criptográfico busca justo lo contrario: ser caro de revertir e imposible de predecir. Mismo nombre, objetivos opuestos.

Por qué las contraseñas se hashean (y no se cifran) #

Para autenticar a un usuario no necesitas conocer su contraseña: solo necesitas comprobar que la que escribe coincide con la que registró. Y eso lo da el hashing sin guardar nunca el original.

1 RegistroHasheas la contraseña y guardas solo el hash
2 LoginHasheas lo que escribe el usuario
3 Comparas hashesCoinciden → acceso. Nunca guardaste el original

Si te roban la base de datos, el atacante se lleva huellas, no contraseñas. Con cifrado, en cambio, robar la clave revelaría todas.

"¿Puedes recuperar mi contraseña?" → no, y es buena señal

Si una web te puede enviar tu contraseña actual por email, es que la guarda de forma reversible: lo hace mal. Una web bien hecha solo puede resetearla, porque no conoce la tuya.

La sal, y por qué MD5/SHA no bastan #

Un hash a secas tiene dos debilidades para contraseñas. Primera: es demasiado rápido. MD5 o SHA-256 se diseñaron para ser veloces, y eso le viene de perlas a un atacante que prueba miles de millones de combinaciones por segundo con una GPU (fuerza bruta).

Segunda: al ser deterministas, dos usuarios con la misma contraseña tienen el mismo hash. Un atacante puede precalcular los hashes de las contraseñas más comunes (una rainbow table) y cruzarlos con tu base de datos de golpe.

La sal (salt) resuelve lo segundo: es un valor aleatorio único por contraseña que se añade antes de hashear. Así, dos contraseñas iguales producen hashes distintos y las tablas precalculadas dejan de servir. La sal se guarda junto al hash (no es secreta; su valor está en ser única).

Función ¿Para contraseñas? Por qué
md5, sha1 Nunca Rotas (colisiones) y rapidísimas. Sin sal, además.
sha256 No Segura como huella, pero demasiado rápida para frenar fuerza bruta.
bcrypt Sí Lenta a propósito y con coste ajustable; sal incorporada.
argon2id Sí (recomendada) Lenta y, además, dura en memoria: frena también a las GPU.

La idea clave: lentitud deliberada

bcrypt y argon2 son funciones de derivación de clave: hashes pensados para ser caros. Un login legítimo que tarda 100 ms no te molesta, pero multiplica por mil millones el coste de un ataque por fuerza bruta. Y el «coste» es un parámetro que subes con los años, según mejora el hardware.

Hacerlo bien en PHP #

La buena noticia: no tienes que elegir algoritmo, generar la sal ni gestionar el formato a mano. PHP trae password_hash y password_verify, que hacen lo correcto por defecto.

php · PasswordService.php 
<?php

declare(strict_types=1);

final class PasswordService
{
    public function hash(string $plainPassword): string
    {
        return password_hash($plainPassword, PASSWORD_ARGON2ID);
    }

    public function isValid(string $plainPassword, string $storedHash): bool
    {
        return password_verify($plainPassword, $storedHash);
    }

    public function needsRehash(string $storedHash): bool
    {
        return password_needs_rehash($storedHash, PASSWORD_ARGON2ID);
    }
}

Tres detalles que marcan la diferencia:

  • La sal va dentro del hash. El resultado de password_hash ya incluye algoritmo, coste y sal. No guardes una columna de sal aparte: guarda la cadena entera y password_verify sabe leerla.
  • Compara con password_verify, nunca con ===. La función compara en tiempo constante para no filtrar información por el tiempo de respuesta (timing attack).
  • Re-hashea al iniciar sesión. Si subes el coste con el tiempo, password_needs_rehash te dice cuándo regenerar el hash de un usuario que acaba de demostrar su contraseña.

No inventes tu propio esquema

«Hago SHA-256 dos veces y le concateno un secreto» es la receta clásica del desastre. La criptografía hecha en casa falla por sitios que no ves. Usa password_hash/argon2 y céntrate en el resto de tu aplicación.

Resumen #

Lo que te llevas

  • Cifrado es reversible con una clave (confidencialidad); hashing es de un solo sentido (verificación).
  • Las contraseñas se hashean, no se cifran: solo necesitas comprobar coincidencia, y así una brecha no revela las claves.
  • La sal (única por contraseña) mata las rainbow tables; el coste alto frena la fuerza bruta.
  • MD5/SHA no valen para contraseñas (rápidas); usa bcrypt o, mejor, argon2id, lentos a propósito.
  • En PHP: password_hash + password_verify (sal incluida, comparación en tiempo constante). No inventes tu propio esquema.
