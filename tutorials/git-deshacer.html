Deshacer y rescatar

La parte que más miedo y más alivio da: cómo volver atrás cuando algo sale mal. Descartar cambios, deshacer commits, guardar trabajo a medias, corregir el último commit y —el truco que casi nadie conoce— recuperar lo que creías perdido para siempre. Con esto, Git deja de dar respeto.

«Deshacer» en Git no es un solo botón: depende de dónde está lo que quieres deshacer (working directory, staging o ya commiteado) y de si ese commit es tuyo o ya lo has compartido. Aclarar ese mapa es la mitad de la batalla; la otra mitad es saber que en Git casi nada se pierde de verdad.

Descartar cambios sin commitear #

Has tocado un fichero y quieres volver a como estaba en el último commit, o sacar algo de staging que añadiste por error. Eso es git restore.

bash · tirar cambios o vaciar staging 
# Descartar los cambios de un fichero (vuelve al último commit)
git restore src/Factura.php

# Sacar un fichero de staging, pero conservar el cambio
git restore --staged src/Factura.php

Descartar cambios sin commitear no tiene vuelta atrás

Lo que aún no ha pasado por un commit (ni por stash) Git no lo conoce, así que restore lo borra de verdad. Es el único rincón realmente peligroso: en cuanto algo está commiteado, casi siempre se puede recuperar (lo verás al final con reflog).

reset vs revert #

Las dos deshacen commits, pero de forma opuesta. reset mueve la rama hacia atrás, como si los commits no hubieran existido: para tu trabajo local. revert crea un commit nuevo que anula otro anterior, sin borrar nada: para deshacer algo ya compartido.

reset tiene tres sabores según qué hace con tus cambios:

ModoQué pasa con los cambios de los commits deshechos
--softVuelven a staging, listos para recommitear.
--mixed (por defecto)Vuelven al working directory, sin stage.
--hardSe borran. El único reset peligroso.
bash · deshacer commits, local vs compartido 
# LOCAL: deshacer el último commit, conservando los cambios sin stage
git reset HEAD~1

# LOCAL: deshacer y tirar también los cambios (cuidado)
git reset --hard HEAD~1

# COMPARTIDO: anular un commit ya subido con uno nuevo que lo revierte
git revert abc123

La misma lógica que el rebase

reset reescribe la historia, así que vale la misma regla de oro: úsalo solo en commits locales que no has compartido. Para deshacer algo que ya está en el remoto, revert es la opción segura, porque añade en vez de reescribir.

Corregir el último commit #

Te has dejado un fichero o has escrito mal el mensaje del commit que acabas de hacer. --amend rehace ese último commit en vez de crear uno nuevo.

bash · arreglar el commit recién hecho 
# Cambiar solo el mensaje del último commit
git commit --amend -m "Mensaje corregido"

# Añadir un fichero olvidado al último commit
git add archivo-olvidado.php
git commit --amend --no-edit

Como amend también reescribe, solo en el último commit y solo si no lo has compartido todavía.

Guardar para luego (stash) #

Estás a medias de algo y necesitas cambiar de rama ya (un bug urgente en producción), pero no quieres hacer un commit a medio cocer. git stash aparca tus cambios en un cajón y te deja el directorio limpio; luego los recuperas.

bash · aparcar y recuperar trabajo 
# Guardo lo que tengo a medias y el directorio queda limpio
git stash

# ...cambio de rama, arreglo el bug, vuelvo...

# Recupero lo aparcado
git stash pop

# Ver qué tengo aparcado
git stash list

Rescatar lo perdido (reflog) #

El as en la manga. Git apunta en un registro privado, el reflog, cada vez que HEAD se mueve: cada commit, cambio de rama, reset… Aunque hagas un reset --hard y creas haber perdido commits, casi siempre siguen ahí, y el reflog te da su identificador para volver.

bash · recuperar un commit "perdido" 
# Ver todo lo que ha hecho HEAD, incluido lo que "borraste"
git reflog

# Volver al estado de antes del reset desafortunado
git reset --hard abc123

La gran lección: en Git casi nada se pierde

Una vez que algo ha pasado por un commit, vive en el historial aunque ya no lo veas, y el reflog es tu red. Por eso conviene commitear a menudo: cada commit es un punto de guardado al que siempre podrás volver. El miedo a «romper algo» se desvanece cuando sabes que todo está recuperable.

Resumen #

Lo que te llevas

  • restore descarta cambios sin commitear (lo único sin vuelta atrás); amend corrige el último commit.
  • reset deshace commits locales (soft/mixed/hard según qué pasa con los cambios); revert deshace lo ya compartido sin reescribir.
  • stash aparca trabajo a medias para cambiar de contexto y recuperarlo con stash pop.
  • El reflog recupera casi cualquier cosa commiteada: en Git, lo que pasó por un commit no se pierde. Commitea a menudo y pierde el miedo.
