Números: enteros, complemento a dos y coma flotante

0.1 + 0.2 no es 0.3, y no es un bug de tu lenguaje: es física de cómo caben los números en un número fijo de bits. Entender enteros y flotantes te ahorra el peor tipo de error, el que parece imposible y aparece justo en la factura del cliente.

Un número en un ordenador no es el número abstracto de las matemáticas: es un patrón de bits de tamaño fijo. Esa frontera —cuántos bits— define qué valores caben, qué pasa al pasarse y por qué los decimales mienten. Todo lo de aquí sale de la misma idea: el espacio es finito.

Enteros: tamaño fijo #

Un entero ocupa un número fijo de bits (hoy casi siempre 32 o 64), y eso fija cuántos valores distintos puede representar: 2ⁿ. Con 8 bits hay 256 patrones. Si solo guardas positivos (sin signo), van del 0 al 255. Si necesitas negativos (con signo), ese mismo rango se reparte entre positivos y negativos.

BitsSin signoCon signo
80 … 255−128 … 127
160 … 65.535−32.768 … 32.767
320 … ~4,29 mil millones±~2,14 mil millones
640 … ~1,8·10¹⁹±~9,2·10¹⁸

PHP solo tiene enteros con signo

En PHP no existe el entero sin signo: el tamaño es PHP_INT_SIZE (8 bytes en sistemas de 64 bits) y el máximo, PHP_INT_MAX. En SQL sí existe UNSIGNED, que dobla el techo positivo de una columna a cambio de no admitir negativos: útil para un id autoincremental que nunca será negativo.

php · los límites del entero 
<?php

declare(strict_types=1);

echo PHP_INT_SIZE;       // 8  -> 8 bytes = entero de 64 bits
echo PHP_INT_MAX;        // 9223372036854775807

$overflow = PHP_INT_MAX + 1;
var_dump($overflow);     // float(9.2233720368548E+18) -> deja de ser int

Negativos: complemento a dos #

¿Cómo se guarda un −5 si solo hay ceros y unos? La idea ingenua (reservar un bit para el signo) trae dos ceros (+0 y −0) y complica la resta. La solución universal es el complemento a dos: el negativo de un número se obtiene invirtiendo todos sus bits y sumando 1.

PasoBits (8)Valor
50000 01015
Invertir1111 1010—
Sumar 11111 1011−5

El bit más a la izquierda acaba siendo el signo: si es 1, el número es negativo. La gracia es que sumar y restar funcionan con el mismo circuito sin tratar el signo aparte: 5 + (−5) en estos 8 bits da 1 0000 0000, el 1 se desborda fuera y queda 0. Exacto.

Overflow: cuando no cabe #

Si sumas 1 al valor máximo, el resultado no cabe en los bits disponibles. Qué pasa entonces depende del lenguaje, y conviene saber cuál es el tuyo:

Wraparound (C, Java, Go…)

  • El bit que sobra se descarta y el número "da la vuelta".
  • 127 + 1 en un int8 = −128.
  • Silencioso: ni error ni aviso. El bug más traicionero.

Promoción a float (PHP, Python…)

  • Al pasarse, el entero se convierte en coma flotante.
  • No se rompe el orden de magnitud, pero se pierde precisión exacta.
  • Un id gigante puede dejar de ser comparable al céntimo.

El overflow vive en la base de datos, no solo en el código

Una columna INT de MySQL desborda a los ~2.147 millones. Más de un sistema de contadores o de IDs ha muerto ahí: el lenguaje calculaba bien con 64 bits, pero la columna era de 32. Para claves que crecen, BIGINT.

Coma flotante: IEEE-754 #

Para representar decimales y números enormes con los mismos bits se usa la coma flotante, estandarizada como IEEE-754. La idea es la notación científica en binario: en vez de fijar la coma, se guarda un valor y dónde va la coma. Un double (lo que usa PHP siempre) reparte sus 64 bits así:

ParteDouble (64 bits)Qué guarda
Signo1 bitPositivo o negativo
Exponente11 bitsDónde va la coma (la magnitud)
Mantisa52 bitsLos dígitos significativos (la precisión)

De ahí salen dos límites: el rango (enorme, gracias al exponente) y la precisión (finita: unos 15–17 dígitos decimales, por la mantisa). Un double representa de maravilla números gigantes… pero solo con 52 bits de detalle. Y ahí está la trampa de la siguiente sección.

Por qué 0.1 + 0.2 ≠ 0.3 #

En base 10, 1/3 es 0,333…: periódico, imposible de escribir exacto con dígitos finitos. En base 2 pasa lo mismo con 0,1: en binario es una fracción periódica que no cabe exacta en 52 bits. El ordenador guarda el valor más cercano que puede, y ese minúsculo redondeo se arrastra en la suma.

php · la imprecisión, al descubierto 
<?php

declare(strict_types=1);

echo 0.1 + 0.2;                  // "0.3"  -> echo redondea a 14 dígitos y engaña
var_dump(0.1 + 0.2);             // float(0.30000000000000004) -> la verdad
var_dump(0.1 + 0.2 === 0.3);     // bool(false)

$epsilon = 0.00001;
var_dump(abs((0.1 + 0.2) - 0.3) < $epsilon);  // bool(true) -> así se compara

Nunca compares floats con ===

Dos cálculos que "deberían" dar lo mismo pueden diferir en el último bit. Compara si la diferencia es menor que un margen pequeño (epsilon), no por igualdad exacta. La igualdad exacta entre floats casi siempre es un bug esperando a ocurrir.

Dinero: nunca en float #

El corolario práctico más importante: no representes dinero con float. Esos redondeos invisibles se acumulan factura a factura hasta que el total no cuadra por un céntimo, y un céntimo descuadrado en contabilidad es un incidente. Hay dos caminos buenos:

php · dinero sin perder un céntimo 
<?php

declare(strict_types=1);

// Opción A: enteros en la unidad mínima (céntimos). Exacto y rápido.
$priceCents = 1999;                         // 19,99 €
$taxCents = (int) round($priceCents * 0.21);
$totalCents = $priceCents + $taxCents;
echo number_format($totalCents / 100, 2, ',', '.');  // "24,19"

// Opción B: aritmética decimal con BCMath, en cadenas.
echo bcadd('0.10', '0.20', 2);              // "0.30" -> exacto, sin float

En la base de datos, lo mismo: una columna DECIMAL(10,2) guarda el valor exacto, frente a FLOAT/DOUBLE, que arrastran el mismo error que en el código. Para importes, DECIMAL o enteros; float jamás.

Errores comunes #

ErrorPor qué duele
Dinero en float / DOUBLE Los redondeos se acumulan y el total descuadra céntimos. Usa enteros o DECIMAL.
Comparar floats con === El último bit puede diferir entre cálculos equivalentes. Compara con un epsilon.
Columna INT para algo que crece sin fin Desborda a los ~2.147 millones. Para IDs y contadores grandes, BIGINT.
Asumir wraparound en PHP (o no asumirlo en C) PHP promociona a float; C/Java dan la vuelta en silencio. Saber cuál usas evita sorpresas.
(int) sobre un float "redondo" (int)(0.1 + 0.7) * 10 puede dar 7, no 8: truncas un 7,999…. Redondea antes con round.

Resumen #

Lo que te llevas

  • Un entero ocupa bits fijos, así que tiene un techo: el rango sale de cuántos bits y de si lleva signo.
  • Los negativos se guardan en complemento a dos (invertir y sumar 1); el bit alto hace de signo y la resta sale gratis.
  • Al pasarse hay overflow: wraparound silencioso (C/Java) o promoción a float (PHP). En BBDD, BIGINT para lo que crece.
  • La coma flotante (IEEE-754) tiene rango enorme pero precisión finita: 0.1 no cabe exacto, por eso 0.1 + 0.2 ≠ 0.3.
  • Dinero nunca en float: enteros de céntimos o DECIMAL/BCMath; y compara floats con epsilon, no con ===.
