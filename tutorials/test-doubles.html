Dobles de test

Para probar una clase aislada necesitas sustituir sus colaboradores por impostores controlados. «Mock» se usa para todos, pero hay cinco tipos con propósitos distintos: dummy, stub, spy, mock y fake. Saber cuál toca evita tests frágiles y mentirosos.

Un test unitario quiere probar una clase sola, pero esa clase casi nunca está sola: depende de un repositorio, un cliente HTTP, un reloj. Un doble de test es un objeto que ocupa el sitio de esa dependencia real, pero bajo tu control absoluto. Como en el cine: un doble que hace la escena peligrosa por el actor.

Por qué falsear dependencias #

Si la clase que pruebas llama a la base de datos real, el test deja de ser unitario: se vuelve lento, depende del estado de la base y falla por motivos ajenos a la lógica que querías comprobar. Sustituyendo esa dependencia por un doble, aíslas la unidad y controlas el escenario: «cuando el repositorio devuelva esto, mi clase debe hacer aquello».

Esto necesita inyección de dependencias

Solo puedes sustituir una dependencia si tu clase la recibe en vez de fabricarla. Por eso los dobles y la inyección por constructor van de la mano: el test pasa el doble por donde en producción entra el objeto real.

La familia de dobles #

«Doble de test» (test double) es el término paraguas que acuñó Gerard Meszaros. Por debajo hay cinco tipos, ordenados de menos a más listos:

Tipo Qué hace
Dummy Solo rellena un hueco. Se pasa para cumplir la firma pero nunca se usa.
Stub Devuelve respuestas predefinidas a las llamadas. No comprueba nada.
Spy Como un stub, pero además registra cómo lo llamaron, para revisarlo luego.
Mock Tiene expectativas: sabe qué llamadas debe recibir y falla si no ocurren.
Fake Una implementación real pero ligera (ej. un repositorio en memoria en vez de la base de datos).

La distinción que más importa es verificación de estado vs de comportamiento. Stubs y fakes te dejan comprobar el resultado (qué devolvió tu clase). Mocks y spies comprueban la interacción (a quién llamó y cómo). No son intercambiables: cada estilo de test prefiere uno.

Stub: dar respuestas #

Un stub responde lo que tú le digas, para montar el escenario. Aquí probamos que un servicio de saludo usa el nombre que devuelve el repositorio, sin tocar base de datos:

php · stub con PHPUnit 
<?php

declare(strict_types=1);

public function test_saluda_con_el_nombre_del_usuario(): void
{
    $users = $this->createStub(UserRepository::class);
    $users->method('nameOf')->willReturn('Ana');

    $greeter = new Greeter($users);

    $this->assertSame('Hola, Ana', $greeter->greet(42));
}

Fíjate en qué comprobamos: el resultado ('Hola, Ana'). El stub solo está para que el escenario exista; no nos importa cómo se usó.

Mock y spy: verificar llamadas #

A veces lo que importa no es el valor que devuelves, sino que hayas provocado un efecto: enviar un email, escribir en la cola. Como no hay valor que mirar, compruebas la interacción. Un mock declara por adelantado la expectativa y falla si no se cumple:

php · mock con expectativa 
<?php

declare(strict_types=1);

public function test_envia_un_email_al_registrarse(): void
{
    $mailer = $this->createMock(Mailer::class);

    $mailer->expects($this->once())
        ->method('send')
        ->with('ana@example.com');

    $service = new SignUp($mailer);
    $service->register('ana@example.com');
}

La diferencia entre mock y spy es cuándo verificas. El mock declara la expectativa antes (estilo «espera-actúa»); el spy te deja actuar primero y preguntar después («¿se llamó a send?»). PHPUnit unifica ambos en createMock; otras librerías los separan.

Fake: una implementación de verdad #

Un fake funciona, solo que de forma simplificada. El ejemplo canónico es un repositorio en memoria que implementa la misma interfaz que el real pero guarda en un array. No devuelve respuestas guionizadas: se comporta, guarda y recupera de verdad.

php · fake en memoria 
<?php

declare(strict_types=1);

final class InMemoryUserRepository implements UserRepository
{
    private array $users = [];

    public function save(User $user): void
    {
        $this->users[$user->id()] = $user;
    }

    public function byId(int $id): ?User
    {
        return $this->users[$id] ?? null;
    }
}

El fake suele envejecer mejor

Un fake reutilizable da tests legibles y robustos: pruebas contra algo que se comporta como el real, sin guionizar cada llamada. Por eso muchos equipos prefieren un buen fake a llenar los tests de mocks que hay que reajustar a cada refactor.

Cuál usar #

Una guía rápida que evita la mayoría de los líos:

  • ¿Solo necesitas una respuesta para montar el caso? Stub (o fake si hay estado).
  • ¿Lo que importa es que ocurrió un efecto sin valor de retorno (enviar, publicar)? Mock/spy.
  • ¿La dependencia se usa mucho y guionizarla cansa? Fake reutilizable.
  • ¿Solo rellenas un parámetro que no se toca? Dummy.

Errores típicos #

No mockees lo que no es tuyo

Falsear una librería de terceros o el cliente de la base de datos te ata a cómo crees que se comporta, no a cómo se comporta de verdad. Mejor envuelve esa dependencia en una interfaz tuya (un adaptador) y mockea esa; la librería real se prueba en un test de integración.

El otro gran error es sobre-mockear: tests llenos de expectativas sobre cada llamada interna. Acaban probando la implementación en vez del comportamiento, así que cualquier refactor —aunque no cambie nada visible— los rompe. Son tests que estorban; el siguiente tutorial va justo de eso. Regla sana: mockea las fronteras (lo que sale del sistema), no las tripas.

Resumen #

Lo que te llevas

  • Un doble de test sustituye una dependencia real para aislar la unidad. Requiere inyección de dependencias.
  • Stub da respuestas, mock/spy verifican llamadas, fake es una implementación ligera real, dummy solo rellena.
  • Comprueba estado (stub/fake) cuando hay un resultado; comportamiento (mock/spy) cuando lo que importa es un efecto.
  • No mockees lo ajeno ni sobre-mockees: atan el test a la implementación y lo vuelven frágil.
