Tests que no estorban

Una mala suite de tests es peor que ninguna: te frena en cada cambio, se rompe sin que nada falle de verdad y acaba ignorada. La diferencia entre tests que ayudan y tests que estorban está en qué pruebas y cómo.

Cada test que escribes es código que hay que leer, entender y mantener. Si ese coste supera lo que el test protege, has creado un pasivo, no un activo. Escribir buenos tests es, sobre todo, saber qué no hacer.

El test también es código #

Solemos tratar los tests como una zona de segunda donde todo vale: duplicación, nombres crípticos, montajes enormes. Pero un test vive tanto como el código que prueba y se lee más veces de las que se escribe. Un test ilegible que falla a las 3 de la madrugada no ayuda: confunde.

La pregunta que ordena todo este tutorial es: cuando este test falle, ¿me estará diciendo algo útil? Si la respuesta es «se rompió porque toqué un detalle interno que no cambia el comportamiento», el test estorba.

Comportamiento, no implementación #

Esta es la regla madre. Un buen test comprueba qué hace tu código visto desde fuera (entradas y salidas, efectos observables), no cómo lo hace por dentro. El «cómo» es libre de cambiar: ese es el sentido de refactorizar.

Prueba la implementación

  • Verifica que se llamó a tal método privado.
  • Espera un orden interno de llamadas concreto.
  • Se rompe al renombrar o reordenar sin cambiar nada visible.
  • Te desanima a mejorar el código.

Prueba el comportamiento

  • Da una entrada y comprueba la salida o el efecto.
  • No le importa el camino interno.
  • Sobrevive a cualquier refactor que conserve la conducta.
  • Te deja refactorizar con red.

La prueba del refactor

Un buen test sobrevive a un refactor y solo se rompe si cambias el comportamiento. Si reorganizas las tripas sin alterar lo que el código hace y media suite se pone roja, esos tests estaban atados a la implementación. Es la señal más clara de tests que estorban.

Los principios FIRST #

FIRST es un acrónimo que resume las cualidades de un buen test unitario:

Letra Qué pide
Fast Rápido. Milisegundos, para poder correr miles en cada cambio sin pereza.
Independent Independiente. Ningún test depende de otro ni del orden de ejecución.
Repeatable Repetible. Mismo resultado siempre, en tu máquina y en CI, hoy y mañana.
Self-validating Se valida solo. Pasa o falla sin que nadie interprete una salida a ojo.
Timely Oportuno. Escrito junto al código (en TDD, antes), no meses después.

El enemigo de «repetible»: lo no determinista

La fecha actual, números aleatorios, el orden de un array sin ordenar, una llamada de red: cualquiera convierte un test en flaky (falla a ratos sin que nada cambie). Inyecta el reloj y la aleatoriedad como dependencias para poder fijarlos en el test.

Tests frágiles #

Un test frágil se rompe ante cambios que no deberían afectarlo. Las causas más comunes:

  • Sobre-especificación: comprobar más de lo que el test pretende (el objeto entero cuando solo importa un campo).
  • Acoplamiento a la implementación: mocks que verifican cada llamada interna (lo viste en dobles de test).
  • Datos compartidos: tests que dependen de un estado global o del que dejó otro test antes.
  • Aserciones sobre formato volátil: comparar un mensaje de texto exacto o un JSON entero que cambia a menudo.

El antídoto general: asegura una sola cosa por test y hazlo por su resultado observable. Un test que afirma una idea clara solo se rompe cuando esa idea deja de cumplirse.

Qué no testear #

Tan importante como saber qué probar es saber qué dejar fuera. Aporta poco y cuesta mantener:

  • Código sin lógica: getters, setters, un DTO que solo transporta datos. No hay decisión que comprobar.
  • El framework o el lenguaje: que el ORM guarde o que array_map mapee no es tu responsabilidad.
  • Configuración trivial: que una constante valga lo que vale. El test solo duplica el dato.
  • Detalles privados: pruébalos a través de la API pública que los usa, no directamente.

Concentra el esfuerzo donde hay decisiones: reglas de negocio, cálculos, ramas, casos límite. Ahí cada test compra confianza de verdad.

La cobertura engañosa #

La cobertura mide qué porcentaje de líneas ejecutan tus tests. Es útil para encontrar zonas sin probar, pero es una métrica traicionera como objetivo: dice qué se ejecutó, no qué se comprobó.

php · 100% de cobertura, cero valor 
<?php

declare(strict_types=1);

public function test_ejecuta_el_calculo(): void
{
    $calculator = new Calculator();

    // Llama al método (cuenta como cobertura) pero no comprueba nada.
    $calculator->add(2, 2);

    $this->assertTrue(true);
}

// La línea de add() figura como "cubierta". Si add() devuelve 5, nadie se entera.

Cobertura como meta = ley de Goodhart

«Cuando una medida se convierte en objetivo, deja de ser una buena medida.» Exigir un 90% empuja a escribir tests que tocan líneas sin afirmar nada, solo para subir el número. Usa la cobertura para descubrir huecos, nunca como nota del examen.

Resumen #

Lo que te llevas

  • Un test es código que se mantiene: si falla sin que nada se rompa de verdad, es un pasivo.
  • Prueba comportamiento, no implementación: debe sobrevivir a un refactor y romperse solo si cambia la conducta.
  • Sigue FIRST (rápido, independiente, repetible, autovalidable, oportuno) y evita lo no determinista.
  • No testees lo trivial ni el framework; concéntrate en las decisiones. La cobertura es para hallar huecos, no un objetivo.
