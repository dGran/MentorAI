S — Una sola razón para cambiar

El principio de responsabilidad única se malinterpreta como «una clase hace una cosa». La definición precisa es más útil: una clase debe tener una sola razón para cambiar. Es decir, debe responder ante un único interesado del negocio. Aquí ves por qué eso importa tanto.

«Razón para cambiar» significa actor: la persona o el área del negocio que pediría una modificación. Si una clase cambia cuando lo pide contabilidad y cuando lo pide el equipo de infraestructura y cuando cambia el formato del informe, tiene tres razones para cambiar y, por tanto, tres jefes. SRP dice: dale un solo jefe a cada clase.

Una razón para cambiar #

La formulación de Uncle Bob es «una clase debe tener una sola razón para cambiar», y la aclaración clave: una razón equivale a un actor, un grupo de interés del software. El problema no es que una clase tenga varios métodos, es que esos métodos respondan a motivos de cambio distintos que evolucionan por separado y se pisarán.

La god class #

El antiejemplo clásico: una Factura que calcula sus importes, se guarda a sí misma en la base de datos y se imprime en PDF. Tres responsabilidades, tres actores distintos.

php · tres responsabilidades en una clase 
<?php

declare(strict_types=1);

final class Factura
{
    public function __construct(private array $lineas)
    {
    }

    public function total(): int          // regla de negocio (contabilidad)
    {
        return array_sum(array_map(fn ($l) => $l['precio'] * $l['cantidad'], $this->lineas));
    }

    public function guardar(PDO $db): void // persistencia (infraestructura)
    {
        $db->prepare('INSERT INTO facturas ...')->execute([$this->total()]);
    }

    public function aPdf(): string         // presentación (formato/diseño)
    {
        return '%PDF-1.4 ... ' . $this->total();
    }
}

Tres actores, tres motivos de cambio

total() cambia si cambian las reglas fiscales (contabilidad). guardar() cambia si migras de base de datos (infraestructura). aPdf() cambia si rediseñan el documento (presentación). Tres áreas que no se coordinan entre sí, tocando la misma clase.

Por qué duele #

Juntar responsabilidades crea acoplamientos invisibles. Un cambio en el formato del PDF te obliga a recompilar, retestear y redesplegar una clase que también contiene las reglas fiscales: riesgo gratis. Y al revés, dos equipos que tocan la misma clase por motivos distintos acaban pisándose en los merges. La clase se vuelve un cuello de botella: todos pasan por ella, nadie la entiende entera.

El refactor #

Separa cada responsabilidad en su clase. La Factura se queda con lo suyo —la regla de negocio—, y la persistencia y la presentación se van a piezas dedicadas que reciben la factura y hacen su trabajo.

php · cada clase con un solo jefe 
<?php

declare(strict_types=1);

final class Factura
{
    public function __construct(private array $lineas)
    {
    }

    public function total(): int
    {
        return array_sum(array_map(fn ($l) => $l['precio'] * $l['cantidad'], $this->lineas));
    }
}

final class RepositorioFacturas
{
    public function __construct(private PDO $db)
    {
    }

    public function guardar(Factura $factura): void
    {
        $this->db->prepare('INSERT INTO facturas ...')->execute([$factura->total()]);
    }
}

final class FacturaPdf
{
    public function generar(Factura $factura): string
    {
        return '%PDF-1.4 ... ' . $factura->total();
    }
}

Ahora un cambio fiscal toca solo Factura; uno de base de datos, solo el repositorio; un rediseño del documento, solo FacturaPdf. Cada equipo trabaja en su clase sin pisar al resto, y puedes testear la regla de negocio sin levantar una base de datos.

2 RepositorioFacturaspersistencia · infraestructura
3 FacturaPdfpresentación · diseño

Una responsabilidad por clase, un único motivo de cambio en cada una.

Cohesión, no microclases #

SRP no significa «una clase, un método». Significa alta cohesión: que lo que vive junto cambie junto. Calcular el total y calcular los impuestos de la factura sí pertenecen a la misma clase, porque responden al mismo actor. Partir por partir, hasta tener clases anémicas de un método, es el extremo opuesto y también hace daño.

La pregunta que lo decide

Ante la duda de si algo va junto, pregunta: ¿quién pediría cambiar esto? Si la respuesta es el mismo actor, va junto; si son actores distintos, sepáralo. Es más fiable que contar métodos o líneas. Conecta con las entidades pequeñas de Object Calisthenics: pequeñas porque están enfocadas, no porque sí.

Resumen #

Lo que te llevas

  • SRP: una clase, una sola razón para cambiar. Una razón = un actor del negocio.
  • La god class que calcula, persiste e imprime mezcla tres actores y se convierte en un cuello de botella frágil.
  • Separa por motivo de cambio: negocio, persistencia y presentación en clases distintas que colaboran.
  • Es cohesión, no fabricar microclases: «¿quién pediría cambiar esto?» decide qué va junto.
