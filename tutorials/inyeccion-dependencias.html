Inyección de dependencias en PHP

La diferencia entre una clase que fabrica sus herramientas y una que las recibe. Un patrón pequeño que decide si tu código es rígido y opaco o flexible y testeable.

Una clase rara vez trabaja sola: necesita un logger, un mailer, un repositorio. La pregunta no es si usa colaboradores, sino quién decide cuáles y de dónde salen. Inyectar dependencias es responder: "yo no los fabrico, me los dan hechos".

El problema: el new que ata #

Mira este servicio de pedidos. Funciona, pero esconde un problema que no se ve hasta que intentas cambiarlo o probarlo:

php · OrderService.php 
<?php

declare(strict_types=1);

final class OrderService
{
    public function placeOrder(array $items): void
    {
        $mailer = new SmtpMailer('smtp.acme.test', 587);
        $logger = new FileLogger('/var/log/orders.log');

        // ... validar stock, calcular total, persistir ...

        $mailer->send('Tu pedido está confirmado');
        $logger->info('Pedido creado');
    }
}

OrderService hace tres trabajos a la vez: decide qué implementación usa (SmtpMailer, FileLogger), decide cómo se configura (el host, la ruta del log) y, de paso, gestiona el pedido. Eso tiene consecuencias concretas:

Lo que este código te impide

No puedes probar placeOrder sin enviar correo real y escribir en disco. No puedes reutilizar el servicio con otro mailer (una cola, un servicio externo) sin editarlo. Y para saber qué necesita esta clase tienes que leerte su cuerpo entero: sus dependencias están escondidas.

La raíz de todo es ese new dentro del método: la clase fabrica sus colaboradores en lugar de recibirlos. Está acoplada a clases concretas que ella misma elige.

Qué es inyectar dependencias #

Inyección de dependencias es justo lo contrario de ese new: una clase declara qué necesita y alguien externo se lo entrega ya construido. La clase deja de preguntarse "¿de dónde saco un mailer?" y pasa a asumir "me darán uno".

✗ Sin inyecciónla clase hace new SmtpMailer(...) dentro
✓ Con inyecciónla clase recibe un Mailer ya listo desde fuera

El cambio es de responsabilidad: quién decide y construye las dependencias deja de ser la propia clase.

A esto se le llama inversión del control: el control sobre la creación de los colaboradores se mueve de dentro de la clase hacia el exterior (quien la usa, o un contenedor). La inyección de dependencias es la forma concreta de aplicar esa inversión.

Inyección por constructor #

Hay tres vías para inyectar, pero una es la que debes usar por defecto: el constructor. Si una dependencia es obligatoria para que el objeto funcione, exígela al construirlo. Así el objeto nace siempre completo y válido.

php · OrderService.php 
<?php

declare(strict_types=1);

final class OrderService
{
    public function __construct(
        private readonly Mailer $mailer,
        private readonly Logger $logger,
    ) {
    }

    public function placeOrder(array $items): void
    {
        // ... validar stock, calcular total, persistir ...

        $this->mailer->send('Tu pedido está confirmado');
        $this->logger->info('Pedido creado');
    }
}

El método placeOrder ya no sabe de SMTP ni de ficheros: usa lo que le hayan dado. Y la firma del constructor es ahora la lista de la compra de la clase: de un vistazo sabes que necesita un mailer y un logger. El cableado vive fuera:

php · arranque de la aplicación 
<?php

$orderService = new OrderService(
    new SmtpMailer('smtp.acme.test', 587),
    new FileLogger('/var/log/orders.log'),
);

$orderService->placeOrder($items);

Crear dentro (acoplado)

  • La clase elige la implementación concreta
  • Configuración incrustada en la lógica
  • Dependencias ocultas en el cuerpo
  • Imposible sustituir sin editar la clase

Recibir por constructor (desacoplado)

  • Quien la usa decide la implementación
  • Configuración en el borde de la app
  • Dependencias explícitas en la firma
  • Sustituible sin tocar la clase

Constructor frente a setter

Inyección por constructor para lo obligatorio: el objeto nunca existe a medias. La inyección por setter (un setLogger() posterior) resérvala para lo verdaderamente opcional; si una dependencia se puede olvidar, tendrás objetos en estados inválidos.

Depender de abstracciones #

Fíjate en que el constructor pedía Mailer y Logger, no SmtpMailer ni FileLogger. Eso es deliberado y es la parte que de verdad desacopla: la clase depende de una interfaz (un contrato), no de una implementación concreta.

php · Mailer.php 
<?php

declare(strict_types=1);

interface Mailer
{
    public function send(string $message): void;
}

final class SmtpMailer implements Mailer
{
    public function __construct(
        private readonly string $host,
        private readonly int $port,
    ) {
    }

    public function send(string $message): void
    {
        // envío real por SMTP usando $this->host y $this->port
    }
}

Ahora OrderService funciona con cualquier cosa que cumpla el contrato Mailer: el SMTP de producción, una cola que envía en diferido, un servicio externo. La clase no se entera ni le importa. Esto es la D de SOLID, la inversión de dependencias: depende de abstracciones, no de concreciones.

Interfaz solo cuando aporta

No conviertas cada clase en interfaz por reflejo. La abstracción paga cuando esperas más de una implementación o quieres un doble en los tests. Para un value object o una clase con una única forma estable, el type-hint a la propia clase es legítimo.

El contenedor: quién arma el grafo #

Inyectar por constructor está muy bien hasta que tu grafo crece: el controlador necesita el OrderService, que necesita el Mailer y el Logger, que a su vez necesitan su configuración. Cablear todo eso a mano en cada punto de entrada es tedioso y repetitivo. Para eso existe el contenedor de inyección de dependencias.

1 Peticiónpides OrderService al contenedor
2 El contenedor construyecrea Mailer y Logger, luego el servicio
3 Objeto listorecibes el servicio con todo inyectado

El contenedor es una fábrica que conoce cómo construir cada pieza y en qué orden encajarlas.

php · configuración del contenedor 
<?php

$container->bind(Mailer::class, fn () => new SmtpMailer('smtp.acme.test', 587));
$container->bind(Logger::class, fn () => new FileLogger('/var/log/orders.log'));

// El contenedor ve que OrderService necesita Mailer y Logger,
// los resuelve y los inyecta por ti.
$orderService = $container->get(OrderService::class);

En la práctica no escribes tu propio contenedor: lo trae el framework. Symfony, Laravel y librerías como PHP-DI hacen autowiring: leen los type-hints del constructor y resuelven el grafo solos. Para que el código sea portable entre contenedores existe el estándar PSR-11, que define una interfaz común:

php · PSR-11 
<?php

use Psr\Container\ContainerInterface;

$mailer = $container->get(Mailer::class); // get() y has(), nada más

No inyectes el contenedor

Pasarle el contenedor entero a una clase para que ella saque lo que quiera ($container->get(...) dentro del servicio) es el anti-patrón Service Locator: vuelves a esconder las dependencias y a acoplarte, ahora al contenedor. El contenedor solo debe usarse en el borde de la aplicación para arrancar el grafo, no dentro de tus servicios.

Por qué tus tests lo agradecen #

Aquí es donde la inyección de dependencias deja de ser teoría elegante y se vuelve práctica. Como OrderService recibe un Mailer, en un test le pasas un doble que no envía nada y que puedes inspeccionar:

php · OrderServiceTest.php 
<?php

declare(strict_types=1);

final class FakeMailer implements Mailer
{
    public array $sentMessages = [];

    public function send(string $message): void
    {
        $this->sentMessages[] = $message;
    }
}

// en el test
$mailer = new FakeMailer();
$service = new OrderService($mailer, new NullLogger());

$service->placeOrder($items);

self::assertSame(['Tu pedido está confirmado'], $mailer->sentMessages);

Sin red, sin disco, sin SMTP: el test es rápido, determinista y comprueba exactamente lo que importa. Con el primer diseño (el del new dentro) esto era imposible sin levantar infraestructura. La testeabilidad no es un extra de la inyección de dependencias: es su consecuencia más directa.

Errores comunes #

SíntomaPor qué duele
Inyectar el contenedor entero Service Locator: dependencias ocultas y acoplamiento al contenedor.
Constructor con 7 dependencias La clase hace demasiado; viola responsabilidad única. Divídela.
new de un colaborador con lógica dentro de la clase Vuelves a acoplar. Solo construye dentro value objects sin dependencias.
Setter para algo obligatorio El objeto puede usarse a medias y reventar en runtime.
Interfaz para todo "por si acaso" Abstracción sin segunda implementación: solo añade ruido.

Una regla práctica

Si un constructor empieza a pedir muchas dependencias, no busques cómo inyectarlas mejor: es la señal de que la clase tiene demasiadas responsabilidades. La inyección de dependencias no solo desacopla, también te avisa cuando un diseño se está yendo de las manos.

Resumen #

Lo que te llevas

  • Una clase no debería fabricar sus colaboradores; debería recibirlos ya construidos.
  • Inyecta por constructor lo obligatorio: el objeto nace completo y sus dependencias quedan explícitas en la firma.
  • Type-hint contra interfaces cuando esperes varias implementaciones o quieras dobles de test (la D de SOLID).
  • El contenedor automatiza el cableado del grafo; úsalo solo en el borde de la app, nunca lo inyectes dentro de un servicio.
  • El beneficio que más se nota: tests rápidos y deterministas inyectando fakes en lugar de infraestructura real.
