Comentarios: por qué un buen comentario es raro

Suena provocador en un curso de buenas prácticas, pero es así: casi todos los comentarios son una disculpa por un código que no se explica solo. El objetivo no es comentar más, es escribir un código tan claro que el comentario sobre. Y reservar los pocos que de verdad aportan.

«No comentes el mal código, reescríbelo», dice Uncle Bob. La regla mental que conviene adoptar: cada vez que vas a escribir un comentario, pregúntate si podrías evitarlo mejorando el código. La mayoría de las veces sí, y el resultado es mejor que el comentario.

El comentario como disculpa #

Un comentario que explica qué hace el código suele ser señal de que el código no lo dice por sí mismo. La solución no es el comentario, es un buen nombre o una función extraída. El comentario que repite el código en castellano no aporta nada y, peor, añade algo que mantener.

Comentario que sustituye al código

  • El comentario explica una línea oscura.
  • Hay que leer las dos cosas.
  • Si el código cambia, el comentario miente.

Código que se explica

  • Un nombre o una función con intención.
  • Se lee una sola cosa.
  • No hay nada que se quede desactualizado.
php · el comentario sobra si el código habla 
<?php

// Comprueba si el usuario es mayor de edad y está verificado
if ($u->edad >= 18 && $u->v === 1) { /* ... */ }

// Mejor: el código dice lo que el comentario decía
if ($usuario->puedeContratar()) { /* ... */ }

Comentarios que mienten #

El peor defecto de los comentarios es que no se compilan: nada obliga a actualizarlos cuando el código cambia. Con el tiempo, el código evoluciona y el comentario se queda contando una versión que ya no existe. Y un comentario que miente es más dañino que la ausencia de comentario, porque el lector se fía de él.

El código comentado: bórralo

Dejar bloques de código «por si acaso», comentados, es basura que confunde: nadie sabe si importa, por qué está ahí o si se puede borrar. Para eso está el control de versiones —git recuerda todo—. Si no se usa, fuera.

Comentarios que suplen un mal nombre #

Un caso muy común: la variable o función tiene un nombre pobre y el comentario viene a rescatarla. Es esfuerzo en el sitio equivocado. Gasta esa energía en el nombre: que la cosa se llame como lo que el comentario iba a explicar, y el comentario desaparece sin pérdida.

php · sube el comentario al nombre 
<?php

$d = 30; // días que dura el periodo de prueba

// Mejor, sin comentario:
$diasDePrueba = 30;

Cuándo sí comentar #

Hay comentarios valiosos, y todos comparten una cosa: explican lo que el código no puede decir por sí mismo. El código dice qué hace; un buen comentario dice por qué.

Comentario que aporta Por qué
El porqué de una decisión no obvia «Usamos este orden porque el proveedor exige el campo X primero». El código no puede contar la razón.
Un aviso de consecuencias «No paralelizar: el servicio externo limita a 1 req/s». Evita que alguien rompa algo sin saberlo.
Una referencia externa Enlace a un ticket, a una norma legal o a un workaround documentado de un bug ajeno.
Documentación pública de una API Un docblock en una librería que otros consumen sin ver el código.

La convención de este proyecto

MentorAI sigue justo esta filosofía: nada de comentarios de prosa dentro de las funciones —el naming se explica solo— y se conservan solo las cabeceras de sección como navegación del fichero. Es la regla de esta lección aplicada a la práctica: comentar el porqué y la estructura, nunca el qué.

Resumen #

Lo que te llevas

  • La mayoría de comentarios son una disculpa por un código que no se explica; mejóralo en vez de comentarlo.
  • Los comentarios mienten con el tiempo (no se compilan); el código comentado «por si acaso» se borra: para eso está git.
  • Si un comentario suple un mal nombre, arregla el nombre: el comentario desaparece sin pérdida.
  • Comenta el porqué, no el qué: decisiones no obvias, avisos, referencias externas y APIs públicas.
