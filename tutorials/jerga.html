La jerga del desarrollo: el glosario que nadie te explica

En tu primera semana en un equipo oirás "parsea el payload, cuidado con el race condition y no te metas en un yak shaving". Nadie se para a traducirlo. Aquí tienes el diccionario: las palabras que se sueltan a diario, con su significado y un ejemplo de cómo se usan de verdad.

La jerga no es para presumir: es atajo. Una sola palabra encapsula un concepto que, explicado, son tres frases. El problema es que se aprende por osmosis y nadie la documenta. Este glosario agrupa los términos por temas; cuando alguno tenga tutorial propio, te enlazo para profundizar.

Código y ejecución #

El vocabulario sobre qué le pasa a tu código desde que lo escribes hasta que corre:

Término Qué significa En una frase real
Parsear Analizar un texto para extraer su estructura y darle sentido. "Parsea el JSON y saca el campo email."
Compilar / interpretar Traducir el código fuente a algo ejecutable, de golpe (compilar) o sobre la marcha (interpretar). "Esto no compila, tienes un punto y coma de menos."
Runtime El "tiempo de ejecución": cuando el programa ya está corriendo (frente a cuando se escribe o compila). "Ese error no salta al compilar, solo en runtime."
Build El proceso (y el resultado) de empaquetar el código listo para ejecutar o desplegar. "Se ha roto el build, no mergees todavía."
Deploy Subir una versión nueva del software al servidor donde la usan los usuarios. "Hago deploy a producción después de comer."
Rollback Volver a la versión anterior cuando un deploy sale mal. "Está petando todo, haz rollback ya."
Hotfix Un arreglo urgente que va directo a producción, saltándose el proceso normal. "Mando un hotfix, esto no puede esperar al sprint."
Refactor Reescribir código para que sea mejor por dentro sin cambiar lo que hace por fuera. "No añado nada, solo refactorizo este servicio."
Deprecated Marcado como obsoleto: aún funciona, pero está desaconsejado y desaparecerá. "Esa función está deprecated desde la 8.1."
Boilerplate Código repetitivo y obligatorio que escribes casi igual cada vez. "Genero el boilerplate con un comando y relleno lo mío."

Errores y estados #

Término Qué significa En una frase real
Bug Un fallo en el programa: hace algo distinto de lo que debería. "Hay un bug: suma mal cuando el carrito está vacío."
Edge case Un caso límite, raro pero posible, que rompe la lógica normal. "Funciona, pero falla en el edge case de cero elementos."
Race condition Un bug que aparece según el orden en que se cruzan dos cosas a la vez. "Vendimos stock de más: clásica race condition."
Memory leak Memoria que se reserva y nunca se libera; el consumo crece sin parar. "El worker se cae cada hora, huele a memory leak."
Flaky Un test (o sistema) que a veces pasa y a veces no, sin tocar nada. "Ignora ese fallo, ese test es flaky."

"Stack overflow": la pila, no la web

Antes de ser la web de preguntas, un stack overflow es un error real: la pila de llamadas se desborda, casi siempre por una recursión que no para. La función se llama a sí misma sin fin, la pila se llena y el programa muere. Que la web se llame así es justo el chiste.

Dos de estos tienen tutorial propio si quieres bajar al detalle: las race conditions (el bug que solo pasa en producción) y las fugas de memoria en el runtime de PHP.

Flujo y proceso #

Las palabras del día a día con Git y la organización del trabajo:

Término Qué significa En una frase real
Merge Fusionar los cambios de una rama en otra. "Cuando aprueben la PR, hago merge a main."
Rebase Reescribir el historial de tu rama como si partiera de la última versión. "Haz rebase sobre main antes de pedir review."
Cherry-pick Llevarte un commit concreto de una rama a otra, solo ese. "Cherry-pickea ese fix a la rama de release."
PR / MR Pull/Merge Request: propuesta de cambios para que alguien la revise y apruebe. "Te he dejado dos comentarios en la PR."
Blocker Algo que te impide avanzar hasta que se resuelva. "Estoy bloqueado, necesito el acceso a la BBDD."
WIP Work In Progress: trabajo a medias, todavía no terminado. "Es un WIP, no lo revises aún."
MVP Minimum Viable Product: la versión más pequeña que ya aporta valor. "Saquemos un MVP y ya iteramos."
Tech debt Deuda técnica: atajos de hoy que tendrás que pagar (arreglar) mañana. "Esto acumula tech debt, pero entramos a tiempo."

La metáfora de la deuda técnica

Igual que una deuda financiera: pedir prestado (un atajo) te deja avanzar rápido ahora, pero genera intereses (cada cambio futuro cuesta más). Un poco de deuda es sano si entras a mercado a tiempo; demasiada y acabas pagando solo intereses sin avanzar.

Datos y red #

Término Qué significa En una frase real
Payload Los datos "útiles" que viajan en una petición o mensaje (sin las cabeceras). "El payload del webhook trae el id del pago."
Endpoint Una URL concreta de una API a la que llamas para algo. "Pega al endpoint /users para crear el usuario."
Idempotente Que repetir la operación da el mismo resultado que hacerla una vez. "Haz el endpoint idempotente: que reintentar no cobre dos veces."
Cache hit / miss Que el dato estaba en caché (hit) o no estaba y hubo que buscarlo (miss). "Vamos lentos: demasiados cache miss."
Throughput Cuánto trabajo procesas por unidad de tiempo (peticiones por segundo). "Subimos el throughput a 2.000 req/s."
Latencia Cuánto tarda una operación en responder. "La latencia del p99 se ha disparado."

Varios enlazan con tutoriales del catálogo: el cache hit/miss está en Redis como caché y los términos de red, en HTTP a fondo.

Cultura y memes #

Y el cajón divertido: expresiones que no son técnicas pero que oirás constantemente y conviene pillar al vuelo.

Término Qué significa
Yak shaving Acabar haciendo una tarea absurda y lejana para poder hacer la que querías. "Quería arreglar un botón y llevo dos horas configurando Docker."
Bikeshedding Gastar muchísimo tiempo discutiendo una trivialidad (el color de un botón) y poco en lo importante.
Rubber duck Explicar tu problema en voz alta (aunque sea a un patito de goma) hasta que tú solo ves la solución.
Foo / bar / baz Nombres de relleno para ejemplos, como el "fulano y mengano" del código.
"Funciona en mi máquina" La coartada clásica cuando algo falla en producción pero no en tu portátil. El origen de medio Docker.

El patito de goma funciona de verdad

Suena a broma, pero el rubber duck debugging es una técnica real y eficaz. Al forzarte a explicar el problema paso a paso, en voz alta y para alguien que no sabe nada, tu cerebro tiene que ordenar lo que asumía por sabido… y ahí suele saltar el error. No necesitas el pato; necesitas explicarlo.

Resumen #

Lo que te llevas

  • La jerga es atajo, no postureo: una palabra que resume un concepto. Pillarla acelera entenderte con el equipo.
  • Ejecución: parsear, runtime, build, deploy, rollback, refactor, deprecated.
  • Errores: edge case, race condition, memory leak, flaky; y "stack overflow" es la pila desbordada, no solo la web.
  • Proceso y datos: merge/rebase, PR, tech debt, payload, endpoint, idempotente, latencia.
  • Cultura: yak shaving, bikeshedding, rubber duck y el eterno "funciona en mi máquina".
