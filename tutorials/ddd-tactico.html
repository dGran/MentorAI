DDD táctico (building blocks)

Con las fronteras ya trazadas, el DDD táctico da las piezas para modelar dentro de cada bounded context: entidades, value objects, agregados, repositorios. No son clases vacías con getters: son patrones que ponen las reglas del negocio donde deben estar.

Estos patrones son herramientas, no obligaciones. La pregunta no es «¿qué patrón aplico aquí?», sino «¿cuál de estas piezas captura mejor esta parte del negocio?». Mal usados, son ceremonia; bien usados, hacen que el código exprese el dominio.

Los building blocks #

El DDD táctico vive dentro de un bounded context y de la capa de dominio de la hexagonal. Su objetivo es un modelo rico: objetos que contienen sus reglas, no sacos de datos manipulados desde fuera. La primera distinción, y la más útil, es entre objetos con identidad (entidades) y objetos sin ella (value objects).

Value object #

Un value object se define por su valor, no por una identidad. Dos billetes de 10 € son intercambiables: lo que importa es «10 €», no cuál es cuál. Email, dinero, un rango de fechas, una dirección: candidatos perfectos. Son inmutables (si cambias algo, creas otro) y validan su propia corrección al nacer.

php · value object Email 
<?php

declare(strict_types=1);

final class Email
{
    public function __construct(private string $value)
    {
        if (!filter_var($value, FILTER_VALIDATE_EMAIL)) {
            throw new InvalidEmail($value);
        }
    }

    public function value(): string
    {
        return $this->value;
    }

    public function equals(Email $other): bool
    {
        return $this->value === $other->value;
    }
}

Los value objects matan las primitivas obsesivas

Pasar emails como string y dinero como float esparce validaciones por todo el código y permite errores absurdos (sumar un id con un precio). Un value object encierra la regla en un sitio: si tienes un Email, es válido por construcción. Empieza por aquí; es el patrón con mejor relación valor/esfuerzo.

Entidad #

Una entidad tiene identidad propia que persiste en el tiempo aunque cambien sus atributos. Un usuario sigue siendo el mismo usuario aunque cambie de email; un pedido sigue siendo ese pedido aunque cambie de estado. Se distinguen por su id, no por sus valores, y encapsulan las operaciones que cambian su estado.

php · entidad con comportamiento, no setters 
<?php

declare(strict_types=1);

final class Subscription
{
    public function __construct(
        private SubscriptionId $id,
        private Status $status
    ) {}

    public function cancel(): void
    {
        if ($this->status === Status::Cancelled) {
            throw new AlreadyCancelled($this->id);
        }

        $this->status = Status::Cancelled;
    }
}

Fíjate: no hay setStatus(). Hay cancel(), que es lo que el negocio sabe hacer, con su regla incluida. Una entidad con solo getters y setters es un modelo anémico: los datos por un lado y las reglas dispersas por servicios, justo lo que DDD quiere evitar.

Agregado y su raíz #

Un agregado es un grupo de entidades y value objects que se tratan como una unidad de consistencia. Tiene una raíz (aggregate root): la única entidad por la que se accede al grupo desde fuera. Un pedido (raíz) con sus líneas (dentro): nadie toca una línea directamente, todo pasa por el pedido, que vela por las invariantes («el total cuadra con las líneas»).

→ Acceso solo por la raízEl exterior habla con Order, nunca con OrderLine directamente
◆ Agregado OrderOrder (raíz) + OrderLine[] + Money. Consistente como un todo.

La raíz es la frontera de consistencia: dentro, siempre coherente.

Un agregado no es una tabla

Es el error que más confunde a quien viene de modelar en relacional. El agregado es una frontera de consistencia y comportamiento, no de almacenamiento. Puede mapear a varias tablas, y no toda relación de claves foráneas es un agregado. Diseña el agregado por las reglas que debe proteger, no por cómo lo guardarás.

Regla práctica: mantén los agregados pequeños y referénciate entre ellos por id, no por objeto. Un pedido no contiene al cliente entero; guarda un CustomerId. Agregados gigantes que se arrastran medio sistema al cargarse son un antipatrón clásico.

Repositorio #

Un repositorio es la abstracción que da y guarda agregados como si fueran una colección en memoria, ocultando la persistencia. Uno por agregado raíz: el repositorio trabaja con la raíz completa, no con las piezas internas. Es justo el puerto que viste en la hexagonal.

php · repositorio = puerto, una por raíz 
<?php

declare(strict_types=1);

interface OrderRepository
{
    public function save(Order $order): void;   // guarda la raíz completa

    public function byId(OrderId $id): ?Order;   // reconstruye el agregado
}

Servicio de dominio y factory #

Cuando una operación de negocio no pertenece de forma natural a ninguna entidad —porque involucra a varias—, vive en un servicio de dominio: una clase sin estado que expresa esa lógica en el lenguaje del negocio (por ejemplo, una transferencia que toca dos cuentas). No lo confundas con el servicio de aplicación: el de dominio contiene reglas; el de aplicación solo orquesta.

Una factory encapsula la creación compleja de un agregado cuando construirlo bien tiene reglas (varios pasos, invariantes que asegurar al nacer). Si crear el objeto es trivial, el constructor basta; la factory es para cuando la construcción es en sí misma una operación de negocio.

Resumen #

Lo que te llevas

  • Value object: se define por su valor, inmutable y autovalidado (Email, Money). Empieza por aquí.
  • Entidad: tiene identidad propia y comportamiento, no setters anémicos.
  • Agregado: unidad de consistencia con una raíz; pequeño y referenciado por id. No es una tabla.
  • Repositorio (uno por raíz) = puerto; servicio de dominio para lo que no cae en una entidad; factory para creación con reglas.
