Clean Code: escribir para quien lee

El código no se escribe para la máquina —a ella le da igual—, se escribe para la próxima persona que lo lea, y casi siempre eres tú dentro de seis meses. El código limpio es una cuestión de empatía con ese lector futuro, y resulta que esa empatía sale rentable cada día.

«Cualquiera puede escribir código que una máquina entienda; los buenos programadores escriben código que los humanos entienden». La frase de Martin Fowler resume el curso entero. Limpiar el código no es estética ni purismo: es bajar el coste de leerlo, entenderlo y cambiarlo, que es donde se va la mayor parte del tiempo de un proyecto.

Se lee más de lo que se escribe #

La proporción que cita Robert C. Martin en Clean Code es de unas diez veces: pasamos diez veces más tiempo leyendo código que escribiéndolo. Cada vez que añades una función, lees la de al lado para entender el contexto. Cada arreglo empieza por leer lo que ya hay. Si optimizas algo, optimiza la lectura: es la operación más frecuente.

El lector es el cliente del código

Escribir «rápido» código que cuesta leer no es rápido: traslada el coste, multiplicado por diez, a todas las lecturas futuras. El código limpio invierte un poco más al escribir para ahorrar mucho más al leer. La cuenta sale casi siempre.

Qué es código limpio #

No hay una definición única, pero todas las buenas apuntan a lo mismo: código que se lee como si alguien te lo estuviera explicando. Se entiende sin esfuerzo, no esconde sorpresas, cada nombre dice la verdad y cada pieza hace lo que su nombre promete. Algunas señales concretas:

Código limpio

  • Se lee de arriba abajo como una explicación.
  • Los nombres revelan la intención.
  • Cada función hace una cosa.
  • No te sorprende: hace lo que parece.
  • Es fácil de cambiar y de testear.

Código sucio

  • Hay que descifrarlo, no leerlo.
  • Nombres como $data, $tmp, $flag.
  • Funciones que hacen de todo.
  • Efectos ocultos que muerden por detrás.
  • Tocarlo da miedo.

El coste del código sucio #

El código sucio se paga como una deuda, y los intereses se cobran a diario. Cada feature tarda un poco más porque primero hay que entender el desorden. Cada bug es más difícil de localizar. Los nuevos en el equipo tardan semanas en orientarse. Y lo peor: el desorden invita a más desorden —si ya está hecho un lío, una chapuza más no se nota—, así que se realimenta.

«Ya lo limpio luego» casi nunca llega

La velocidad que parece ganar el código sucio es un espejismo de las primeras semanas. A medida que el proyecto crece, el equipo desordenado se frena hasta casi pararse, mientras que mantener la casa limpia conserva el ritmo. Ir despacio para ir rápido no es un eslogan: es la curva real de productividad.

La regla del boy scout #

La práctica más útil para no acumular deuda cabe en una frase: deja el código un poco más limpio de como lo encontraste. No hace falta una gran refactorización ni pedir permiso para «una semana de limpieza». Cada vez que tocas un archivo, mejora un nombre, extrae una función, borra un comentario que miente. Pequeñas mejoras continuas mantienen el código sano sin proyectos heroicos.

Limpieza segura = limpieza con tests

Mejorar código existente sin una red de tests es arriesgado: no sabes si has roto algo. Por eso Clean Code y testing van de la mano; lo veremos a fondo en la última lección, sobre code smells y refactoring.

Lo que viene en el curso #

Este curso baja del «por qué» al «cómo», adaptado a PHP y a las convenciones de naming semántico. El recorrido:

  • Nombres — el 80% de la legibilidad.
  • Funciones limpias — pequeñas, una cosa, un nivel de abstracción.
  • Comentarios — por qué un buen comentario es raro.
  • Manejo de errores — excepciones específicas, fail fast, nada de null.
  • Code smells y refactoring — detectar y limpiar con seguridad.

Clean Code y SOLID se complementan

SOLID trata de cómo repartir responsabilidades entre clases (la estructura); Clean Code, de cómo se escribe cada pieza para que se lea bien (el detalle). Uno sin el otro cojea: estructura impecable con nombres crípticos sigue siendo difícil de leer, y nombres preciosos en una god class no la salvan.

Resumen #

Lo que te llevas

  • El código se lee unas diez veces más de lo que se escribe: optimiza la lectura, es la operación más frecuente.
  • Código limpio es el que se lee como una explicación: nombres honestos, funciones enfocadas, sin sorpresas.
  • El código sucio es deuda con intereses diarios; la velocidad que aparenta es un espejismo que se paga después.
  • La regla del boy scout: deja cada archivo un poco más limpio de como lo encontraste, con la red de los tests.
