Texto: de ASCII a Unicode y UTF-8

"Es solo texto" es la frase que precede a casi todos los bugs de codificación: la é que sale como Ã©, el emoji que tu base de datos rechaza, el strlen que devuelve un número raro. Todo eso se entiende —y se evita— en cuanto separas dos ideas que casi todos mezclan: el carácter y sus bytes.

Un ordenador solo guarda bytes (esto es justo el tutorial anterior). El texto es un acuerdo sobre qué carácter representa cada número. La historia de ese acuerdo —de 128 símbolos en inglés a cualquier signo de cualquier idioma más los emojis— explica casi todos los problemas de codificación que te encontrarás.

ASCII: el punto de partida #

En 1963, ASCII fijó una tabla de 128 caracteres numerados del 0 al 127: letras inglesas, dígitos, signos de puntuación y unos códigos de control. 128 valores entran en 7 bits, así que cada carácter cabía de sobra en un byte y aún sobraba un bit. La A es el 65, la a el 97, el espacio el 32.

Por qué mayúsculas y minúsculas se llevan 32

A = 65 y a = 97: difieren exactamente en 32, que es 2⁵, un único bit. Pasar de una a otra es cambiar ese bit. No es casualidad: la tabla se diseñó así para que cambiar de caja fuera una operación de bits trivial.

El problema: más allá del inglés #

ASCII deja libre el octavo bit, así que durante años cada región lo usó para meter sus 128 caracteres extra: del 128 al 255. El resultado fue un caos de páginas de código incompatibles: en Latin-1 (Europa occidental) el byte 233 es é; en otra página, el mismo byte es otra cosa. El texto no llevaba dentro qué tabla usar, así que abrir un fichero con la página equivocada lo convertía en jeroglíficos.

1 ASCII128 caracteres, 7 bits, solo inglés
2 Páginas de código256 por región, incompatibles entre sí
3 Unicode + UTF-8un número por carácter, bytes variables

El salto clave: dejar de atar el carácter a un único byte de una tabla regional.

Y aun con 256 valores no caben a la vez el español, el ruso, el griego y el chino, no digamos todos juntos. Un byte se queda corto. Hacía falta separar dos cosas que las páginas de código mezclaban: qué carácter es y con qué bytes se guarda.

Unicode: un número por carácter #

Unicode resuelve la primera mitad: asigna a cada carácter imaginable un número único y permanente llamado code point, que se escribe U+ y su valor en hex. La A es U+0041 (el mismo 65 de ASCII, por compatibilidad), la é es U+00E9, el euro € es U+20AC y la cara sonriente 😀 es U+1F600. Hay espacio para más de un millón de code points; se usan unos 150.000.

Unicode es el catálogo, no los bytes

Unicode dice "la é es el número U+00E9", pero no cómo guardarlo en disco. Esa es la otra mitad del problema, y la resuelve una codificación como UTF-8: el puente entre el code point y los bytes reales.

UTF-8: del número a los bytes #

UTF-8 es la codificación que ganó (es la del 98% de la web), y su diseño es elegante: usa de 1 a 4 bytes según el code point, y los primeros bits de cada byte dicen cuántos bytes ocupa el carácter. Lo mejor: los code points de 0 a 127 se codifican en un solo byte idéntico a ASCII, así que todo fichero ASCII ya es UTF-8 válido. Por eso se adoptó sin romper nada.

Texto en la práctica #

La consecuencia diaria: en UTF-8 un carácter no es un byte. Cualquier función que cuente, corte o recorra texto tiene dos versiones: la que trabaja con bytes y la que respeta caracteres (en PHP, las mb_*). Confundirlas es el origen del 90% de los bugs de acentos.

php · bytes vs caracteres 
<?php

declare(strict_types=1);

$text = 'áéí';

echo strlen($text);        // 6  -> cuenta BYTES (cada vocal son 2)
echo mb_strlen($text);     // 3  -> cuenta CARACTERES

echo bin2hex('é');         // "c3a9"     -> 2 bytes
echo bin2hex('€');         // "e282ac"   -> 3 bytes
echo bin2hex('😀');        // "f09f9880" -> 4 bytes

Del code point a los bytes y vuelta, en código: mb_ord da el number Unicode y mb_chr hace el camino inverso.

php · code points 
<?php

declare(strict_types=1);

echo mb_ord('€');              // 8364
echo dechex(mb_ord('€'));      // "20ac"  -> U+20AC
echo mb_chr(0x1F600);          // "😀"    -> de code point a carácter

Y la trampa clásica: cortar por bytes parte un carácter por la mitad y produce basura. Usa siempre la versión que respeta caracteres.

php · cortar sin romper 
<?php

declare(strict_types=1);

$text = 'café';            // 5 bytes: c a f + (é = 2 bytes)

echo substr($text, 0, 4);     // "caf" + medio byte -> carácter roto
echo mb_substr($text, 0, 4);  // "café" -> respeta el carácter completo

El "utf8" de MySQL no es UTF-8

Durante años, el charset utf8 de MySQL guardaba como mucho 3 bytes por carácter, así que los emojis y otros caracteres de 4 bytes (U+10000 en adelante) reventaban con un error o se truncaban. El UTF-8 completo es utf8mb4: úsalo siempre en tablas y conexión. Es el error de codificación más caro de descubrir en producción.

Resumen #

Lo que te llevas

  • ASCII (128 caracteres, 7 bits) solo cubría el inglés; las páginas de código de 1 byte fueron un caos incompatible.
  • Separa dos ideas: Unicode da un número (code point U+…) a cada carácter; una codificación decide sus bytes.
  • UTF-8 usa 1–4 bytes, es compatible con ASCII en el rango 0–127 y se autosincroniza. Por eso ganó.
  • Un carácter no es un byte: cuenta y corta con funciones que respetan caracteres (mb_strlen, mb_substr).
  • En MySQL usa utf8mb4, no utf8, o los emojis (4 bytes) se romperán.
