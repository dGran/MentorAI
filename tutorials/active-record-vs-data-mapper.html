Active Record vs Data Mapper

Todos los ORM parecen iguales hasta que descubres que hay dos filosofías debajo. ¿La entidad sabe guardarse a sí misma, o es un objeto puro que otro persiste? Esa decisión, que parece de matiz, marca cómo se siente todo tu código de dominio.

La diferencia entre Eloquent y Doctrine no es de sintaxis ni de marca: es de filosofía. Active Record y Data Mapper responden distinto a una pregunta de fondo —¿quién es responsable de persistir una entidad?— y de esa respuesta cuelgan el acoplamiento, la testabilidad y cómo de "puro" puede ser tu dominio.

Dos filosofías de mapeo #

Los dos patrones los catalogó Martin Fowler, y se distinguen por dónde vive la lógica de persistencia:

Active Record

  • La entidad es la fila y sabe guardarse.
  • $pedido->save(), $pedido->delete().
  • Objeto y persistencia, juntos.
  • Encarnación típica: Eloquent (Laravel).

Data Mapper

  • La entidad es un objeto puro, no sabe nada de la BD.
  • Un mapper aparte la lee y la escribe.
  • Objeto y persistencia, separados.
  • Encarnación típica: Doctrine.

Las dos columnas están marcadas como válidas a propósito: no hay una "buena" y una "mala". Son compromisos distintos. Veámoslas con código.

Active Record #

En Active Record, la clase de la entidad extiende de una clase base del ORM y hereda de ella los métodos para consultarse y persistirse. El objeto y su tabla son la misma cosa: cada instancia es una fila, y guardar es un método del propio objeto.

php · estilo Active Record (Eloquent) 
<?php

class Pedido extends Model
{
}

$pedido = Pedido::find(1001);
$pedido->estado = 'enviado';
$pedido->save();

$nuevos = Pedido::where('estado', 'pendiente')->get();

Es directísimo y productivo: con muy poco código tienes consultas, guardado y relaciones. Por eso Active Record domina en frameworks orientados a entregar rápido, como Laravel. El precio es el acoplamiento: tu entidad es el ORM, hereda de él. No puedes tener un Pedido sin arrastrar toda la maquinaria de persistencia, y eso complica aislar la lógica de negocio o testearla sin base de datos.

Data Mapper #

Data Mapper invierte la responsabilidad. La entidad es una clase tuya, sin herencias del ORM ni conocimiento de la base de datos: solo propiedades y comportamiento de negocio. Una pieza separada —el entity manager y sus mappers— se encarga de leerla y escribirla.

php · estilo Data Mapper (Doctrine) 
<?php

class Pedido
{
    private string $estado = 'pendiente';

    public function marcarEnviado(): void
    {
        $this->estado = 'enviado';
    }
}

$pedido = $entityManager->find(Pedido::class, 1001);
$pedido->marcarEnviado();
$entityManager->flush();

Fíjate en que Pedido no extiende nada y no tiene save(): solo expresa reglas de negocio (marcarEnviado()). Es el $entityManager quien persiste. Esa separación es justo lo que pide el diseño orientado al dominio: una entidad limpia, testeable sin base de datos, ignorante de cómo se guarda. El coste es más ceremonia y una curva de aprendizaje más empinada (mapeo explícito, el concepto de unit of work).

Por qué el dominio agradece Data Mapper

Una entidad que no depende de la infraestructura es la base de la arquitectura hexagonal y del DDD táctico: el dominio en el centro, la persistencia como un detalle de fuera. El patrón Repository que ves en esos cursos encaja de forma natural sobre un Data Mapper.

Cuándo cada uno #

La elección sigue a la complejidad del dominio, no a la moda:

Active Record encaja si…

  • El proyecto es un CRUD rico, entregar rápido manda.
  • La lógica vive cerca de los datos, sin un dominio complejo.
  • El equipo ya va con Laravel y su ecosistema.

Data Mapper encaja si…

  • El dominio es rico y las reglas de negocio pesan.
  • Quieres aislar y testar el dominio sin infraestructura.
  • Apuntas a hexagonal, DDD o una vida larga del proyecto.

Ninguno es "más profesional". Active Record mal usado acaba con lógica de negocio repartida en modelos gigantes acoplados a la BD; Data Mapper mal usado es sobreingeniería ceremoniosa para un CRUD que no la necesitaba. La pregunta útil no es cuál es mejor, sino cuánta complejidad de dominio tiene tu proyecto de verdad: esa respuesta elige por ti.

El patrón no salva un mal diseño

Puedes escribir un dominio limpio sobre Active Record (delegando la lógica a servicios) y un desastre sobre Data Mapper. El patrón inclina la balanza, pero la disciplina de separar responsabilidades la pones tú.

Resumen #

Lo que te llevas

  • Debajo de todo ORM hay una de dos filosofías: Active Record (la entidad se persiste a sí misma) o Data Mapper (un mapper la persiste).
  • Active Record (Eloquent): la entidad extiende del ORM, $pedido->save(). Muy productivo, pero acoplado a la BD.
  • Data Mapper (Doctrine): la entidad es pura y un entity manager la persiste con flush(). Más ceremonia, dominio limpio y testeable.
  • La elección sigue a la complejidad del dominio: AR para CRUD y rapidez; DM para dominios ricos, DDD y hexagonal.
  • Ninguno es "más pro": el patrón inclina la balanza, pero el buen diseño lo pones tú separando responsabilidades.
