Qué es REST de verdad

Casi todas las APIs se llaman REST y casi ninguna lo es del todo. Detrás de la etiqueta hay un puñado de ideas sencillas —recursos, representaciones, sin estado— que explican por qué una API se siente predecible o un caos. Esto es lo que de verdad significa.

REST es un estilo de arquitectura, no una librería que instalas ni un protocolo que hablas. Es un conjunto de restricciones que, si las respetas, te dan APIs predecibles, cacheables y que escalan. Entender el porqué de cada restricción es lo que separa "monto endpoints con JSON" de "diseño una API".

REST no es un protocolo #

REST significa Representational State Transfer. Lo definió Roy Fielding en el año 2000 en su tesis doctoral, describiendo las propiedades que hacían que la Web escalara a millones de máquinas sin coordinarse entre ellas. No inventó una tecnología nueva: puso nombre a las ideas que ya hacían funcionar HTTP.

Eso tiene una consecuencia importante: REST se apoya en HTTP, pero HTTP con JSON no es automáticamente REST. Puedes mandar JSON por HTTP y violar todas las ideas de REST a la vez. La etiqueta "API REST" se ha vuelto un sinónimo perezoso de "API web con JSON"; aquí vamos a recuperar lo que de verdad quiere decir.

Esto se apoya en HTTP

REST hereda los métodos, los códigos de estado y las cabeceras de HTTP. Si ese terreno te suena flojo, el tutorial HTTP a fondo es el cimiento de todo este curso.

Todo es un recurso #

La pieza central de REST es el recurso: cualquier cosa con nombre sobre la que quieras hablar. Un usuario, un pedido, una factura, la lista de comentarios de un artículo. Si tu negocio lo menciona como un sustantivo, probablemente es un recurso.

Cada recurso vive en una URL que lo identifica de forma única y estable. La URL es el nombre del recurso, no la acción que haces con él:

Recursos (sustantivos)

  • /usuarios
  • /usuarios/42
  • /usuarios/42/pedidos
  • /pedidos/1001

Acciones disfrazadas de URL

  • /getUsuario?id=42
  • /crearPedido
  • /usuarios/borrar/42
  • /listarTodo

La diferencia es de raíz. En la columna mala, el verbo va en la URL y cada operación inventa su propio nombre, así que para usar la API hay que memorizar un catálogo de funciones. En la columna buena, la URL solo nombra la cosa; el verbo lo pone el método HTTP (GET para leer, POST para crear, DELETE para borrar). El mismo puñado de verbos sirve para todos los recursos, y eso hace la API predecible. Lo desarrollamos en la lección de verbos y códigos de estado.

Recurso ≠ representación #

Aquí está la "R" de REST, y es la idea que más se pasa por alto. El recurso es una entidad conceptual: "el usuario 42". Lo que viaja por el cable no es el usuario, es una representación de su estado en un momento dado y en un formato concreto.

El mismo recurso puede representarse de varias formas. Un cliente pide JSON; otro, el mismo recurso en XML o en HTML. La negociación se hace con la cabecera Accept, y el servidor responde diciendo qué formato eligió con Content-Type:

http · una representación JSON del usuario 42 
GET /usuarios/42 HTTP/1.1
Host: api.ejemplo.com
Accept: application/json

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 42,
  "nombre": "Marina",
  "email": "marina@ejemplo.com"
}

Que recurso y representación sean cosas distintas tiene consecuencias prácticas: la URL identifica el recurso para siempre aunque cambies el formato de la respuesta; puedes versionar la representación sin mover el recurso de sitio; y la cache guarda representaciones con su Content-Type, no objetos de tu base de datos.

Por qué "transfer of state"

El nombre completo encaja ahora: el cliente y el servidor se pasan representaciones del estado de los recursos. El cliente lee el estado actual (GET) o manda un nuevo estado deseado (PUT/PATCH). Toda la conversación es transferir estado de un lado a otro.

Sin estado: cada petición se basta #

Una API REST es stateless: el servidor no recuerda nada entre una petición y la siguiente. Cada petición llega con todo lo necesario para entenderla —a qué recurso apunta, quién la hace (su token), qué formato acepta—, y el servidor la atiende sin apoyarse en una "conversación" previa.

Esto no significa que no haya sesiones de usuario; significa que el estado de la sesión viaja en cada petición (típicamente en un token de la cabecera Authorization), no en la memoria del servidor.

1 Cada petición es autónomaLleva su token, su recurso, su Accept
2 El servidor no guarda contextoNo hay "memoria" de la petición anterior
3 Escala en horizontalPones 10 servidores detrás de un balanceador y da igual cuál responda

El precio de no recordar nada: poder repartir la carga entre máquinas que no se coordinan.

Esa es la razón de ser de la restricción. Si el servidor guardara el estado de la sesión en su memoria, la segunda petición tendría que volver a la misma máquina que atendió la primera. Al no guardar nada, cualquier instancia sirve cualquier petición: puedes escalar añadiendo servidores, reiniciar uno sin tirar sesiones y cachear con criterio. La Web entera funciona así.

El error clásico contra stateless

Guardar en memoria del servidor un "carrito en curso" o un "asistente paso a paso" atado a la conexión. En cuanto hay más de una instancia, el paso 2 puede caer en otra máquina que no sabe nada del paso 1. El estado del flujo debe vivir en el cliente o persistido como un recurso (un /carritos/77 real), no en una variable del proceso.

La interfaz uniforme #

Es la restricción que hace que REST "se sienta" REST. La idea: que todos los recursos se manejen con la misma interfaz genérica en lugar de una a medida por entidad. En la práctica son cuatro convenciones:

PiezaQué aporta
Recursos identificados por URLCada cosa tiene un nombre estable. /pedidos/1001 es ese pedido, hoy y dentro de un año.
Manipulación por representacionesModificas el recurso mandando una representación de su nuevo estado, no llamando a un método propietario.
Mensajes autodescriptivosCada petición/respuesta dice cómo interpretarse: el método, el Content-Type, las cabeceras de cache. No hace falta contexto externo.
Hipermedia (HATEOAS)Las respuestas incluyen enlaces a las siguientes acciones posibles, como una página web enlaza a otras.

La uniformidad es lo que te deja aprender una API REST por inferencia: si sabes que GET /usuarios/42 devuelve el usuario 42, ya intuyes que GET /pedidos/1001 devuelve el pedido 1001 y que DELETE /pedidos/1001 lo borra. No hay que leer un manual por cada operación; el patrón se repite.

Los niveles de Richardson #

Leonard Richardson propuso una escala —el modelo de madurez de Richardson— para medir cuánto aprovecha una API las ideas de REST. Es la mejor forma de situar dónde está la tuya:

NivelQué usaSíntoma
0 — El pantano del POSTHTTP solo como túnel. Un único endpoint, todo por POST.POST /api con {"accion":"getUsuario"} dentro. Es RPC envuelto en HTTP.
1 — RecursosVarias URLs, una por recurso.POST /usuarios/42 y POST /pedidos/1001, pero todo sigue siendo POST.
2 — Verbos HTTPRecursos + métodos con su significado (GET, POST, PUT, DELETE) y códigos de estado correctos.Donde vive el 95 % de las "APIs REST" reales. Predecibles y suficientes.
3 — HipermediaNivel 2 + enlaces en las respuestas (HATEOAS).El cliente descubre qué puede hacer a continuación siguiendo enlaces, no cableando URLs.

El nivel 3 es HATEOAS: la respuesta no solo trae datos, trae también las acciones disponibles como enlaces. Un pedido pendiente de pago te dice, en la propia respuesta, dónde pagarlo o cancelarlo:

json · respuesta con hipermedia (nivel 3) 
{
  "id": 1001,
  "estado": "pendiente_de_pago",
  "total": 49.90,
  "_links": {
    "self":     { "href": "/pedidos/1001" },
    "pagar":    { "href": "/pedidos/1001/pago" },
    "cancelar": { "href": "/pedidos/1001" }
  }
}

Sobre el papel HATEOAS es elegante: el cliente no necesita conocer de antemano la estructura de URLs, la descubre. En la práctica casi nadie llega al nivel 3 puro —construir clientes que naveguen por hipermedia es costoso y aporta poco en APIs internas— y el sector ha aceptado que el nivel 2 es "ser REST" para casi todos los efectos. Conviene conocer el 3 para no confundirlo con un fallo de diseño cuando lo veas.

Dato de purista

Para Fielding, una API solo es REST de verdad en el nivel 3: sin hipermedia, no hay "hypermedia as the engine of application state". Es técnicamente correcto y a la vez una batalla que el uso cotidiano del término ya perdió. Apunta a un nivel 2 sólido y bien hecho; el 3, cuando aporte.

REST en la práctica, sin dogma #

REST es un estilo, no una ley. Su objetivo es que tu API sea predecible, cacheable y fácil de evolucionar; cuando una regla deje de servir a ese objetivo, el objetivo gana. Hay operaciones que no encajan limpias como "un sustantivo con un verbo" —una búsqueda compleja, un login, un "recalcula todos los precios"— y forzarlas a un molde REST perfecto produce URLs peores, no mejores.

La meta no es marcar todas las casillas de la tesis de Fielding. Es que cualquiera que llegue a tu API entienda, sin manual, qué recursos hay y cómo se manejan. Las siguientes lecciones bajan esto a decisiones concretas: qué verbo y qué código usar, cómo nombrar y paginar las URLs y cómo devolver errores que se entiendan.

Resumen #

Lo que te llevas

  • REST es un estilo de arquitectura sobre HTTP, no un protocolo ni una librería. HTTP con JSON no es automáticamente REST.
  • El recurso es la entidad (un sustantivo en una URL estable); el verbo lo pone el método HTTP, no la URL.
  • Recurso y representación son distintos: lo que viaja es una representación de su estado en un formato (JSON, XML…) negociado por Accept.
  • Stateless: cada petición se basta a sí misma. Es lo que permite escalar en horizontal poniendo más servidores detrás de un balanceador.
  • Los niveles de Richardson miden la madurez REST (0 túnel → 1 recursos → 2 verbos → 3 hipermedia). El nivel 2 es "ser REST" para casi todos.
