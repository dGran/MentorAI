docker-compose: el entorno local

Una app real no es un contenedor: son varios —la aplicación, su base de datos, una cache— que tienen que arrancar juntos y hablarse. Orquestar eso a mano con comandos sueltos es un suplicio. Compose lo declara todo en un fichero y lo levanta con una orden.

Docker Compose es la herramienta para definir y arrancar aplicaciones de varios contenedores. En vez de lanzar cada contenedor con su docker run lleno de flags, describes todos los servicios en un fichero compose.yaml y los levantas —o los tiras— de golpe. Es el corazón del entorno de desarrollo local containerizado.

Varios contenedores a la vez #

Tu aplicación necesita PHP, pero también MySQL para los datos y quizá Redis para la cache. Son tres contenedores que deben arrancar, conectarse entre sí y configurarse con sus puertos y credenciales. Hacerlo a mano significa tres docker run largos, en el orden correcto, recordando cada flag, cada vez. Inviable.

Compose convierte todo eso en una descripción declarativa: dices qué servicios quieres y cómo se relacionan, y él se encarga del cómo. Y como es un fichero, vive en el repositorio: quien clona el proyecto tiene el entorno entero a un comando de distancia.

El fichero compose #

Es un YAML con una sección services, donde cada entrada es un contenedor. Un entorno típico de app PHP + base de datos:

compose.yaml · app + base de datos 
services:
  app:
    build: .
    ports:
      - "8080:9000"
    volumes:
      - .:/app
    depends_on:
      - db

  db:
    image: mysql:8.3
    environment:
      MYSQL_DATABASE: tienda
      MYSQL_ROOT_PASSWORD: secreto
    volumes:
      - datos-db:/var/lib/mysql

volumes:
  datos-db:

Se lee de un vistazo: dos servicios, app (construido desde tu Dockerfile) y db (la imagen oficial de MySQL). El app publica un puerto, monta el código y declara que depende de db. Abajo, un volumen con nombre para los datos de la base de datos. Todo el entorno, en veinte líneas versionadas.

Se hablan por su nombre #

Una de las cosas que Compose hace por ti sin que la pidas: crea una red privada para tus servicios y los conecta a ella. Dentro de esa red, cada servicio es localizable por el nombre que le diste. Tu app no se conecta a la base de datos por una IP, sino por db:

bash · la app encuentra la BD por el nombre del servicio 
# dentro del contenedor app, el host de la BD es "db"
DATABASE_URL="mysql://root:secreto@db:3306/tienda"

Compose monta un pequeño DNS interno que resuelve db a la IP del contenedor de la base de datos. Esto es enorme: no dependes de IPs que cambian en cada arranque, sino de nombres estables que tú eliges. Es el mismo principio de nombres sobre direcciones del que habla URL a fondo, aquí dentro de tu red de contenedores.

Puerto interno ≠ puerto publicado

En "8080:9000", el primero es el puerto en tu máquina y el segundo el del contenedor. Entre contenedores de la misma red se hablan por el puerto interno (db:3306) sin necesidad de publicar nada. Solo publicas (ports) lo que quieres alcanzar desde fuera, como tu navegador.

Volúmenes: persistir y editar #

Recordemos que un contenedor es efímero: lo que escribe dentro muere con él. Los volúmenes son el mecanismo para que algo sobreviva, y en el ejemplo cumplen dos papeles distintos:

LíneaTipoPara qué
datos-db:/var/lib/mysqlVolumen con nombrePersistencia: los datos de MySQL sobreviven aunque borres y recrees el contenedor.
.:/appBind mountDesarrollo en vivo: monta tu carpeta de código dentro del contenedor; editas en tu IDE y el cambio se ve al instante, sin reconstruir.

El bind mount es lo que hace que trabajar containerizado sea cómodo: tu código vive en tu máquina (lo editas con tu editor de siempre) pero se ejecuta dentro del contenedor. No reconstruyes la imagen por cada cambio de una línea. El volumen con nombre, en cambio, deja que Docker gestione dónde guardar los datos que deben perdurar entre reinicios.

Comandos del día a día #

Con el fichero escrito, el entorno entero se maneja con un puñado de comandos:

bash · levantar, mirar, entrar y tirar el entorno 
# levantar todos los servicios en segundo plano
docker compose up -d

# ver los logs (en vivo con -f)
docker compose logs -f app

# abrir una shell dentro del contenedor de la app
docker compose exec app bash

# parar y eliminar contenedores y redes
docker compose down

Cuidado con down -v

docker compose down normal borra contenedores y redes pero conserva los volúmenes, así que tus datos siguen ahí al volver a levantar. Añadir -v (down -v) borra también los volúmenes: te cargas la base de datos. Útil para empezar de cero a propósito, peligroso por error.

Una nota sobre el entorno: Compose es la herramienta estrella en desarrollo local. Para producción con muchas máquinas y alta disponibilidad, el relevo lo toma un orquestador como Kubernetes; pero ese es otro escalón, y para construir y operar tu servicio en local, Compose es exactamente lo que necesitas.

Resumen #

Lo que te llevas

  • Compose declara una app de varios contenedores en un YAML y la levanta o tira de golpe. El fichero vive en el repo.
  • Cada entrada de services es un contenedor; build usa tu Dockerfile, image usa una imagen ya hecha.
  • Los servicios se encuentran por su nombre en una red privada (db:3306), sin depender de IPs.
  • Volúmenes con nombre persisten datos (la BD sobrevive); los bind mounts montan tu código para editarlo en vivo.
  • up -d, logs -f, exec, down cubren el día a día. Ojo con down -v: borra los volúmenes.
