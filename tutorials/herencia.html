Herencia: reutilizar y especializar

La herencia deja que una clase parta del trabajo de otra y lo especialice. Es potente y se abusa de ella a partes iguales. Aquí ves cómo funciona, cuándo de verdad expresa un «es un» y por qué la regla veterana es preferir la composición.

Heredar es decir «esta clase es como aquella, pero con un añadido o un cambio». La subclase recibe gratis las propiedades y métodos de la superclase y solo escribe lo que la diferencia. Bien usada ahorra repetición; mal usada crea unas jerarquías que se vuelven una jaula. El objetivo de esta lección es que sepas distinguir los dos casos.

Qué es heredar #

La herencia modela una relación «es un». Un Administrador es un Usuario con permisos extra; un CuadradoTest es un TestCase. La subclase hereda todo lo de la superclase (la clase padre o base) y puede añadir lo suyo o cambiar parte. La prueba del algodón antes de usar extends es decir la frase en voz alta: si «X es un Y» suena raro, la herencia es el martillo equivocado.

«Es un» vs «tiene un»

Es un sugiere herencia; tiene un sugiere composición. Un coche tiene un motor (composición), no es un motor. Confundir las dos es el origen de la mayoría de jerarquías retorcidas.

extends y parent:: #

En PHP una clase hereda de otra con extends. La subclase puede usar lo public y protected del padre (no lo private: eso es solo del padre). Y con parent:: llama a la versión del padre de un método, algo típico en el constructor para no repetir su inicialización.

php · extends y parent:: 
<?php

declare(strict_types=1);

class Usuario
{
    public function __construct(protected string $nombre)
    {
    }

    public function saludar(): string
    {
        return "Hola, soy {$this->nombre}";
    }
}

class Administrador extends Usuario
{
    public function __construct(string $nombre, private array $permisos)
    {
        parent::__construct($nombre);
    }

    public function puede(string $accion): bool
    {
        return in_array($accion, $this->permisos, true);
    }
}

$admin = new Administrador('Ana', ['borrar']);
echo $admin->saludar();        // heredado: "Hola, soy Ana"
var_dump($admin->puede('borrar')); // propio: true

Administrador no reescribe saludar(): lo hereda tal cual. Solo añade puede() y guarda los permisos. El protected $nombre es accesible desde la subclase; si fuera private, no. Esa es la razón principal de protected: «esto es para mí y para quien herede».

Sobrescribir métodos #

Especializar de verdad es sobrescribir: declarar en la subclase un método con el mismo nombre que el del padre para cambiar su comportamiento. Quien tenga un Usuario en la mano y llame a saludar() recibirá la versión que corresponda al objeto real. Esto es la antesala del polimorfismo.

php · sobrescribir y extender el padre 
<?php

class Administrador extends Usuario
{
    public function saludar(): string
    {
        return parent::saludar() . ' (administrador)';
    }
}

echo (new Administrador('Ana', []))->saludar();
// "Hola, soy Ana (administrador)"

Aquí la subclase no descarta al padre: lo reutiliza con parent::saludar() y le añade un matiz. Sobrescribir no obliga a empezar de cero; muchas veces es «lo del padre, más esto».

El peligro de las jerarquías #

La herencia engancha porque al principio ahorra código. El problema llega cuando la jerarquía crece. Cada subclase queda acoplada a las tripas del padre: si cambias un método de la base, puedes romper a todos sus descendientes sin saberlo. Y como PHP solo permite heredar de una clase, tarde o temprano dos ejes de variación no caben en una sola línea.

El clásico: la jerarquía que explota

Empiezas con Empleado. Luego EmpleadoPorHoras y EmpleadoFijo. Después necesitas distinguir por departamento, y por país, y por tipo de contrato. Cada eje nuevo multiplica las subclases (EmpleadoFijoVentasEspaña…). La herencia solo da un eje; la realidad tiene varios.

A esto se suma el problema de la clase base frágil: la superclase no puede evolucionar con libertad porque no sabe en qué se apoyaron sus hijas. Una jerarquía profunda es un contrato implícito y rígido entre clases que casi nunca cambian al mismo ritmo.

Herencia vs composición #

La alternativa es la composición: en vez de ser otra clase, tu objeto tiene otros objetos y les delega trabajo. En lugar de heredar la capacidad de notificar, le pasas un notificador. Ganas flexibilidad: cambias la pieza sin tocar la jerarquía, y combinas comportamientos sin la explosión de subclases.

Herencia rígida

  • InformeConEmail extends Informe.
  • ¿Y por SMS? ¿Y por email + SMS? Más subclases.
  • Cambiar el envío toca la jerarquía.

Composición flexible

  • Informe recibe un Notificador.
  • Email, SMS o ambos: distintos objetos, misma clase.
  • Cambiar el envío es pasar otra pieza.
php · componer en vez de heredar 
<?php

declare(strict_types=1);

final class Informe
{
    public function __construct(private Notificador $notificador)
    {
    }

    public function publicar(string $contenido): void
    {
        // ...genera el informe...
        $this->notificador->enviar($contenido);
    }
}

$porEmail = new Informe(new NotificadorEmail());
$porSms = new Informe(new NotificadorSms());

Informe no es un notificador: tiene uno. Ese Notificador que recibe es, idealmente, una interfaz, lo que conecta directamente con la inyección de dependencias. La máxima «favorece la composición sobre la herencia» no prohíbe heredar: te recuerda que la composición suele envejecer mejor.

¿Cuándo sí heredar?

Cuando hay un «es un» genuino y estable, la subclase no rompe lo que el padre promete (eso es el principio de sustitución de Liskov) y la jerarquía es plana. Para compartir un contrato sin código, la clase abstracta y la interfaz suelen ser mejores que una base concreta.

Resumen #

Lo que te llevas

  • La herencia (extends) modela un «es un»: la subclase recibe lo del padre y añade o sobrescribe.
  • parent:: reutiliza la versión del padre; protected es lo que el padre comparte con sus subclases.
  • Las jerarquías profundas acoplan y se vuelven frágiles; la herencia solo ofrece un eje de variación.
  • Favorece la composición («tiene un») sobre la herencia: cambias y combinas piezas sin reorganizar una jerarquía.
