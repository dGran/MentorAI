Transacciones y ACID: todo o nada

Transfieres 100 € de una cuenta a otra: restar de una y sumar a la otra son dos pasos, pero el mundo no debe verlos nunca por separado. Si el servidor se cae entre ambos, el dinero no puede evaporarse. De eso van las transacciones, y ACID es la promesa que las respalda.

Una transacción agrupa varias operaciones en una sola unidad indivisible: o se aplican todas, o ninguna. Es la herramienta con la que la base de datos te protege de dos enemigos: los fallos a mitad de camino y la concurrencia de muchos usuarios a la vez.

El problema: dos pasos, un resultado #

La transferencia bancaria es el ejemplo canónico porque deja el problema desnudo: restar de la cuenta A y sumar a la B son dos escrituras distintas. Si el proceso muere justo entre ambas, el dinero desaparece de A sin llegar a B. La cuenta no cuadra, y «a veces» no es una opción cuando hablamos de dinero, reservas o inventario.

Sin transacción, cada UPDATE se confirma por su cuenta y no hay vuelta atrás. Con transacción, los dos pasos viven dentro de un paréntesis: hasta que no dices COMMIT, nada es definitivo, y siempre puedes ROLLBACK para dejarlo como estaba.

ACID: las cuatro garantías #

ACID es el acrónimo de las cuatro propiedades que una transacción promete. No son jerga académica: cada letra resuelve un fallo concreto del mundo real.

LetraGarantíaQué evita
AtomicidadTodo o nada: o se aplican todos los pasos o ninguno.Que la transferencia se quede a medias.
ConsistenciaLa transacción lleva de un estado válido a otro válido (respeta reglas y claves).Saldos negativos prohibidos, claves foráneas rotas.
IslamientoLas transacciones concurrentes no se pisan: cada una se ejecuta como si estuviera sola.Que otra transacción vea tus datos a medio escribir.
DurabilidadTras el COMMIT, el cambio sobrevive a un corte de luz.Perder una venta confirmada por un reinicio.

La «I» es la que da guerra

Atomicidad y durabilidad son casi binarias: las tienes o no. El aislamiento, en cambio, es un dial: más aislamiento significa menos sorpresas pero también menos concurrencia. El resto del tutorial gira en torno a ese dial.

Una transacción en SQL #

En SQL, una transacción se abre con BEGIN y se cierra con COMMIT (confirmar) o ROLLBACK (deshacer). Entre medias, ningún otro cliente ve tus cambios a medias.

sql · la transferencia, atómica 
BEGIN;

UPDATE cuentas SET saldo = saldo - 100 WHERE id = 1;
UPDATE cuentas SET saldo = saldo + 100 WHERE id = 2;

-- Si algo falla entre medias, una sola orden lo deshace todo:
ROLLBACK;   -- nadie pierde 100 sin que el otro los reciba

-- Si las dos escrituras fueron bien:
COMMIT;     -- ahora sí, permanente y visible para todos

Lo que sale mal en concurrencia #

Cuando dos transacciones se solapan, aparecen tres anomalías clásicas. El estándar SQL las define para luego decir, por niveles, cuáles permite cada uno:

AnomalíaQué pasa
Lectura sucia (dirty read)Lees un cambio de otra transacción que aún no ha hecho COMMIT… y que puede acabar en ROLLBACK.
Lectura no repetible (non-repeatable read)Lees una fila dos veces en tu transacción y obtienes valores distintos porque otra la modificó y confirmó en medio.
Lectura fantasma (phantom read)Repites una consulta con WHERE y aparecen filas nuevas que otra transacción insertó.

Niveles de aislamiento #

El nivel de aislamiento es el dial que decide qué anomalías toleras a cambio de más concurrencia. De menos a más estricto:

NivelLectura suciaNo repetibleFantasma
READ UNCOMMITTEDposibleposibleposible
READ COMMITTEDnoposibleposible
REPEATABLE READnonoposible*
SERIALIZABLEnonono

* En el estándar, REPEATABLE READ permite fantasmas; el InnoDB de MySQL los evita con su mecanismo de snapshots.

sql · fijar el nivel de la transacción 
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;

BEGIN;
SELECT saldo FROM cuentas WHERE id = 1;   -- 500
-- ... otra transacción cambia el saldo y hace COMMIT ...
SELECT saldo FROM cuentas WHERE id = 1;   -- sigue 500: lectura repetible
COMMIT;

Más aislamiento no es gratis

SERIALIZABLE elimina toda anomalía, pero a costa de más bloqueos, menos paralelismo y más probabilidad de que una transacción tenga que reintentarse. Por eso casi nadie lo usa por defecto: PostgreSQL y Oracle vienen en READ COMMITTED; MySQL/InnoDB, en REPEATABLE READ. Sube el dial solo donde el dato lo exija.

Locks y deadlocks #

Para aislar, la base de datos usa bloqueos (locks): mientras una transacción modifica una fila, otra que quiera tocarla espera. Puedes pedir el bloqueo por adelantado con FOR UPDATE, y así reservar las filas que vas a cambiar.

sql · bloqueo pesimista con FOR UPDATE 
BEGIN;

-- Reserva las dos filas: nadie más las modifica hasta tu COMMIT
SELECT * FROM cuentas WHERE id IN (1, 2) FOR UPDATE;

UPDATE cuentas SET saldo = saldo - 100 WHERE id = 1;
UPDATE cuentas SET saldo = saldo + 100 WHERE id = 2;

COMMIT;

El riesgo de los bloqueos es el deadlock: dos transacciones esperando, cada una, una fila que la otra tiene bloqueada. Ninguna avanza. El motor lo detecta, mata a una (verás un error de deadlock) y deja seguir a la otra; tu código debe reintentar la que murió.

El antídoto del deadlock: orden consistente

Si todas tus transacciones bloquean las filas en el mismo orden (por ejemplo, siempre por id ascendente), el ciclo de espera no puede formarse. La mayoría de los deadlocks de producción se curan ordenando los accesos, no subiendo el aislamiento.

Frente al bloqueo pesimista (reservo antes de tocar) existe el optimista: no bloqueo, pero al guardar compruebo que la fila no ha cambiado desde que la leí (con una columna de versión). Cada uno encaja en un escenario distinto:

Pesimista (FOR UPDATE)

  • Bloquea desde el principio; cero conflictos al guardar.
  • Bueno con mucha contención sobre las mismas filas.
  • Coste: más esperas y riesgo de deadlock.

Optimista (versión)

  • No bloquea; valida al hacer COMMIT.
  • Bueno con poca contención: máxima concurrencia.
  • Coste: si hubo conflicto, toca reintentar.

Errores comunes #

ErrorPor qué duele
Transacciones largas y «gordas» Mantienen bloqueos mucho tiempo y frenan a todos. Abre tarde, confirma pronto.
No reintentar tras un deadlock El motor mata una transacción por diseño; si no la reintentas, pierdes la operación.
Subir a SERIALIZABLE «por seguridad» Estrangulas la concurrencia. Elige el nivel mínimo que tu caso tolera.
Lógica de negocio entre BEGIN y COMMIT Llamar a una API o esperar entrada del usuario con la transacción abierta bloquea filas eternamente.
Confiar el «todo o nada» a la aplicación Sin transacción real, un fallo a media tanda deja datos inconsistentes que nadie deshace.

Resumen #

Lo que te llevas

  • Una transacción agrupa varios pasos en una unidad atómica: COMMIT confirma, ROLLBACK deshace.
  • ACID = atomicidad, consistencia, aislamiento y durabilidad; cada letra cubre un fallo concreto.
  • La concurrencia provoca lecturas sucias, no repetibles y fantasma; el nivel de aislamiento decide cuáles toleras.
  • Más aislamiento = menos anomalías pero menos concurrencia. Usa el mínimo que tu caso tolere.
  • Los deadlocks se previenen bloqueando siempre en el mismo orden; cuando ocurren, se reintenta.
