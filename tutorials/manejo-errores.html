Manejo de errores: excepciones con criterio

El manejo de errores ensucia el código más que casi nada: si cada llamada va seguida de un «¿salió bien?», la lógica de verdad se ahoga entre comprobaciones. Las excepciones, los tipos honestos y el fail fast separan lo que importa de lo que puede fallar, y dejan respirar al código.

Un error es algo excepcional: lo normal es que la operación salga bien. El código debería leerse pensando en ese camino feliz, y desviar el manejo del fallo a un lado. Las herramientas para lograrlo son viejas conocidas de PHP —excepciones tipadas, tipos nullable explícitos, validación temprana—, bien usadas.

Excepciones, no códigos #

Devolver un código de error (un false, un -1) obliga a quien llama a comprobarlo justo ahí, en plena lógica, y a propagar la comprobación hacia arriba. El resultado es un código tejido de if defensivos. Una excepción separa el qué del fallo: el camino normal queda limpio y el error se captura donde tenga sentido tratarlo.

Códigos de error

  • Cada llamada va seguida de un if.
  • La lógica se mezcla con la comprobación.
  • Es fácil ignorar un código y seguir como si nada.

Excepciones

  • El camino feliz se lee de un tirón.
  • El error no se puede ignorar en silencio.
  • Se captura donde de verdad se puede actuar.

Excepciones específicas #

Lanzar un \Exception genérico o un \RuntimeException a secas obliga a quien captura a leer el mensaje para saber qué pasó. Una excepción específica —SaldoInsuficiente, UsuarioNoEncontrado— es autoexplicativa, se puede capturar de forma selectiva y documenta los modos de fallo del código por su propio tipo.

php · una excepción que se nombra a sí misma 
<?php

declare(strict_types=1);

final class SaldoInsuficiente extends \DomainException
{
    public static function para(int $saldo, int $solicitado): self
    {
        return new self("Saldo {$saldo}, se pidieron {$solicitado}");
    }
}

// Quien captura puede ser selectivo
try {
    $cuenta->retirar($importe);
} catch (SaldoInsuficiente $error) {
    // tratar SOLO este caso, sin tragarte otros fallos
}

Usa la jerarquía estándar de PHP

Extiende de \LogicException (errores de programación que deberían pillarse en desarrollo: argumentos inválidos, estado imposible) o de \RuntimeException (fallos que solo se conocen en ejecución: red, base de datos). Esa raíz da semántica a tu excepción antes de leer el nombre.

No devolver ni pasar null #

Devolver null «cuando no hay» reparte la responsabilidad de comprobarlo por todo el código que llama; basta que uno lo olvide para tener el clásico error de método sobre null. Y pasar null como argumento es aún peor: mete ramas ocultas dentro de la función. Alternativas más limpias:

En vez de devolver null… Mejor
Cuando «no encontrar» es un caso esperado Tipo nullable explícito (?Usuario): el tipo avisa y obliga a tratarlo.
Cuando «no encontrar» es un error de verdad Lanzar una excepción específica (UsuarioNoEncontrado).
Una colección vacía Devuelve un array vacío o un objeto nulo, nunca null.

La clave es que el null, cuando exista, sea visible en el tipo (?Usuario) y no una sorpresa: así PHP y tu IDE te obligan a contemplarlo.

Fail fast #

Falla pronto y ruidosamente. Valida las precondiciones al principio del método y lanza en cuanto algo no cuadre, en lugar de arrastrar un estado inválido que reventará tres capas más allá, donde será imposible saber de dónde vino. Un error que salta cerca de su causa se arregla en minutos; uno que salta lejos, en horas.

php · valida y lanza al entrar 
<?php

declare(strict_types=1);

public function __construct(private string $email)
{
    if (!filter_var($email, FILTER_VALIDATE_EMAIL)) {
        throw new EmailInvalido($email);
    }
}

Validar en el constructor garantiza que un objeto inválido no llega a existir: el resto del código puede confiar en que cualquier Email que reciba es válido, sin volver a comprobarlo. La invariante se protege en un solo sitio.

Separar lógica y manejo #

Como dice Clean Code, «el manejo de errores es una cosa». Una función que mezcla la lógica de negocio con grandes bloques try/catch hace dos cosas a la vez. Extrae el cuerpo a una función y deja la de fuera dedicada solo a capturar y reaccionar: cada una se lee a su nivel.

No te tragues las excepciones

El antipatrón más dañino es el catch vacío o que solo registra y sigue: el error desaparece y el bug se vuelve invisible. Si capturas, es para hacer algo (reintentar, traducir el error, informar al usuario). Capturar para silenciar es esconder la suciedad bajo la alfombra.

Resumen #

Lo que te llevas

  • Usa excepciones, no códigos de error: el camino feliz queda limpio y el fallo no se ignora en silencio.
  • Lanza excepciones específicas (no genéricas), apoyadas en la jerarquía Logic/Runtime de PHP.
  • Evita null: hazlo explícito en el tipo (?Usuario) o lanza; nunca lo pases como argumento.
  • Fail fast: valida al entrar y protege las invariantes en el constructor; separa la lógica del manejo y nunca te tragues una excepción.
