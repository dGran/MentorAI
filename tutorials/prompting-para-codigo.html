Prompting para código

La calidad de lo que te devuelve un modelo depende casi por completo de cómo se lo pides. Prompting no es magia de palabras secretas: es comunicar un problema con la misma claridad con que se lo explicarías a un colega que acaba de llegar.

Recuerda del tutorial anterior que el modelo solo ve su ventana de contexto y completa lo más probable. El prompt es tu única palanca: es el contexto y la instrucción a la vez. Un prompt vago produce código vago; uno preciso, código preciso.

El prompt es una especificación #

Piensa en el prompt como en un mini documento de requisitos. Si a un programador humano le dijeras solo «hazme el login», te haría veinte preguntas. El modelo no las hace: rellena los huecos con lo que le parece más probable. Cada decisión que no tomas tú, la toma él, y no siempre como querías.

La regla del colega nuevo

Antes de enviar, léelo como si se lo dieras a alguien que entró hoy al equipo y no conoce el proyecto. ¿Tendría todo lo que necesita para no preguntarte? Si la respuesta es no, al modelo también le falta.

Dar contexto #

Lo que más mejora una respuesta es el contexto correcto: el código que se va a tocar, el error exacto que ves, la versión del lenguaje, las convenciones del proyecto. No describas el código de memoria: pégalo. El modelo razona muchísimo mejor sobre lo que tiene delante que sobre lo que le cuentas.

Sin contexto

  • «No me funciona el guardado, arréglalo.»
  • El modelo inventa cómo es tu código.
  • Te da una solución genérica que no encaja.

Con contexto

  • Pegas el método, el error y la versión.
  • Dices qué esperabas y qué pasó.
  • El modelo trabaja sobre tu realidad.

Ser específico #

Di el lenguaje, el estilo, las restricciones y la forma de la salida. Un detalle que cambia todo: indicar qué no quieres (sin librerías externas, sin cambiar la firma pública, sin comentarios). Las restricciones acotan el espacio de respuestas y te acercan a lo que de verdad buscas.

prompt · vago vs específico 
VAGO:
  "Valida un email en PHP."

ESPECÍFICO:
  "Escribe una función PHP 8.2 que valide un email.
   - Strict types, sin librerías externas.
   - Devuelve bool, no lanza excepciones.
   - Usa filter_var con FILTER_VALIDATE_EMAIL.
   - Nombre: isValidEmail(string $email): bool.
   Solo el código, sin explicación."

Dar ejemplos #

Un ejemplo vale más que un párrafo de descripción. Si quieres una salida con cierto formato, enseña una entrada y su salida deseada. Si quieres que siga un patrón de tu código, pega un trozo que ya lo siga. Esta técnica —darle un par de ejemplos— se llama few-shot y es de las más potentes.

Muéstrale tu estilo

Para que el código generado parezca tuyo, dale una clase de referencia y di «sigue este estilo y estas convenciones». El modelo imita patrones muy bien: aprovéchalo para que respete tu naming, tu manejo de errores y tu estructura sin tener que explicarlos.

Descomponer #

Pedir «toda la funcionalidad» de golpe invita al modelo a improvisar y a ti a no poder revisar. Trocea: primero el diseño de la interfaz, luego una pieza, luego la siguiente. Es la misma lógica de los baby steps de TDD: pasos pequeños que puedes verificar uno a uno.

Para problemas con razonamiento, pedirle que piense paso a paso antes de escribir el código («primero explica el plan, luego impleméntalo») suele dar mejores resultados que ir directo a la solución. Le das espacio para ordenar el problema antes de comprometerse.

Iterar sobre el resultado #

El primer intento rara vez es el final, y eso está bien. El prompting es una conversación: lee lo que devolvió, señala lo que falla con precisión y pide el ajuste. «Funciona, pero no contempla el caso de la lista vacía; añádelo y mantén el resto» avanza mucho más que empezar de cero.

El valor no está en el prompt perfecto, sino en el bucle de refinamiento.

Pedir el porqué #

Un uso infravalorado: no pidas solo el código, pide la explicación. «¿Por qué esta forma y no la otra?», «¿qué casos límite no cubre?», «¿qué riesgos tiene este enfoque?». Te obliga a entender lo que vas a integrar y a menudo destapa que el modelo tomó una decisión discutible que conviene cambiar.

No delegues el entendimiento

Pedir el porqué no es para que el modelo decida por ti, sino para que tú decidas con más información. Si integras código que no entiendes, la deuda es tuya. El prompting termina cuando entiendes lo que aceptas, no cuando compila.

Resumen #

Lo que te llevas

  • El prompt es una especificación: cada decisión que no tomas tú, la toma el modelo.
  • Da contexto (pega el código real), sé específico (incluye qué no quieres) y da ejemplos (few-shot).
  • Descompón en pasos verificables e itera: el valor está en el bucle de refinamiento, no en el prompt perfecto.
  • Pide el porqué para decidir con más información. No integres lo que no entiendes.
