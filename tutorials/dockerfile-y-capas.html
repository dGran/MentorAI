Dockerfile y capas

El Dockerfile es la receta que construye tu imagen, y cada línea deja una capa apilada sobre la anterior. Entender ese apilado —y la caché que lo acompaña— es la diferencia entre un build de tres segundos y uno de tres minutos en cada cambio.

Una imagen no aparece por magia: se construye paso a paso a partir de un Dockerfile, un fichero de texto con instrucciones que Docker ejecuta de arriba abajo. Cada instrucción añade una capa, y esas capas son la clave tanto del tamaño de la imagen como de lo rápido que se reconstruye.

La receta de una imagen #

El Dockerfile describe, de forma declarativa, cómo pasar de una imagen base a la imagen de tu aplicación. Se lee como una receta: parte de algo, copia ingredientes, ejecuta pasos, define cómo arrancar. Un ejemplo mínimo para una app PHP:

Dockerfile · una receta mínima 
FROM php:8.3-fpm

WORKDIR /app

COPY composer.json composer.lock ./
RUN composer install --no-dev --optimize-autoloader

COPY . .

EXPOSE 9000
CMD ["php-fpm"]

Se construye con un comando, dándole una etiqueta (un nombre) a la imagen resultante:

bash · construir la imagen desde el Dockerfile 
docker build -t mi-app:1.0 .

Las instrucciones clave #

No necesitas saberlas todas; con un puñado construyes el 95 % de las imágenes:

InstrucciónQué hace
FROMLa imagen base de la que partes. Siempre la primera línea.
WORKDIREl directorio de trabajo dentro de la imagen; los pasos siguientes corren ahí.
COPYCopia ficheros de tu máquina a la imagen.
RUNEjecuta un comando durante la construcción (instalar dependencias, compilar).
ENVDefine una variable de entorno dentro de la imagen.
EXPOSEDocumenta qué puerto usa el contenedor (informativo).
CMDEl comando que se ejecuta al arrancar el contenedor. Define el proceso principal.

RUN es en build; CMD es en run

La confusión típica: RUN se ejecuta mientras construyes la imagen (instalar paquetes, una sola vez); CMD se ejecuta cada vez que arranca un contenedor (lanzar tu app). Uno congela trabajo en la imagen; el otro define qué proceso vive en el contenedor.

Cada instrucción es una capa #

Aquí está el modelo mental que lo explica todo: cada instrucción del Dockerfile crea una capa, una porción de sistema de ficheros apilada sobre la anterior. La imagen final es la pila de todas las capas, de solo lectura. Cuando arrancas un contenedor, Docker le pone encima una capa fina de escritura.

1 FROM php:8.3-fpmCapa base: PHP y su sistema
2 RUN composer installCapa con las dependencias
3 COPY . .Capa con tu código

La imagen es la suma de capas inmutables; el contenedor añade una capa de escritura efímera encima.

Las capas se comparten entre imágenes: si diez imágenes parten de php:8.3-fpm, esa capa base se guarda una sola vez en disco. Es la misma idea de no duplicar que ya viste en otros sitios, aplicada al almacenamiento de imágenes.

La caché de capas #

Y ahora la consecuencia práctica más importante. Al reconstruir, Docker reutiliza las capas que no han cambiado: recorre el Dockerfile de arriba abajo y, mientras la entrada de una instrucción sea idéntica a la del build anterior, usa la capa cacheada en vez de rehacerla. En cuanto una instrucción cambia, esa capa y todas las de debajo se reconstruyen.

Una capa que cambia invalida todas las de abajo

La caché es secuencial: si modificas la instrucción nº 3, Docker rehace la 3, la 4, la 5… aunque su contenido no haya cambiado. Por eso lo que cambia a menudo debe ir lo más abajo posible en el Dockerfile, y lo estable lo más arriba.

El orden lo decide todo #

De esa regla nace el patrón más rentable al escribir Dockerfiles. Tu código cambia en cada commit; tus dependencias, casi nunca. Si copias todo junto y luego instalas, cualquier cambio de código invalida la caché y reinstalas las dependencias desde cero cada vez. La solución: copiar primero solo el manifiesto de dependencias, instalarlas, y copiar el código después.

Orden ingenuo

  • COPY . .
  • RUN composer install
  • Cualquier cambio de código → reinstala dependencias.
  • Build lento en cada commit.

Orden óptimo

  • COPY composer.json composer.lock ./
  • RUN composer install
  • COPY . .
  • Cambias código → solo se rehace la última capa.

Mira de nuevo la receta del principio: copia composer.json y composer.lock, instala, y luego hace COPY . .. Así, mientras no toques las dependencias, la capa pesada del composer install sale de caché y el build vuela; solo se rehace la capa ligera de tu código. Ese pequeño reordenamiento convierte minutos en segundos.

Regla mnemotécnica

De arriba abajo en el Dockerfile: de lo más estable a lo más cambiante. Imagen base → librerías del sistema → dependencias → código de la app. Cuanto más arriba, más veces sale de caché.

Resumen #

Lo que te llevas

  • El Dockerfile es la receta declarativa que construye una imagen; docker build la ejecuta de arriba abajo.
  • Pocas instrucciones bastan: FROM, WORKDIR, COPY, RUN, ENV, EXPOSE, CMD. RUN corre en build; CMD, al arrancar el contenedor.
  • Cada instrucción crea una capa inmutable; la imagen es la pila, y las capas se comparten entre imágenes.
  • La caché reutiliza capas sin cambios, pero una capa modificada invalida todas las de debajo.
  • Ordena de lo estable a lo cambiante: copia e instala dependencias antes de copiar el código. Builds de segundos, no minutos.
