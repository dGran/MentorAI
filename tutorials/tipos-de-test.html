La pirámide de tests

No todos los tests cuestan lo mismo ni dan la misma confianza. La pirámide ordena tres niveles —unitarios, de integración y end-to-end— según ese trade-off, y te dice cuántos poner de cada uno para que la suite sea rápida y fiable a la vez.

Un test te da confianza en que algo funciona, pero esa confianza tiene precio: velocidad de ejecución, fragilidad y coste de mantenerlo. La pirámide de tests es una forma visual de equilibrar esos tres ejes para que tu suite no sea ni inútil ni insufrible.

Por qué hay niveles #

Un test puede comprobar una función aislada o el sistema entero de punta a punta. Cuanto más abarca, más confianza da («esto funciona de verdad para el usuario») pero más cuesta: tarda más, se rompe por más motivos y es más difícil saber qué falló cuando falla. Los niveles existen para poner cada comprobación al precio justo.

Tres ejes en tensión

Velocidad (¿corre en milisegundos o en minutos?), confianza (¿se parece a lo que hace el usuario real?) y coste (¿cuánto duele escribirlo y mantenerlo?). Ningún test gana en los tres a la vez; eliges según lo que estés cubriendo.

La pirámide #

La metáfora, de Mike Cohn, es un triángulo con tres pisos. La base, ancha, son muchos tests unitarios; el centro, menos, de integración; la cúspide, pocos, end-to-end. La forma no es decorativa: cuanto más arriba, más caro y lento, así que pones menos.

▲ End-to-end (pocos)El sistema entero como lo usa un usuario. Lentos, frágiles, máxima confianza.
■ Integración (algunos)Varias piezas juntas: código + base de datos, código + API.
▬ Unitarios (muchos)Una unidad aislada. Rápidos, baratos, precisos al señalar el fallo.

Más abajo = más cantidad. Más arriba = más confianza por test, pero más caro.

Tests unitarios #

Prueban una unidad de código aislada —normalmente una clase o función— sin tocar base de datos, red ni ficheros. Las dependencias se sustituyen por dobles de test. Corren en milisegundos, así que puedes tener miles y ejecutarlos en cada guardado.

Su superpoder es la precisión: cuando uno falla, sabes exactamente qué función se rompió. Su límite es que prueban las piezas por separado; que cada engranaje gire no garantiza que el reloj dé la hora.

php · unitario (sin dependencias externas) 
<?php

declare(strict_types=1);

public function test_calcula_el_iva_de_un_importe(): void
{
    $calculator = new TaxCalculator();

    $this->assertSame(121.0, $calculator->withVat(100.0, 0.21));
}

Tests de integración #

Comprueban que varias piezas colaboran bien: tu repositorio contra una base de datos real, tu cliente contra una API de pruebas, tu código contra la cola de mensajes. Aquí ya no sustituyes la dependencia: usas una de verdad (a menudo en un contenedor, una base de datos efímera).

Dan una confianza que el unitario no puede: que tu SQL es válido, que el mapeo objeto-tabla cuadra, que la transacción hace lo que crees. A cambio son más lentos (arrancar y poblar una base de datos cuesta) y se rompen por más motivos.

Aquí vive el valor real en backend

En una aplicación de backend, mucha de la lógica que importa es «pedir esto a la base de datos y transformarlo». Los tests de integración cubren justo esa frontera, que los unitarios fingen con dobles. No los descuides por estar en mitad de la pirámide.

Tests end-to-end #

Ejercitan el sistema completo como lo haría un usuario: lanzan la aplicación de verdad y simulan clics en el navegador o peticiones HTTP reales que recorren toda la pila. Son la única prueba de que «el conjunto funciona», incluida la configuración, las rutas y el despliegue.

También son los más caros: lentos (segundos por test), frágiles (un cambio de maquetación los rompe) y difíciles de depurar (¿falló el front, el back, la red?). Por eso van en la cúspide: unos pocos que cubran los caminos críticos (registrarse, pagar), no cada variación.

El cono de helado #

El antipatrón clásico es la pirámide al revés: el cono de helado. Muchos tests end-to-end, pocos unitarios y, coronándolo, un montón de pruebas manuales. Pasa cuando se testea «por la interfaz» porque parece más realista.

Pirámide

  • Base ancha de unitarios rápidos.
  • La suite corre en segundos.
  • Un fallo señala la pieza exacta.
  • Refactorizas sin miedo.

Cono de helado

  • Casi todo es e2e lento y frágil.
  • La suite tarda y la gente la ignora.
  • Cuando falla, no sabes dónde.
  • Tests «flaky» que fallan a ratos sin motivo.

Una suite lenta es una suite muerta

Si los tests tardan diez minutos, la gente deja de ejecutarlos antes de subir código. Una suite que no se corre no protege nada. La velocidad de la base unitaria no es un lujo: es lo que mantiene viva la práctica.

Cuánto de cada #

Las proporciones famosas (70/20/10) son una guía, no una ley. La regla útil de verdad es: empuja cada test al nivel más bajo que pueda probar lo que te importa. Si una regla de negocio se puede comprobar en un unitario, no la pruebes por la interfaz. Reserva el e2e para lo que solo se ve integrando todo.

Hay quien prefiere hablar de un panal (más integración que pirámide pura) para backends donde la lógica vive en la frontera con la base de datos. El reparto exacto depende de tu sistema; el principio no cambia: lo barato y preciso abajo, lo caro y realista arriba, y poco.

Resumen #

Lo que te llevas

  • Cada test equilibra velocidad, confianza y coste: ninguno gana en los tres.
  • Unitarios: muchos, rápidos, una unidad aislada. Integración: piezas reales juntas. E2e: pocos, el sistema entero.
  • El cono de helado (al revés) da suites lentas y frágiles que la gente acaba ignorando.
  • Empuja cada comprobación al nivel más bajo que la pueda probar. Lo caro y realista, arriba y poco.
