Clases abstractas: plantilla a medio hacer

Una clase abstracta es un molde incompleto: trae código común ya hecho y deja huecos marcados que sus hijas deben rellenar. Es perfecta para el patrón «el esqueleto lo pongo yo, los pasos concretos los pones tú». Y es el momento de resolver la pregunta del millón: ¿abstracta o interfaz?

Entre la interfaz (todo contrato, cero código) y la clase normal (todo implementado) hay un punto medio: la clase abstracta. Define parte del comportamiento ya resuelto y declara otros métodos como huecos que las subclases están obligadas a rellenar. No se puede instanciar directamente, porque está incompleta a propósito.

Qué es una clase abstracta #

Se marca con abstract class. La diferencia visible con una clase normal es que no se puede hacer new de ella: solo existe para que otras la extiendan. Tiene sentido cuando «X en general» no es algo concreto —no hay un MedioDePago a secas, hay tarjetas, transferencias, PayPal— pero todos comparten parte del trabajo.

Por qué no se instancia

Si una clase tiene métodos sin cuerpo (abstractos), instanciarla dejaría llamadas sin destino. PHP lo prohíbe: la clase abstracta es un contrato con código común, no un objeto usable por sí mismo.

Métodos abstractos #

Dentro de una clase abstracta puedes mezclar dos cosas: métodos normales (con cuerpo, heredados tal cual) y métodos abstractos (solo la firma, sin cuerpo). Cada método abstracto es un hueco obligatorio: toda subclase concreta debe implementarlo o no compila.

php · código común + huecos obligatorios 
<?php

declare(strict_types=1);

abstract class MedioDePago
{
    public function __construct(protected string $referencia)
    {
    }

    abstract public function cobrar(int $centimos): Recibo;

    public function referencia(): string
    {
        return $this->referencia;
    }
}

referencia() ya está hecho y lo heredan todas; pero cobrar() es abstracto: cada medio de pago cobra de una forma distinta y tiene que decir cómo.

El patrón plantilla #

El uso estrella de las clases abstractas es el template method (método plantilla): la clase base define el esqueleto de un proceso —el orden de los pasos— y deja los pasos que varían como métodos abstractos. La subclase no controla el flujo, solo rellena los huecos. El «qué orden» está fijo arriba; el «cómo de cada paso», abajo.

php · el esqueleto manda, los pasos se rellenan 
<?php

declare(strict_types=1);

abstract class ImportadorCsv
{
    final public function importar(string $ruta): void
    {
        $filas = $this->leer($ruta);

        foreach ($filas as $fila) {
            $this->procesar($fila);
        }

        $this->finalizar();
    }

    abstract protected function procesar(array $fila): void;

    protected function finalizar(): void
    {
        // por defecto no hace nada; la subclase puede sobrescribir
    }

    private function leer(string $ruta): iterable { /* ... */ }
}

importar() es final: define el flujo y nadie lo cambia. Cada importador concreto solo implementa procesar(). Así el orden (leer → procesar → finalizar) se garantiza una vez y no se puede romper por descuido en una subclase.

Abstracta vs interfaz #

Las dos definen un contrato y permiten polimorfismo, así que la duda es legítima. La diferencia de fondo: la interfaz solo promete métodos; la clase abstracta promete métodos y además aporta código y estado compartidos, a cambio de gastar tu única herencia.

Clase abstracta Interfaz
¿Aporta código? Sí (métodos con cuerpo). No, solo firmas.
¿Estado/propiedades? Sí. No.
¿Cuántas por clase? Una (gasta la herencia). Varias.
Relación «es un, y comparte tripas». «se comporta como».

Cómo decidir #

La heurística práctica, en dos preguntas:

Interfaz cuando…

  • Solo necesitas el contrato, sin código compartido.
  • Tipos sin parentesco real («cualquier cosa exportable»).
  • Una clase debe cumplir varios roles a la vez.
  • Por defecto: empieza aquí, es lo menos comprometedor.

Clase abstracta cuando…

  • Hay implementación común real que no quieres repetir.
  • Quieres fijar un flujo (template method) y dejar huecos.
  • Las subclases son variantes genuinas de un mismo concepto.

No es o lo uno o lo otro

Un patrón muy común y sano: una interfaz define el contrato público, y una clase abstracta que la implementa ofrece una base con lo común para quien quiera partir de ahí. El cliente depende de la interfaz; las implementaciones eligen si reutilizan la base abstracta. Lo mejor de ambos mundos.

La trampa a evitar: usar herencia abstracta solo para «ahorrar» unas líneas compartidas entre clases que en realidad no son la misma familia. Si el único motivo es reutilizar código, casi siempre es mejor componer (pasar una pieza) que heredar de una base abstracta forzada.

Resumen #

Lo que te llevas

  • Una clase abstracta trae código común y deja métodos abstractos como huecos obligatorios; no se instancia.
  • El template method fija el esqueleto del proceso en la base y deja que las subclases rellenen los pasos que varían.
  • Interfaz = solo contrato, varias a la vez; abstracta = contrato + código + estado, pero gasta tu única herencia.
  • Por defecto, empieza por la interfaz; sube a abstracta solo cuando haya implementación común real que compartir.
