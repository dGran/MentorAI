Los tipos de clases de PHP y cuándo usar cada uno

PHP moderno tiene un pequeño zoo de declaraciones: clase normal, abstract, final, interface, trait, enum, readonly y clases anónimas. Cada una resuelve un problema concreto. Esta lección las pone en fila y te da una tabla para elegir sin dudar.

Ya conoces las piezas grandes de las lecciones anteriores. Aquí las juntamos con las que faltan —final, trait, enum, readonly, anónimas— para tener el mapa completo de cómo se declara una unidad de comportamiento en PHP 8, y sobre todo cuándo tirar de cada una.

Clase normal y final #

La clase normal es el caso por defecto: se instancia, se puede heredar de ella. La novedad sana de PHP moderno es marcarla final cuando no esperas que nadie herede: cierra la puerta a que la extiendan y te deja cambiar sus tripas sin miedo a romper subclases ajenas. Encaja con la máxima de preferir composición a herencia.

final por defecto, herencia por decisión

Muchos equipos declaran sus clases final salvo que haya un motivo explícito para extenderlas. Heredar deja de ser el reflejo automático y pasa a ser una decisión consciente. Como dependes de interfaces para el polimorfismo, perder la herencia no duele.

abstract e interface #

Las dos protagonistas de las lecciones anteriores, en una frase cada una. La interfaz es puro contrato: qué métodos hay, cero implementación, varias por clase. La clase abstracta es contrato más código y estado común, pero gasta tu única herencia y no se instancia. Interfaz para desacoplar y dar roles; abstracta para compartir un esqueleto entre variantes de un mismo concepto.

Traits #

Un trait es un bloque de métodos (y propiedades) para copiar y pegar dentro de varias clases sin herencia. Resuelve la limitación de la herencia única de PHP: cuando un mismo comportamiento transversal hace falta en clases sin parentesco. Con use, la clase incorpora ese código como si lo hubiera escrito ella.

php · un trait reutilizado sin herencia 
<?php

trait RegistraEventos
{
    /** @var object[] */
    private array $eventos = [];

    protected function registrar(object $evento): void
    {
        $this->eventos[] = $evento;
    }

    public function eventos(): array
    {
        return $this->eventos;
    }
}

final class Pedido
{
    use RegistraEventos;
    // ...
}

Con moderación

El trait no es polimorfismo: no hay contrato, el tipo no «es» nada. Abusar de ellos esparce estado oculto y dificulta seguir de dónde sale un método. Úsalos para utilidades transversales claras; para colaboración entre objetos, casi siempre es mejor componer con una pieza inyectada.

Enums #

Los enum (PHP 8.1) son un tipo con un conjunto cerrado y fijo de valores posibles. Sustituyen a las constantes sueltas o a los strings mágicos («pending», «paid»…) por un tipo de verdad: el sistema de tipos garantiza que un EstadoPedido solo puede ser uno de los casos declarados, ni un typo más. Pueden tener métodos, lo que los hace ideales para encapsular las reglas de cada caso.

php · enum con valor y comportamiento 
<?php

declare(strict_types=1);

enum EstadoPedido: string
{
    case Pendiente = 'pending';
    case Pagado = 'paid';
    case Enviado = 'shipped';

    public function permiteCancelar(): bool
    {
        return $this === self::Pendiente;
    }
}

$estado = EstadoPedido::from('paid');   // EstadoPedido::Pagado

readonly y value objects #

Una propiedad readonly (PHP 8.1) solo se puede asignar una vez, normalmente en el constructor; después es inmutable. Es la herramienta perfecta para los value objects: objetos pequeños definidos por su valor (un Dinero, un Email) que, una vez creados, no cambian. PHP 8.2 permite marcar toda la clase como readonly de una vez.

php · value object inmutable 
<?php

declare(strict_types=1);

final readonly class Dinero
{
    public function __construct(
        public int $centimos,
        public string $moneda,
    ) {
    }

    public function sumar(Dinero $otro): self
    {
        return new self($this->centimos + $otro->centimos, $this->moneda);
    }
}

Como es inmutable, sumar() no muta: devuelve un Dinero nuevo. Aquí las propiedades públicas no rompen la encapsulación porque no hay estado que proteger: el objeto no cambia nunca. Es la excepción razonable a «propiedades siempre privadas» de la primera lección.

Clases anónimas #

Una clase anónima se define y se instancia en el sitio, sin nombre, con new class { ... }. Sirve para crear una implementación de usar y tirar de una interfaz, sobre todo en tests (un doble rápido) o para un objeto puntual que no merece su propio fichero.

php · doble de test al vuelo 
<?php

$relojFijo = new class implements Reloj {
    public function ahora(): DateTimeImmutable
    {
        return new DateTimeImmutable('2026-01-01 00:00:00');
    }
};

Tabla de decisión #

Necesitas… Usa
Un objeto normal que nadie debería extender final class
Un contrato puro para desacoplar / dar roles interface
Un esqueleto común con huecos para variantes abstract class
Reutilizar métodos transversales sin herencia trait (con moderación)
Un conjunto cerrado y fijo de valores enum
Un valor pequeño e inmutable readonly / value object
Una implementación puntual de usar y tirar clase anónima

El hilo común de todo el curso

Todas estas herramientas existen para lo mismo que viste en la primera lección: poner cada cosa tras una frontera clara. Con esta base sólida, el salto natural es SOLID, que son cinco principios para decidir cómo repartir responsabilidades y dependencias entre estas clases.

Resumen #

Lo que te llevas

  • final cierra la herencia (sana por defecto); interface y abstract dan contrato (con o sin código común).
  • trait reutiliza código transversal sin herencia, pero no es un contrato: úsalo con moderación.
  • enum da un conjunto cerrado de valores con tipo; readonly habilita value objects inmutables.
  • Las clases anónimas sirven para implementaciones puntuales; la tabla de decisión te da el atajo para elegir.
