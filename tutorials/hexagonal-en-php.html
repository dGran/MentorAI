Hexagonal en PHP

La teoría de puertos y adaptadores baja aquí a tierra: qué carpetas creas, qué va en cada capa, cómo un mismo puerto tiene un adaptador de MySQL para producción y uno en memoria para los tests, y cómo eso te deja probar el dominio sin levantar nada.

La arquitectura no se ve en un diagrama, se ve en el árbol de carpetas y en los use de cada fichero. Vamos a montar un caso de uso pequeño —registrar un usuario— respetando la regla de oro: las dependencias apuntan siempre hacia el dominio.

De la teoría a las carpetas #

La hexagonal se suele materializar en tres capas concéntricas: dominio en el centro, aplicación alrededor e infraestructura en el borde. No es la única forma de organizarla, pero es la más extendida en PHP y deja la regla de dependencias muy visible: nadie de dentro mira hacia fuera.

Estructura por capas #

árbol de carpetas (módulo User) 
src/User/
├── Domain/
│   ├── User.php                 # Entidad
│   ├── UserId.php               # Value object
│   ├── Email.php                # Value object
│   └── UserRepository.php       # PUERTO (interfaz)
├── Application/
│   └── RegisterUser/
│       ├── RegisterUser.php     # Caso de uso (puerto driving)
│       └── RegisterUserCommand.php
└── Infrastructure/
    ├── Persistence/
    │   ├── MySqlUserRepository.php      # ADAPTADOR driven
    │   └── InMemoryUserRepository.php   # ADAPTADOR para tests
    └── Http/
        └── RegisterUserController.php   # ADAPTADOR driving

Organiza por módulo, no por tipo

Fíjate en que la raíz es User/, no Controllers/ o Models/. Agrupa por concepto de negocio y, dentro, por capa. Así cada funcionalidad vive junta y se entiende sola, en vez de estar desperdigada por carpetas técnicas.

Qué va en cada capa #

Capa Contiene Puede importar de…
Domain Entidades, value objects, puertos, reglas de negocio. Nada externo. Solo de sí misma.
Application Casos de uso que orquestan el dominio. Domain.
Infrastructure Adaptadores: ORM, HTTP, colas, email. Domain y Application.

La columna de la derecha es la regla de oro hecha tabla: las flechas van hacia el centro. Domain no importa de nadie; Infrastructure importa de los dos de dentro, nunca al revés. Si te ves escribiendo un use de Infrastructure dentro de Domain, algo se torció.

Un puerto, dos adaptadores #

El caso de uso depende del puerto. En producción le inyectas el adaptador de MySQL; en los tests, el de memoria. La lógica del caso de uso es idéntica en ambos casos: no sabe ni le importa cuál recibió.

php · Application/RegisterUser/RegisterUser.php 
<?php

declare(strict_types=1);

final class RegisterUser
{
    public function __construct(
        private UserRepository $users
    ) {}

    public function __invoke(RegisterUserCommand $command): void
    {
        $email = new Email($command->email);

        if ($this->users->byEmail($email) !== null) {
            throw new EmailAlreadyInUse($email);
        }

        $this->users->save(
            User::register(UserId::next(), $email)
        );
    }
}

En el arranque de la aplicación (el cableado, donde vive la inyección de dependencias) decides qué adaptador entra:

php · cableado: producción vs test 
<?php

declare(strict_types=1);

// Producción: el puerto se resuelve al adaptador de MySQL.
$useCase = new RegisterUser(
    new MySqlUserRepository($pdo)
);

// Test: el mismo caso de uso, con el adaptador en memoria.
$useCase = new RegisterUser(
    new InMemoryUserRepository()
);

Testear el dominio aislado #

Aquí se cobra la inversión. Como el caso de uso solo conoce el puerto, el test usa el fake en memoria y corre en milisegundos, sin base de datos, sin contenedor, sin red:

php · test del caso de uso (sin infraestructura) 
<?php

declare(strict_types=1);

public function test_registra_un_usuario_nuevo(): void
{
    $users = new InMemoryUserRepository();
    $register = new RegisterUser($users);

    $register(new RegisterUserCommand('ana@example.com'));

    $this->assertNotNull($users->byEmail(new Email('ana@example.com')));
}

public function test_rechaza_un_email_repetido(): void
{
    $users = new InMemoryUserRepository();
    $register = new RegisterUser($users);
    $register(new RegisterUserCommand('ana@example.com'));

    $this->expectException(EmailAlreadyInUse::class);
    $register(new RegisterUserCommand('ana@example.com'));
}

El adaptador real también se prueba

El fake cubre la lógica del caso de uso, pero el MySqlUserRepository necesita su propio test de integración contra una base de datos real: que el SQL es válido y el mapeo cuadra. Cada cosa en su nivel de la pirámide.

Errores al aplicarla #

La entidad de dominio no es la tabla

El error más común es dejar que el ORM mande: anotar la entidad de dominio con mapeos de la base de datos. Eso vuelve a atar el centro a la infraestructura. El dominio debe ser PHP puro; el adaptador de persistencia es quien traduce entre el objeto y la tabla.

Otros tropiezos habituales:

  • Adaptadores anémicos que filtran tecnología: un puerto que devuelve un ResultSet del driver en vez de objetos de dominio. El puerto debe hablar el idioma del dominio.
  • Aplicar la artillería a un CRUD: tres capas y cuatro interfaces para guardar un formulario sin reglas. Reserva la hexagonal para módulos con negocio de verdad.
  • Lógica de negocio en el controlador: el adaptador driving solo traduce la petición a un comando y llama al caso de uso. Si hay reglas ahí, se han escapado del dominio.

Resumen #

Lo que te llevas

  • Tres capas: Domain (centro, sin dependencias), Application (casos de uso) e Infrastructure (adaptadores).
  • Organiza por módulo de negocio y, dentro, por capa. Las dependencias apuntan hacia dentro.
  • Un puerto, dos adaptadores (MySQL y en memoria): el dominio se testea aislado y rapidísimo.
  • La entidad de dominio no es la tabla; sin lógica en el controlador y sin matar moscas a cañonazos en un CRUD.
