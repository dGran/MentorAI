El contenedor de DI por dentro

Sabes inyectar dependencias por el constructor; pero ¿quién construye el árbol entero de objetos y los pasa en orden? El contenedor. Abrirlo por dentro —autowiring por reflexión, ciclo de vida, compilación, memoria— desmitifica la «magia» de Symfony y Laravel y te enseña a usarla con cabeza.

La inyección de dependencias resuelve un problema de diseño: que cada clase reciba sus colaboradores en vez de fabricarlos. Pero eso traslada el trabajo a alguien: si nadie fabrica nada, ¿quién arma el grafo de objetos con sus decenas de niveles? Ese alguien es el contenedor, y aquí lo abrimos.

Qué resuelve el contenedor #

Una clase de aplicación moderna depende de un servicio que depende de un repositorio que depende de una conexión que depende de una configuración. Construir eso a mano —new A(new B(new C(new D(...))))— en cada sitio es insostenible. El contenedor es un registro que sabe construir cualquier servicio bajo demanda, resolviendo sus dependencias recursivamente.

Lo configuras con definiciones de servicio: el mapa de qué hace falta para construir cada cosa. Puede ser explícito (esta interfaz se resuelve con esta clase; este parámetro vale esto) o, en gran parte, deducido automáticamente.

php · el contrato mínimo de un contenedor (PSR-11) 
<?php

declare(strict_types=1);

interface ContainerInterface
{
    public function get(string $id): mixed;     // dame el servicio id, construido

    public function has(string $id): bool;       // ¿sabes construir id?
}

Autowiring por reflexión #

La parte que parece magia. El autowiring es la capacidad del contenedor de averiguar solo qué necesita un constructor y resolverlo, sin que se lo digas. ¿Cómo? Con reflexión: PHP puede inspeccionar una clase en ejecución y leer los tipos de los parámetros de su constructor. Por cada parámetro con tipo de clase, el contenedor se llama a sí mismo para construir esa dependencia. Es recursión sobre el grafo de tipos.

php · un autowiring mínimo, didáctico 
<?php

declare(strict_types=1);

final class Container implements ContainerInterface
{
    /** @var array<string, object> */
    private array $resueltos = [];

    public function get(string $id): mixed
    {
        if (isset($this->resueltos[$id])) {
            return $this->resueltos[$id];
        }

        return $this->resueltos[$id] = $this->construir($id);
    }

    private function construir(string $clase): object
    {
        $reflexion = new \ReflectionClass($clase);
        $constructor = $reflexion->getConstructor();

        if ($constructor === null) {
            return new $clase();
        }

        $argumentos = [];
        foreach ($constructor->getParameters() as $parametro) {
            $tipo = $parametro->getType();
            $argumentos[] = $this->get($tipo->getName());   // recursión
        }

        return $reflexion->newInstanceArgs($argumentos);
    }

    public function has(string $id): bool
    {
        return class_exists($id);
    }
}

Eso es, en esencia, lo que hace un contenedor real: leer el constructor, resolver cada dependencia por su tipo y construir de abajo arriba. Lo de verdad añade lo que falta aquí: resolver interfaces a implementaciones, inyectar parámetros escalares (un DSN, una API key), manejar argumentos opcionales y, sobre todo, no morir ante un ciclo.

La reflexión no es gratis

Inspeccionar clases en ejecución tiene un coste. Hacerlo en cada petición, para todo el grafo de servicios, se nota. Es precisamente el motivo por el que existe la compilación del contenedor, que veremos más abajo: reflexionar una vez y guardar el resultado.

Detectar ciclos #

Si A necesita B y B necesita A, la resolución recursiva entra en bucle infinito hasta agotar la pila. Un contenedor serio lleva la cuenta de qué está construyendo ahora mismo: si vuelve a pedir algo que ya está en esa pila, lanza una excepción de dependencia circular en vez de colgarse.

php · marcar lo que está en construcción 
<?php

private array $construyendo = [];

private function construir(string $clase): object
{
    if (isset($this->construyendo[$clase])) {
        throw new DependenciaCircular(
            implode(' -> ', array_keys($this->construyendo)) . " -> {$clase}"
        );
    }

    $this->construyendo[$clase] = true;
    // ... reflexión y resolución recursiva ...
    unset($this->construyendo[$clase]);

    return $objeto;
}

Un ciclo casi siempre es un olor de diseño

Que A y B se necesiten mutuamente suele indicar que tienen una responsabilidad compartida mal repartida (un problema de SRP). Antes de buscar trucos para romper el ciclo, plantéate si falta una tercera clase que ambas usen, o si una dependencia debería ser un evento.

Ciclo de vida y lazy #

¿El contenedor crea un objeto nuevo cada vez que lo pides, o reutiliza el mismo? Es el ciclo de vida del servicio, y se elige por servicio:

Modo Comportamiento Para
Shared (singleton) Se construye una vez y se cachea; las siguientes llamadas devuelven la misma instancia. Servicios sin estado: la mayoría. Es el valor por defecto.
Factory (no compartido) Una instancia nueva en cada get(). Objetos con estado propio que no debe compartirse.

En el contenedor mínimo de arriba, la línea $this->resueltos[$id] = ... ya implementa el modo shared: se cachea la instancia. Un modo factory simplemente no cachearía.

El matiz fino son los lazy services. A veces un servicio es caro de construir (abre una conexión, carga un fichero) pero casi nunca se usa en una petición dada. La solución es inyectar un proxy: un objeto ligero que finge ser el servicio real y lo construye de verdad solo en la primera llamada a un método. Así el coste se paga únicamente si se usa.

1 Se inyecta un proxyligero, no construye nada aún
2 El proxy construye el realaquí se paga el coste
3 Servicio real respondelas siguientes ya van directas

Un lazy service difiere su coste de construcción hasta el primer uso real.

Compilado vs runtime #

Aquí está la diferencia entre un contenedor de juguete y uno de producción. Resolver todo por reflexión en cada petición es cómodo en desarrollo pero caro en producción. Por eso Symfony compila el contenedor: en el build, hace la reflexión una sola vez y genera código PHP plano —una clase enorme con un método por servicio que hace los new en el orden correcto, sin reflexión—. En cada request se ejecuta ese PHP directo, rapidísimo.

Runtime (reflexión cada vez)

  • Inspecciona constructores en cada petición.
  • Flexible: cambias config y va al vuelo.
  • Cómodo en desarrollo, caro en producción.

Compilado (PHP generado)

  • Reflexiona una vez en el build.
  • Genera un contenedor en PHP plano, cacheado.
  • Cada request ejecuta new directos.

Cada framework, su filosofía

Symfony compila el contenedor a PHP en el warmup de la caché: máximo rendimiento, configuración resuelta de antemano. Laravel resuelve más en runtime con su container, priorizando flexibilidad y expresividad. Dos puntos distintos del mismo compromiso entre velocidad y dinamismo.

Memoria y GC #

¿Cuánto vive un servicio shared? La respuesta depende por completo del modelo de ejecución, y aquí se conecta todo con cómo corre PHP.

En PHP-FPM, cada petición es un proceso (o un worker reiniciado por request): el contenedor y todos sus servicios se construyen, viven la petición y se tiran al final. La arena de memoria de Zend se libera entera de golpe, así que el ciclo de vida shared es «gratis»: no hay que preocuparse de fugas entre peticiones porque no hay entre; el siguiente request empieza con el contenedor limpio.

En workers long-running (Swoole, RoadRunner, Octane), el proceso persiste entre miles de peticiones. Ahí el contenedor y sus servicios shared sobreviven, y eso cambia las reglas:

Lo que en FPM era inofensivo, en long-running muerde

Un servicio shared que acumula estado (un array que va creciendo, una colección de eventos que no se vacía) es una fuga de memoria que crece petición a petición hasta tumbar el worker. Y un servicio con estado del usuario A puede filtrarlo al usuario B si se reutiliza sin resetear. En long-running, los servicios shared deben ser de verdad sin estado.

Sobre el recolector de ciclos: PHP libera la mayoría de objetos por conteo de referencias en cuanto dejan de usarse, pero los ciclos de referencias (el servicio apunta al contenedor y el contenedor al servicio) necesitan al recolector de ciclos para liberarse. En FPM da igual —todo muere al final del request—; en long-running, esos ciclos retienen memoria hasta que el GC pasa, y por eso frameworks como Octane reinician el worker cada N peticiones como red de seguridad.

El service locator #

Un último aviso. Es tentador inyectar el contenedor entero en una clase y pedirle servicios dentro ($this->container->get(Mailer::class)). Eso es el service locator, y es un antipatrón: la clase oculta sus dependencias reales (ya no las declara en el constructor), depende de todo el contenedor y se vuelve difícil de testear, porque hay que simular el contenedor completo.

Service locator

  • Recibe el contenedor y pide dentro.
  • Las dependencias quedan ocultas.
  • Acoplado a todo; test costoso.

Inyección por constructor

  • Recibe exactamente lo que usa.
  • Las dependencias son visibles en la firma.
  • Acoplado a lo justo; test trivial.

El contenedor se usa en la frontera, no en el dominio

La regla práctica: el contenedor solo aparece en el punto de entrada (el front controller, la configuración del framework), donde arma el grafo. Tu código de negocio nunca debería conocer el contenedor: recibe sus dependencias ya construidas y no sabe que existe. Eso mantiene el dominio limpio y testeable.

Resumen #

Lo que te llevas

  • El contenedor construye el grafo de objetos bajo demanda; el autowiring lo deduce por reflexión del constructor, recursivamente.
  • Detecta ciclos marcando lo que construye; el ciclo de vida es shared (cacheado, por defecto) o factory, con lazy + proxies para diferir lo caro.
  • Symfony compila el contenedor a PHP plano para no reflexionar en cada request; Laravel resuelve más en runtime.
  • En FPM los servicios mueren con el request (shared «gratis»); en long-running, el estado compartido es fuga y filtración: servicios sin estado y reinicio periódico. Y nunca uses el contenedor como service locator.
