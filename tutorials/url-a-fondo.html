Qué pasa al escribir una URL

Pulsas Enter y la página aparece en medio segundo. Entremedias han pasado cuatro etapas bien distintas: traducir el nombre a una dirección, abrir una conexión, cifrarla y, por fin, pedir el contenido. Es la pregunta de entrevista de toda la vida, y entenderla de verdad ordena media carrera.

Una URL es solo texto. Para convertir ese texto en una página hace falta encontrar la máquina correcta, hablar con ella por un canal fiable y seguro, y usar un idioma común para pedir el recurso. Cada etapa la resuelve una pieza distinta, y verlas en orden quita la magia.

El viaje en cuatro etapas #

Cuando escribes https://www.ejemplo.com/precios y pulsas Enter, el navegador recorre cuatro pasos antes de tener un solo byte de la página. Este es el mapa; el resto del tutorial es cada parada con detalle.

1 DNSwww.ejemplo.com → 93.184.216.34
2 TCPabrir conexión con esa IP, puerto 443
3 TLScifrar el canal y verificar el certificado
4 HTTPGET /precios → la respuesta con el HTML

DNS → TCP → TLS → HTTP. Cada capa se apoya en la anterior.

Anatomía de una URL #

Antes de viajar, el navegador descompone la URL en sus partes. Cada una le dice algo distinto: el protocolo a usar, con qué máquina hablar y qué pedirle.

ParteEjemploPara qué sirve
EsquemahttpsProtocolo: HTTP sobre TLS
Hostwww.ejemplo.comEl nombre a resolver por DNS
Puerto443 (implícito)La puerta en esa máquina (80 para http, 443 para https)
Ruta/preciosEl recurso concreto que pides
Query?orden=ascParámetros para el servidor
Fragmento#planesPosición dentro de la página; no se envía al servidor

Un detalle que sorprende: el fragmento (lo que va tras #) nunca sale del navegador. Es cosa del cliente: a qué sección saltar. El servidor no lo ve.

DNS: del nombre a la IP #

Las máquinas no se localizan por nombre sino por dirección IP. El DNS (Domain Name System) es la agenda de internet: traduce www.ejemplo.com a algo como 93.184.216.34. Sin esta traducción, el navegador no sabría a dónde conectarse.

La resolución va por niveles de caché: primero el propio navegador y el sistema operativo, luego el resolver de tu proveedor, y si nadie lo sabe, se pregunta a los servidores autoritativos del dominio. Como cambia poco, se cachea con un TTL; por eso un cambio de DNS tarda en propagarse. Puedes verlo a mano:

bash · terminal 
dig +short www.ejemplo.com

# 93.184.216.34

"No me funciona, pero hago ping y va"

Muchos fallos "de red" son en realidad de DNS: el nombre no resuelve. Si dig no devuelve IP pero la IP directa sí responde, el problema está en la resolución, no en la conexión.

TCP: abrir la conexión #

Con la IP en mano, el navegador abre una conexión TCP con esa máquina en el puerto correspondiente (443 para HTTPS). TCP es el protocolo que garantiza que los bytes llegan completos y en orden, aunque la red pierda paquetes por el camino.

Esa conexión no es gratis: requiere un saludo de ida y vuelta entre cliente y servidor —el three-way handshake— antes de poder enviar nada. Es una de las razones por las que la primera petición a un servidor lejano "tarda en arrancar". Cómo funciona ese handshake por dentro es el tema del tutorial de TCP/IP.

TLS: cifrar el canal #

Si la URL es https, sobre la conexión TCP se monta otra capa: TLS (lo que pone la "S" en HTTPS y el candado en el navegador). Antes de intercambiar datos, cliente y servidor hacen el handshake TLS, que resuelve dos cosas a la vez.

Primero, identidad: el servidor presenta un certificado firmado por una autoridad de confianza, y el navegador comprueba que de verdad es ejemplo.com y no un impostor. Segundo, secreto: ambos acuerdan una clave para cifrar todo lo que venga después, de modo que quien espíe la red solo vea ruido. A partir de aquí, el canal es privado y autenticado.

El candado no dice que el sitio sea legítimo

TLS garantiza que hablas con quien dice el certificado y que nadie espía, pero no que ese sitio sea de fiar. Un phishing también puede tener su candado. El candado es sobre el canal, no sobre las intenciones.

HTTP: pedir el contenido #

Recién ahora, sobre el canal seguro, viaja la petición HTTP: un mensaje de texto con un método (GET), la ruta (/precios) y unas cabeceras. El servidor responde con un código de estado y el cuerpo (el HTML). Puedes ver el viaje entero con curl:

bash · terminal 
curl -v https://www.ejemplo.com/precios

# * Resolviendo www.ejemplo.com (DNS)
# * Conectado a 93.184.216.34 puerto 443 (TCP)
# * Handshake TLS correcto, certificado verificado
# > GET /precios HTTP/2
# < HTTP/2 200

Ahí están las cuatro etapas en una sola orden: resolver, conectar, cifrar, pedir. El navegador hace exactamente esto, y luego lee el HTML, descubre que necesita CSS, imágenes y scripts, y repite el proceso para cada uno (a menudo reutilizando la conexión ya abierta). Qué dice esa petición y esa respuesta por dentro lo desgrana HTTP a fondo.

Errores comunes #

Confundir un fallo de DNS con uno de servidor. Si el nombre no resuelve, nunca llegas a conectar: el servidor puede estar perfecto. Mira primero si hay IP.

Creer que HTTPS protege el contenido en el servidor. TLS cifra los datos en tránsito, entre tu navegador y el servidor. Una vez allí, el servidor los ve en claro; protegerlos en reposo es otra cosa.

Esperar que un cambio de DNS sea instantáneo. El TTL hace que las IP viejas sigan cacheadas un tiempo. Tras mover un dominio, parte del mundo lo verá donde estaba hasta que caduque la caché.

Resumen #

Lo que te llevas

  • Una petición web son cuatro etapas en orden: DNS → TCP → TLS → HTTP, cada una apoyada en la anterior.
  • El DNS traduce el nombre a una IP y se cachea con un TTL; muchos fallos "de red" son en realidad de resolución.
  • TCP abre un canal fiable con un handshake; TLS lo cifra y verifica el certificado del servidor.
  • Solo sobre ese canal seguro viaja la petición HTTP que pide el recurso; curl -v te enseña las cuatro etapas de golpe.
