CQRS y Event Sourcing

¿Y si en vez de guardar el estado actual guardaras todo lo que pasó para llegar a él? Event sourcing convierte la historia en la fuente de verdad: el saldo no se almacena, se calcula sumando movimientos. Es poderoso, encaja con CQRS como un guante y no es para casi nadie.

La contabilidad lleva siglos haciendo event sourcing: un libro mayor no guarda el saldo, guarda cada apunte; el saldo es la suma. Nunca se borra un asiento, se añade uno que lo corrige. Event sourcing trae esa idea milenaria al software.

Guardar eventos, no estado #

En una aplicación normal guardas el estado actual: una fila con el saldo de la cuenta, y cada operación la sobrescribe. Con event sourcing no guardas el saldo: guardas la secuencia de eventos que ocurrieron («ingresados 100», «retirados 30»). El estado actual es el resultado de reproducirlos todos en orden.

Estado (lo habitual)

  • Guardas saldo = 70.
  • Cada cambio sobrescribe el valor.
  • El pasado se pierde: solo ves el ahora.
  • «¿Por qué está así?» no tiene respuesta.

Eventos (event sourcing)

  • Guardas +100, -30.
  • Cada cambio añade un evento; nada se sobrescribe.
  • La historia completa está ahí, auditable.
  • El estado es una proyección de los hechos.

El event store #

El event store es el almacén de eventos: una secuencia ordenada e inmutable, append-only. No se actualiza ni se borra; solo se añade al final. Cada evento pertenece a un stream (típicamente, uno por agregado) y lleva un número de versión para ordenarlos y detectar conflictos.

php · el stream de un agregado, solo añadir 
<?php

declare(strict_types=1);

interface EventStore
{
    // Añade eventos al final del stream del agregado.
    public function append(string $streamId, array $events): void;

    // Devuelve todos los eventos del agregado, en orden.
    public function load(string $streamId): array;
}

Reconstruir el estado #

Para operar sobre un agregado, lo rehidratas: cargas sus eventos y los aplicas uno a uno sobre un objeto vacío hasta llegar al estado actual. Cada evento sabe cómo modifica el estado. Una vez reconstruido, ejecutas el comando, que produce nuevos eventos que se añaden al store.

php · rehidratar reproduciendo los eventos 
<?php

declare(strict_types=1);

final class Account
{
    private int $balance = 0;

    public static function fromHistory(array $events): self
    {
        $account = new self();

        foreach ($events as $event) {
            $account->apply($event);
        }

        return $account;
    }

    private function apply(object $event): void
    {
        $this->balance += match (true) {
            $event instanceof MoneyDeposited => $event->amount,
            $event instanceof MoneyWithdrawn => -$event->amount,
            default => 0,
        };
    }
}

Snapshots para no releer la historia entera

Si un agregado acumula miles de eventos, rehidratarlo desde cero cada vez es caro. La solución son los snapshots: cada N eventos guardas una foto del estado y, al cargar, partes de la última foto y aplicas solo los eventos posteriores. Optimización, no parte esencial del modelo.

Proyecciones de lectura #

Reproducir eventos está muy bien para escribir, pero es pésimo para consultar: nadie quiere recorrer un millón de eventos para pintar una lista. Por eso el event store alimenta proyecciones: modelos de lectura que se construyen escuchando los eventos y se guardan en un formato cómodo de consultar (una tabla, un índice).

3 QueriesLa pantalla lee de la proyección, no del store

Una proyección es desechable: si la quieres distinta, la borras y la reconstruyes reproduciendo los eventos.

El superpoder: nuevas vistas del pasado

Como tienes todos los eventos, puedes crear una proyección nueva mañana y reconstruirla con datos de hace años. Una métrica que no se te ocurrió capturar entonces está, de hecho, en la historia. Con estado puro, ese pasado se habría perdido para siempre.

Por qué encaja con CQRS #

Event sourcing y CQRS son cosas distintas —puedes hacer CQRS sin event sourcing y viceversa— pero se atraen. El event store es un modelo de escritura perfecto (append-only, sin conflictos de actualización); las proyecciones son modelos de lectura naturales. La separación de CQRS deja de ser una decisión: viene impuesta por el diseño.

Y como las proyecciones se actualizan reaccionando a los eventos, heredas la consistencia eventual: entre que un comando añade un evento y la proyección lo refleja, hay un desfase. La misma tensión de siempre, ahora estructural.

Trade-offs reales #

No es una arquitectura por defecto

Event sourcing es de las decisiones más difíciles de revertir y de las que más complejidad añaden. Cambiar el formato de un evento pasado, depurar un flujo reconstruido, gestionar proyecciones que se desincronizan: todo cuesta. Adóptalo solo cuando la historia es un requisito de negocio, no por elegancia técnica.

Ganas Pagas
Auditoría total: cada cambio queda registrado. Versionado de eventos: el pasado es inmutable y hay que poder leerlo siempre.
Viajar en el tiempo: reconstruir el estado en cualquier instante. Consistencia eventual entre escritura y proyecciones.
Nuevas proyecciones sobre datos históricos. Curva de aprendizaje alta y herramientas menos maduras.
Encaje natural con CQRS y eventos de dominio. Más piezas móviles que depurar y operar.

¿Dónde compensa? Donde la historia es el negocio: banca, contabilidad, trazabilidad legal, dominios donde «cómo se llegó aquí» importa tanto como el estado actual. Para el resto, guardar el estado y, si acaso, un log de auditoría sencillo es muchísimo más barato. Con esto cierras el bloque de diseño y arquitectura: de la inyección de dependencias al event sourcing, siempre con la misma brújula —elegir la complejidad que el problema justifica, ni más ni menos.

Resumen #

Lo que te llevas

  • Event sourcing guarda la secuencia de eventos, no el estado. El estado se calcula reproduciéndolos.
  • El event store es append-only e inmutable; el agregado se rehidrata aplicando sus eventos (con snapshots si hace falta).
  • Las proyecciones son modelos de lectura desechables construidos desde los eventos. Encaja de forma natural con CQRS.
  • Da auditoría y viaje en el tiempo, pero cuesta caro y trae consistencia eventual. Úsalo solo donde la historia es el negocio.
