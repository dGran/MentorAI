Git: comandos del día a día

La chuleta para tener a mano. Aunque trabajes con Git desde el IDE, entender qué hace cada comando por debajo te saca de cualquier apuro y te quita el miedo a la terminal. Aquí están los del día a día, agrupados por tarea, cada uno con qué hace y un ejemplo real.

14 min de lectura Nivel principiante Git

Git es un sistema de control de versiones: guarda fotos (commits) de tu proyecto a lo largo del tiempo para que puedas ver qué cambió, volver atrás y trabajar en paralelo sin pisarte. El IDE te tapa los comandos, pero todos hacen lo mismo por dentro; conocerlos te da control cuando algo no sale como esperabas.

El modelo mental #

Antes de la lista, la idea que lo ordena todo: un cambio en Git viaja por tres zonas. Entender este recorrido hace que casi todos los comandos cobren sentido.

1 Working directorytus ficheros, donde editas
2 Staging arealo que entrará en el próximo commit
3 Repositorioel historial de commits, permanente

Editas en el working directory, eliges qué guardar con add (staging) y lo confirmas con commit (repositorio).

Configurar #

Lo primero al estrenar máquina: decirle a Git quién eres (va en cada commit que hagas).

ComandoQué hace
git config --global user.nameFija tu nombre para todos los repos.
git config --global user.emailFija tu email (el que asocia los commits a tu cuenta).
git config --listMuestra toda la configuración actual.
git config --global init.defaultBranch mainQue los repos nuevos arranquen en main.
bash · configuración inicial 
git config --global user.name "David Gran"
git config --global user.email "david@ejemplo.com"
git config --global init.defaultBranch main

Empezar un repo #

ComandoQué hace
git initConvierte la carpeta actual en un repo Git (crea .git/).
git cloneDescarga una copia completa de un repo remoto, con todo su historial.
bash · arrancar de cero o clonar 
# Empezar un proyecto nuevo
git init

# O traerte uno que ya existe
git clone https://github.com/dGran/MentorAI.git

Los cambios del día a día #

El ciclo que repites cien veces al día: ves qué has cambiado, eliges qué guardar y lo confirmas con un mensaje.

ComandoQué hace
git statusQué has cambiado y qué está en staging. El más usado.
git add archivo.phpPone un fichero en staging (lo marca para el próximo commit).
git add .Pone en staging todos los cambios del directorio.
git commit -m "mensaje"Confirma lo que hay en staging como una foto nueva del proyecto.
git commit -am "mensaje"Atajo: hace add de lo ya rastreado y commitea de un tirón.
git rm archivo.phpBorra el fichero y registra el borrado.
git mv viejo.php nuevo.phpRenombra o mueve un fichero rastreado.
bash · el ciclo básico 
# 1. ¿Qué he tocado?
git status

# 2. Elijo qué guardar
git add src/Factura.php

# 3. Lo confirmo con un mensaje que explique el porqué
git commit -m "Calcula el IVA en la factura"

El .gitignore te ahorra disgustos

Un fichero .gitignore en la raíz le dice a Git qué no rastrear: vendor/, .env (¡nunca subas secretos!), ficheros del IDE como .idea/. Lo que está en .gitignore no aparece en git status ni se puede añadir por error.

Inspeccionar #

ComandoQué hace
git logLista el historial de commits (autor, fecha, mensaje).
git log --onelineEl historial compacto, un commit por línea.
git log --oneline --graphIgual, pero dibujando las ramas y merges.
git diffQué has cambiado en el working directory y aún no está en staging.
git diff --stagedQué hay en staging a la espera de commit.
git show abc123Detalle de un commit concreto (qué cambió exactamente).
git blame archivo.phpQuién y en qué commit tocó cada línea de un fichero.
bash · mirar el historial y los cambios 
# El historial de un vistazo
git log --oneline --graph

# Qué he cambiado desde el último commit
git diff

# Quién escribió cada línea de un fichero
git blame src/Factura.php

Sincronizar con el remoto #

El remoto (normalmente origin, en GitHub o GitLab) es la copia compartida del repo. Estos comandos mueven commits entre tu copia local y la de arriba.

ComandoQué hace
git remote -vMuestra los remotos configurados y sus URLs.
git fetchTrae los commits nuevos del remoto, sin tocar tu trabajo.
git pullTrae los commits nuevos y los integra en tu rama (fetch + merge).
git pushSube tus commits locales al remoto.
git push -u origin mainLa primera vez: sube y enlaza tu rama con la del remoto.
bash · subir y bajar cambios 
# Bajar lo que hayan subido los demás antes de empezar
git pull

# ...trabajas, haces commits...

# Subir lo tuyo
git push

fetch vs pull, la duda eterna

fetch es «mírame qué hay nuevo arriba» sin tocar tu código; pull es «tráelo y mézclalo ya». Si quieres ver antes de integrar, fetch y luego decides. pull es el atajo cómodo del día a día cuando no esperas sorpresas.

Con esto cubres el 90% de tu actividad diaria. Lo que falta —moverte entre ramas, combinar trabajo y deshacer líos— son las dos lecciones siguientes, que ya entran en el «cómo se trabaja» más allá de los comandos sueltos.

Resumen #

Lo que te llevas

  • Un cambio viaja por tres zonas: working directory → staging (add) → repositorio (commit).
  • El ciclo diario es git status → git add → git commit -m, con .gitignore para lo que no se sube.
  • Inspeccionas con git log --oneline, git diff y git blame.
  • Con el remoto: fetch (solo mira), pull (trae e integra), push (sube lo tuyo).
