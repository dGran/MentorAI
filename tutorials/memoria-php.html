Cómo gestiona la memoria PHP

PHP casi nunca te obliga a pensar en la memoria: creas variables y desaparecen solas. Detrás hay un gestor propio, contadores de referencias y un recolector de ciclos, todo apoyado en un truco que lo hace barato: al acabar la petición, tira la memoria entera de golpe.

Antes de empezar: esto es la memoria en concreto dentro de PHP. Los fundamentos generales (stack, heap, memoria virtual, caché) están en el tutorial de memoria. Aquí bajamos al runtime: cómo el motor de PHP pide, reparte y libera esa memoria petición a petición.

La arena: memoria atada al request #

PHP no usa directamente el malloc del sistema para cada variable. Tiene su propio gestor, el Zend Memory Manager, que pide bloques grandes al sistema operativo y luego los reparte él en trocitos con funciones internas como emalloc y efree (la e es de engine).

La clave es que toda esa memoria es por petición: se asocia a un "pool" o arena que nace al empezar el request y se destruye al acabar. Por eso liberar es tan barato.

1 Empieza la peticiónSe abre la arena de memoria
2 Tu código pide memoriaemalloc reparte trozos de la arena
3 Termina la peticiónSe tira la arena ENTERA de golpe

No se liberan los objetos uno a uno al final: se descarta el bloque completo. Limpiar es casi gratis porque no importa qué había dentro.

Por qué PHP "perdona" las fugas

En un lenguaje long-running, olvidarte de liberar memoria es una fuga que crece sin parar. En PHP clásico, da igual: aunque te "olvides", la arena entera muere al acabar el request. Por eso se programa sin pensar en liberar… hasta que pasas a procesos de larga vida (Swoole, workers de cola), donde esa red de seguridad desaparece.

memory_limit: el tope por petición #

memory_limit es cuánta memoria puede consumir una sola petición antes de que PHP la mate con un error fatal. No es la memoria del servidor ni la del pool: es un tope por request, pensado para que un script desbocado no se coma la máquina.

ini · php.ini 
; Tope por petición (web). 128M suele bastar para una request normal
memory_limit = 128M

; En CLI a menudo se deja sin límite para comandos pesados
; memory_limit = -1

"Allowed memory size exhausted"

Casi siempre no es que falte RAM: es que cargaste demasiado en una petición. El clásico: SELECT de medio millón de filas a un array, o leer un fichero gigante de golpe. La solución no es subir memory_limit a ciegas, sino procesar en streaming (cursores, generadores, leer por trozos).

El zval: cómo se guarda un valor #

Cada valor en PHP no es "el número 5" pelado: vive en una estructura interna llamada zval (Zend value). Un zval guarda dos cosas: el tipo (entero, cadena, array, objeto…) y el valor en sí (o un puntero a él, si es grande como un array o un objeto).

Los tipos simples (int, bool, float) caben dentro del propio zval. Los tipos complejos (strings, arrays, objetos) guardan en el zval un puntero a una estructura aparte en el heap, y esa estructura es la que lleva un contador de referencias. Ese contador es la base de todo lo que viene.

Refcounting: contar quién apunta #

PHP libera memoria principalmente por conteo de referencias: cada valor compartible lleva la cuenta de cuántas variables lo están usando. Cuando una variable nueva apunta al valor, el contador sube; cuando una deja de hacerlo (se reasigna o sale de ámbito), baja. Al llegar a cero, ese valor se libera al instante, sin esperar a ningún recolector.

php · refcount.php 
<?php

declare(strict_types=1);

$first = "una cadena larga compartida";
$second = $first;   // ambos apuntan al mismo valor: refcount = 2

unset($second);     // refcount = 1
unset($first);      // refcount = 0: se libera de inmediato

Verlo con tus ojos

debug_zval_refcount() ya no existe, pero xdebug_debug_zval('var') (con Xdebug) te muestra el refcount y si el valor está marcado como is_ref. Es la forma más didáctica de ver subir y bajar el contador.

Copy-on-write: copiar solo cuando hace falta #

Si asignar $b = $a con un array de un millón de elementos lo duplicara en memoria al instante, PHP sería carísimo. No lo hace: aplica copy-on-write (COW). Al asignar, las dos variables comparten el mismo valor y suben el refcount. La copia real solo ocurre cuando una de las dos intenta modificarlo.

1 $b = $aArray de 1M de elementos
2 $a y $b apuntan al mismo arrayrefcount = 2, cero bytes extra
3 Ahora SÍ copia$b se separa con su propia versión

Mientras solo lees, compartir es gratis. La copia (cara) se pospone hasta el primer cambio, y solo entonces.

El mito del paso por referencia "para optimizar"

Pasar un array grande a una función que solo lo lee no lo copia: COW ya lo comparte. Poner & "para no copiar" no aporta nada y puede empeorar las cosas, porque marcar un valor como referencia desactiva justamente la magia del COW. Pasa por valor y deja que el motor optimice.

El recolector de ciclos #

El refcounting tiene un punto ciego: los ciclos. Si el objeto A guarda una referencia a B y B guarda otra a A, sus contadores nunca llegan a cero aunque nadie más los use. Se quedarían colgados en memoria hasta el final del request. Para eso existe el recolector de ciclos (cycle collector), un complemento del refcount, no un sustituto.

php · ciclo.php 
<?php

declare(strict_types=1);

class Node
{
    public ?Node $partner = null;
}

$a = new Node();
$b = new Node();
$a->partner = $b;   // A apunta a B
$b->partner = $a;   // B apunta a A: ciclo

unset($a, $b);      // refcount NO llega a 0: se apuntan entre ellos
// El recolector de ciclos es quien acabará liberándolos

El recolector se dispara solo cuando se acumulan suficientes "candidatos a ciclo" (un buffer interno con un umbral). También puedes forzarlo con gc_collect_cycles(). En código web normal casi nunca te preocupa; en procesos de larga vida sí importa, porque ahí los ciclos no se limpian "gratis" al morir el request.

Refcount + GC, no uno u otro

Es un error pensar que PHP "tiene garbage collector como Java". El 90% del trabajo lo hace el refcount, que libera al instante. El recolector de ciclos solo se ocupa del caso concreto de las referencias circulares. Son dos mecanismos que colaboran.

Qué hacer en la práctica #

Toda esta teoría se traduce en unos pocos hábitos concretos cuando tienes un problema de memoria:

php · medir.php 
<?php

declare(strict_types=1);

// Memoria usada ahora mismo y pico máximo del request
echo memory_get_usage(true), "\n";
echo memory_get_peak_usage(true), "\n";

// Procesar muchas filas SIN cargarlas todas: un generador
function readRows(\PDOStatement $stmt): \Generator
{
    while ($row = $stmt->fetch()) {
        yield $row;
    }
}
  • Mide con memory_get_peak_usage(): el pico es lo que cuenta para memory_limit, no el momento final.
  • Streaming en vez de cargar de golpe: generadores (yield), cursores de BBDD, leer ficheros por trozos.
  • Suelta lo grande cuanto antes: reasignar o unset() un array enorme baja su refcount a cero y lo libera ya.
  • En long-running: rompe ciclos manualmente o llama a gc_collect_cycles() entre tandas; ahí la arena no te salva.

Ese último punto enlaza con el modelo de ejecución: un worker de cola o un proceso Swoole no muere por petición, así que la memoria se acumula. Es justo el tema del siguiente tutorial, workers y modelos de ejecución.

Resumen #

Lo que te llevas

  • El gestor de Zend reparte memoria por arena de request y la libera entera al acabar: limpiar es casi gratis.
  • memory_limit es el tope por petición; "memory exhausted" suele ser cargar demasiado de golpe, no falta de RAM.
  • Los valores viven en zvals con refcount: al llegar a cero referencias, se liberan al instante.
  • Copy-on-write: asignar comparte, copiar solo al modificar. No fuerces & "para optimizar".
  • El recolector de ciclos complementa al refcount para las referencias circulares; importa sobre todo en procesos de larga vida.
