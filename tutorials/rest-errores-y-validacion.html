Errores y validación en una API

El día feliz de una API es el fácil; lo que la hace usable de verdad es cómo trata el error. Un formato de error consistente, una validación que explica qué falló y reintentos que no duplican nada: eso es lo que separa una API que da gusto integrar de una que da miedo.

Una API se integra leyendo sus errores tanto como sus éxitos. Si cada endpoint falla a su manera, quien te consume escribe un parser distinto por caso y acaba odiándote. La meta de esta lección es que todos tus errores tengan la misma forma, digan lo justo y permitan reintentar sin miedo.

Un solo formato de error #

La decisión más rentable sobre errores es la más simple: que todos tengan la misma estructura. Un cliente que aprende a leer un error de tu API ya sabe leerlos todos. Da igual la forma exacta que elijas mientras sea una sola y la respete cada endpoint.

Errores que cambian de forma

  • Un endpoint: {"error":"texto"}
  • Otro: {"mensaje":"...","ok":false}
  • Otro: un string pelado, sin JSON.
  • El cliente parsea cada caso a mano.

Un contrato de error

  • Misma estructura en toda la API.
  • Un campo legible para humanos y uno estable para máquinas.
  • El código de estado HTTP correcto, además del cuerpo.
  • El cliente escribe el manejo una vez.

Fíjate en el último punto de la izquierda: el cuerpo del error acompaña al código de estado, no lo sustituye. El 422 o el 409 ya dijimos que es la señal (lo vimos en verbos y códigos de estado); el cuerpo solo añade el detalle.

problem+json, el estándar #

No hace falta inventar el formato: existe un estándar, Problem Details for HTTP APIs (RFC 9457, antes 7807). Define un cuerpo JSON con un Content-Type propio, application/problem+json, y unos campos acordados:

CampoPara qué
typeURI que identifica el tipo de problema. Estable, pensado para máquinas.
titleResumen legible y corto del problema. No cambia entre ocurrencias.
statusEl código de estado HTTP, repetido en el cuerpo por comodidad.
detailExplicación concreta de esta ocurrencia, para un humano.
instanceURI de la ocurrencia concreta (útil para correlacionar con logs).

Y puedes añadir campos propios. Para validación, lo habitual es sumar un array errors con el detalle por campo:

http · un error de validación en problem+json 
HTTP/1.1 422 Unprocessable Entity
Content-Type: application/problem+json

{
  "type": "https://api.ejemplo.com/errores/validacion",
  "title": "La petición no es válida",
  "status": 422,
  "detail": "Revisa los campos marcados.",
  "errors": [
    { "campo": "email",    "mensaje": "Formato de email no válido" },
    { "campo": "cantidad", "mensaje": "Debe ser mayor que 0" }
  ]
}

Que el type sea una URI no es capricho: es un identificador global y estable que el cliente puede comparar (if type == ".../validacion") sin parsear textos que cambian de idioma o de redacción. El title es para que lo lea una persona; el type es para que lo entienda el código.

Validar la entrada #

Toda entrada que cruza la frontera de tu API es sospechosa hasta que la validas. La regla mental: nunca confíes en el cliente, ni siquiera en tu propio frontend. La validación del navegador es comodidad para el usuario; la del servidor es la que de verdad protege tus datos, porque cualquiera puede saltarse la del navegador y llamar a la API directamente.

Valida en el borde, antes de tocar la lógica de negocio: tipos correctos, campos obligatorios presentes, rangos y formatos, y reglas de negocio (un email no repetido, un stock suficiente). Y distingue dos códigos que se confunden:

SituaciónCódigo
El JSON está roto, falta un campo obligatorio, un tipo no cuadra400 Bad Request
El JSON es correcto pero los valores no pasan las reglas (email inválido, cantidad negativa)422 Unprocessable Entity
Choca con el estado actual (email ya registrado, versión obsoleta)409 Conflict

Validar también es defenderse

La validación de entrada es la primera línea contra inyecciones y datos corruptos. No sustituye a las consultas parametrizadas ni al escapado, pero recorta la superficie de ataque. Es una pieza del cuadro completo de seguridad web.

Devuelve todos los fallos #

Un detalle pequeño con un impacto enorme en la experiencia: cuando validas un formulario con diez campos y fallan tres, devuélvelos los tres a la vez, no el primero y paras. Si fallas de uno en uno, obligas al cliente a un baile de prueba-error: corrige uno, reenvía, descubre el siguiente, corrige, reenvía…

Falla al primer error

  • {"error":"email inválido"}
  • El usuario lo corrige y reenvía.
  • Ahora: "cantidad inválida". Y otra vuelta.
  • Tres envíos para tres errores.

Acumula y devuelve todo

  • El array errors con los tres campos.
  • El usuario los corrige de una.
  • Un solo reenvío.
  • El formulario puede marcar cada campo en rojo.

Qué no contar en un error #

Un error generoso de más es un agujero de seguridad. Lo que mandas en un 5xx lo puede leer un atacante igual que un cliente legítimo, así que cuida el equilibrio entre "útil para depurar" y "regalo para quien ataca".

Nunca filtres el interior por el cuerpo del error

Fuera de la respuesta: stack traces, nombres de tablas y columnas, consultas SQL, rutas del sistema de ficheros, versiones exactas del software. Todo eso es un mapa para un atacante. El cliente recibe un mensaje genérico y un identificador de error; el detalle técnico va a tus logs del servidor, correlacionado por ese identificador.

Hay un matiz fino en los 404 frente a los 403: a veces revelar que un recurso existe pero no tienes permiso (403) ya filtra información. Para recursos sensibles, responder 404 ("aquí no hay nada") puede ser más prudente que 403 ("existe, pero no es para ti"). Es una decisión de diseño, no una regla fija.

Reintentos que no duplican #

La red falla. Un cliente manda POST /pagos, el pago se procesa, pero la respuesta se pierde por el camino. El cliente no sabe si funcionó, así que reintenta… y cobra dos veces. Como vimos, POST no es idempotente, y aquí eso duele de verdad.

La solución acordada es la clave de idempotencia: el cliente genera un identificador único para la operación y lo manda en una cabecera. El servidor recuerda esa clave; si llega repetida, en vez de procesar otra vez devuelve el resultado de la primera.

http · el reintento llega con la misma clave 
POST /pagos HTTP/1.1
Idempotency-Key: 8f3a1c7e-2b4d-49a0-9e21-7c6f0b2d5a11
Content-Type: application/json

{ "pedido": 1001, "importe": 49.90 }
1 Llega una clave¿La he visto antes?
2 Proceso y guardoAsocio el resultado a la clave
3 Devuelvo lo guardadoSin volver a cobrar

El reintento es seguro: la segunda petición con la misma clave no ejecuta de nuevo.

Esto convierte un POST peligroso en un reintento seguro sin cambiar su semántica. El mecanismo completo —cuánto guardar la clave, qué pasa si la operación aún está en curso, cómo lo hacen Stripe y compañía— está en el tutorial dedicado: Idempotencia.

Resumen #

Lo que te llevas

  • Un único formato de error en toda la API: el cliente aprende a leerlo una vez y vale para todos los endpoints.
  • No inventes: application/problem+json (RFC 9457) ya define los campos. Un type estable para máquinas, un title para humanos.
  • Valida en el servidor siempre; nunca confíes en el cliente. 400 para malformado, 422 para semántica inválida, 409 para conflicto.
  • Devuelve todos los errores de validación a la vez, no de uno en uno. Y no filtres stack traces ni internals en el cuerpo.
  • Una clave de idempotencia (Idempotency-Key) convierte un POST en un reintento seguro: la repetición no vuelve a ejecutar.
