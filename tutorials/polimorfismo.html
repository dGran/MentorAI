Polimorfismo: un mensaje, muchas respuestas

Polimorfismo es la palabra que asusta y la idea que libera: llamas al mismo método sobre objetos distintos y cada uno responde a su manera, sin que tú tengas que preguntar de qué tipo es. Es lo que convierte una cascada de if por tipo en una sola línea.

«Polimorfismo» viene del griego: muchas formas. En código significa que una misma llamada —$forma->area(), $pago->cobrar()— toma la forma que corresponde al objeto que la recibe. El que llama no sabe ni le importa el tipo concreto: confía en que el objeto sabe responder. Esa indiferencia es justo lo que da flexibilidad.

La idea en una frase #

Mismo mensaje, distinta respuesta según quién lo recibe. Si tienes un Circulo y un Rectangulo, ambos entienden el mensaje area(), pero cada uno lo calcula a su manera. El código que recorre formas y suma áreas no necesita saber qué forma es cada una: le basta con que todas respondan a area().

El problema: condicionales por tipo #

Sin polimorfismo, el código que trata con varios tipos acaba lleno de ramas que preguntan «¿qué eres?». Cada tipo nuevo obliga a abrir todos esos switch y añadir un caso. Es frágil y se repite por todo el programa.

php · sin polimorfismo: el switch que crece 
<?php

function area(object $forma): float
{
    return match (true) {
        $forma instanceof Circulo => 3.14159 * $forma->radio ** 2,
        $forma instanceof Rectangulo => $forma->ancho * $forma->alto,
        // cada forma nueva = otra rama aquí, y en cada función parecida
    };
}

Polimorfismo por interfaces #

La solución es darle a cada forma la responsabilidad de su propio cálculo y declarar el mensaje común en un contrato. Lo idiomático es una interfaz (también vale una clase abstracta): cada clase la implementa a su manera, y el código cliente habla con el contrato.

php · cada forma sabe su área 
<?php

declare(strict_types=1);

interface Forma
{
    public function area(): float;
}

final class Circulo implements Forma
{
    public function __construct(private float $radio)
    {
    }

    public function area(): float
    {
        return 3.14159 * $this->radio ** 2;
    }
}

final class Rectangulo implements Forma
{
    public function __construct(
        private float $ancho,
        private float $alto,
    ) {
    }

    public function area(): float
    {
        return $this->ancho * $this->alto;
    }
}

Ahora el código que suma áreas no tiene ni un if por tipo:

php · una línea, cualquier forma 
<?php

/** @param Forma[] $formas */
function areaTotal(array $formas): float
{
    return array_sum(array_map(
        fn (Forma $forma) => $forma->area(),
        $formas
    ));
}

echo areaTotal([new Circulo(2), new Rectangulo(3, 4)]);

Añadir un tipo ya no toca el cliente

Para soportar un Triangulo, escribes una clase nueva que implementa Forma y ya. areaTotal() no cambia. Esto es exactamente el principio abierto/cerrado: abierto a extensión (formas nuevas), cerrado a modificación (el cliente intacto).

Type hints como contrato #

La pieza que hace funcionar todo esto es el type hint: cuando declaras Forma $forma, le dices a PHP «aquí cabe cualquier cosa que cumpla el contrato Forma». No pides una clase concreta, pides una capacidad. El código depende de lo que el objeto sabe hacer, no de lo que es. Programar contra ese contrato en vez de contra clases concretas es la base de la inversión de dependencias.

Sustituibilidad #

El polimorfismo solo funciona si los objetos son de verdad intercambiables: cualquier Forma debe poder usarse donde se espera una Forma sin sorpresas. Si una implementación devolviera un área negativa o lanzara una excepción donde las demás devuelven un número, el código cliente —que confiaba en el contrato— se rompería.

Puente con Liskov

Esa exigencia de «sustituibilidad real» tiene nombre propio: el principio de sustitución de Liskov, la L de SOLID. El polimorfismo es la herramienta; Liskov es la regla que evita que una implementación traicione el contrato que las demás respetan.

Resumen #

Lo que te llevas

  • Polimorfismo = mismo mensaje, distinta respuesta según el objeto que lo recibe.
  • Sustituye las cascadas de if/switch por tipo por una llamada al contrato común.
  • El type hint al contrato (interfaz o clase base) hace que el cliente dependa de la capacidad, no del tipo concreto.
  • Exige sustituibilidad: toda implementación debe respetar el contrato (puente al principio de Liskov).
