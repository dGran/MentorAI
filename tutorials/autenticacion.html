Autenticación: sesiones, JWT y OAuth

HTTP no recuerda quién eres entre una petición y la siguiente. Todo el mundo de las sesiones, los tokens y «entrar con Google» existe para resolver ese olvido. Veamos qué problema ataca cada mecanismo, en qué se diferencian de verdad y cuál encaja en tu caso.

Entras a un hotel: en recepción enseñas el DNI una vez (autenticación) y te dan una tarjeta que abre tu habitación, pero no las demás (autorización). No enseñas el DNI en cada puerta: basta la tarjeta. Sesiones, JWT y OAuth son distintas formas de diseñar esa tarjeta y lo que abre.

Autenticar no es autorizar #

Dos palabras que se confunden y conviene separar desde el principio:

Autenticación (authN)

  • ¿Quién eres?
  • Demostrar identidad (contraseña, huella, token)
  • Ocurre una vez, al entrar

Autorización (authZ)

  • ¿Qué puedes hacer?
  • Permisos sobre recursos (roles, scopes)
  • Se comprueba en cada acción

Este tutorial trata sobre todo la primera mitad —cómo el servidor sabe que sigues siendo tú— y cómo ese mecanismo carga también los datos que luego usará la autorización.

El problema: HTTP no recuerda #

Como viste en HTTP a fondo, el protocolo es sin estado: cada petición es independiente y el servidor no recuerda la anterior. Si te autenticas en /login, la siguiente petición a /perfil llega «en blanco»: nada dice que eres tú.

La solución, en todos los esquemas, es la misma idea: tras autenticarte, el servidor te entrega una credencial que el navegador reenvía en cada petición siguiente. Lo que cambia es qué es esa credencial y dónde vive la información de quién eres.

Sesiones con cookie #

El modelo clásico. Al autenticarte, el servidor crea una sesión: guarda en su lado (memoria, Redis, BBDD) los datos de quién eres, bajo un identificador aleatorio. Ese ID viaja al navegador en una cookie, y se reenvía solo en cada petición.

1 Login correctoEl servidor crea la sesión y guarda los datos
★ Cookie con el IDEl navegador la guarda y la reenvía sola
2 El servidor busca la sesiónPor el ID recupera quién eres

La cookie solo lleva un identificador opaco; los datos reales viven en el servidor. El estado es suyo.

Las cookies, bien puestas

Una cookie de sesión debe ir con HttpOnly (que JavaScript no la pueda leer, frena el robo por XSS), Secure (solo por HTTPS) y SameSite (que no se mande desde otros sitios, frena el CSRF). Tres atributos que cierran la mayoría de los ataques contra la cookie.

JWT: el token autocontenido #

Un JWT (JSON Web Token) le da la vuelta a la idea: en lugar de guardar el estado en el servidor, lo mete dentro del propio token, firmado. El token lleva los datos (quién eres, tus roles, cuándo caduca) y una firma que demuestra que no se han manipulado. Son tres partes separadas por puntos:

bash · estructura de un JWT 
# header . payload . firma   (cada parte en Base64Url)
eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiI0MiIsInJvbGUiOiJhZG1pbiJ9.k3nF...

# header  -> { "alg": "HS256", "typ": "JWT" }
# payload -> { "sub": "42", "role": "admin", "exp": 1735689600 }
# firma   -> HMAC del header+payload con una clave secreta del servidor

Firmado no es cifrado

El payload de un JWT va en Base64, no cifrado: cualquiera puede leerlo. La firma solo garantiza que nadie lo ha modificado, no que sea secreto. Nunca metas datos sensibles (contraseñas, datos personales) en un JWT.

La gran ventaja: el servidor no necesita guardar nada ni consultar una base de datos para validarte. Le basta verificar la firma con su clave. Eso lo hace muy cómodo para APIs y para repartir la carga entre muchos servidores sin un almacén de sesiones compartido.

Sesión vs JWT: el verdadero trade-off #

La diferencia de fondo es dónde vive el estado, y de ahí sale todo lo demás, incluida la pega más importante del JWT:

Sesión (servidor) JWT (cliente)
Dónde está el estado En el servidor; la cookie solo lleva un ID Dentro del token, en el cliente
Validar la petición Buscar la sesión (memoria/Redis/BBDD) Verificar una firma, sin tocar BBDD
Revocar acceso (logout, ban) Inmediato: borras la sesión Difícil: el token vale hasta que caduca
Escala horizontal Requiere almacén de sesiones compartido Sin estado compartido: cómodo

El talón de Aquiles del JWT: la revocación

Como el servidor no guarda estado, no puede «cancelar» un JWT emitido: si caduca en una hora, una hora seguirá siendo válido aunque eches al usuario. El patrón habitual lo mitiga: un access token de vida corta (minutos) + un refresh token de vida larga que sí se guarda y se puede revocar. Más piezas, más complejidad.

OAuth 2 y OpenID Connect #

Aquí hay otra confusión muy común. OAuth 2 no es para hacer login: es un protocolo de autorización delegada. Resuelve un problema concreto: «dejar que una app acceda a una parte de tus datos en otro servicio sin darle tu contraseña».

El ejemplo clásico: una app de calendario que quiere leer tus contactos de Google. En vez de pedirte tu contraseña de Google, te redirige a Google, tú apruebas allí, y Google le entrega a la app un token de acceso con permisos limitados (los scopes: «leer contactos», nada más).

1 La app te redirige a GooglePide permiso para ciertos scopes
2 Apruebas en GoogleTu contraseña no sale de Google
3 Google da un token a la appAcceso limitado a lo que aprobaste

La app nunca ve tu contraseña; solo recibe un token acotado que Google puede revocar.

"Entrar con Google" es OpenID Connect

El botón de login social no es OAuth a secas, sino OpenID Connect: una capa fina sobre OAuth 2 que añade lo que falta para autenticar (un id_token, que es un JWT con tu identidad). OAuth te da acceso a recursos; OIDC, encima, te dice quién es el usuario.

Cómo elegir #

Sesiones

  • App web clásica con servidor propio
  • Necesitas logout y revocación inmediatos
  • La opción simple y segura por defecto

JWT

  • API consumida por móvil o SPA
  • Varios servicios sin sesión compartida
  • Asumiendo el coste de la revocación

Usa OAuth/OIDC para identidad de terceros

Si quieres «entrar con Google/GitHub» o que otras apps accedan a tu API en nombre del usuario, ahí van OAuth 2 + OpenID Connect. Para tu propio login interno, casi siempre basta con sesiones o JWT. Y en cualquier caso: no implementes el protocolo a mano, usa una librería probada.

Resumen #

Lo que te llevas

  • Autenticar (¿quién eres?) no es autorizar (¿qué puedes hacer?). Todo nace de que HTTP es sin estado.
  • Sesión: el estado vive en el servidor, la cookie solo lleva un ID. Revocación inmediata; requiere almacén compartido para escalar.
  • JWT: el estado va firmado en el token. Sin tocar BBDD y fácil de escalar, pero difícil de revocar (de ahí access + refresh tokens). Firmado ≠ cifrado.
  • OAuth 2 es autorización delegada (acceso sin compartir contraseña); OpenID Connect añade encima la identidad: es el «entrar con Google».
  • Web clásica → sesiones; API/SPA/móvil → JWT; identidad de terceros → OAuth/OIDC. Con librerías probadas, nunca a mano.
