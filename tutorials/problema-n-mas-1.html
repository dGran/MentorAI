El problema N+1

Una página que iba fina con datos de prueba se arrastra en producción, y nadie tocó el código. Casi siempre es lo mismo: un bucle inocente que, por debajo, dispara una consulta por cada fila. El N+1 es el cuello de botella más común del acceso a datos, y el más fácil de no ver.

El N+1 no es un bug que el ORM tenga: es una trampa que te tiende su comodidad. Como cargar una relación parece gratis —es solo acceder a una propiedad—, es facilísimo escribir un bucle que, sin que se vea, habla con la base de datos en cada vuelta. Reconocerlo y desactivarlo es una habilidad básica del backend.

Qué es el N+1 #

El nombre lo dice todo: haces 1 consulta para traer una lista de N elementos, y luego N consultas más, una por cada elemento, para traer un dato relacionado. Total: N+1 viajes a la base de datos donde bastaban 2 (o incluso 1).

El ejemplo canónico: listar 50 pedidos y mostrar el nombre del cliente de cada uno.

php · el bucle inocente que esconde 51 consultas 
<?php

$pedidos = $repositorioPedidos->findBy(['estado' => 'enviado']);

foreach ($pedidos as $pedido) {
    echo $pedido->getCliente()->getNombre();
}

Se lee como código limpio y trivial. Pero $pedido->getCliente() no tenía el cliente cargado: al pedirlo, el ORM va a la base de datos a buscarlo. La primera consulta trajo los 50 pedidos; el bucle dispara 50 consultas más, una por cliente. 51 en total para algo que una sola consulta con JOIN resolvería.

Por qué lo provoca el ORM #

La causa tiene nombre: carga perezosa (lazy loading). Por defecto, el ORM no trae las relaciones hasta que las pides; así no malgasta cargando datos que quizá no uses. Es una decisión sensata… que se vuelve una trampa dentro de un bucle.

1 SELECT de los 50 pedidos1 consulta. Los clientes aún no están.
N getCliente() en cada vueltaEl lazy loading dispara un SELECT por pedido
51 51 idas y vueltasDonde 1 JOIN bastaba

El lazy loading convierte un acceso a propiedad en una consulta. Multiplicado por el bucle, ahí está el N+1.

La razón de que sea tan invisible es que el código no parece hablar con la base de datos: no hay ningún SELECT a la vista, solo un getCliente(). La consulta está escondida detrás del patrón Active Record o del proxy del Data Mapper. Lo que en SQL plano verías clarísimo (un SELECT dentro de un foreach), el ORM te lo oculta.

Por qué duele tanto #

Cada consulta tiene un coste fijo que no se ve en una sola: ida y vuelta por la red al servidor de BD, planificación, latencia. Da igual que cada una sea rapidísima; lo que mata es multiplicar ese coste fijo por N. Es exactamente el O(n) escondido del que habla Big-O: una operación dentro de un bucle que parecía O(1) y no lo era.

Por eso aparece "de repente" en producción

Con 10 filas de prueba en local, 11 consultas pasan desapercibidas. Con 10 000 filas reales y latencia de red de verdad, son 10 001 consultas y la página se cae. El N+1 escala con tus datos: no lo ves nacer, lo sufres cuando creces.

Cómo detectarlo #

No se caza leyendo el código —recuerda, las consultas son invisibles—, sino contando consultas. Las herramientas:

HerramientaCómo ayuda
Debug bar / profilerEl Symfony Profiler o Laravel Debugbar muestran cuántas consultas lanzó la página. Ver "212 queries" en una lista es la señal.
Query log del ORMActivar el log de SQL en desarrollo y ver la misma consulta repetida con distinto id es el N+1 en flagrante.
Log de consultas lentasEl slow query log de la BD no cuenta el N+1 (cada query es rápida), pero el número total de conexiones sí delata el patrón.
Tests / assertsAlgunos proyectos afirman un máximo de consultas por endpoint en los tests, y fallan si un cambio introduce un N+1.

La regla mental: si el número de consultas crece con el número de filas, tienes un N+1. Una lista bien hecha hace un número constante de consultas tanto para 10 elementos como para 10 000.

La solución: eager loading #

La cura es la carga ansiosa (eager loading): decirle al ORM, por adelantado, que vas a necesitar la relación, para que la traiga toda de golpe en lugar de fila a fila. Por debajo lo resuelve con un JOIN o con una segunda consulta WHERE id IN (...): dos consultas en total, no N+1.

php · pedir la relación por adelantado 
<?php

$pedidos = Pedido::with('cliente')
    ->where('estado', 'enviado')
    ->get();

foreach ($pedidos as $pedido) {
    echo $pedido->getCliente()->getNombre();
}

El with('cliente') (su equivalente en Doctrine es un JOIN explícito en el DQL o un fetch: EAGER) cambia 51 consultas por 2. El bucle es idéntico; lo único que cambió es que los clientes ya venían cargados. Esa es la esencia: el N+1 no se arregla tocando el bucle, sino diciendo de antemano qué vas a necesitar.

sql · lo que ejecuta por debajo: 2 consultas, no 51 
SELECT * FROM pedidos WHERE estado = 'enviado';
SELECT * FROM clientes WHERE id IN (12, 12, 7, 33, ...);

El IN(...) se apoya en un índice

Esa segunda consulta busca muchos clientes por su id de golpe; que sea rápida depende de que la columna esté indexada (la primaria lo está). Es otra vez la conexión con índices y B-tree: el eager loading evita el N+1, y el índice hace que la consulta de relleno vuele.

Sin pasarse al otro lado #

Cuidado con la sobrecorrección. Si la respuesta al N+1 es "pues cargo siempre todas las relaciones de todo, por si acaso", caes en el problema opuesto: traer montañas de datos que no usas, justo el over-fetching que veíamos en las APIs. El equilibrio es cargar por adelantado solo lo que esa vista concreta va a usar.

N+1 (under-fetching)

  • Lazy loading dentro de un bucle.
  • Una consulta por fila.
  • Se hunde al crecer los datos.

Eager de todo (over-fetching)

  • Cargar relaciones que no se usan.
  • Consultas gigantes con JOINs de más.
  • Memoria y ancho de banda malgastados.

La pregunta correcta antes de cada listado: ¿qué datos relacionados va a tocar esta pantalla? Esos, y solo esos, se cargan por adelantado. Ni el N+1 de pedir tarde, ni el derroche de pedir todo.

Resumen #

Lo que te llevas

  • N+1: 1 consulta para la lista + N consultas (una por fila) para una relación. Decenas o miles de viajes donde bastaban 2.
  • Lo provoca la carga perezosa (lazy loading): acceder a una relación en un bucle dispara una consulta oculta cada vuelta.
  • Se detecta contando consultas (profiler, query log), no leyendo el código. Si las queries crecen con las filas, es un N+1.
  • Se cura con eager loading (with(...) / JOIN): pides la relación por adelantado y el ORM la trae en 2 consultas.
  • Sin pasarte: carga por adelantado solo lo que la vista usa. Ni el N+1 de pedir tarde, ni el over-fetching de pedir todo.
