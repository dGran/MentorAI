Redis a fondo: estructuras de datos en memoria

Casi todo el mundo conoce Redis como "esa caché rápida". Pero su gran idea es otra: es un servidor de estructuras de datos en memoria. Saber qué tipos ofrece y cuándo usar cada uno convierte a Redis en una navaja suiza, no solo en un almacén de claves.

Imagina una pizarra enorme, en memoria, a la que todos tus servidores pueden escribir y leer en microsegundos. Eso es Redis. Lo interesante no es solo que sea rápida: es que la pizarra entiende de listas, conjuntos, rankings y contadores, no solo de "clave = valor".

Qué es Redis y por qué vuela #

Redis (REmote DIctionary Server) guarda los datos en RAM, no en disco. Ahí está el primer motivo de su velocidad: leer de memoria es órdenes de magnitud más rápido que de disco. El segundo es más sutil: el núcleo que procesa los comandos es de un solo hilo.

¿Un solo hilo y aun así rápido?

Parece contradictorio, pero es una ventaja. Sin varios hilos no hay bloqueos ni condiciones de carrera dentro de Redis: cada comando se ejecuta entero, de principio a fin, sin que otro se cuele en medio. Eso hace que las operaciones sean atómicas por diseño y elimina todo el coste de sincronización. La velocidad sale de hacer cada cosa pequeña y de no esperar nunca a disco.

La forma mental correcta: Redis es un diccionario gigante donde cada clave apunta a un valor que tiene tipo. Y según el tipo, dispones de comandos distintos. Vamos uno a uno.

Strings: el tipo base (y los contadores) #

El string es el valor más simple: una clave guarda un texto o un número. Sirve para cachear el HTML de una página, guardar un token o, gracias a comandos atómicos, llevar contadores fiables.

redis 
# Guardar y leer
SET user:42:name "Ana"
GET user:42:name

# Contador atómico: ideal para "visitas" o rate limiting
INCR page:home:views
INCRBY page:home:views 10

# Guardar con expiración de una vez (segundos)
SETEX session:abc 3600 "datos-de-sesion"

INCR es más que un +1: es atómico. Dos peticiones simultáneas que incrementan el mismo contador nunca pierden una cuenta, justo el tipo de condición de carrera que en una base de datos relacional tendrías que proteger con bloqueos.

Listas: colas y pilas #

Una lista es una secuencia ordenada de strings en la que insertas y sacas por los extremos. Con eso montas una cola (entras por un lado, sales por el otro) o una pila. Es la base de muchas colas de trabajos sencillas.

redis 
# Encolar trabajos por la izquierda
LPUSH queue:emails "enviar:1001"
LPUSH queue:emails "enviar:1002"

# Un worker los saca por la derecha (FIFO)
RPOP queue:emails

# BRPOP bloquea esperando si la cola está vacía (sin hacer polling)
BRPOP queue:emails 5

La versión bloqueante evita el polling

BRPOP (la B es de blocking) deja al worker dormido hasta que llega un elemento, en lugar de preguntar "¿hay algo?" mil veces por segundo. Es la diferencia entre una cola eficiente y quemar CPU sondeando.

Sets y sorted sets #

Un set es una colección de strings únicos y sin orden. Brilla cuando la pregunta es "¿está este elemento?" o cuando necesitas operaciones de conjuntos: intersección, unión, diferencia. Comprobar pertenencia es O(1).

redis 
# Etiquetas únicas de un artículo
SADD post:99:tags php redis cache
SISMEMBER post:99:tags redis   # ¿tiene la etiqueta? -> 1

# Usuarios online (únicos por naturaleza)
SADD online:users 42 17 88
SCARD online:users            # cuántos hay

# Amigos en común: intersección de dos sets
SINTER user:1:friends user:2:friends

El sorted set (o zset) añade lo que al set le falta: cada miembro lleva un score numérico, y Redis los mantiene ordenados por ese score. Es la estructura perfecta para rankings y leaderboards: insertar y consultar el top-N es muy barato.

redis 
# Marcador de un juego: miembro + puntuación
ZADD leaderboard 1500 "ana"
ZADD leaderboard 980 "luis"
ZINCRBY leaderboard 50 "ana"   # suma 50 a Ana

# Top 3 de mayor a menor, con sus puntos
ZREVRANGE leaderboard 0 2 WITHSCORES

# ¿En qué puesto va un jugador?
ZREVRANK leaderboard "luis"

Hashes: objetos #

Un hash es un mapa de campos y valores dentro de una sola clave: justo un objeto. En vez de guardar un usuario como un JSON en un string (que tendrías que leer y reescribir entero para cambiar un campo), lo guardas como hash y tocas campos sueltos.

redis 
# Un usuario como objeto
HSET user:42 name "Ana" age 30 plan "pro"

# Leer un campo o todo el objeto
HGET user:42 plan
HGETALL user:42

# Incrementar un contador dentro del objeto, atómico
HINCRBY user:42 logins 1

Hash vs JSON en un string

Si solo lees el objeto entero, un JSON en un string va de sobra. Si actualizas campos sueltos a menudo o quieres incrementar uno sin leer el resto, el hash gana: menos datos movidos y operaciones atómicas por campo.

Bitmaps, HyperLogLog y streams #

Más allá de los cuatro grandes, Redis trae tipos especializados que resuelven problemas concretos con muy poca memoria:

Tipo Para qué Ejemplo
Bitmap Banderas por ID en bits sueltos. ¿Entró el usuario N hoy? (un bit por usuario)
HyperLogLog Contar únicos aproximados con KB fijos. Visitantes únicos de una web enorme
Stream Log de eventos append-only con consumidores. Cola de eventos tipo Kafka ligero
Pub/Sub Mensajes a suscriptores en tiempo real. Notificaciones, chats

El truco del HyperLogLog

Contar visitantes únicos de verdad exige guardar cada identificador visto: en un sitio enorme, gigas de memoria. El HyperLogLog estima el número de únicos con un error de ~0,8% usando unos 12 KB fijos, vea los que vea. Cambias precisión exacta por memoria casi nula; para métricas, un trato excelente.

TTL y nombrado de claves #

Cualquier clave puede tener una expiración (TTL): pasado ese tiempo, Redis la borra sola. Es la base de las sesiones, los tokens temporales y la caché que debe caducar.

redis 
# Caducar una clave existente en 60 segundos
EXPIRE cart:42 60

# ¿Cuánto le queda de vida?
TTL cart:42      # 60, o -1 si no expira, o -2 si no existe

# Quitar la expiración
PERSIST cart:42

El nombrado de claves es tu esquema

Redis no tiene tablas: la convención de nombres es tu única estructura. El estándar de facto son los dos puntos como separador: user:42:cart, session:abc, page:home:views. Un buen esquema de claves hace que Redis sea navegable; uno malo, un cajón de sastre imposible de mantener.

Atomicidad, MULTI y pipelines #

Cada comando de Redis es atómico por sí solo (recuerda: un solo hilo). Cuando necesitas que varios comandos se ejecuten como un bloque, tienes MULTI/EXEC: encolan los comandos y se ejecutan juntos, sin que nadie se cuele en medio.

redis 
# Transacción: los dos se ejecutan juntos
MULTI
INCRBY account:1:balance -100
INCRBY account:2:balance 100
EXEC

No confundas dos cosas que suenan parecido. MULTI es atomicidad (se ejecutan sin intercalarse). El pipeline es otra cosa: enviar muchos comandos de golpe sin esperar la respuesta de cada uno, para ahorrarte los viajes de ida y vuelta por la red. Uno resuelve corrección; el otro, latencia.

Persistencia: RDB vs AOF #

"En memoria" asusta: ¿si Redis se reinicia, lo pierdo todo? No necesariamente. Redis puede volcar su estado a disco de dos formas, combinables:

RDB (snapshot)

  • Foto del dataset cada cierto tiempo
  • Fichero compacto, arranque rápido
  • Ideal para backups
  • Riesgo: pierdes lo de entre snapshots

AOF (append-only)

  • Registra cada comando de escritura
  • Mucha menos pérdida ante caída
  • Fichero mayor, restauración más lenta
  • Se reescribe periódicamente para compactarse

La elección depende del papel de Redis

Si Redis es solo caché (todo está también en tu BBDD), perder datos al reiniciar no es grave: incluso puedes desactivar la persistencia. Si Redis es la fuente de verdad de algo (un contador, una cola), querrás AOF para minimizar la pérdida. No es "uno u otro" siempre: muchos montajes usan los dos.

Con las estructuras claras, el siguiente paso natural es usarlas para el caso más común: caché de aplicación, con sus estrategias de invalidación y TTL. Eso es justo el tutorial de Redis como caché.

Resumen #

Lo que te llevas

  • Redis es un servidor de estructuras de datos en memoria y de un solo hilo: por eso es rápido y atómico por diseño.
  • Elige el tipo por la pregunta: string (contadores), list (colas), set (únicos), zset (rankings), hash (objetos).
  • Tipos especiales para problemas concretos: bitmaps, HyperLogLog (únicos aproximados), streams y pub/sub.
  • El TTL caduca claves solo; el nombrado con : es tu única estructura. MULTI da atomicidad; el pipeline, latencia.
  • RDB (snapshots) y AOF (log de escrituras) le dan persistencia; cuánta necesitas depende de si Redis es caché o fuente de verdad.
