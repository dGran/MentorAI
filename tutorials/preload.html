Preload: precargar clases al arrancar PHP

OPcache evita recompilar tu código, pero cada worker todavía lo carga y enlaza la primera vez que lo necesita. Preload da el siguiente paso: deja tu framework ya cargado en memoria desde que arranca PHP, disponible para todas las peticiones sin pasar por el autoload. Veamos cómo, y cuándo de verdad merece la pena.

Imagina un restaurante que cada noche, antes de abrir, deja las salsas base ya hechas y calientes. Cuando entra un cliente, no se empieza de cero: la mitad del trabajo está hecho desde antes de que llegara nadie. Eso es preload para PHP: preparar las clases antes de la primera petición, no durante ella.

Lo que OPcache deja sin resolver #

Con OPcache activo, PHP ya no recompila tus ficheros a opcodes en cada petición: los compila una vez y los guarda en memoria compartida. Es la mayor ganancia que puedes darle a una app PHP con un solo ajuste. Pero queda un coste que OPcache no elimina.

Cuando un worker de PHP-FPM atiende su primera petición, todavía tiene que cargar y enlazar las clases que esa petición usa: resolver el autoload (encontrar el fichero de cada clase), ejecutar su definición, vincular cada clase con su clase padre, sus interfaces y sus traits. OPcache le ahorra recompilar, pero el trabajo de «traer esta clase a la memoria de este proceso y dejarla lista para usar» sigue ocurriendo, clase a clase, la primera vez que cada worker la necesita.

Cachear opcodes ≠ tener la clase lista

OPcache guarda los opcodes de un fichero. Pero una clase no «existe» para el proceso hasta que ese fichero se ejecuta y la clase se enlaza con sus dependencias. Preload se ocupa justamente de ese segundo paso.

Qué es preload #

Preload, introducido en PHP 7.4, le dice al motor: «al arrancar, antes de atender ninguna petición, carga este conjunto de ficheros y deja sus clases, funciones e interfaces permanentemente disponibles». Esas clases pasan a formar parte del propio motor: están en memoria desde el minuto cero y todas las peticiones las ven sin tener que cargarlas.

La diferencia clave con una caché normal es cuándo y para quién:

OPcache (solo opcodes)

  • Cachea el resultado de compilar cada fichero
  • La clase se carga y enlaza por petición, la primera vez que se usa
  • Vive en memoria compartida como bytecode
  • No necesita que toques tu código

Preload

  • Carga y enlaza las clases al arrancar PHP
  • Quedan listas para todas las peticiones, sin autoload
  • Pasan a ser parte del motor, como si fueran clases internas
  • Exige un fichero de preload y reiniciar para cambiarlas

Por eso preload se construye encima de OPcache: lo necesita para funcionar. No es un sustituto, es la capa siguiente.

Cómo funciona por dentro #

Cuando PHP-FPM arranca, el proceso maestro ejecuta una sola vez el fichero que le indiques en opcache.preload. Ese fichero carga las clases que quieras precargar. El motor las compila, las enlaza con sus dependencias y las guarda en la memoria compartida de OPcache, ya vinculadas. A partir de ahí, cada worker que el maestro crea las hereda listas para usar.

1 Arranca PHP-FPMEl proceso maestro inicia
2 Ejecuta el fichero de preloadUna sola vez, antes de aceptar peticiones
★ Clases enlazadas en memoria compartidaListas y vinculadas, no solo compiladas
3 Los workers atienden peticionesSin autoload para lo precargado

El trabajo se hace una vez al arrancar, no en la primera petición de cada worker. Cambiar lo precargado obliga a repetir el paso 2: hay que reiniciar.

El enlazado es la clave

Una clase precargada no necesita resolver su jerarquía en cada petición: ya sabe quién es su padre, qué interfaces implementa y qué traits usa. Ese vínculo, que normalmente se rehace por proceso, queda hecho de una vez para siempre.

El fichero de preload #

El fichero que apuntas con opcache.preload es PHP normal. Su trabajo es decidir qué clases entran. Hay dos formas de cargarlas, con una diferencia importante:

Mecanismo Qué hace Cuándo usarlo
opcache_compile_file() Compila y precarga el fichero, pero no ejecuta su código de nivel superior. Para clases sueltas cuyas dependencias quizá aún no están cargadas.
require_once Ejecuta el fichero de verdad, resolviendo y enlazando sus dependencias en el acto. Cuando controlas el orden y quieres el enlazado completo.

Un fichero de preload mínimo que recorre un directorio y precarga sus clases podría verse así:

php · preload.php 
<?php

declare(strict_types=1);

$directory = new RecursiveDirectoryIterator(__DIR__ . '/src');
$iterator = new RecursiveIteratorIterator($directory);

foreach ($iterator as $file) {

    if ($file->getExtension() !== 'php') {

        continue;
    }

    opcache_compile_file($file->getRealPath());
}

Dependencias sin resolver se quedan fuera

Si precargas una clase pero su clase padre o una interfaz que implementa no está disponible, PHP no puede enlazarla y la omite, avisando por el log al arrancar. Por eso lo habitual es precargar el proyecto entero (o apoyarse en el fichero que genera el framework), no clases sueltas a mano.

Configuración #

Preload se activa con dos directivas en php.ini (o en un fichero de conf.d/). Necesita OPcache activo de base:

ini · 11-preload.ini 
; Preload necesita OPcache activo
opcache.enable=1

; Fichero que se ejecuta al arrancar para precargar clases
opcache.preload=/var/www/app/preload.php

; Usuario con el que se ejecuta el preload (no puede ser root)
opcache.preload_user=www-data
Directiva Qué hace
opcache.preload Ruta absoluta al fichero PHP que se ejecuta al arrancar. Vacío = preload desactivado.
opcache.preload_user Usuario bajo el que corre el preload. Obligatorio si arrancas como root; suele ser el mismo del pool (www-data).

Para comprobar que funciona, opcache_get_status() incluye una sección preload_statistics con las clases, funciones y scripts precargados y la memoria que ocupan:

php · preload-status.php 
<?php

declare(strict_types=1);

$status = opcache_get_status();
$preload = $status['preload_statistics'];

$memoryMb = round($preload['memory_consumption'] / 1024 / 1024, 1);

echo "Clases precargadas:   " . count($preload['classes']) . "\n";
echo "Funciones precargadas: " . count($preload['functions']) . "\n";
echo "Memoria de preload:    {$memoryMb} MB\n";

Limitaciones y la gran pega #

Preload tiene una contrapartida que define cuándo usarlo: lo precargado se congela al arrancar.

Cambiar el código exige reiniciar PHP-FPM

A diferencia de OPcache con validate_timestamps=1, aquí no hay revalidación por fecha: las clases precargadas no se refrescan solas. Si editas una clase que está en el preload, el cambio no se ve hasta que reinicies (no recargues) PHP-FPM. Por eso preload es una técnica de producción, no de desarrollo: en local te haría la vida imposible.

Reload no basta: tiene que ser restart

Un reload de PHP-FPM no vuelve a ejecutar el preload. Para que entre el código nuevo necesitas systemctl restart php8.3-fpm. Tenlo en cuenta en el script de despliegue, justo como ya haces con OPcache.

Otras cosas que conviene saber:

  • No precargues lo que cambia mucho: el código de tu app evoluciona en cada deploy, pero el del framework y las librerías es estable. Cuanto más estable, mejor candidato.
  • Cuesta memoria por proceso compartida: las clases ocupan sitio en la memoria de OPcache desde el arranque, las use o no esta petición. Es un intercambio: memoria a cambio de latencia.
  • La ganancia es real pero acotada: ahorra el autoload y el enlazado, no la lógica de tu petición. Brilla cuando cargas muchas clases por request, típico de frameworks grandes.

Preload en frameworks reales #

La buena noticia: rara vez escribirás el fichero de preload a mano. Los frameworks modernos lo generan por ti, resolviendo el orden de dependencias para que no se quede nada fuera.

Stack Cómo lo da
Symfony Al compilar el contenedor genera un var/cache/prod/App_KernelProdContainer.preload.php. Apuntas opcache.preload ahí y listo.
Composer El classmap optimizado (composer dump-autoload --optimize) es la base para saber qué ficheros precargar.
Laravel No trae preload propio; el camino habitual hacia el «framework en memoria» es Octane (Swoole/RoadRunner), que mantiene la app viva entre peticiones.

Preload no es lo mismo que «long-running»

Preload sigue el modelo clásico de PHP: cada petición empieza limpia, solo que con las clases ya cargadas. Swoole, RoadRunner u Octane van más allá y mantienen la aplicación entera viva entre peticiones (estado incluido), lo que es más rápido pero rompe el shared-nothing de PHP y exige escribir el código con cuidado. Son herramientas distintas para problemas distintos.

Resumen #

Lo que te llevas

  • OPcache cachea opcodes; preload va más allá y deja las clases cargadas y enlazadas en memoria desde el arranque.
  • Se configura con opcache.preload (el fichero) y opcache.preload_user, y se ejecuta una vez al arrancar PHP-FPM.
  • Su gran pega: lo precargado se congela. Cambiar el código exige reiniciar (restart, no reload) FPM. Es técnica de producción.
  • Precarga lo estable (framework, librerías), no lo que cambia cada deploy. Brilla cuando cada petición carga muchas clases.
  • Symfony genera el fichero de preload por ti; Laravel busca el mismo objetivo por otra vía (Octane, modelo long-running).
