Interfaces: el contrato que desacopla

Una interfaz es una promesa sin cuerpo: dice qué métodos existen, no cómo se cumplen. Es la herramienta más afilada de la OOP para desacoplar, y la que te abre la puerta a programar contra contratos en vez de contra clases concretas. Aquí ves por qué cambia tanto el diseño.

Si una clase es «qué soy y cómo lo hago», una interfaz es solo «qué prometo poder hacer». No tiene propiedades ni cuerpo de métodos: es una lista de firmas. Cualquier clase que la implemente se compromete a ofrecer esos métodos. Suena a poco, pero es el contrato que permite cambiar una pieza por otra sin que el resto se entere.

Qué es una interfaz #

Piensa en el enchufe de la pared. La interfaz es la forma del enchufe: define cómo conectarse. Lo que haya detrás —una central de gas, una placa solar, una batería— da igual mientras respete la forma. Tus aparatos dependen del enchufe (el contrato), no de la central eléctrica (la implementación). En código:

php · una interfaz es solo firmas 
<?php

declare(strict_types=1);

interface RepositorioUsuarios
{
    public function porId(int $id): ?Usuario;

    public function guardar(Usuario $usuario): void;
}

No hay cómo: ni una línea de implementación. Solo el compromiso de que quien firme este contrato sabrá buscar por id y guardar. Los métodos de una interfaz son siempre public (es lo que se promete al exterior).

implements #

Una clase se compromete con implements. A partir de ahí, PHP la obliga a tener todos los métodos del contrato con firmas compatibles; si falta uno, no compila. Puedes tener muchas implementaciones del mismo contrato: una contra base de datos, otra en memoria para los tests.

php · dos implementaciones del mismo contrato 
<?php

final class RepositorioMySql implements RepositorioUsuarios
{
    public function porId(int $id): ?Usuario { /* SELECT ... */ }
    public function guardar(Usuario $usuario): void { /* INSERT ... */ }
}

final class RepositorioEnMemoria implements RepositorioUsuarios
{
    /** @var Usuario[] */
    private array $usuarios = [];

    public function porId(int $id): ?Usuario
    {
        return $this->usuarios[$id] ?? null;
    }

    public function guardar(Usuario $usuario): void
    {
        $this->usuarios[$usuario->id()] = $usuario;
    }
}

Programar contra interfaces #

Aquí está el premio. El código que usa el repositorio escribe su dependencia como la interfaz, no como una clase concreta. Así puede recibir cualquier implementación: en producción la de MySQL, en los tests la de memoria, sin cambiar una línea.

php · depender del contrato, no de la clase 
<?php

final class RegistrarUsuario
{
    public function __construct(
        private RepositorioUsuarios $repositorio,
    ) {
    }

    public function __invoke(string $email): void
    {
        $this->repositorio->guardar(Usuario::nuevo($email));
    }
}

Esto es la inversión de dependencias

RegistrarUsuario no conoce MySQL: depende de una abstracción que tú le inyectas. Ese giro —el código de alto nivel depende del contrato, no del detalle— es la D de SOLID y el corazón de la inyección de dependencias. Las interfaces son lo que lo hace posible.

Implementar varias #

A diferencia de la herencia (una sola clase padre), una clase puede implementar varias interfaces. Eso permite componer capacidades por rol: una clase puede ser a la vez Comparable y Serializable sin forzar ninguna jerarquía. Es la base del principio de segregación de interfaces: muchos contratos pequeños y específicos en vez de uno gigante.

php · varias capacidades por rol 
<?php

final class Factura implements Exportable, Firmable
{
    public function aPdf(): string { /* ... */ }
    public function firmar(Certificado $cert): void { /* ... */ }
}

Interfaz vs herencia #

Las dos permiten polimorfismo, pero comprometen cosas distintas. La herencia comparte implementación (código real del padre) y ata a una jerarquía. La interfaz comparte solo el contrato y deja a cada clase su libertad. Por eso, cuando solo quieres un punto común para el polimorfismo sin imponer cómo se hace, la interfaz suele ganar.

Interfaz Herencia (clase)
Qué comparte Solo el contrato (firmas). Implementación y estado.
Cuántas a la vez Varias. Una sola clase padre.
Acoplamiento Mínimo: «sé hacer X». Alto: «soy un Y por dentro».
Relación «se comporta como». «es un».

Resumen #

Lo que te llevas

  • Una interfaz es un contrato: dice qué métodos hay, no cómo se cumplen.
  • Programar contra la interfaz deja intercambiar implementaciones (MySQL, memoria) sin tocar el cliente.
  • Una clase puede implementar varias: capacidades por rol, base de la segregación de interfaces.
  • Frente a la herencia, la interfaz comparte solo el contrato: menos acoplamiento, «se comporta como» en vez de «es un».
