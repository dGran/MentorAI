Funciones limpias: pequeñas y de una sola cosa

La primera regla de las funciones es que sean pequeñas. La segunda, que sean más pequeñas todavía. Detrás de esa boutade de Uncle Bob hay un principio sólido: una función debe hacer una cosa, hacerla bien y leerse a un solo nivel de abstracción. Aquí ves cómo se consigue.

Una función limpia se lee como un párrafo bien escrito: una idea por frase, sin incisos que te pierdan. Cuando una función crece, no es que haga «mucho», es que hace varias cosas, y cada una pide salir a su propia función con nombre. El tamaño es la consecuencia, no el objetivo.

Hacer una sola cosa #

La regla de oro: una función hace una sola cosa. ¿Cómo saber si cumple? Dos pruebas. Una: si puedes extraer otra función con un nombre que no sea solo reformular su cuerpo, es que hacía más de una cosa. Dos: si para describir qué hace necesitas la palabra «y», son dos funciones.

php · de un cajón de sastre a pasos con nombre 
<?php

declare(strict_types=1);

// Hace tres cosas: validar, calcular y notificar
public function procesarPedido(Pedido $pedido): void
{
    if ($pedido->lineas() === []) {
        throw new PedidoVacio();
    }

    $total = 0;
    foreach ($pedido->lineas() as $linea) {
        $total += $linea->precio() * $linea->cantidad();
    }
    $pedido->asignarTotal($total);

    $this->mailer->enviar($pedido->cliente(), "Pedido confirmado");
}
php · cada paso, su función 
<?php

public function procesarPedido(Pedido $pedido): void
{
    $this->garantizarNoVacio($pedido);
    $this->calcularTotal($pedido);
    $this->notificarConfirmacion($pedido);
}

La segunda versión se lee como un índice: sabes qué hace sin entrar en el detalle, y si necesitas el detalle, bajas a la función concreta. Eso es una función que hace una cosa: orquestar tres pasos con nombre.

Un nivel de abstracción #

El olor que delata una función que hace varias cosas es mezclar niveles de abstracción: en la misma función conviven conceptos de negocio de alto nivel (notificarConfirmacion) con detalles de bajo nivel (concatenar strings, manejar índices de un array). Leerla es como un texto que salta de la idea general al pie de página sin avisar. Mantén cada función a un solo nivel: o cuenta el qué, o resuelve el cómo, no ambos.

Pocos argumentos #

Cuantos más argumentos, más difícil es entender y probar una función. El ideal es cero, uno o dos; tres ya escuece; cuatro o más pide a gritos un objeto que los agrupe. Y ojo con el argumento booleano: casi siempre significa que la función hace dos cosas según el flag, y es mejor partirla en dos.

Lista larga y flag

  • crearUsuario($nombre, $email, $calle, $cp, $ciudad, $esAdmin)
  • ¿Qué era el cuarto argumento al llamarla?
  • El $esAdmin esconde dos caminos.

Agrupar y separar

  • crearUsuario($nombre, $email, Direccion $direccion)
  • La dirección es un concepto, no cuatro sueltos.
  • crearAdmin(...) y crearUsuario(...) separados.

Sin efectos secundarios #

Un efecto secundario es algo que la función hace además de lo que su nombre promete: un comprobarPassword() que de paso inicia la sesión. El lector confía en el nombre y no espera la sorpresa, así que el bug aparece donde nadie mira. Una función debe hacer lo que dice y solo eso.

De ahí sale la separación comando-consulta (CQS): una función o hace algo (cambia estado, devuelve void) o responde algo (devuelve un valor sin cambiar nada), pero no las dos. Así $cuenta->saldo() nunca tendrá la sorpresa de modificar el saldo al consultarlo.

Funciones predecibles = tests fáciles

Una función pequeña, sin efectos ocultos y con pocos argumentos es trivial de testear: le das entradas, compruebas la salida. Las funciones que hacen de todo y tocan medio mundo son justo las que cuesta probar. Funciones limpias y TDD se empujan mutuamente.

Early return en vez de anidar #

La anidación profunda de if es enemiga de la lectura: cada nivel obliga a recordar una condición mientras lees el interior. La cura es el early return: resuelve los casos borde arriba y sal, dejando el camino principal sin sangrar. Evita el else y aplana el código.

php · de la pirámide al camino plano 
<?php

// Anidado: hay que sostener tres condiciones en la cabeza
public function precio(Cliente $cliente): int
{
    if ($cliente->estaActivo()) {
        if ($cliente->tieneSaldo()) {
            return $this->base;
        } else {
            throw new SinSaldo();
        }
    } else {
        throw new ClienteInactivo();
    }
}

// Early return: guardas arriba, camino principal limpio
public function precio(Cliente $cliente): int
{
    if (!$cliente->estaActivo()) {
        throw new ClienteInactivo();
    }

    if (!$cliente->tieneSaldo()) {
        throw new SinSaldo();
    }

    return $this->base;
}

Es la regla de Object Calisthenics

«Un nivel de indentación por método» y «no uses else» son dos de las nueve reglas de Object Calisthenics. El early return las cumple las dos de golpe, y por eso el código resultante se lee como una lista de guardas en vez de como un árbol.

Resumen #

Lo que te llevas

  • Una función hace una sola cosa; si para describirla necesitas «y», son dos. El tamaño pequeño es la consecuencia.
  • Mantén un solo nivel de abstracción por función: o el qué, o el cómo, no ambos mezclados.
  • Pocos argumentos (agrupa en objetos, evita flags) y sin efectos secundarios ocultos (separa comando de consulta).
  • Early return para los casos borde: aplana la anidación y deja el camino principal limpio.
