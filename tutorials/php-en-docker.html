PHP en Docker: nginx, php-fpm y extensiones

Containerizar PHP tiene una peculiaridad que sorprende a todos: PHP no trae servidor web propio para producción, así que el entorno típico son dos contenedores hablándose. Entender por qué —y cómo se montan— es lo que cierra el círculo de construir tu servicio.

Esta lección junta todo lo del curso en el caso real que el backend PHP se encuentra a diario: un servidor web (nginx) sirviendo lo estático y delegando el PHP a un intérprete (php-fpm), cada uno en su contenedor, más la base de datos. Es el esqueleto de prácticamente cualquier app PHP containerizada.

Por qué dos contenedores #

Viniendo de un PHP "tradicional" donde Apache y PHP iban juntos, choca: el patrón moderno separa el servidor web (nginx) del intérprete de PHP (php-fpm) en dos contenedores. No es capricho; responde a la regla de oro de Docker: un proceso principal por contenedor.

1 NavegadorPetición HTTP al puerto 80
2 nginxSirve estáticos; el PHP lo delega
3 php-fpmEjecuta el PHP y devuelve el resultado

nginx recibe la petición; si es PHP, se la pasa a php-fpm por FastCGI y devuelve su respuesta.

¿Por qué separarlos? Porque cada uno tiene un ciclo de vida propio: puedes escalar el número de procesos PHP sin tocar nginx, actualizar uno sin el otro, reutilizar la misma imagen de nginx para varias apps. Meter ambos en un contenedor obligaría a un proceso supervisor y rompería el modelo de "un contenedor, una responsabilidad".

Qué es php-fpm #

php-fpm (FastCGI Process Manager) es PHP funcionando como un servicio que se queda esperando peticiones, con un pool de procesos workers listos para ejecutar tu código. No es un servidor web: no habla HTTP con el navegador. Habla un protocolo distinto, FastCGI, pensado para que un servidor web le pase trabajo.

Esto ya lo viste por dentro

El modelo de php-fpm —un gestor con un pool de workers que atienden peticiones— es justo el que explican php-fpm a fondo y los workers de PHP. Aquí lo containerizamos: ese mismo motor, metido en su contenedor. La imagen oficial php:8.3-fpm trae fpm ya configurado.

nginx habla con fpm #

nginx recibe la petición HTTP. Si es un fichero estático (una imagen, un CSS), lo sirve él mismo. Si es una ruta que debe ejecutar PHP, se la pasa a php-fpm por FastCGI, espera el resultado y se lo devuelve al navegador. La configuración de nginx que lo conecta es breve:

nginx · delegar el PHP a php-fpm por FastCGI 
location ~ \.php$ {
    fastcgi_pass   app:9000;
    fastcgi_index  index.php;
    include        fastcgi_params;
}

La línea clave es fastcgi_pass app:9000: nginx envía el trabajo al servicio llamado app (el contenedor de php-fpm) en su puerto 9000. Reconoces el patrón de la lección de compose: se localizan por nombre de servicio dentro de la red de Docker, no por IP. El 9000 es el puerto estándar de FastCGI de php-fpm.

Instalar extensiones #

La imagen base de PHP trae lo esencial, pero tu app casi seguro necesita extensiones que no vienen activadas: pdo_mysql para hablar con MySQL, intl, gd, opcache… Las imágenes oficiales incluyen un ayudante, docker-php-ext-install, justo para esto:

Dockerfile · añadir las extensiones que la app necesita 
FROM php:8.3-fpm

RUN docker-php-ext-install pdo_mysql opcache

RUN pecl install redis && docker-php-ext-enable redis

Dos vías: docker-php-ext-install para las extensiones que vienen con el código fuente de PHP (como pdo_mysql u opcache), y pecl install + docker-php-ext-enable para las de terceros (como redis). La idea de fondo conecta con cómo funcionan las extensiones de PHP: aquí las congelas en la imagen, declaradas, en lugar de instalarlas a mano en cada servidor.

No olvides OPcache en producción

En una imagen de producción, activar y configurar OPcache es de lo que más rendimiento da por el esfuerzo: PHP compila tu código una vez y reutiliza el bytecode. Es una extensión que instalas y configuras en el Dockerfile como cualquier otra.

El compose completo #

Juntando las tres piezas —servidor web, intérprete y base de datos— el entorno de desarrollo de una app PHP queda así:

compose.yaml · nginx + php-fpm + base de datos 
services:
  web:
    image: nginx:1.27-alpine
    ports:
      - "8080:80"
    volumes:
      - .:/app
      - ./nginx.conf:/etc/nginx/conf.d/default.conf
    depends_on:
      - app

  app:
    build: .
    volumes:
      - .:/app
    depends_on:
      - db

  db:
    image: mysql:8.3
    environment:
      MYSQL_DATABASE: tienda
      MYSQL_ROOT_PASSWORD: secreto
    volumes:
      - datos-db:/var/lib/mysql

volumes:
  datos-db:

Léelo con todo lo aprendido: web (nginx) publica el 8080 hacia tu navegador y delega el PHP a app; app (php-fpm, construido desde tu Dockerfile con sus extensiones) ejecuta el código y habla con db; db guarda sus datos en un volumen que persiste. Los tres comparten la carpeta del código por un bind mount para desarrollar en vivo. Un docker compose up y tienes el servicio entero corriendo.

Local y producción se parecen, no son idénticos

Este compose con bind mounts es ideal para desarrollar. Para producción, el código va dentro de la imagen (no montado desde fuera), con OPcache activo, sin dependencias de desarrollo y corriendo como usuario sin privilegios, como vimos en buenas prácticas. Misma arquitectura, distinto afinado.

Resumen #

Lo que te llevas

  • El patrón PHP en Docker separa nginx (servidor web) y php-fpm (intérprete) en contenedores distintos: un proceso por contenedor.
  • php-fpm no es un servidor web: es PHP como servicio con un pool de workers, que habla FastCGI, no HTTP.
  • nginx sirve lo estático y delega el PHP a fpm con fastcgi_pass app:9000, localizándolo por nombre de servicio.
  • Las extensiones se declaran en el Dockerfile: docker-php-ext-install para las del core, pecl para las de terceros.
  • El compose de web + app + db levanta el servicio entero con un comando. Local usa bind mounts; producción mete el código en la imagen.
