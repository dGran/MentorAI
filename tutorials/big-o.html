Big-O sin matemáticas: cómo saber si tu código escala

El código que vuela con 100 registros puede agonizar con 100.000, y la diferencia no está en la máquina: está en cómo crece el coste al crecer los datos. Big-O es el vocabulario para ver eso de un vistazo, sin cronómetro y sin fórmulas.

Big-O no mide segundos: mide cómo se multiplica el trabajo cuando los datos se multiplican. Por eso es independiente de tu CPU, de tu lenguaje y de lo cargado que esté el servidor. Es la pregunta correcta: no "¿cuánto tarda?", sino "¿qué pasa cuando esto crece 10 veces?".

La idea: cómo crece el coste #

Llamamos n al tamaño de la entrada: el número de filas, de elementos, de usuarios. Big-O describe la forma en que el número de operaciones crece en función de n. No te dice que algo tarde 3 milisegundos; te dice si, al doblar n, el trabajo se mantiene, se dobla o se cuadruplica.

Esa forma es lo que decide si un sistema aguanta el crecimiento. Una función que dobla su trabajo al doblar los datos escala de maravilla; una que lo cuadruplica acabará cayéndose, por muy rápida que sea la máquina. La constante (los milisegundos por operación) importa poco al lado de la forma.

Leer la notación #

Se escribe O(...) con dentro la forma de crecimiento: O(n) (lineal), O(n²) (cuadrático), O(1) (constante). Dos reglas de lectura que despejan casi todo:

  • Se ignoran las constantes y los términos menores. Un bucle que hace 3n + 5 operaciones es O(n): al crecer n, el 3 y el 5 dejan de importar frente a la forma. O(2n) y O(n) son lo mismo.
  • Por defecto se habla del peor caso. Big-O describe la cota superior: lo máximo que puede costar. Buscar en una lista es O(n) porque el elemento podría ser el último (o no estar).

Por qué se tiran las constantes

Porque Big-O compara tendencias, no implementaciones. Una operación O(n) con una constante grande puede ser más lenta que una O(n²) cuando n es diminuto; pero en cuanto n crece, la forma manda y la O(n²) pierde siempre. Big-O es sobre el "a lo grande".

Ahí se ve el abismo: pasar de O(n) a O(n²) en un millón de elementos es pasar de un millón de operaciones a un billón. Por eso elegir la estructura de datos correcta no es pedantería: es la diferencia entre responder al instante o no responder.

En código real #

La forma se lee en el código casi a ojo: cuántas veces tocas los datos. Sin bucle sobre n, suele ser O(1); un bucle, O(n); un bucle dentro de otro sobre lo mismo, O(n²).

php · O(n) frente a O(1) 
<?php

declare(strict_types=1);

// O(n): in_array recorre el array hasta encontrar (o hasta el final)
$found = in_array('target', $haystack, true);

// O(1): el acceso por clave no depende del tamaño del array
$index = array_flip($haystack);     // se prepara una vez
$exists = isset($index['target']);  // cada consulta, coste constante

La búsqueda binaria es el ejemplo canónico de O(log n): cada comparación descarta la mitad de lo que queda, así que de un millón de elementos llega al objetivo en unas 20 comparaciones. Requiere que los datos estén ordenados; ese es el truco que el índice de una base de datos explota.

php · búsqueda binaria, O(log n) 
<?php

declare(strict_types=1);

function binarySearch(array $sorted, int $needle): int
{
    $low = 0;
    $high = count($sorted) - 1;

    while ($low <= $high) {
        $mid = intdiv($low + $high, 2);

        if ($sorted[$mid] === $needle) {
            return $mid;
        }

        if ($sorted[$mid] < $needle) {
            $low = $mid + 1;

            continue;
        }

        $high = $mid - 1;
    }

    return -1;
}

El O(n²) escondido #

El error de rendimiento más común no es escribir dos bucles anidados a la vista: es esconder un bucle dentro de otro sin darte cuenta. Un in_array dentro de un foreach es O(n²), porque in_array ya recorre por dentro.

php · de O(n²) a O(n) cambiando la estructura 
<?php

declare(strict_types=1);

// O(n²): por cada usuario, in_array recorre todos los baneados
foreach ($users as $user) {
    if (in_array($user->id, $bannedIds, true)) {
        continue;
    }

    $active[] = $user;
}

// O(n): un set indexado por clave; comprobar es O(1)
$bannedSet = array_flip($bannedIds);

foreach ($users as $user) {
    if (isset($bannedSet[$user->id])) {
        continue;
    }

    $active[] = $user;
}

El N+1 es un O(n²) con disfraz de base de datos

Recorrer 100 pedidos y, dentro del bucle, lanzar una consulta por cada uno para traer su cliente son 101 consultas: el clásico problema N+1. El coste crece con los datos igual que un bucle anidado. La solución es la misma idea: traer todo de una (un JOIN o un WHERE IN) en vez de consultar dentro del bucle.

No solo tiempo: la memoria #

Big-O también describe el espacio: cuánta memoria extra necesita un algoritmo según n. Hay un canje constante entre los dos. En el ejemplo de los baneados, pasamos de O(n²) en tiempo a O(n) a cambio de gastar O(n) de memoria en el set. Casi siempre vale la pena: la memoria es barata, el tiempo cuadrático no.

Cambiar tiempo por memoria es el truco más usado

Un índice, una caché, un hashmap precalculado: todos gastan memoria para convertir una búsqueda O(n) en O(1). Reconocer cuándo aplicar ese canje es media optimización de rendimiento; la otra media es medir antes de tocar.

Errores comunes #

MalentendidoLa realidad
"O(1) significa rápido" Significa que no crece con n. Una constante grande puede ser lenta; Big-O no habla de velocidad absoluta.
Optimizar la forma con n diminuto Con 10 elementos, O(n²) va sobrado. Vale la pena pelear la forma cuando n es o será grande.
No ver el bucle escondido in_array, array_search o una consulta dentro de un foreach añaden un factor n invisible.
Olvidar el coste de ordenar "Ordeno y luego busco binario" incluye un O(n log n) previo: solo compensa si buscas muchas veces.
Razonar en vez de medir Big-O guía, pero el cuello de botella real se confirma con un profiler. La forma orienta; la medición decide.

Resumen #

Lo que te llevas

  • Big-O mide cómo crece el coste con n, no los segundos. La pregunta es "¿y si esto crece 10×?".
  • Se ignoran constantes y términos menores, y se habla del peor caso: importa la forma, no la implementación.
  • Memoriza la escala: O(1) < O(log n) < O(n) < O(n log n) < O(n²). El salto a cuadrático es un abismo.
  • El O(n²) suele esconderse en un in_array dentro de un bucle o en un N+1 de consultas.
  • Cambiar tiempo por memoria (un set, un índice, una caché) convierte búsquedas O(n) en O(1). Y mide antes de optimizar.
