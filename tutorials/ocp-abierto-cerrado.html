O — Abierto a extensión, cerrado a modificación

Deberías poder añadir comportamiento nuevo sin abrir y editar el código que ya funciona. Suena a contradicción —¿cómo extiendo sin tocar?— y la respuesta es una de las más elegantes de la OOP: con polimorfismo sobre una abstracción. Lo nuevo se enchufa; lo viejo ni se entera.

La formulación original (Bertrand Meyer) suena a koan: un módulo debe estar abierto a extensión pero cerrado a modificación. Traducido: cuando llega un caso nuevo, lo ideal es añadir código, no editar el que ya está probado y funcionando. Cada vez que tocas algo que funcionaba, arriesgas romperlo.

Abierto y cerrado a la vez #

«Cerrado a modificación» no quiere decir que el código sea inmutable para siempre; quiere decir que el flujo estable —la parte que orquesta— no debería cambiar cada vez que aparece una variante. «Abierto a extensión» significa que esas variantes entran por un punto previsto: una abstracción que el flujo ya conoce. El truco es identificar qué eje va a variar y dejar ahí una costura.

El switch que crece #

El síntoma de violar OCP es el condicional por tipo que engorda con cada feature. Un calculador de comisiones según el método de pago:

php · cada método de pago nuevo edita esta clase 
<?php

declare(strict_types=1);

final class CalculadoraComision
{
    public function comision(string $metodo, int $importe): int
    {
        return match ($metodo) {
            'tarjeta' => (int) round($importe * 0.015),
            'paypal' => (int) round($importe * 0.025) + 30,
            'transferencia' => 0,
            // ¿bizum? ¿cripto? -> editar AQUÍ cada vez
        };
    }
}

Cada medio de pago nuevo te obliga a abrir esta clase, añadir una rama y volver a probarla entera. Y normalmente este match no está solo: el mismo «por tipo» se repite para validar, para mostrar el icono, para el informe… Un cambio, muchos sitios.

Cerrarlo con polimorfismo #

Conviertes cada rama en una clase que implementa una interfaz común. El calculador deja de saber de tipos: habla con la abstracción y delega.

php · una abstracción, una clase por variante 
<?php

declare(strict_types=1);

interface MetodoPago
{
    public function comision(int $importe): int;
}

final class Tarjeta implements MetodoPago
{
    public function comision(int $importe): int
    {
        return (int) round($importe * 0.015);
    }
}

final class PayPal implements MetodoPago
{
    public function comision(int $importe): int
    {
        return (int) round($importe * 0.025) + 30;
    }
}
php · el flujo queda cerrado al cambio 
<?php

final class Cobro
{
    public function total(MetodoPago $metodo, int $importe): int
    {
        return $importe + $metodo->comision($importe);
    }
}

Añadir Bizum no toca nada existente

Para soportar Bizum escribes class Bizum implements MetodoPago y la registras donde construyes los objetos. Cobro no cambia, no se reprueba, no se rompe. Esa es la promesa de OCP: el cambio es aditivo.

El patrón estrategia #

Lo que acabas de hacer tiene nombre: el patrón estrategia. Encapsulas cada variante de un algoritmo (aquí, el cálculo de comisión) tras una interfaz común e intercambias la implementación desde fuera. OCP es el principio; estrategia es el patrón que lo materializa más a menudo. Quién decide qué estrategia usar suele ser el contenedor de inyección de dependencias.

El matiz: no abstraer de más #

OCP tienta a crear costuras para ejes que quizá varíen. No lo hagas el primer día. La heurística sana es la regla de los tres: con la primera variante, escribe el caso directo; con la segunda, fíjate en el patrón; con la tercera, abstrae. Predecir qué va a variar es difícil, y una abstracción equivocada estorba más que un if honesto.

No todo switch es un pecado

Un match sobre un enum cerrado y estable (los días de la semana, los estados de un semáforo) está perfectamente bien: el conjunto no va a crecer. OCP ataca el condicional que sabes que crecerá con cada feature, no cualquier ramificación.

Resumen #

Lo que te llevas

  • OCP: añade comportamiento nuevo extendiendo, sin modificar el código estable que ya funciona.
  • El síntoma de violarlo es el condicional por tipo que crece (y se repite) con cada feature.
  • Se cierra con polimorfismo sobre una interfaz (patrón estrategia): cada variante es una clase que se enchufa.
  • No abstraigas de más: regla de los tres. Un match sobre un conjunto cerrado y estable no viola OCP.
