Conceptos de IA: LLM, agente, MCP y RAG

El mundo de la IA para programar llega lleno de siglas: LLM, tokens, tools, agente, MCP, RAG, fine-tuning. Detrás de cada una hay una idea simple. Este es el mapa del territorio antes de adentrarnos: qué es cada pieza y cómo encajan.

Casi todas estas siglas giran alrededor de una sola pieza —el modelo de lenguaje— y describen formas de darle información, manos o memoria. Si entiendes esa pieza central y qué le añade cada capa, el vocabulario deja de intimidar y se vuelve un mapa que sabes leer.

El mapa del ecosistema #

En el centro está el LLM: el modelo que genera texto. Por sí solo, solo habla. Todo lo demás son capas que lo hacen útil para trabajo real: el prompt y el contexto le dan información; las herramientas le dan manos; el agente lo pone a actuar en bucle; MCP estandariza cómo se conecta a tus sistemas; RAG le acerca tu documentación; el fine-tuning reentrena el propio modelo. Vamos pieza a pieza.

El LLM: el cerebro #

Un LLM (Large Language Model, modelo grande de lenguaje) es una máquina de predecir la siguiente palabra. Aprendió de cantidades enormes de texto y, dado lo que tiene delante, calcula la continuación más probable, una y otra vez. De ese mecanismo simple sale todo: responder, razonar, escribir código. Claude, GPT y Gemini son ejemplos de LLM.

El LLM solo es texto que entra y texto que sale

No ejecuta tu código, no consulta tu base de datos, no navega. Eso lo entendemos a fondo en cómo «piensa» un LLM. Lo que importa aquí: todo lo demás del ecosistema existe para superar ese límite —darle datos frescos o capacidad de actuar— sin cambiar el modelo en sí.

Prompt, tokens y contexto #

El prompt es lo que le envías: tu pregunta, las instrucciones, el código que pegas. El modelo no ve letras ni palabras exactas, sino tokens: trozos de texto (a menudo fragmentos de palabra) que son su unidad de medida y de precio.

La ventana de contexto es el máximo de tokens que el modelo puede tener delante a la vez: tu prompt, los ficheros que le das, su respuesta y, en un chat, todo lo dicho antes. Lo que no cabe ahí no existe para él. Por eso «dar contexto» es la habilidad central de trabajar con IA.

Término Qué es
Prompt El texto que envías al modelo: instrucciones, pregunta, datos.
System prompt Instrucciones de fondo que fijan el papel y las reglas (el «quién eres y cómo actúas»).
Token La unidad en que el modelo trocea el texto; se cuenta para límites y coste.
Ventana de contexto El máximo de tokens que el modelo ve de una vez. Lo que no entra, se ignora.

Herramientas: darle manos #

Un LLM solo escribe texto, pero puede pedir que se ejecute una acción. Una herramienta (o tool, también llamada function calling) es una función que tú defines —con un nombre, una descripción y unos parámetros— y que el modelo puede solicitar. El modelo no la ejecuta: dice «llama a buscar_vuelo con estos datos», tu programa la ejecuta y le devuelve el resultado.

Eso convierte al modelo de hablador en hacedor: consultar el tiempo, buscar en una base de datos, enviar un correo. La herramienta es el puente entre el texto que genera el LLM y el mundo real que tú controlas.

1 El modelo pide una herramienta«Llama a buscar_vuelo(Tokio, 2 personas)»
2 Tú la ejecutasLlamas a la API de vuelos de verdad
3 El modelo continúaUsa el resultado para responder

El modelo decide qué herramienta usar; tu código la ejecuta y le pasa la respuesta.

Agente: el modelo en bucle #

Un agente es un LLM envuelto en un bucle con herramientas y autonomía para usarlas. En vez de responder una vez y parar, observa el resultado de cada acción y decide la siguiente: lee un fichero, ejecuta los tests, ve que fallan, corrige, vuelve a probar. Sigue así hasta cumplir el objetivo o quedarse sin saber qué hacer.

La diferencia con un chat

Un chat te da texto que tú trasladas al sistema. Un agente actúa: lee, ejecuta y modifica por su cuenta dentro de los permisos que le des. Claude Code es un agente de código. Entramos a fondo en flujo de trabajo con agentes.

MCP: el estándar para conectar #

Si cada herramienta y cada fuente de datos se conectara al modelo de una forma distinta, integrarlas sería un caos de adaptadores a medida. MCP (Model Context Protocol, protocolo de contexto del modelo) es un estándar abierto —impulsado por Anthropic— que define una forma común de conectar modelos y agentes con herramientas y datos externos.

La analogía habitual es el puerto USB-C: antes cada dispositivo traía su propio cable; con un estándar, cualquier cosa que lo respete encaja. Un servidor MCP expone capacidades (consultar GitHub, leer tu sistema de ficheros, hablar con tu base de datos) de forma estándar; cualquier agente que «hable MCP» puede usarlas sin integración a medida.

Tools vs MCP, sin confusión

Una herramienta es qué puede hacer el modelo (una función concreta). MCP es cómo se le ofrecen esas herramientas y datos de forma estándar y reutilizable. MCP no sustituye a las tools: las empaqueta para que cualquier agente las consuma sin código a medida.

RAG: darle tu información #

El modelo no conoce tu documentación interna ni tus datos privados, y su conocimiento está congelado a una fecha. RAG (Retrieval-Augmented Generation, generación aumentada por recuperación) resuelve esto: antes de responder, busca los fragmentos relevantes en tus documentos y los mete en el contexto para que el modelo conteste basándose en ellos.

Para buscar por significado y no solo por palabras exactas se usan embeddings: representaciones numéricas del texto donde lo que significa cosas parecidas queda «cerca». Guardas esos vectores en una base de datos vectorial y, ante una pregunta, recuperas los trozos más cercanos. Es como darle al modelo el manual abierto por la página correcta justo antes de preguntarle.

1 Pregunta«¿Cómo se factura un cliente moroso?»
2 RecuperaLos fragmentos relevantes de tu documentación
3 El modelo respondeBasándose en tus documentos, no en su memoria

RAG no cambia el modelo: cambia lo que le pones delante antes de preguntar.

Fine-tuning vs contexto #

Hasta aquí, todo lo demás (prompt, RAG, herramientas) deja el modelo intacto y solo cambia lo que recibe. El fine-tuning es distinto: reentrena el propio modelo con tus ejemplos para ajustar su comportamiento o su estilo. Es más caro, más lento y rara vez la primera opción.

Contexto / RAG (empieza aquí)

  • El modelo no cambia; cambias lo que le das.
  • Barato, rápido, fácil de actualizar.
  • Ideal para datos que cambian y conocimiento propio.

Fine-tuning (solo si hace falta)

  • Reentrenas el modelo con tus ejemplos.
  • Caro, lento, hay que rehacerlo al cambiar los datos.
  • Para un estilo o formato muy específico y estable.

La regla práctica

Casi siempre el problema se resuelve dando mejor contexto (prompt, RAG, herramientas), no reentrenando. Reserva el fine-tuning para cuando ya has agotado lo anterior y necesitas un comportamiento muy concreto que el contexto no logra.

Resumen #

Lo que te llevas

  • El LLM es el centro: predice texto. Solo habla; todo lo demás le da datos, manos o memoria.
  • El prompt y la ventana de contexto (medida en tokens) son lo que ve; las herramientas le dan manos; un agente lo pone a actuar en bucle.
  • MCP es el estándar abierto para conectar modelos con herramientas y datos; RAG le acerca tu información buscándola con embeddings.
  • El fine-tuning reentrena el modelo (caro, último recurso); casi siempre basta con dar mejor contexto.
