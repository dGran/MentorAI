Puertos y adaptadores

La arquitectura hexagonal no va de dibujar un hexágono: va de proteger tu lógica de negocio del mundo que la rodea. La base de datos, el framework, las APIs externas son detalles intercambiables; tu dominio no debería enterarse de cuál usas.

Alistair Cockburn la propuso para resolver un dolor concreto: la lógica de negocio acababa enredada con la interfaz de usuario y con la base de datos, imposible de probar y de cambiar sin romperlo todo. La cura es trazar una frontera nítida entre lo que es tuyo y lo que es del exterior.

El problema del acoplamiento #

En una aplicación sin arquitectura, la regla de negocio vive mezclada con el detalle técnico: el controlador HTTP llama al cálculo, que a su vez ejecuta SQL directamente, que devuelve filas que se pintan en la vista. Todo está pegado. Cambiar la base de datos, exponer la misma lógica por una cola en vez de por HTTP, o testear sin levantar la base entera se vuelve un suplicio.

La pregunta que lo ordena todo

¿Qué es esencial de tu aplicación y qué es un detalle? Calcular un precio, validar un pedido: esencial. Que los datos estén en MySQL o que la petición llegue por HTTP: detalles. La hexagonal pone lo esencial en el centro y empuja los detalles a los bordes.

Puertos y adaptadores #

El nombre técnico de la arquitectura hexagonal es justo ese: puertos y adaptadores. La metáfora es la de un aparato electrónico.

  • Un puerto es un punto de conexión definido por el dominio: una interfaz que dice «necesito poder guardar un pedido» o «alguien puede pedirme que registre un usuario». Es un contrato, no una implementación.
  • Un adaptador es lo que se enchufa a ese puerto: el código concreto que cumple el contrato usando una tecnología real (un adaptador de MySQL, uno de Redis, uno de HTTP).

Como un puerto USB: el dominio define la forma del enchufe; cualquier dispositivo que respete esa forma encaja. Puedes cambiar el dispositivo sin tocar el ordenador.

→ Adaptador drivingControlador HTTP, comando CLI, consumidor de cola
◆ Dominio + aplicaciónLa lógica esencial, sin saber quién la llama ni qué hay detrás
→ Adaptador drivenRepositorio MySQL, cliente de la API de pagos, envío de email

El centro no conoce los bordes; los bordes conocen el centro a través de los puertos.

Lado driving y lado driven #

Hay dos tipos de puerto según quién manda a quién:

Lado Quién manda Ejemplos
Driving (primario) El mundo llama al dominio. HTTP, CLI, consumidor de mensajes, tests.
Driven (secundario) El dominio llama al mundo. Base de datos, email, API de pago, reloj.

El puerto driving es la API que tu aplicación ofrece (un caso de uso: «registrar usuario»). El puerto driven es lo que tu aplicación necesita del exterior («guardar este usuario en algún sitio»). La distinción importa porque la dirección de la dependencia se trata distinto en cada lado.

El motor: inversión de dependencias #

Aquí está el truco que hace posible todo. Lo natural sería que el dominio, para guardar un pedido, dependiera de la clase de MySQL. Eso lo ataría a la base de datos. La inversión de dependencias le da la vuelta: el dominio define la interfaz (el puerto) y el adaptador de MySQL la implementa. La flecha de dependencia apunta hacia el dominio, no desde él.

php · el puerto lo define el dominio 
<?php

declare(strict_types=1);

// PUERTO (en el dominio): un contrato, ninguna tecnología.
interface OrderRepository
{
    public function save(Order $order): void;

    public function byId(OrderId $id): ?Order;
}

// El servicio de dominio depende del PUERTO, no de MySQL.
final class PlaceOrder
{
    public function __construct(
        private OrderRepository $orders
    ) {}

    public function handle(Cart $cart): OrderId
    {
        $order = Order::fromCart($cart);
        $this->orders->save($order);

        return $order->id();
    }
}

El adaptador concreto vive fuera del dominio y cumple el contrato:

php · el adaptador implementa el puerto 
<?php

declare(strict_types=1);

// ADAPTADOR (infraestructura): conoce MySQL; el dominio no lo conoce a él.
final class MySqlOrderRepository implements OrderRepository
{
    public function __construct(private PDO $connection) {}

    public function save(Order $order): void
    {
        // ... INSERT / UPDATE con consultas preparadas
    }

    public function byId(OrderId $id): ?Order
    {
        // ... SELECT y reconstrucción del objeto de dominio
        return null;
    }
}

Esto es DIP en acción

Es exactamente la inversión de dependencias que viste en inyección de dependencias: depender de abstracciones, no de concreciones. La hexagonal lleva esa idea al nivel de la arquitectura entera, no de una clase suelta.

El dominio en el centro #

El resultado es un dominio que no importa nada del exterior: ni el ORM, ni el framework, ni el protocolo. Eso permite probarlo con un adaptador en memoria (un fake), sin base de datos, en milisegundos. Y permite añadir un segundo adaptador driving —exponer la misma lógica por una cola además de por HTTP— sin tocar el centro.

La regla de oro: las flechas hacia dentro

Una sola regla resume la disciplina: el código de dentro (dominio) nunca importa código de fuera (infraestructura). Si tu entidad Order tiene un use del ORM o del framework HTTP, la frontera está rota. Las dependencias siempre apuntan hacia el centro.

Qué ganas y qué cuesta #

Ganas

  • Dominio testeable sin infraestructura.
  • Cambiar de tecnología sin tocar el negocio.
  • Varias entradas (HTTP, CLI, cola) a la misma lógica.
  • Un código que expresa el negocio, no la fontanería.

Cuesta

  • Más capas, más interfaces, más ficheros.
  • Mapear entre objetos de dominio y de persistencia.
  • Sobra para un CRUD simple sin reglas.
  • Curva de aprendizaje para el equipo.

La hexagonal rinde cuando hay lógica de negocio que proteger y vida por delante. Para un CRUD sin reglas, sus capas son burocracia. Resumen #

Lo que te llevas

  • La hexagonal separa lo esencial (negocio) de los detalles (base de datos, framework, protocolo).
  • Un puerto es un contrato del dominio; un adaptador lo implementa con una tecnología concreta.
  • Lado driving (el mundo llama al dominio) y driven (el dominio llama al mundo). El motor es la inversión de dependencias.
  • Regla de oro: las dependencias apuntan hacia dentro. El dominio nunca importa infraestructura.
