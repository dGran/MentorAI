L — Que un subtipo no traicione al base

Si tu código funciona con un tipo, debería seguir funcionando con cualquiera de sus subtipos, sin enterarse del cambiazo. Cuando una subclase rompe esa promesa —lanza donde no se espera, hace menos de lo prometido— el polimorfismo se vuelve una trampa. Liskov pone el listón.

Barbara Liskov lo formuló en 1987 con rigor matemático, pero la versión de andar por casa basta: si S es un subtipo de T, debes poder usar un S donde se espera un T sin que el programa se entere ni se rompa. El contrato del tipo base es una promesa, y un subtipo que lo incumple es una herencia que miente.

La promesa del subtipo #

LSP es la regla que hace que el polimorfismo sea fiable. Sin ella, cada vez que recibes un tipo base tendrías que preguntarte «¿y si es esta subclase rara que se comporta distinto?», y volverías a los instanceof que tanto trabajo costó eliminar. LSP exige que todas las implementaciones respeten el mismo contrato de comportamiento, no solo la misma firma.

Cuadrado y rectángulo #

El ejemplo de manual, porque revienta una intuición. En geometría un cuadrado es un rectángulo. Tentador: Cuadrado extends Rectangulo. Pero el rectángulo promete que ancho y alto se mueven por separado; el cuadrado, para mantenerse cuadrado, fuerza que cambiar uno cambie el otro. Esa diferencia rompe a quien confiaba en la promesa del rectángulo.

php · la subclase rompe la expectativa del base 
<?php

declare(strict_types=1);

class Rectangulo
{
    public function __construct(protected int $ancho, protected int $alto)
    {
    }

    public function setAncho(int $ancho): void { $this->ancho = $ancho; }
    public function setAlto(int $alto): void { $this->alto = $alto; }
    public function area(): int { return $this->ancho * $this->alto; }
}

class Cuadrado extends Rectangulo
{
    public function setAncho(int $ancho): void
    {
        $this->ancho = $ancho;
        $this->alto = $ancho;   // efecto sorpresa para quien usa un Rectangulo
    }

    public function setAlto(int $alto): void
    {
        $this->ancho = $alto;
        $this->alto = $alto;
    }
}
php · el cliente que confiaba en el contrato falla 
<?php

function redimensionar(Rectangulo $r): void
{
    $r->setAncho(5);
    $r->setAlto(4);
    assert($r->area() === 20);   // cierto para Rectangulo, FALSO para Cuadrado (16)
}

redimensionar() es correcta para todo rectángulo… hasta que le pasas un cuadrado. La firma encaja (un cuadrado es un rectángulo a ojos del compilador), pero el comportamiento traiciona la expectativa. Eso es violar LSP.

Precondiciones y postcondiciones #

LSP se concreta en reglas sobre el contrato de los métodos al sobrescribir:

Regla Qué significa
No endurecer precondiciones El subtipo no puede exigir más que el base (p. ej. rechazar entradas que el base aceptaba).
No debilitar postcondiciones El subtipo no puede prometer menos (devolver algo más pobre, o no cumplir lo garantizado).
Respetar invariantes Las verdades que siempre se cumplen en el base deben seguir cumpliéndose.
No lanzar excepciones nuevas El subtipo no debería lanzar errores que el contrato del base no contempla.

La herencia que miente #

El caso más común en la práctica no es geométrico: es la subclase que «casi» encaja y resuelve el hueco lanzando una excepción. El típico PatoDeGoma extends Pato que en volar() lanza NoPuedeVolar. Eso rompe a quien tenía un Pato y esperaba poder llamar a volar() sin sorpresas.

php · el síntoma: sobrescribir para no hacer nada (o lanzar) 
<?php

class CuentaAhorro
{
    public function retirar(int $importe): void { /* ... */ }
}

class CuentaBloqueada extends CuentaAhorro
{
    public function retirar(int $importe): void
    {
        throw new OperacionNoPermitida();   // endurece la precondición: viola LSP
    }
}

El olor: comprobar el tipo antes de usar

Si tu código empieza a hacer if (!$cuenta instanceof CuentaBloqueada) antes de llamar a un método, es señal de que la jerarquía viola LSP: los subtipos ya no son intercambiables y el cliente tiene que conocerlos. La abstracción está mintiendo.

La salida #

Cuando un subtipo no puede cumplir el contrato, la herencia es el modelo equivocado. Las salidas habituales:

Forzar la herencia

  • Cuadrado extends Rectangulo con efectos raros.
  • Subclases que lanzan en métodos heredados.
  • El cliente acaba comprobando el tipo.

Modelar honestamente

  • Tipos hermanos bajo una interfaz común (Forma), sin herencia entre ellos.
  • Inmutabilidad: un Cuadrado sin setters no tiene el problema.
  • Segregar capacidades: solo lo que vuela implementa Volador.

LSP y la segregación van de la mano

Muchas violaciones de Liskov se evaporan al partir una interfaz gorda en capacidades pequeñas: si Volador es su propia interfaz, el pato de goma simplemente no la implementa, y nadie le pide volar(). Esa es justo la siguiente letra, la segregación de interfaces.

Resumen #

Lo que te llevas

  • LSP: un subtipo debe poder sustituir al base sin que el cliente se entere ni se rompa.
  • No basta con que la firma encaje: el comportamiento debe respetar el contrato (cuadrado/rectángulo).
  • No endurezcas precondiciones ni debilites postcondiciones; no lances excepciones que el base no contempla.
  • Si el cliente comprueba el tipo antes de usar, la jerarquía miente: modela con interfaces hermanas o inmutabilidad.
