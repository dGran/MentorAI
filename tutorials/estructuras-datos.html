Estructuras de datos: cuál usar y cuándo

No hay una estructura "mejor": hay una que hace barata la operación que tú repites y cara el resto. Elegir bien es casi todo el rendimiento, y se reduce a una pregunta: ¿qué voy a hacer con estos datos mil veces?

Una estructura de datos es una forma de organizar valores en memoria que hace ciertas operaciones rápidas a cambio de otras lentas. Conocer ese canje —qué abarata y qué encarece cada una— es lo que convierte el Big-O del tutorial anterior en decisiones concretas.

Elegir por la operación que domina #

La pregunta no es "¿cuál es la mejor estructura?", sino "¿qué operación voy a hacer muchas veces?". ¿Accedo por una posición concreta? ¿Busco por un identificador? ¿Necesito que estén ordenados? ¿Inserto y borro por el medio constantemente? Cada respuesta señala una estructura distinta, y la equivocada convierte un bucle inocente en O(n²).

Array: acceso por índice #

El array clásico guarda los elementos en memoria contigua, uno detrás de otro. Eso hace que acceder a la posición i sea O(1): la dirección se calcula con una multiplicación. El precio lo pagas al insertar o borrar por el medio: hay que desplazar todo lo que viene detrás, O(n).

  • Barato: acceso por índice, recorrer en orden, añadir al final.
  • Caro: insertar o borrar por el medio; buscar un valor (hay que recorrer).

Lista enlazada: insertar y borrar #

La lista enlazada invierte el canje: cada elemento es un nodo que guarda su valor y un puntero al siguiente. No están contiguos, así que no hay acceso por índice barato (para llegar al elemento 500 recorres 500 punteros, O(n)). A cambio, insertar o borrar es O(1)… si ya tienes el nodo: solo reenlazas punteros, sin mover nada.

En la práctica, el array suele ganar igualmente

El Big-O dice que la lista enlazada inserta mejor, pero la realidad tiene una palabra: la localidad de caché. Un array contiguo se lee del caché de la CPU a ráfagas; una lista enlazada salta por la memoria persiguiendo punteros y falla el caché una y otra vez. Para la mayoría de cargas, el array es más rápido aunque la teoría diga lo contrario. La memoria manda, y eso es un tutorial aparte.

Hashmap y set: buscar por clave #

El hashmap (mapa, diccionario, tabla hash) asocia claves a valores y los busca en O(1) de media: una función de hash convierte la clave en una posición directa, sin recorrer nada. Es la estructura que vuelve a O(n) tantos bucles O(n²). Su pariente es el set (conjunto): un mapa que solo guarda claves, perfecto para unicidad y para preguntar "¿está esto?" en tiempo constante.

php · array como lista y como mapa 
<?php

declare(strict_types=1);

// Como lista: índices 0,1,2... acceso por posición O(1)
$list = ['rojo', 'verde', 'azul'];
echo $list[1];                       // "verde"

// Como mapa: clave -> valor, acceso por clave O(1)
$ages = ['ana' => 30, 'leo' => 25];
echo $ages['ana'];                   // 30
php · set: unicidad y pertenencia O(1) 
<?php

declare(strict_types=1);

$tags = ['php', 'sql', 'php', 'redis', 'sql'];

// Las claves no se repiten: el array se vuelve un set
$set = array_flip($tags);

$hasRedis = isset($set['redis']);    // true, en O(1)
$unique = array_keys($set);          // ['php', 'sql', 'redis']

El "array" de PHP no es un array

Lo que PHP llama array es en realidad un mapa ordenado: por dentro es una tabla hash que además recuerda el orden de inserción. Por eso sirve a la vez de lista, de mapa, de set, de pila y de cola. Cómodo, pero engañoso: si esperas el acceso contiguo de un array clásico, su coste de memoria y su comportamiento te sorprenderán. Una clave numérica no garantiza contigüidad.

Árbol: mantener el orden #

El hashmap es imbatible buscando por clave exacta, pero no sabe nada de orden: no puede darte "todos los valores entre 100 y 200" ni "el siguiente más grande". Para eso está el árbol de búsqueda balanceado: mantiene los elementos ordenados y permite buscar, insertar y borrar en O(log n), además de recorrer rangos de forma eficiente.

Esa propiedad —orden + O(log n)— es exactamente la que usan los índices de las bases de datos (un B-tree, primo de estos árboles) para resolver tanto WHERE id = 42 como WHERE fecha BETWEEN ... sin leer toda la tabla. Es el puente directo con el pilar de bases de datos.

Grafo: relaciones #

Cuando lo que importa no son los elementos sino cómo se conectan entre sí, la estructura es el grafo: nodos (vértices) unidos por relaciones (aristas). Amigos en una red social, dependencias entre tareas, rutas entre ciudades, permisos que heredan de otros: todo eso son grafos. La forma habitual de guardarlos en código es la lista de adyacencia, un mapa de cada nodo a sus vecinos.

php · grafo como lista de adyacencia 
<?php

declare(strict_types=1);

// Cada nodo apunta a la lista de sus vecinos
$graph = [
    'a' => ['b', 'c'],
    'b' => ['d'],
    'c' => ['d'],
    'd' => [],
];

foreach ($graph['a'] as $neighbor) {
    echo $neighbor;                  // vecinos directos de 'a': b, c
}

La chuleta de costes #

Todo lo anterior cabe en una tabla. Es la que conviene tener en la cabeza al diseñar:

Regla de bolsillo

¿Buscas por una clave? Hashmap. ¿Necesitas orden o rangos? Árbol. ¿Solo recorres y accedes por posición? Array. ¿Te importan las conexiones? Grafo. La lista enlazada pura, casi nunca: el array de tu lenguaje ya cubre ese hueco mejor.

Errores comunes #

ErrorPor qué duele
Buscar valores recorriendo un array in_array repetido es O(n) cada vez. Si buscas mucho, indexa por clave en un mapa.
Usar un array para garantizar unicidad a mano Comprobar "¿ya está?" recorriendo es O(n²) en total. Un set lo hace en O(1) por elemento.
Pedirle orden a un hashmap El hash no conserva el orden por valor. Si necesitas rangos u ordenación, es un árbol (o un ORDER BY en BBDD).
Modelar relaciones con tablas planas "Amigos de amigos" o dependencias encadenadas piden un grafo; forzarlo en filas lleva a consultas imposibles.
Elegir por el Big-O ignorando el caché La lista enlazada "gana" en teoría y pierde en la práctica por localidad. Mide con datos reales.

Resumen #

Lo que te llevas

  • No hay estructura "mejor": elige por la operación que repites mil veces. Cada una abarata unas cosas y encarece otras.
  • Array: acceso por índice O(1), pero buscar e insertar por el medio es O(n).
  • Hashmap / set: buscar por clave y comprobar pertenencia en O(1); sin orden. La herramienta contra el O(n²).
  • Árbol: orden y rangos en O(log n) (es lo que hace un índice). Grafo: cuando importan las relaciones.
  • El array de PHP es un mapa ordenado, no un array contiguo: hace de todo, pero no des por hecha su contigüidad ni su coste.
