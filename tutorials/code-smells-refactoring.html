Code smells y refactoring seguro

Un code smell es un olorcillo: no es un bug, pero avisa de que algo se está pudriendo en el diseño. Aprender a olerlos es media batalla; la otra media es saber refactorizar sin romper nada, apoyado en una red de tests. Aquí juntas las dos cosas y cierras el curso.

El término lo popularizó Kent Beck con Martin Fowler en Refactoring. Un code smell es una característica del código que sugiere un problema más profundo. No te obliga a actuar —algunos olores son tolerables—, pero te dice dónde mirar. Y a cada olor le corresponde, casi siempre, un refactor conocido que lo quita.

Qué es un code smell #

Es una pista, no un veredicto. Un método de cincuenta líneas probablemente hace demasiado, pero quizá esté bien así. El valor del catálogo es darte un vocabulario compartido —«esto huele a feature envy»— y un mapa de qué refactor aplicar. Con la práctica, los hueles antes de poder explicarlos.

Catálogo de olores #

Smell Qué es Suele pedir
Código duplicado La misma lógica en varios sitios. El olor número uno. Extraer método/clase
Método largo Una función que hace varias cosas y mezcla niveles. Extraer método
Clase grande Una god class con demasiadas responsabilidades. Extraer clase
Lista de parámetros larga Cuatro o más argumentos que viajan juntos. Agrupar en un objeto
Feature envy Un método que usa más datos de otra clase que de la suya. Mover el método
Obsesión por primitivos Strings e ints sueltos para conceptos (un email como string). Crear un value object
Switch por tipo El mismo match sobre un tipo, repetido. Reemplazar por polimorfismo

Los olores apuntan a los principios

No es casualidad que estos olores remitan a lo que ya viste: la clase grande viola la S, el switch por tipo viola la O, la obsesión por primitivos la cura un value object. Los smells son la cara visible de los principios incumplidos.

Qué es refactorizar #

Refactorizar es cambiar la estructura interna del código sin cambiar su comportamiento externo. Es la palabra clave: el comportamiento se queda igual, solo mejora la forma. Por eso refactorizar y añadir features son dos sombreros distintos que no se llevan a la vez: o cambias la forma manteniendo el comportamiento, o cambias el comportamiento; nunca ambos en el mismo paso.

Refactors comunes #

El más frecuente con diferencia es extraer método: coger un trozo de una función larga, ponerle nombre y sacarlo. Quita duplicación, acorta el método y a menudo el nombre nuevo documenta lo que antes pedía un comentario.

El más vistoso es reemplazar condicional por polimorfismo, que ya viste en OCP: el switch por tipo se convierte en una jerarquía de clases bajo una interfaz.

php · extraer método: de duplicado a reutilizado 
<?php

// Antes: el cálculo del IVA, duplicado y sin nombre
$totalA = $a->base + $a->base * 0.21;
$totalB = $b->base + $b->base * 0.21;

// Después: un método con nombre, una sola verdad
private function conIva(int $base): int
{
    return (int) round($base * (1 + self::TIPO_IVA));
}

$totalA = $this->conIva($a->base);
$totalB = $this->conIva($b->base);

Refactorizar con seguridad #

Aquí está la regla que lo hace viable: no refactorices sin tests. Si cambias la estructura sin una red que confirme que el comportamiento sigue igual, no estás refactorizando, estás reescribiendo a ciegas y rezando. Los tests son lo que convierte el refactoring en una operación segura y reversible.

1 Tests en verdela red está puesta
2 Un refactorextraer, renombrar, mover
3 Tests en verde otra vezrepite

Pasos pequeños, tests en verde entre cada uno. Si uno se pone rojo, deshaces solo ese paso.

Pasos pequeños, no la gran reescritura

El error clásico es parar el mundo para «reescribirlo bien de una vez»: meses sin entregar y un riesgo enorme. El refactoring sano es continuo y en pasos diminutos —la regla del boy scout—, con los tests en verde todo el rato. Mejora el código mientras entregas, no en vez de entregar.

Resumen #

Lo que te llevas

  • Un code smell es una pista, no un veredicto: te da vocabulario y te dice dónde mirar.
  • El catálogo (duplicación, método largo, clase grande, feature envy, obsesión por primitivos…) es la cara visible de los principios incumplidos.
  • Refactorizar es cambiar la forma sin cambiar el comportamiento; extraer método y reemplazar condicional por polimorfismo son los más útiles.
  • No refactorices sin tests: pasos pequeños, en verde entre cada uno, de forma continua y no como gran reescritura.
