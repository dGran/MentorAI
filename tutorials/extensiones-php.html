Extensiones de PHP: qué son y cuáles importan

PHP "de fábrica" sabe hacer sorprendentemente poco. Conectar con MySQL, manejar UTF-8, hablar HTTPS o leer una imagen no viene en el lenguaje: viene en extensiones. Veamos qué son, cómo se cargan y cuáles necesitas de verdad antes de desplegar nada.

Piensa en PHP como un móvil recién comprado: el sistema funciona, pero sin apps no hace gran cosa. Las extensiones son esas apps. El núcleo del lenguaje sabe ejecutar bucles, funciones y clases; todo lo demás (bases de datos, criptografía, imágenes, fechas internacionales) llega como piezas que se enchufan al motor.

Qué es una extensión #

Una extensión es un módulo escrito en C que añade funciones, clases o constantes al intérprete de PHP. No es código PHP: es una librería compilada (.so en Linux, .dll en Windows) que el motor carga al arrancar y que pasa a formar parte del lenguaje como si siempre hubiera estado ahí.

Por eso cuando llamas a json_encode(), mb_strlen() o curl_init() no estás usando "PHP a secas": estás usando las extensiones json, mbstring y curl. Están tan integradas que parecen parte del lenguaje, pero técnicamente son módulos aparte.

¿Por qué en C y no en PHP?

Porque una extensión necesita hablar directamente con el sistema operativo (sockets, memoria, librerías como OpenSSL o libcurl) y correr a máxima velocidad. Implementar el cifrado o el parseo de imágenes en PHP puro sería lento y tedioso; en C es rápido y reutiliza librerías que ya existen.

Compiladas en el core vs de PECL #

No todas las extensiones llegan igual. Hay dos grandes familias, y la diferencia importa a la hora de instalarlas:

Del core (bundled)

  • Vienen con el código fuente de PHP
  • Muchas se activan al compilar PHP
  • json, pdo, mbstring, opcache…
  • En distros, suelen ir en paquetes oficiales

De PECL

  • Repositorio externo de extensiones de la comunidad
  • Se instalan aparte (pecl install)
  • redis, imagick, xdebug, apcu…
  • No las trae PHP; las añades tú

PECL (PHP Extension Community Library) es el repositorio donde vive el código C de las extensiones que no se distribuyen con el core. En la práctica, en una distro moderna casi nunca compilas a mano: instalas un paquete del sistema (php8.3-redis) que por debajo es esa extensión de PECL ya compilada para tu versión de PHP.

Cómo se cargan: el php.ini #

Una extensión instalada no sirve de nada hasta que PHP la carga. Eso se controla con una línea en la configuración:

ini · 20-redis.ini 
; Carga la extensión por su nombre
extension=redis

; Las de tipo Zend (cambian el motor) usan otra directiva
zend_extension=opcache

Fíjate en el detalle: opcache y xdebug se cargan con zend_extension, no con extension. Son extensiones de Zend: no solo añaden funciones, sino que se enganchan en el motor de ejecución para alterar cómo corre el código (cachear opcodes, depurar paso a paso). El resto, las normales, usan extension=.

El patrón conf.d/

En distros tipo Debian/Ubuntu no se toca un php.ini gigante: cada extensión trae su propio ficherito en conf.d/ (20-redis.ini). El número del principio fija el orden de carga: si una extensión depende de otra (PDO antes que pdo_mysql), el número menor se carga primero.

Ver qué extensiones tienes activas #

Antes de pelearte con un "función no definida", comprueba qué hay cargado de verdad. La forma más directa desde consola:

bash 
# Lista todas las extensiones cargadas
php -m

# ¿Está cargada una concreta?
php -m | grep -i redis

# Versión y config de una extensión
php --ri opcache

CLI y FPM no comparten configuración

php -m te muestra lo que ve la línea de comandos. PHP-FPM (el que sirve la web) usa su propio php.ini y su propio conf.d/. Una extensión puede estar en CLI y faltar en FPM. Para ver lo que carga la web de verdad, usa una página con phpinfo() o revisa la config del pool de FPM.

Las imprescindibles y para qué sirven #

Hay un puñado que aparecen en casi cualquier proyecto serio. Si falta alguna, lo normal es que un framework ni arranque. Estas son las que de verdad conviene conocer:

Extensión Para qué Si falta…
opcache Cachea el código compilado en memoria. Todo va mucho más lento.
pdo + pdo_mysql Acceso a base de datos con una API uniforme. No te conectas a MySQL/MariaDB.
mbstring Cadenas multibyte (UTF-8) bien contadas y cortadas. Acentos y emojis se rompen.
intl Fechas, monedas y ordenación internacionales. Formatos y NumberFormatter fallan.
curl Cliente HTTP para hablar con otras APIs. No haces peticiones salientes.
json Codificar/decodificar JSON. (Ya va integrada desde PHP 8.)
openssl TLS, cifrado y firmas. Sin HTTPS ni cifrado seguro.
gd / imagick Crear y manipular imágenes. Sin thumbnails ni recortes.
redis Cliente del servidor Redis (caché, colas). No usas Redis desde PHP.
sodium Criptografía moderna (la recomendada hoy). Sin cifrado/hash modernos a mano.
bcmath Aritmética de precisión arbitraria. Cuentas con dinero pierden precisión.

¿gd o imagick?

Las dos manipulan imágenes. gd va incluida y cubre lo básico (redimensionar, recortar). imagick envuelve ImageMagick: más formatos y más potencia, pero más pesada. Para thumbnails sencillos, gd sobra; para PDF, TIFF o efectos serios, imagick.

Importante: php-fpm NO es una extensión #

Es una confusión habitual, así que vamos a deshacerla. Una extensión añade capacidades al lenguaje (funciones, clases). PHP-FPM no añade funciones: es la interfaz por la que PHP corre detrás de un servidor web. En la jerga, es un SAPI (Server API), no una extensión.

Extensión (p. ej. curl)

  • Te da funciones nuevas
  • Se carga con extension=
  • Aparece en php -m
  • Responde a la pregunta "¿qué sé hacer?"

SAPI (php-fpm, CLI, mod_php)

  • Define cómo arranca PHP
  • No añade funciones de usuario
  • Es un proceso/binario, no un módulo
  • Responde a "¿quién me ejecuta?"

Lo dejamos aquí solo para quitar la confusión; los SAPIs y PHP-FPM en detalle (pools, pm.max_children, modelo shared-nothing) tienen su propio tutorial. Quédate con la frase: las extensiones son lo que PHP sabe hacer; el SAPI es cómo se arranca.

Instalar y habilitar una extensión #

En una distro Debian/Ubuntu con los paquetes oficiales de PHP, el camino corto es instalar el paquete del sistema: trae la extensión ya compilada para tu versión y deja el .ini en conf.d/ por ti.

bash 
# Paquete del sistema (recomendado)
sudo apt install php8.3-redis php8.3-intl

# Recarga PHP-FPM para que cargue lo nuevo
sudo systemctl reload php8.3-fpm

# Comprueba que aparece
php -m | grep -iE "redis|intl"

En un contenedor Docker con la imagen oficial de PHP el patrón es distinto: existen unos helpers para compilar extensiones del core y para instalar las de PECL.

bash · Dockerfile 
# Extensiones del core: configurar (si hace falta) e instalar
docker-php-ext-install pdo_mysql intl bcmath

# Extensiones de PECL: instalar y luego habilitar
pecl install redis
docker-php-ext-enable redis

"Instalé la extensión y sigue sin funcionar"

Casi siempre es una de tres: (1) la instalaste para una versión de PHP distinta de la que corre; (2) la habilitaste en CLI pero no en FPM; o (3) no recargaste FPM tras añadirla. Comprueba la versión (php -v), revisa el conf.d/ correcto y recarga el servicio.

Resumen #

Lo que te llevas

  • Una extensión es un módulo en C que añade funciones y clases al motor; gran parte de "lo que sabe PHP" son extensiones.
  • Las hay del core (vienen con PHP) y de PECL (comunidad, se instalan aparte). Se cargan con extension= o zend_extension=.
  • Comprueba lo cargado con php -m, pero recuerda que CLI y FPM tienen config distinta.
  • Las de andar por casa: opcache, pdo_mysql, mbstring, intl, curl, openssl, redis.
  • PHP-FPM no es una extensión, es un SAPI: define cómo se ejecuta PHP, no qué funciones tiene.
