Object Calisthenics: 9 reglas para mejor diseño OO

Nueve reglas deliberadamente extremas que, aplicadas como ejercicio, reeducan tus dedos para escribir objetos de verdad: pequeños, cohesionados y sin lógica desperdigada. No son ley para producción; son gimnasio para tu diseño orientado a objetos.

Las propuso Jeff Bay en The ThoughtWorks Anthology: nueve restricciones para aplicar sobre un programa de unas mil líneas, todas a la vez. Suenan absurdas por separado, y esa es la gracia. Al prohibirte los atajos que te alejan de la orientación a objetos, te empujan a descubrir el diseño que de otro modo no buscarías.

Qué es Object Calisthenics #

Calistenia son esos ejercicios con el propio peso del cuerpo: flexiones, sentadillas. Object Calisthenics es lo mismo para el código: rutinas de diseño que fortalecen un músculo concreto, el de programar con objetos en vez de con estructuras de datos disfrazadas de clases. No buscas que el resultado sea «bonito», buscas que el gesto se te quede.

Son nueve reglas. La clave es aplicarlas juntas y al extremo en un ejercicio acotado, no negociarlas una a una. Cuando las nueve aprietan a la vez, el código no tiene más remedio que partirse en objetos pequeños con responsabilidades claras. Las agrupamos en cuatro familias para entender qué cura cada una.

El enemigo a batir: el modelo anémico

Muchas «clases» son sacos de datos con getters y setters que otro código manipula desde fuera. Eso es programación procedural con sintaxis de objetos. Object Calisthenics ataca justo ese hábito: que los datos y el comportamiento que los gobierna vivan juntos.

Reglas de flujo #

Las dos primeras atacan los métodos enredados, llenos de ramas y niveles.

  • 1. Un solo nivel de indentación por método. Si tienes un bucle con un if dentro, ya vas por dos niveles. La regla te obliga a extraer el interior a un método con nombre. El efecto: métodos diminutos que hacen una cosa y se leen como una frase.
  • 2. No uses la palabra else. El else esconde complejidad y crea caminos paralelos. Sustitúyelo por early return (sal pronto del método) o, en casos de negocio, por polimorfismo. El código pasa a leerse como una lista de guardas, no como un árbol.

Con anidación y else

  • Bucle → if → else → otro if.
  • Cuesta seguir qué pasa en cada rama.
  • El método hace de todo.

Plano, con early return

  • Guardas arriba, salida temprana.
  • El cuerpo principal sin sangrías profundas.
  • Cada trozo extraído tiene nombre.
php · sin else, con early return 
<?php

declare(strict_types=1);

public function priceFor(Customer $customer): Money
{
    if ($customer->isVip()) {
        return $this->base->withDiscount(20);
    }

    if ($customer->isNew()) {
        return $this->base->withDiscount(5);
    }

    return $this->base;
}

Reglas de tipos #

Las dos siguientes combaten las primitivas sueltas y las colecciones desnudas que arrastran lógica por todas partes.

  • 3. Envuelve todos los primitivos y strings (cuando tengan comportamiento o reglas). Un importe no es un int, es un Money. Un email no es un string, es un Email que se valida al nacer. Es el patrón value object llevado al hábito.
  • 4. Colecciones de primera clase. Si una clase tiene un array, ese array es su única propiedad y merece su propia clase. Un OrderLines en vez de un array $lines: ahí viven el total, el filtrado y las reglas de la colección, en vez de esparcidas por quien la use.
php · colección de primera clase 
<?php

declare(strict_types=1);

final class OrderLines
{
    /** @var OrderLine[] */
    private array $lines;

    public function __construct(OrderLine ...$lines)
    {
        $this->lines = $lines;
    }

    public function total(): Money
    {
        return array_reduce(
            $this->lines,
            fn (Money $sum, OrderLine $line) => $sum->add($line->subtotal()),
            Money::zero()
        );
    }
}

El puente con DDD

Las reglas 3 y 4 son, literalmente, los value objects y las colecciones de dominio que viste en DDD táctico. Object Calisthenics es un buen calentamiento para llegar a DDD: te acostumbra a no dejar primitivas sueltas mucho antes de hablar de agregados.

Reglas de acoplamiento #

Estas dos cortan las cadenas que atan unos objetos a las tripas de otros.

  • 5. Un punto por línea (la ley de Demeter, «no hables con extraños»). $pedido->cliente()->direccion()->ciudad() te ata a tres niveles de estructura ajena; si cambia cualquiera, te rompes. La regla te empuja a pedirle al objeto lo que necesitas, no a navegar por sus entrañas.
  • 9. No uses getters, setters ni properties públicas. La más radical. En vez de $cuenta->setSaldo(...), di $cuenta->ingresar($importe). Cuando no puedes leer ni escribir el estado desde fuera, te ves obligado a mover el comportamiento dentro del objeto. Es la cura directa del modelo anémico.

«Tell, don't ask»

Las reglas 5 y 9 son dos caras de la misma máxima: dile al objeto qué hacer, no le preguntes por sus datos para decidir tú. Un getter invita a sacar el estado y operar fuera; un método con intención (confirmar(), cancelar()) mantiene la decisión donde viven los datos.

Reglas de tamaño #

Las tres restantes mantienen todo pequeño y nombrado con honestidad.

Regla Qué te obliga a hacer
6. No abrevies Si un nombre te pide abreviarse, suele ser que la clase o el método hacen demasiado. Nombres completos y semánticos delatan responsabilidades hinchadas.
7. Mantén pequeñas las entidades Clases de pocas líneas, paquetes de pocas clases. Lo grande se parte; lo que no se deja partir es que estaba mal modelado.
8. No más de dos variables de instancia La más dura junto a la 9. Te fuerza a agrupar datos que van juntos en objetos propios, descubriendo conceptos que estaban escondidos.

La regla 8 parece imposible —¿una clase con solo dos campos?— y por eso enseña tanto: cuando un objeto tiene cinco propiedades, casi siempre hay dos o tres conceptos pidiendo salir. Forzar el límite los saca a la luz.

Es un ejercicio, no dogma #

Aquí está el matiz que evita malentendidos: no se trata de cumplir las nueve reglas en tu código de producción. Algunas (la 8, la 9 al extremo) producirían una explosión de clases que en un proyecto real no compensa. El valor está en practicarlas hasta que el instinto que entrenan se te quede.

No lo conviertas en una regla de linter

Imponer «cero getters» o «máximo dos variables» como check obligatorio en un equipo produce código retorcido para contentar la norma. Object Calisthenics es una kata: la haces para aprender el movimiento, no para vivir dentro de ella. En producción te quedas con el criterio, no con la restricción literal.

Cómo practicarlo: coge un trozo de código tuyo de ~1000 líneas, o un ejercicio pequeño desde cero, y aplícalas todas a la vez. Vas a sufrir, y ese sufrimiento es la lección: cada vez que una regla te bloquea, te está señalando un punto donde tu diseño se apoyaba en un atajo procedural. Lo que te llevas a tu trabajo real es ese olfato, no las nueve reglas literales.

Resumen #

Lo que te llevas

  • Object Calisthenics son 9 reglas extremas para entrenar el diseño OO; su enemigo es el modelo anémico.
  • Flujo (un nivel de indentación, sin else) y tipos (envolver primitivos, colecciones de primera clase) parten los métodos y dan nombre a los datos.
  • Acoplamiento (un punto por línea, sin getters/setters) mueve el comportamiento dentro del objeto: «tell, don't ask».
  • Es una kata, no una norma de producción: practícalas todas a la vez para quedarte con el criterio, no con la restricción literal.
