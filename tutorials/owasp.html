OWASP imprescindible: inyección, XSS y CSRF

La mayoría de las brechas web no usan magia: explotan tres o cuatro errores que se repiten desde hace décadas. Si entiendes inyección, XSS y CSRF —los más frecuentes del Top 10 de OWASP— cierras la puerta a la mayor parte de los ataques reales contra una aplicación.

Casi todos los ataques web famosos se reducen a la misma jugada: colar instrucciones donde el programa esperaba datos. El sistema confunde «lo que el usuario escribió» con «lo que debe ejecutar». Una vez ves ese patrón, las tres vulnerabilidades estrella dejan de ser magia y se vuelven la misma idea con tres disfraces.

Qué es el OWASP Top 10 #

OWASP (Open Worldwide Application Security Project) es una organización sin ánimo de lucro que publica, entre otras cosas, el Top 10: una lista de las categorías de riesgo más críticas en aplicaciones web, actualizada cada pocos años a partir de datos reales. No es una norma legal, pero es el punto de partida que todo el sector usa.

No vamos a recorrer las diez. Vamos a las tres que más vas a tocar como backend y que comparten una misma raíz: inyección, XSS y CSRF.

La raíz común: confiar en la entrada #

La regla de oro de la seguridad web cabe en una frase: nunca confíes en datos que vienen de fuera. Parámetros de la URL, campos de un formulario, cabeceras, cookies, respuestas de otra API: todo eso lo controla potencialmente un atacante. El fallo aparece cuando esos datos se mezclan, sin separación, con código que se ejecuta o se interpreta.

Datos vs instrucciones

Inyección SQL y XSS son la misma enfermedad en distintos órganos: el dato del usuario acaba interpretándose como instrucción (SQL en la base de datos, HTML/JS en el navegador). La cura siempre es la misma idea: mantener separados los datos del código.

Inyección SQL #

Ocurre cuando construyes una consulta SQL pegando texto del usuario directamente en la cadena. El atacante escribe algo que no es un dato, sino más SQL, y tu base de datos lo ejecuta como propio.

php · vulnerable 
<?php

$email = $_POST['email'];

$sql = "SELECT * FROM users WHERE email = '" . $email . "'";
$database->query($sql);

// Si el atacante manda:  ' OR '1'='1
// la consulta pasa a ser:
// SELECT * FROM users WHERE email = '' OR '1'='1'
// ...que devuelve TODOS los usuarios.

La defensa no es «filtrar comillas»: es no construir nunca la consulta por concatenación. Con consultas preparadas, el SQL y los datos viajan por canales separados; el motor trata el parámetro siempre como un valor, jamás como instrucción.

php · seguro (consulta preparada) 
<?php

declare(strict_types=1);

$statement = $database->prepare(
    'SELECT * FROM users WHERE email = ?'
);

$statement->execute([$_POST['email']]);

$user = $statement->fetch();

El mismo patrón, otros intérpretes

La inyección no es solo SQL: existe inyección de comandos del sistema (shell_exec con datos del usuario), de LDAP, de plantillas… Siempre que pasas entrada a un intérprete, usa la API parametrizada de ese intérprete en vez de armar la cadena a mano.

XSS: inyección en el navegador #

El Cross-Site Scripting es inyección, pero el intérprete es el navegador de otra víctima. Si guardas un comentario con <script> dentro y luego lo muestras tal cual, ese script se ejecuta en el navegador de quien lo lee, con su sesión: puede robar su cookie, actuar por él o modificar la página.

1 El atacante publica un comentarioCon un <script> dentro del texto
2 Lo guardas y lo renderizas sin escaparEl navegador no distingue tu HTML del suyo
3 Se ejecuta en otra víctimaRoba su cookie de sesión, actúa por ella

El servidor reparte el ataque a cada visitante. Por eso este tipo (almacenado) es el más peligroso.

La defensa central es escapar la salida según el contexto: al meter un dato en HTML, conviértelo para que sus caracteres especiales se muestren como texto, no como etiquetas.

php · escapar al renderizar 
<?php

// Convierte < > & " en entidades: el navegador lo ve como texto
echo htmlspecialchars($comment, ENT_QUOTES, 'UTF-8');

// Resultado de <script>alert(1)</script> ->
// &lt;script&gt;alert(1)&lt;/script&gt;  (se muestra, no se ejecuta)

Escapa al salir, no al entrar

El escapado depende del contexto donde pintas el dato (HTML, atributo, JavaScript, URL son distintos). Por eso se escapa al renderizar, no al guardar: el mismo dato puede acabar en sitios diferentes. Los motores de plantillas (Twig, Blade) escapan por defecto justo por esto. Refuerza con una cabecera Content-Security-Policy.

CSRF: usar tu sesión sin permiso #

El Cross-Site Request Forgery es distinto: no roba datos, abusa de tu sesión ya iniciada. Como el navegador envía tus cookies automáticamente (lo viste en sesiones), una web maliciosa puede provocar que tu navegador haga una petición a tu banco —donde sigues logueado— sin que tú lo sepas.

El ejemplo clásico: estás logueado en tu banco y visitas una página trampa que contiene un formulario oculto que se autoenvía a tu-banco.com/transferir. Tu navegador adjunta tu cookie, y el banco cree que la orden la das tú.

La diferencia con XSS

XSS ejecuta código del atacante en tu página; CSRF no ejecuta nada en tu sitio: solo lanza una petición «legítima» desde fuera aprovechando que tu cookie viaja sola. Por eso la defensa es distinta: demostrar que la petición salió de tu propia página.

Dos defensas que se combinan:

  • Token anti-CSRF: tu formulario incluye un valor secreto e impredecible que el atacante no puede conocer. El servidor lo exige y lo valida; una petición desde fuera no lo lleva.
  • Cookie SameSite: le dice al navegador que no envíe la cookie en peticiones originadas por otros sitios. Con SameSite=Lax o Strict cortas la mayoría de los CSRF de raíz.

Principios que cubren el resto #

Las demás categorías del Top 10 se controlan con unos pocos hábitos transversales:

Principio Qué evita
Mínimo privilegio Cada usuario y servicio solo con los permisos que necesita: limita el daño de un fallo (control de acceso roto).
Validar en el servidor La validación del navegador es comodidad, no seguridad: el atacante salta el front. Revalida siempre detrás.
Dependencias al día Componentes con vulnerabilidades conocidas: actualiza y audita (composer audit).
Secretos fuera del código Claves y contraseñas en variables de entorno, nunca en el repositorio.
Errores sin filtrar detalles No muestres trazas ni SQL al usuario: regalan información al atacante.

La seguridad es una capa, no un parche

Ninguna defensa basta sola: consultas preparadas + escapado de salida + tokens CSRF + cabeceras + mínimo privilegio se refuerzan entre sí. Es la defensa en profundidad: si una capa falla, otra te cubre.

Resumen #

Lo que te llevas

  • El OWASP Top 10 recoge los riesgos web más críticos; casi todos nacen de confiar en la entrada del usuario.
  • Inyección SQL: el dato se vuelve SQL. Defensa: consultas preparadas, nunca concatenar.
  • XSS: el dato se vuelve HTML/JS en otro navegador. Defensa: escapar la salida según contexto + CSP.
  • CSRF: abusan de tu sesión desde otro sitio. Defensa: token anti-CSRF + cookie SameSite.
  • El resto se cubre con hábitos: mínimo privilegio, validar en servidor, dependencias al día, secretos fuera del código. Defensa en profundidad.
