D — Depender de abstracciones, no de detalles

La última letra es la que cose el resto: que la lógica importante no dependa de los detalles concretos (qué base de datos, qué proveedor de email), sino de abstracciones que tú controlas. Es un principio breve de enunciar y enorme de consecuencias, y ya lo tienes medio andado.

DIP se enuncia en dos frases: los módulos de alto nivel no deben depender de los de bajo nivel; ambos deben depender de abstracciones, y las abstracciones no deben depender de los detalles; los detalles deben depender de las abstracciones. Esta lección es deliberadamente corta: el principio ya lo has visto en acción en interfaces y OCP.

El enunciado #

«Alto nivel» es la política, la lógica de negocio que da valor (registrar un usuario, calcular un pedido). «Bajo nivel» son los detalles mecánicos (MySQL, el SMTP de un proveedor, el sistema de ficheros). Lo natural y erróneo es que el negocio dependa directamente del detalle. DIP dice: mete una abstracción en medio y que ambos dependan de ella.

Qué se invierte #

El nombre «inversión» despista. Lo que se invierte es la dirección de la dependencia en el código respecto al flujo de ejecución. En tiempo de ejecución, el negocio llama a la base de datos. Pero en el código, la base de datos depende de una interfaz que define el negocio, no al revés. La flecha de dependencia apunta «hacia dentro», hacia la política.

1 Negocio (alto nivel)define la interfaz que necesita
2 Abstraccióninterface RepositorioUsuarios
3 Detalle (bajo nivel)RepositorioMySql

Las dos puntas dependen del centro. El detalle apunta hacia arriba, hacia la abstracción que define el negocio.

El ejemplo mínimo #

Sin DIP

  • El servicio hace new RepositorioMySql() dentro.
  • Atado a MySQL: no se puede testear sin base de datos.
  • Cambiar de almacén = reescribir el servicio.

Con DIP

  • El servicio recibe una interfaz RepositorioUsuarios.
  • En tests, un doble en memoria; en prod, MySQL.
  • Cambiar de almacén = otra implementación, servicio intacto.
php · el alto nivel depende de la abstracción 
<?php

declare(strict_types=1);

// Abstracción definida por el negocio (alto nivel)
interface RepositorioUsuarios
{
    public function guardar(Usuario $usuario): void;
}

// Política de alto nivel: no menciona MySQL por ningún sitio
final class RegistrarUsuario
{
    public function __construct(private RepositorioUsuarios $repositorio)
    {
    }

    public function __invoke(string $email): void
    {
        $this->repositorio->guardar(Usuario::nuevo($email));
    }
}

// Detalle de bajo nivel: depende de la abstracción, no al revés
final class RepositorioMySql implements RepositorioUsuarios
{
    public function guardar(Usuario $usuario): void { /* INSERT ... */ }
}

Quién posee la abstracción #

El detalle que hace que DIP funcione de verdad: la interfaz pertenece al alto nivel, no a la infraestructura. La define el negocio según lo que necesita, y la capa de detalle se adapta a ella. Por eso conviene que RepositorioUsuarios viva junto al dominio, y RepositorioMySql en la capa de infraestructura que depende del dominio. Si la interfaz la dictara MySQL, no habrías invertido nada.

DIP no es lo mismo que inyección de dependencias

DIP es el principio (depende de abstracciones); la inyección de dependencias es la técnica de pasar esas dependencias por el constructor en vez de crearlas dentro; y el contenedor es la herramienta que las cablea. Se confunden porque casi siempre aparecen juntas.

Para profundizar #

Esta lección es la puerta, no la casa entera. Lo que sigue a DIP ya tiene su propio espacio en MentorAI:

  • Inyección de dependencias en PHP — la técnica al detalle: constructor, contenedor, testing con fakes.
  • Arquitectura hexagonal — DIP elevado a arquitectura: el dominio en el centro, los detalles como adaptadores enchufados por puertos.

Resumen #

Lo que te llevas

  • DIP: el alto nivel (negocio) y el bajo nivel (detalles) dependen ambos de una abstracción, no uno del otro.
  • Lo que se «invierte» es la dirección de la dependencia en el código: el detalle apunta a la abstracción que define el negocio.
  • La interfaz pertenece al alto nivel; la infraestructura se adapta a ella. Si la dicta el detalle, no has invertido nada.
  • DIP (principio) ≠ inyección de dependencias (técnica) ≠ contenedor (herramienta). Profundiza en esas dos páginas.
