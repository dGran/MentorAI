ORM, query builder o SQL plano

Entre tu objeto PHP y una fila de la base de datos hay un abismo, y tienes tres formas de cruzarlo. Cada una te sube un peldaño de comodidad a cambio de un peldaño de control. Saber cuál usar —y cuándo bajar— es media batalla del acceso a datos.

Tu aplicación piensa en objetos: un Pedido con su Cliente y sus líneas. La base de datos piensa en tablas y filas. Alguien tiene que traducir entre los dos mundos, y la pregunta es cuánto de esa traducción haces tú a mano y cuánto delegas. Esa decisión tiene tres respuestas.

El abismo objeto-tabla #

El problema tiene nombre: desajuste de impedancia objeto-relacional. Los objetos tienen referencias, herencia y colecciones anidadas; las tablas tienen filas planas, claves foráneas y JOINs. No encajan de forma natural, y todo el código de acceso a datos vive en esa costura, traduciendo de un lado a otro.

Las tres herramientas se diferencian en quién escribe esa traducción:

1 SQL planoTú escribes el SQL y mapeas las filas a mano. Control total.
2 Query builderCompones el SQL con métodos; el mapeo sigue siendo tuyo.
3 ORMTrabajas con objetos; la herramienta genera el SQL y mapea.

Más arriba, más productividad; más abajo, más control. No hay un nivel "correcto": hay un nivel adecuado a cada consulta.

SQL plano #

El nivel más bajo: escribes el SQL tú mismo y lo ejecutas con el driver (PDO en PHP). Tienes el control absoluto de cada consulta —ves exactamente qué se manda a la base de datos— a cambio de escribir y mantener cada línea, incluido el mapeo de la fila al objeto.

php · SQL plano con PDO (siempre parametrizado) 
<?php

$statement = $pdo->prepare(
    'SELECT id, total FROM pedidos WHERE cliente_id = :cliente AND estado = :estado'
);
$statement->execute(['cliente' => 42, 'estado' => 'enviado']);

$pedidos = $statement->fetchAll(PDO::FETCH_ASSOC);

Nunca concatenes valores en el SQL

Con SQL plano, la inyección SQL es tu responsabilidad. Jamás metas una variable dentro del string con interpolación: usa siempre parámetros (:cliente) y deja que el driver los escape. Lo desarrolla OWASP Top 10; aquí solo el recordatorio de que el control total incluye el control del riesgo.

SQL plano brilla cuando la consulta es compleja o crítica para el rendimiento: un informe con varios JOIN, funciones de ventana, o una agregación pesada que quieres afinar al milímetro. Pierde cuando tienes cincuenta CRUD sencillos: escribir a mano el INSERT/UPDATE/SELECT de cada entidad es repetitivo y propenso a erratas.

Query builder #

Un peldaño arriba: en vez de escribir el SQL como un string, lo compones encadenando métodos. El builder genera el SQL final por ti, parametrizado, y a menudo con autocompletado del IDE. El mapeo a objetos sigue siendo cosa tuya (o de una capa fina encima).

php · la misma consulta con un query builder 
<?php

$pedidos = $queryBuilder
    ->select('id', 'total')
    ->from('pedidos')
    ->where('cliente_id = :cliente')
    ->andWhere('estado = :estado')
    ->setParameters(['cliente' => 42, 'estado' => 'enviado'])
    ->fetchAllAssociative();

La ventaja real aparece con el SQL dinámico: cuando los filtros dependen de lo que pida el usuario, concatenar WHEREs a mano sobre un string es un infierno de comas y espacios; con un builder, añades un ->andWhere(...) dentro de un if y ya está. Es el punto dulce entre el control del SQL plano y la comodidad del ORM: hablas casi en SQL, pero programáticamente.

ORM #

El nivel más alto. Un ORM (Object-Relational Mapper) mapea tus clases a tablas y trabajas con objetos; él genera el SQL y traduce las filas a entidades en ambos sentidos. Defines el mapeo una vez y la herramienta se encarga del resto.

php · con un ORM piensas en objetos, no en SQL 
<?php

$pedidos = $repositorioPedidos->findBy([
    'cliente' => $cliente,
    'estado'  => 'enviado',
]);

foreach ($pedidos as $pedido) {
    echo $pedido->getTotal();
}

No hay SQL a la vista: pides objetos Pedido y recibes objetos Pedido. Para el 80 % del CRUD del día a día, esto es una ganancia enorme de productividad y de legibilidad. El coste es que cedes el control: el ORM decide qué SQL ejecuta, y a veces decide mal (consultas de más, el problema N+1) sin que lo veas a simple vista. Y debajo de los nombres comerciales hay dos filosofías distintas —Active Record y Data Mapper— que cambian cómo se siente: es la siguiente lección.

Cuándo cada uno #

El error de principiante es pensar que hay que elegir uno para todo el proyecto. No: conviven. Lo sano es un nivel por defecto y bajar de nivel en los casos puntuales que lo pidan.

SituaciónNivel adecuado
CRUD de entidades, lógica de dominio, el 80 % del códigoORM
Filtros dinámicos, búsquedas que se montan según la entradaQuery builder
Informe pesado, agregación compleja, consulta a optimizar al milímetroSQL plano
Operación masiva (actualizar un millón de filas de golpe)SQL plano

La buena noticia: las herramientas serias lo asumen. Doctrine trae su query builder y un modo de SQL nativo; Eloquent te deja caer a la fachada DB:: con SQL plano cuando hace falta. Usar el ORM por defecto no te encierra: la escotilla de escape a un nivel inferior está siempre ahí.

La abstracción tiene fugas #

Hay una tentación peligrosa: creer que el ORM te exime de saber SQL. Es justo al revés. Toda abstracción "tiene fugas" —acaba dejando ver la complejidad que oculta— y la del ORM gotea precisamente cuando algo va lento.

El ORM no te libera de entender la BD

Cuando una página va lenta, abres el SQL que genera el ORM y necesitas saber leerlo: si usa el índice, si hace un JOIN de más, si dispara cien consultas donde debería una. Sin esa base —índices, modelado, transacciones— el ORM es una caja negra que no sabes depurar.

Por eso el orden de este temario no es casual: primero la base de datos por dentro, ahora cómo tu código habla con ella. El ORM es una herramienta excelente encima de saber SQL, nunca un sustituto de saberlo.

Resumen #

Lo que te llevas

  • Entre objetos y tablas hay un desajuste de impedancia; las tres herramientas se diferencian en quién escribe la traducción.
  • SQL plano: control total, máxima verbosidad. Para consultas complejas, informes y operaciones masivas. Siempre parametrizado.
  • Query builder: compones el SQL con métodos. Su punto fuerte es el SQL dinámico (filtros que dependen de la entrada).
  • ORM: trabajas con objetos y genera el SQL. Gran productividad para el CRUD; cedes control y puede generar SQL ineficiente.
  • No es uno para todo: conviven. ORM por defecto, baja a query builder o SQL plano donde haga falta. Y el ORM no te exime de saber SQL.
