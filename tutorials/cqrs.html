CQRS: separar lecturas de escrituras

La mayoría de las aplicaciones usan el mismo modelo para escribir y para leer. CQRS propone separarlos: un modelo optimizado para cambiar estado y otro para consultarlo. Es potente y peligroso a partes iguales; lo más importante que aprenderás aquí es cuándo NO usarlo.

CQRS son las siglas de Command Query Responsibility Segregation: separar la responsabilidad de los comandos (que cambian estado) de la de las queries (que leen). Suena académico; la idea de fondo es que escribir y leer tienen necesidades tan distintas que forzar un solo modelo para ambos perjudica a los dos.

Comando vs query #

La base es un principio más viejo, el CQS de Bertrand Meyer: un método o cambia estado (comando, no devuelve datos) o devuelve datos (query, no cambia nada), pero no las dos cosas. CQRS lleva esa separación del nivel de método al nivel de modelo y arquitectura.

Comando (escritura) Query (lectura)
Intención Cambiar estado, aplicar reglas. Devolver datos para mostrar.
Le importa Invariantes, consistencia, validación. Velocidad, forma cómoda para la vista.
Modelo ideal Agregados ricos del dominio. Vistas planas, justo lo que pinta la pantalla.

Separar los dos modelos #

En CQRS, los comandos pasan por el modelo de escritura —los agregados con sus reglas— y las queries van por un camino aparte que devuelve datos listos para mostrar, a menudo sin pasar por los agregados en absoluto. Son dos rutas distintas para dos propósitos distintos.

C ComandoCreateOrder → modelo de escritura (agregados, reglas)
DB AlmacenamientoUno o dos modelos de datos según el nivel
Q QueryOrderSummary → modelo de lectura (vistas planas)

Dos caminos: escribir pasa por el dominio; leer toma un atajo hacia la vista.

php · dos rutas, dos modelos 
<?php

declare(strict_types=1);

// ESCRITURA: pasa por el agregado y sus reglas.
final class ConfirmOrderHandler
{
    public function __construct(private OrderRepository $orders) {}

    public function __invoke(ConfirmOrder $command): void
    {
        $order = $this->orders->byId($command->orderId);
        $order->confirm();
        $this->orders->save($order);
    }
}

// LECTURA: atajo directo a datos para la pantalla, sin agregados.
final class OrderSummaryQuery
{
    public function __construct(private PDO $connection) {}

    public function forCustomer(string $customerId): array
    {
        // SELECT optimizado que devuelve justo lo que pinta la vista.
        return [];
    }
}

Por qué ayuda #

Separar trae ventajas reales cuando lectura y escritura tienen presiones distintas:

  • Optimizar cada lado: el modelo de lectura puede ser una vista desnormalizada, una tabla pensada para esa pantalla, o incluso otra base de datos pensada para consultas.
  • Escalar por separado: si lees mil veces más de lo que escribes (lo normal), puedes replicar y escalar la lectura sin tocar la escritura.
  • Modelos más limpios: el de escritura no se contamina con campos que solo existen para mostrar; el de lectura no arrastra la complejidad de las reglas.

Niveles de CQRS #

CQRS no es todo o nada. Hay un gradiente, y casi siempre el nivel ligero es suficiente:

Nivel Qué separa
Ligero Solo el código: comandos y queries distintos, pero la misma base de datos. Barato y muy útil.
Medio Modelos de lectura propios (vistas/tablas desnormalizadas) en la misma base.
Completo Almacenes separados para lectura y escritura, sincronizados de forma asíncrona.

Empieza por el nivel ligero

Separar comandos de queries en el código —sin tocar la base de datos— ya ordena mucho y casi no cuesta. La mayoría de los beneficios que la gente atribuye a CQRS vienen de este nivel. Sube de nivel solo cuando una presión concreta (escala de lectura, una query imposible) lo justifique.

Sincronizar los modelos #

En el nivel completo, con dos almacenes, surge la pregunta clave: ¿cómo se entera el modelo de lectura de los cambios que hizo la escritura? La respuesta habitual son los eventos de dominio: la escritura emite eventos y un proceso actualiza el modelo de lectura al recibirlos.

El precio: consistencia eventual

Si la sincronización es asíncrona, durante un instante el modelo de lectura está desactualizado: escribes algo y al recargar todavía no aparece. Es consistencia eventual en tu propia aplicación. A veces es aceptable; a veces arruina la experiencia. Decídelo a conciencia, no por accidente.

Cuándo NO usarlo #

Esta es la sección más importante. CQRS añade complejidad real: duplica modelos, complica el flujo y, en su forma completa, introduce consistencia eventual. Para la mayoría de las aplicaciones, eso es un coste sin beneficio.

No lo uses si…

  • Es un CRUD donde leer y escribir usan los mismos datos.
  • Lo aplicas «por defecto» a todo el sistema.
  • No tienes una presión real de lectura o de modelo.
  • El equipo no domina sus implicaciones.

Puede compensar si…

  • Lectura y escritura escalan muy distinto.
  • Las consultas piden una forma muy ajena al modelo de dominio.
  • Un bounded context concreto lo necesita (no todos).
  • Vas a usar event sourcing (encaja de forma natural).

La regla de oro: CQRS se aplica a un bounded context concreto que lo pida, no a la aplicación entera por moda. Empieza simple; introdúcelo cuando el dolor sea real y localizado. En el siguiente tutorial lo verás llevado a su extremo natural: el event sourcing.

Resumen #

Lo que te llevas

  • CQRS separa el modelo de escritura (comandos, agregados, reglas) del de lectura (queries, vistas planas).
  • Permite optimizar y escalar cada lado por separado y mantener ambos modelos limpios.
  • Hay niveles: empieza por el ligero (solo código). El completo (almacenes separados) trae consistencia eventual.
  • No lo uses en un CRUD ni «por defecto»: aplícalo a un bounded context que lo pida. La complejidad es real.
